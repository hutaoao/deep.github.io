## 一句话概括

浏览器提供了三种本地存储方案：`localStorage`（持久化键值对）、`sessionStorage`（会话级键值对）和 `IndexedDB`（异步事务型数据库），掌握它们的能力与边界是前端存储选型的基础。

## 核心知识点

### 1. localStorage — 持久化键值存储

```javascript
// 存储（仅支持字符串，对象需 JSON 序列化）
localStorage.setItem('theme', 'dark');
localStorage.setItem('prefs', JSON.stringify({ lang: 'zh', fontSize: 14 }));

// 读取（注意：取出来永远是字符串）
const theme = localStorage.getItem('theme');          // 'dark'
const prefs = JSON.parse(localStorage.getItem('prefs'));

// 遍历所有键
for (let i = 0; i < localStorage.length; i++) {
  const key = localStorage.key(i);
  console.log(key, localStorage.getItem(key));
}

// 监听其他同源标签页的存储变化（当前页面自己的修改不会触发）
window.addEventListener('storage', (e) => {
  console.log(e.key, e.oldValue, e.newValue);
});
```

**关键数据：容量 ~5MB，同源共享，同步 API（会阻塞主线程），关闭浏览器不丢失。**

### 2. sessionStorage — 会话级隔离存储

```javascript
// API 与 localStorage 完全一致
sessionStorage.setItem('draft', JSON.stringify({ title: '未完成' }));
const draft = JSON.parse(sessionStorage.getItem('draft'));

// 场景演示：标签页 A
sessionStorage.setItem('tab', 'A');
// 标签页 B（同源，但完全独立）
console.log(sessionStorage.getItem('tab')); // null
```

**关键数据：容量 ~5MB，每个标签页独立一份，关闭标签页即清除。** 从 `window.open()` 或链接打开的新标签页会**复制**一份初始数据，但之后各自独立。

### 3. IndexedDB — 浏览器内置 NoSQL

```javascript
// 打开数据库
const dbReq = indexedDB.open('AppDB', 1);
dbReq.onupgradeneeded = (e) => {
  const db = e.target.result;
  const store = db.createObjectStore('users', { keyPath: 'id' });
  store.createIndex('nameIdx', 'name', { unique: false });
};
dbReq.onsuccess = async (e) => {
  const db = e.target.result;
  // 写入
  const tx = db.transaction('users', 'readwrite');
  tx.objectStore('users').add({ id: 1, name: 'Alice', age: 28 });
  tx.oncomplete = () => console.log('写入完成');
};
```

**关键数据：容量可达磁盘可用空间的 60%-80%，异步不阻塞，支持事务 + 索引 + 游标，可存 File/Blob。**

### 4. 三剑客对比速查表

| 维度 | localStorage | sessionStorage | IndexedDB |
|------|-------------|----------------|-----------|
| 容量 | ~5MB | ~5MB | 数百MB~GB |
| 生命周期 | 永久 | 标签页关闭 | 永久 |
| 作用域 | 同源共享 | 同源+同标签页 | 同源共享 |
| 线程模型 | 同步 | 同步 | 异步 |
| 数据类型 | 仅字符串 | 仅字符串 | 结构化克隆支持的类型 |
| 适合场景 | 用户偏好、主题 | 表单草稿、临时 token | 离线数据、大文件缓存 |

### 5. 带过期时间的封装

```javascript
// 面试高频点：localStorage 本身不支持过期，需要自己封装
const storage = {
  set(key, value, ttlMs) {
    const item = { value, expiry: Date.now() + ttlMs };
    localStorage.setItem(key, JSON.stringify(item));
  },
  get(key) {
    const raw = localStorage.getItem(key);
    if (!raw) return null;
    const { value, expiry } = JSON.parse(raw);
    if (Date.now() > expiry) { localStorage.removeItem(key); return null; }
    return value;
  }
};

storage.set('token', 'abc123', 3600_000); // 1 小时后过期
```

## 其实你每天都在用

- **主题切换**：「暗色/亮色」偏好存在 localStorage，刷新页面不用重新设
- **表单草稿**：写了半天的长文章，浏览器崩了？sessionStorage 救不回来，但 localStorage 可以
- **大文件 PWA 离线**：Google Docs / Figma 用 IndexedDB 缓存文档，断网还能继续编辑
- **跨标签页同步**：A 标签页「退出登录」，B 标签页通过 `storage` 事件自动跳转登录页
- **性能监控数据暂存**：页面埋点数据先写 IndexedDB，攒够一批再上报，避免频繁请求

## 常见误解

- **❌ 误区：「sessionStorage 能在多个标签页之间共享」** 不能。每个标签页有自己独立的 sessionStorage，哪怕 URL 一模一样。这是设计如此——否则"会话"概念就失去意义了。

- **❌ 误区：「localStorage 可以安全存敏感数据」** 不能。任何 JS 脚本（包括 XSS 注入）都能读取 localStorage。token、密码等敏感信息应放在 HttpOnly Cookie 或内存中。

- **❌ 误区：「IndexedDB 太复杂了，不如直接用 localStorage」** 当你需要存 10MB+ 的结构化数据、需要按索引查询、需要事务保证数据一致性时，IndexedDB 是唯一正确选择。可以直接用 `idb` 这类 Promise 封装库，API 体验不输 localStorage。

- **❌ 误区：「localStorage 容量不够就用 IndexedDB，不需要考虑 quota 管理」** IndexedDB 也有配额限制（通常按源分配），且浏览器在存储压力大时会清除"非持久化"数据。应调用 `navigator.storage.estimate()` 和 `persist()` 主动管理配额。

## 一句话总结

选存储方案很简单：小且永久 → localStorage，临时且隔离 → sessionStorage，大且结构化 → IndexedDB，需要随请求自动发送 → Cookie。
