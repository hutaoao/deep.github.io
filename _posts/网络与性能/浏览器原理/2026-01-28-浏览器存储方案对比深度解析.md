## 一句话概括

Cookie、localStorage、sessionStorage、IndexedDB —— 四个存储方案管的不是"存数据"这一件事，而是四种完全不同的场景：身份传递、持久化偏好、会话隔离、海量结构化存储。面试问你区别，本质是考你有没有理解每种方案的**设计意图**。

## 核心知识点

### 1. Cookie — 自动携带的身份牌（4KB）

```javascript
// 后端下发（推荐）：身份信息走 HttpOnly，前端拿不到
// Set-Cookie: sessionId=abc123; HttpOnly; Secure; SameSite=Lax; Max-Age=3600

// 前端只能操作非 HttpOnly 的
document.cookie = 'theme=dark; Max-Age=2592000; Path=/; Secure';

// ⚠️ 面试杀手题：document.cookie 读出来是什么？
console.log(document.cookie); // "theme=dark; lang=zh"—— 是所有非 HttpOnly cookie 拼成的字符串
// 不是对象！要解析：
const parseCookie = str =>
  Object.fromEntries(str.split('; ').map(p => p.split('=')));
```

**核心定位：** Cookie 的设计目的从来不是"存数据"，而是解决 HTTP 无状态的致命缺陷——每次请求自动把自己带过去让服务器认人。4KB 够存一个 sessionId 了，再多就是对请求体积的污染。

### 2. localStorage — 持久化键值对（~5MB）

```javascript
// 基础操作：只能存字符串
localStorage.setItem('prefs', JSON.stringify({ theme: 'dark', fontSize: 16 }));
const prefs = JSON.parse(localStorage.getItem('prefs'));

// 面试高频：手写带过期时间的封装
const storage = {
  set(key, value, ttlMs) {
    const item = { value, expiry: Date.now() + ttlMs };
    localStorage.setItem(key, JSON.stringify(item));
  },
  get(key) {
    const raw = localStorage.getItem(key);
    if (!raw) return null;
    const { value, expiry } = JSON.parse(raw);
    if (Date.now() > expiry) { localStorage.removeItem(key); return null; }
    return value;
  }
};

// storage 事件 —— 跨标签页同步
window.addEventListener('storage', e => {
  if (e.key === 'auth_token') location.reload(); // 其他标签页退出登录，本页也刷新
});
```

**关键心智模型：** 同步 API、同源共享、持久化。注意 `storage` 事件只在**其他**同源页面触发存储变化时发送——自己的页面改了 localStorage 自己是收不到这个事件的。

### 3. sessionStorage — 标签页级隔离（~5MB）

```javascript
sessionStorage.setItem('draft', JSON.stringify({ title: '未完成' }));

// 核心行为：两个同源标签页之间 sessionStorage 相互完全隔离
// 标签页 A：
sessionStorage.setItem('key', 'A');
// 标签页 B（同源）：
console.log(sessionStorage.getItem('key')); // null —— 拿不到！

// 特殊情况：window.open() 和 location.href = '_blank' 会复制当前快照
// Duplicate Tab 不会复制 —— 新建一个干净的 session
```

**一句话：** 一个标签页 = 一个独立 session。表单草稿、一次性验证码、多开同一应用的会话隔离，sessionStorage 是天然选择。

### 4. IndexedDB — 浏览器内置异步数据库（数百MB+）

```javascript
// 用 idb 库（indexedDB 的原生 API 是回调地狱别直接用）
import { openDB } from 'idb';

const db = await openDB('MyAppDB', 1, {
  upgrade(db) {
    const store = db.createObjectStore('messages', { keyPath: 'id', autoIncrement: true });
    store.createIndex('timestamp', 'timestamp');
  }
});

await db.add('messages', { text: 'Hello', timestamp: Date.now() });
const recent = await db.getAllFromIndex('messages', 'timestamp',
  IDBKeyRange.lowerBound(Date.now() - 86400000)); // 最近24小时
```

**为什么不是 localStorage 的替代品：** 异步（不卡 UI）、支持索引查询、能存 Blob/File、有事务特性。但异步也意味着代码复杂度成倍增加——5MB 以内的小数据用 localStorage 封装一下就够了。

### 5. 四合一决策速查

| | Cookie | localStorage | sessionStorage | IndexedDB |
|---|---|---|---|---|
| 容量 | 4KB | 5~10MB | 5~10MB | 数百MB~磁盘的60% |
| 生命周期 | 可设过期/会话 | 永久（手动清） | 标签页关闭即销毁 | 永久 |
| 随请求自动发送 | ✅ | ❌ | ❌ | ❌ |
| 同步/异步 | 同步 | 同步 | 同步 | 异步 |
| 支持事务 | ❌ | ❌ | ❌ | ✅ |
| 结构化数据 | ❌ | ❌ | ❌ | ✅ |

## 其实你每天都在用

- **登录态维持**：`sessionId` 存在 HttpOnly Cookie 里，每次请求后端自动识别——你从未手动传过 sessionId
- **暗色模式开关**：`theme: 'dark'` 写 localStorage，刷新页面从 localStorage 先读主题再渲染，避免闪烁
- **表单草稿自动保存**：写三十分钟的评论，突然关标签页——localStorage 草稿恢复功能救你一命（注意 sessionStorage 做不到：关了标签页就没了）
- **Google Docs 离线编辑**：所有文档内容存在 IndexedDB，断网正常编辑，联网后增量同步
- **性能埋点批量上报**：用户行为日志先写 IndexedDB，攒够 50 条或每 30 秒批量 POST，减少请求次数

## 常见误解

- **❌ 误区：「sessionStorage 多个标签页同源就能共享」** 完全不能。这是 sessionStorage 最核心的设计——每个标签页独立的 session。需要在标签页间通信用 `BroadcastChannel` API 或 `storage` 事件。

- **❌ 误区：「token 存 localStorage 最方便」** 方便但不安全——任何 XSS 注入的 `<script>` 都能 `localStorage.getItem('token')` 偷走。token 的正确做法：HttpOnly Cookie（防 XSS 读）或仅存在内存闭包里（刷新重新登录）。

- **❌ 误区：「localStorage 放不下就上 IndexedDB」** 别一刀切。IndexedDB 的异步模型引入了大量 then/callback，5MB 以内用 localStorage + 过期封装 + LRU 淘汰方案完全够用。需要索引查询、Blob/File 存储、大量结构性数据时才切 IndexedDB。

- **❌ 误区：「sessionStorage 关闭浏览器就没了」** 准确说是关闭**标签页**。浏览器崩溃恢复标签页时 sessionStorage 可能还在——各浏览器对 crash recovery 的行为不一致，不要依赖这个特性做数据恢复。

## 一句话总结

选存储方案不是比谁容量大——Cookie 管身份，localStorage 管偏好，sessionStorage 管临时隔离，IndexedDB 管海量数据。真正上线的应用，四个通常全用，各司其职。
