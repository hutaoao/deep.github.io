## 一句话概括

Cookie、localStorage、sessionStorage、IndexedDB 四个浏览器存储方案的本质区别不是容量大小，而是**设计意图**：Cookie 解决 HTTP 无状态（自动传身份），localStorage 管持久化偏好，sessionStorage 管标签页隔离，IndexedDB 管海量结构化数据。真正上线的应用，四个全用，各司其职。

## 核心知识点

### 1. Cookie — HTTP 无状态问题的补丁（4KB）

```js
// Cookie 不是"前端存储"，而是 HTTP 协议的扩展——浏览器自动在请求头里带过去
// 后端种：身份 token 走 HttpOnly + Secure + SameSite
// Set-Cookie: session_id=abc123; HttpOnly; Secure; SameSite=Lax; Max-Age=3600; Path=/

// 前端只能操作非 HttpOnly 的 Cookie
document.cookie = 'theme=dark; Max-Age=2592000; Path=/; Secure; SameSite=Lax';

// ⚠️ 面试必问：document.cookie 返回的不是对象，是所有非 HttpOnly 的键值对拼成的字符串
console.log(document.cookie); // "theme=dark; lang=zh"
// 要解析得自己写：
const parseCookies = str =>
  Object.fromEntries(str.split('; ').map(pair => pair.split('=')));
```

**核心定位：** Cookie 诞生时就一个使命——让无状态的 HTTP 协议"记住"用户。每次请求自动携带是它的**特性**而非缺陷。4KB 存一个 sessionId 绰绰有余，再多就污染请求体积了。三个关键安全属性：
- `HttpOnly`：JS 无法读取，防 XSS 窃取
- `Secure`：仅 HTTPS 传输，防中间人嗅探
- `SameSite`（Lax/Strict/None）：防 CSRF，现代浏览器默认 Lax

### 2. localStorage — 持久化键值对（~5MB）

```js
// 同源所有页面共享、永不过期、同步 API
localStorage.setItem('prefs', JSON.stringify({ theme: 'dark', fontSize: 16 }));
const prefs = JSON.parse(localStorage.getItem('prefs'));

// 面试高频：给 localStorage 加过期时间
const smartStorage = {
  set(key, value, ttlMs = Infinity) {
    localStorage.setItem(key, JSON.stringify({ value, expires: Date.now() + ttlMs }));
  },
  get(key) {
    const raw = localStorage.getItem(key);
    if (!raw) return null;
    const { value, expires } = JSON.parse(raw);
    if (Date.now() > expires) { localStorage.removeItem(key); return null; }
    return value;
  }
};

// storage 事件：跨标签页同步（只在**其他**同源页面写时触发，自己写自己不触发）
window.addEventListener('storage', ({ key, newValue }) => {
  if (key === 'auth_token' && !newValue) location.reload(); // 另一个标签页退登了
});
```

**心智模型：** 存键值对、同源共享、永久保留。适合主题偏好、用户设置、不敏感的 token 缓存。`storage` 事件的关键细节：A 页面写 localStorage，B 页面（同源）收到事件，A 自己收不到。

### 3. sessionStorage — 标签页级沙箱（~5MB）

```js
sessionStorage.setItem('draft', JSON.stringify({ title: '草稿', content: '' }));

// 核心：两个同源标签页的 sessionStorage 完全隔离
// 标签页 A：sessionStorage.setItem('x', 'A');
// 标签页 B（同源）：sessionStorage.getItem('x'); // null！
//
// 注意：window.open() / <a target="_blank"> 会复制当前快照，但后续各自独立
// Duplicate Tab（右键复制标签页）不复制，完全干净的 session
```

**一句话：** 一个标签页 = 一个独立 session。天然适合表单草稿、多开同一应用时的会话隔离、一次性验证码。关标签页即销毁（浏览器崩溃恢复行为各浏览器不一致，别依赖）。

### 4. IndexedDB — 浏览器里的异步数据库（数百 MB+）

```js
// 原生 API 回调地狱，生产代码用 idb 库（1KB 的封装）
import { openDB } from 'idb';

const db = await openDB('MyApp', 1, {
  upgrade(db) {
    const store = db.createObjectStore('messages', { keyPath: 'id', autoIncrement: true });
    store.createIndex('by_time', 'timestamp');
  }
});

// 写入
await db.add('messages', { text: 'Hello', timestamp: Date.now() });

// 索引查询（最近 24h 消息）
const recent = await db.getAllFromIndex('messages', 'by_time',
  IDBKeyRange.lowerBound(Date.now() - 86400000));
```

**什么时候用 IndexedDB 而非 localStorage：**
- 数据量超过几十 MB（localStorage 5MB 是硬上限）
- 需要按条件查询（localStorage 只能全量遍历）
- 存储 Blob / File / ArrayBuffer（localStorage 只存字符串）
- 需要事务保证（多次写操作原子性）

⚠️ 别一来就上 IndexedDB——异步模型让代码复杂度陡增，5MB 以内用 localStorage + 过期封装完全够用。

### 5. 💡 进阶：CacheStorage 和 OPFS

面试加分项——两个现代浏览器存储方案：

```js
// CacheStorage：Service Worker 专用，缓存网络请求（HTML/JS/CSS/字体/图片）
// 你访问过的 PWA 应用秒开，靠的就是这个
const cache = await caches.open('v1');
await cache.addAll(['/', '/styles.css', '/app.js']);

// OPFS（Origin Private FileSystem）：大文件高性能读写
// 适合视频编辑、3D 模型、SQLite（WASM 版）等场景
const root = await navigator.storage.getDirectory();
const file = await root.getFileHandle('data.bin', { create: true });
const writable = await file.createSyncAccessHandle(); // 同步写，极快
```

### 决策速查

| | Cookie | localStorage | sessionStorage | IndexedDB |
|---|---|---|---|---|
| 容量 | 4KB | ~5MB | ~5MB | 数百MB+ |
| 生命周期 | 可设过期 | 永久 | 标签页关闭 | 永久 |
| 随请求发送 | ✅ 自动 | ❌ | ❌ | ❌ |
| API 类型 | 同步 | 同步 | 同步 | 异步 |
| 事务 | ❌ | ❌ | ❌ | ✅ |
| 存 Blob/File | ❌ | ❌ | ❌ | ✅ |

## 其实你每天都在用

- **登录态**：sessionId 在 HttpOnly Cookie 里，你从未手动传过它，后端却每次都能识别你
- **暗色模式无闪烁**：刷新页面前先从 localStorage 读 `theme`，设置 `<html data-theme>` 再渲染，否则会闪一下白屏
- **评论草稿救场**：写了半小时的评论不小心关了标签页——localStorage 里的草稿帮你恢复
- **Google Docs 离线编辑**：所有文档内容存在 IndexedDB 里，断网照写不误，联网后增量同步
- **PWA 秒开**：Service Worker + CacheStorage 把整个应用的 HTML/JS/CSS 缓存到本地，第二次打开不需要网络

## 常见误解

- **❌ 误区：「sessionStorage 同源标签页就能共享」** 绝对不能。这是它最核心的设计——每个标签页独立 session。跨标签页通信请用 `BroadcastChannel` 或 `storage` 事件。

- **❌ 误区：「token 存 localStorage 最方便，大家都这么干」** 方便但危险。任何 XSS 注入的 `<script>` 一行 `localStorage.getItem('token')` 就能偷走。正确姿势：access token 存内存闭包（刷新需重新登录），refresh token 存 HttpOnly + Secure + SameSite=Strict Cookie。

- **❌ 误区：「SameSite=Strict 一定比 Lax 安全」** Strict 确实更安全，但有个致命问题：用户从邮件/GitHub 点链接跳转到你的网站时，Cookie 不会被携带，导致用户永远是"未登录"状态。Lax 在"安全"和"可用"之间取得了平衡——允许 GET 导航请求携带 Cookie，阻止 POST 等危险方法的跨站请求。

- **❌ 误区：「sessionStorage 关闭浏览器就清空」** 准确说是关闭**标签页**。浏览器崩溃后恢复标签页时，不同浏览器对 sessionStorage 的保留策略不一致（Chrome 会恢复，Firefox 不一定），不要依赖它做崩溃恢复。

## 一句话总结

选存储方案不是比谁容量大——Cookie 补 HTTP 无状态的坑，localStorage 存偏好，sessionStorage 管隔离，IndexedDB 当本地数据库。面试官问你"区别"，真正想听的是你有没有理解每种方案**为什么被设计出来**。
