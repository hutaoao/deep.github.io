## 一句话概括

Cookie 是浏览器端最老的存储机制（4KB），通过 `Set-Cookie` 响应头下发、每次 HTTP 请求自动携带，核心能力在于精细的安全属性控制——`HttpOnly`、`Secure`、`SameSite` 构成前端安全的三道防线。

## 核心知识点

### 1. Cookie 的基本玩法

```javascript
// 服务端设置（推荐方式）
// Set-Cookie: sessionId=abc123; HttpOnly; Secure; SameSite=Lax; Max-Age=3600; Path=/

// 前端读取（只能读到非 HttpOnly 的 Cookie）
console.log(document.cookie); // "theme=dark; lang=zh"

// 前端设置
document.cookie = "theme=dark; Max-Age=2592000; Path=/; Secure";

// 删除（设置过期时间为过去）
document.cookie = "theme=; Max-Age=0; Path=/";
```

### 2. HttpOnly — 防 XSS 窃取

标记 `HttpOnly` 的 Cookie **无法通过 `document.cookie` 读取**，但浏览器发请求时依然自动携带。

```javascript
// Set-Cookie: sessionId=abc123; HttpOnly

// 即使 XSS 注入成功，恶意脚本也拿不到 sessionId
console.log(document.cookie); // 看不到 sessionId

// 但正常请求依然会带上
// GET /api/user → Cookie: sessionId=abc123
```

**这是防御 XSS 窃取会话的最关键手段。** 凡是只给服务器用的 cookie（sessionId、csrf token），务必加 `HttpOnly`。

### 3. SameSite — 防 CSRF 攻击

控制**跨站请求**时是否发送 Cookie：

```javascript
// Strict：最严格，从任何其他站点跳过来都不带 Cookie
// 场景：点击邮件里的链接进入银行网站，需要重新登录
Set-Cookie: id=xxx; SameSite=Strict

// Lax（默认值，推荐）：允许「安全」的跨站 GET 请求
// ✅ 点击 <a> 链接跳转 → 带 Cookie
// ✅ GET 表单提交 → 带 Cookie
// ❌ <img> 加载、iframe、AJAX 跨站请求 → 不带 Cookie
Set-Cookie: id=xxx; SameSite=Lax

// None：完全放开跨站，必须搭配 Secure（即 HTTPS）
Set-Cookie: tracking=xxx; SameSite=None; Secure
```

**面试重点：Chrome 80+ 将 `SameSite` 默认值从 `None` 改为 `Lax`，这是近年来最大的 Cookie 行为变化，影响所有跨站场景。**

### 4. Domain 与 Path — 控制可见范围

```javascript
// 不设 Domain：仅当前域名可见（不含子域名）
// 设在 www.example.com，api.example.com 看不到
Set-Cookie: name=value

// 设 Domain=.example.com：所有子域名共享
// api.example.com、admin.example.com 都能访问
Set-Cookie: name=value; Domain=.example.com

// Path：限定 URL 路径前缀
// 仅 /admin 及子路径生效
Set-Cookie: token=xxx; Path=/admin
```

### 5. Cookie 与 Storage 的对比

| 维度 | Cookie | localStorage/sessionStorage |
|------|--------|---------------------------|
| 容量 | 4KB | 5-10MB |
| 自动发送 | ✅ 每次请求 | ❌ |
| HttpOnly | ✅ 防 XSS | ❌ |
| SameSite | ✅ 防 CSRF | ❌ |
| 过期控制 | ✅ Max-Age/Expires | localStorage 无，需手动封装 |
| 性能影响 | 每次请求都带，多了拖慢网络 | 不影响网络请求 |

## 其实你每天都在用

- **登录态**：`sessionId` 存 HttpOnly Cookie，JS 读不到，但每次请求自动带上，后端就知道你是谁
- **记住密码**：勾了「两周内免登录」，后端设一个长有效期的 Cookie 来识别你
- **第三方追踪**：A 网站嵌了 B 网站的 iframe，B 用 `SameSite=None` 的 Cookie 跟踪你的浏览行为
- **多语言偏好**：`lang=zh-CN` 存 Cookie，后端直接渲染对应语言页面，不用前端异步切换
- **购物车未登录态**：Cookie 里存个 `cartId`，后端根据它查到你的购物车数据

## 常见误解

- **❌ 误区：「Cookie 容量 4KB 太小了，用不上了」** Cookie 的定位从来不是「存数据」，而是「传递身份」。4KB 存 sessionId + 少量偏好足够了。大数据存在 localStorage 或 IndexedDB，只把 ID 放 Cookie。

- **❌ 误区：「只要设了 HttpOnly 就安全了」** HttpOnly 只防 XSS 窃取 Cookie，防不了中间人攻击（需搭配 `Secure`）和 CSRF（需搭配 `SameSite` 或 CSRF Token）。三重保护缺一不可。

- **❌ 误区：「SameSite=Lax 之后就不需要 CSRF Token 了」** Lax 只防跨站的非 GET 请求（POST/PUT/DELETE），对于 GET 请求的 CSRF 仍然存在风险（虽然 GET 不应有副作用，但实际很多老接口用 GET 做操作）。敏感操作建议依然保留 CSRF Token 作为纵深防御。

- **❌ 误区：「前端设 Cookie 不就行了吗，为什么要后端设？」** 前端 `document.cookie` 无法设置 `HttpOnly`（这是有意为之），这意味着前端的 Cookie 永远能被 JS 读取。会话相关的 Cookie 应该由后端通过 `Set-Cookie` 响应头设置。

## 一句话总结

Cookie 不是用来存数据的——它是 HTTP 无状态协议的身份补丁；`HttpOnly + Secure + SameSite=Lax` 三件套上齐，才算是安全生产。
