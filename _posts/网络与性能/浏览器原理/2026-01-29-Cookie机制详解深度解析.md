## 一句话概括

Cookie 是 HTTP 无状态协议的身份补丁——每次请求自动带上是它的超能力，但自动带上也意味着每次请求都被拖慢。`HttpOnly` + `Secure` + `SameSite` 三道防线不是可选配置，是底线。

## 核心知识点

### 1. Cookie 的写入和读取 —— 两种方式截然不同

```javascript
// ═══ 服务端 Set-Cookie（推荐，所有敏感 Cookie 的唯一方式）═══
// Set-Cookie: sessionId=abc123; HttpOnly; Secure; SameSite=Lax; Max-Age=3600; Path=/

// ═══ 前端 document.cookie（只能操作非 HttpOnly 的）═══
document.cookie = 'theme=dark; Max-Age=2592000; Path=/; Secure';

// ⚠️ 常见坑：读出来的是一整坨字符串
console.log(document.cookie); // "theme=dark; lang=zh"
// 解析工具函数：
const parseCookie = str =>
  Object.fromEntries(str.split('; ').map(pair => {
    const idx = pair.indexOf('=');
    return [pair.slice(0, idx), pair.slice(idx + 1)];
  }));
// → { theme: 'dark', lang: 'zh' }

// 删除 = 设过期时间为过去
document.cookie = 'theme=; Max-Age=0; Path=/';
```

**关键区分：** 前端 `document.cookie` **永远**设不了 `HttpOnly`。所以需要保密的（session id、csrf token）必须由后端通过 `Set-Cookie` 响应头下发——这个方向是单向的。

### 2. Cookie 属性详解 —— 五个控制字段

```javascript
// Set-Cookie 完整的属性面板：
// Set-Cookie: name=value; Domain=site.com; Path=/app;
//             Max-Age=3600; Secure; HttpOnly; SameSite=Lax

// Domain — 决定哪些子域名能看到
// 不设：仅当前域名（不含子域名），最安全
// Domain=.example.com：api.example.com、admin.example.com 全能看到

// Path — 限定 URL 路径前缀
// Path=/admin — 只有 /admin 及子路径请求才带这个 Cookie

// Max-Age — 相对过期时间（秒），优先级高于 Expires

// Secure — 只在 HTTPS 连接中传输，中间人看不到明文

// HttpOnly — JS 完全不可读，只有浏览器发 HTTP 请求时自动带上
```

### 3. SameSite — Chrome 80 以来最重要的 Cookie 变化

```typescript
// Strict：任何跨站请求都不带 Cookie
// 场景：从邮件点链接进银行 → Cookie 不发送 → 需要重新登录（用户体验差但安全）
Set-Cookie: token=xxx; SameSite=Strict

// Lax：Chrome 80+ 的默认值，折中方案
// ✅ <a> 点击跳转 → 带 Cookie
// ✅ 顶层 GET 表单提交 → 带 Cookie
// ❌ <img src="跨站"> / iframe / fetch 跨站 → 不带 Cookie
Set-Cookie: token=xxx; SameSite=Lax

// None：完全放开跨站发送（必须搭配 Secure）
Set-Cookie: tracking=xxx; SameSite=None; Secure
```

**为什么这个变化影响巨大：** Chrome 80 将默认行为从 `SameSite=None` 改成了 `SameSite=Lax`。很多第三方登录、iframe 内嵌支付、跨域 SSO 突然全部失效——不是因为代码有 bug，是因为没有显式写 `SameSite=None; Secure`。

### 4. Cookie 安全三角 —— 缺一不可

| 攻击类型 | 防线 | 效果 |
|---|---|---|
| XSS 窃取 Cookie | `HttpOnly` | `document.cookie` 读不到 |
| 中间人拦截（HTTP 明文） | `Secure` | Cookie 只在 HTTPS 上传输 |
| CSRF 跨站伪造请求 | `SameSite=Lax` | 跨站 POST/PUT/DELETE 不带 Cookie |

三道防线是 **AND** 关系，不是 **OR**。只设 HttpOnly 不管中间人，只设 SameSite 不管 XSS —— 安全木桶的短板效应在这里极度残酷。

## 其实你每天都在用

- **登录态 "两周内免登录"** ：勾选后后端设 `Max-Age=1209600`，两周内每次请求自动带 sessionId 识别身份
- **多语言偏好无感切换**：`lang=zh-CN` 存在 Cookie 里，后端直接返回对应语言 HTML，用户无感知
- **第三方广告追踪**：网站 A 嵌入的广告 iframe 用 `SameSite=None` 的 Cookie 跨站追踪你的浏览行为
- **CSRF Token 双重验证**：后端下发 CSRF Token 到 Cookie（不设 HttpOnly），前端 JS 读出放进 `X-CSRF-Token` 请求头，服务器比对 Cookie 里和 Header 里是否一致
- **购物车未登录状态**：未登录用户的购物车数据存 Cookie（加密后的商品 ID 列表），下次访问还在

## 常见误解

- **❌ 误区：「Cookie 只有 4KB 太小，没什么用了」** 把 Cookie 当存储用本身就是用错了方向。Cookie 的价值是 **Transport**（自动随请求传输），不是 Storage。session id 只需要几十个字节就够，4KB 绰绰有余。

- **❌ 误区：「SameSite=Lax 能完全取代 CSRF Token」** Lax 解决了 POST/PUT/DELETE 的跨站 CSRF，但 GET 请求的 CSRF 依然可能——比如某个老接口用 `GET /api/delete?id=123` 做删除操作。CSRF Token + SameSite 是纵深防御，不是二选一。

- **❌ 误区：「前端设 Cookie 和后端设一样」** 完全不一样。`document.cookie` 无法设 `HttpOnly`，所以前端写入的任何 Cookie 都能被 JS 读走。真正需要保密的（session id、csrf token）必须由服务端 `Set-Cookie` 下发——这是单行道。

- **❌ 误区：「Domain 设 .example.com 向下兼容无所谓」** `Domain` 越宽，Cookie 暴露面越大。如果只有 `www.example.com` 需要用到某个 Cookie，不设 Domain 是最安全的——设成 `.example.com` 后，你的 staging 环境、CDN 域名、甚至恶意拼接子域名都能拿到这个 Cookie。

## 一句话总结

Cookie 三板斧：`HttpOnly` 让脚本看不见，`Secure` 让中间人截不到，`SameSite=Lax` 让跨站请求带不走——三条全忘的安全事故比三条全记得的多 100 倍。
