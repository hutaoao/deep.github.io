## 一句话概括

Cookie 是 HTTP 无状态协议的身份补丁——每次请求自动带上是它的超能力，但也意味着每次请求都被它拖慢。`HttpOnly` + `Secure` + `SameSite` 不是"建议"，是**底线**。丢一条，安全体系就缺一块。

## 核心知识点

### 1. Cookie 的两种写入方式——本质完全不同

```js
// ========== 方式一：后端 Set-Cookie（敏感数据的唯一通道）==========
// HTTP 响应头：
// Set-Cookie: session_id=abc123; HttpOnly; Secure; SameSite=Lax; Max-Age=3600; Path=/

// ========== 方式二：前端 document.cookie（永远设不了 HttpOnly）==========
document.cookie = 'theme=dark; Max-Age=2592000; Path=/; Secure';

// ⚠️ 读的时候是一整坨字符串，不是对象！
console.log(document.cookie); // "theme=dark; lang=zh"
// 手动解析：
const parseCookie = str =>
  Object.fromEntries(
    str.split('; ').map(pair => {
      const i = pair.indexOf('=');
      return [pair.slice(0, i), pair.slice(i + 1)];
    })
  );

// 删除 = 设过期时间为过去
document.cookie = 'theme=; Max-Age=0; Path=/';
```

**关键区分：** 前端 `document.cookie` **永远**设不了 `HttpOnly`。需要保密的内容（session id、refresh token）必须由后端 `Set-Cookie` 下发——这是单向通道。

### 2. 五大属性——每一个都影响安全面

```js
// 完整 Cookie 属性面板：
// Set-Cookie: key=val; Domain=site.com; Path=/app;
//             Max-Age=3600; Secure; HttpOnly; SameSite=Lax

// Domain：决定哪些子域名能拿到这个 Cookie
// 不设 → 仅当前域名（最安全，staging 环境拿不到）
// .example.com → api.example.com、admin.example.com 都能看到，攻击面扩大

// Path：限定 URL 前缀，Path=/admin 的 Cookie 只在 /admin/* 路径携带

// Max-Age vs Expires：Max-Age 是秒数（相对），Expires 是绝对时间
// Max-Age 优先级更高，推荐用 Max-Age（不依赖客户端时钟）

// Secure：只在 HTTPS 连接中传输
// HttpOnly：JS 完全不可读，浏览器只在发 HTTP 请求时自动带
```

### 3. SameSite——Chrome 80 以来最被低估的安全特性

```js
// Strict：最严，任何跨站场景都不带 Cookie
// 用户从 GitHub 邮件点链接跳转到你的网站 → Cookie 不发送 → 永远是"未登录"
Set-Cookie: token=xxx; SameSite=Strict

// Lax：Chrome 80+ 默认值，安全与体验的平衡点
// ✅ <a> 点击跳转、顶层 GET 表单 → 带 Cookie
// ❌ <img src="跨站"> / iframe / fetch(POST) → 不带
Set-Cookie: token=xxx; SameSite=Lax

// None：完全放开跨站携带 → 必须搭配 Secure
Set-Cookie: tracker=xxx; SameSite=None; Secure
```

**面试加分点：** Chrome 80 把默认从 `None` 改为 `Lax` 后，无数第三方登录、iframe 支付、跨域 SSO 突然挂掉——不是代码有 bug，是没显式声明 `SameSite=None; Secure`。这也是为什么面试官喜欢问"你遇到过 Cookie 莫名其妙丢失的情况吗？"

### 4. 三道防线——AND 关系，不是 OR

| 攻击 | 防线 | 原理 |
|---|---|---|
| XSS 窃取 | `HttpOnly` | `document.cookie` 读不到 |
| 中间人嗅探 | `Secure` | Cookie 只在 HTTPS 传输 |
| CSRF 伪造 | `SameSite=Lax` | 跨站 POST 不带 Cookie |

安全木桶原理在这里极其残酷：三条只做了两条 = 白做。只设 HttpOnly 不管中间人？HTTP 公网 WiFi 上明文被抓。只设 SameSite 不管 XSS？注入一个 script 照样偷。

## 其实你每天都在用

- **"两周内免登录"**：勾选后后端设 `Max-Age=1209600`，两周内每次请求自动带 sessionId，再也不用输密码
- **多语言站点无感切换**：`lang=zh-CN` 存 Cookie，后端直接返回对应语言页面，URL 都不用改
- **CSRF Token 双写验证**：后端下发 CSRF Token 到 Cookie（不设 HttpOnly 让 JS 可读），前端读出放进 `X-CSRF-Token` 请求头，服务器比对 Cookie 和 Header 里的是否一致
- **第三方广告追踪**：网站 A 里嵌的广告 iframe 用 `SameSite=None` Cookie 跨站追踪你浏览了哪些商品
- **未登录购物车**：商品 ID 加密后存 Cookie，登录前加入购物车的东西登录后还在

## 常见误解

- **❌ 误区：「Cookie 只有 4KB，太废了」** 把 Cookie 当存储用本身就是方向性错误。Cookie 的价值在于 **Transport**（自动随请求传输），不是 Storage。sessionId 只需要几十字节，4KB 绰绰有余。存数据请用 localStorage/IndexedDB。

- **❌ 误区：「SameSite=Lax 可以完全替代 CSRF Token」** Lax 解决了 POST/PUT/DELETE 的跨站伪造，但 `GET /api/delete?id=123` 这样的老接口依然可以跨站触发（link 导航带 Cookie）。CSRF Token + SameSite 是纵深防御，不是二选一。

- **❌ 误区：「Domain=.example.com 方便，向下兼容」** Domain 越宽暴露面越大。如果只有 `www.example.com` 需要这个 Cookie，不设 Domain 最安全。设了 `.example.com` 后，你的 staging、CDN 域名甚至攻击者拼接的子域名都能拿到。

- **❌ 误区：「document.cookie 跟后端 Set-Cookie 一样，都能设置 Cookie」** 前端只能设置能被 JS 读写的 Cookie（无 HttpOnly），且操作的是字符串拼接而非结构化 API。敏感内容（session id、refresh token）必须走 `Set-Cookie` 响应头单行道。

## 一句话总结

`HttpOnly` 让脚本看不见，`Secure` 让中间人截不到，`SameSite=Lax` 让跨站请求带不走——三条全忘的事故比三条全记得的多一百倍。
