## 一句话概括

React 多节点 Diff 的第一轮遍历是一个"从左到右"的线性扫描过程，依次比较新 React Element 数组和旧 Fiber 链表中对应位置的节点，如果 key 和 type 都匹配则复用并继续，一旦出现 key 不匹配或 type 变化则立即跳出第一轮，进入第二轮处理。

## 背景与意义

### 为什么 Diff 需要多轮？

在 Stack Reconciler 时代，React 对列表的 Diff 采用**完整的双端比较算法**——同时从两端向中间逼近，在一次遍历中完成所有操作。这种方式很强大，但也很复杂且容易产生误判。

Fiber 架构重构了 Diff 策略，采用**三轮遍历**的分治思路：

1. **第一轮**：处理"从头开始匹配"的部分（最常见的批处理场景）
2. **第二轮**：处理剩余的新老节点，包括新增、删除、移动
3. **第三轮**：处理剩余标记为待删除的节点

这种三轮策略是"基于统计优化的设计"——在实际应用中，**大多数列表更新是追加或仅头部变化**，第一轮遍历就能处理大多数情况，进入第二三轮的场景相对较少。

### 数据驱动：为什么第一轮就跳出？

假设一个 Todo 列表：

```jsx
// 旧列表（Fiber 链表）
[Todo_1, Todo_2, Todo_3, Todo_4, Todo_5]

// 新列表（React Elements）
[Todo_1, Todo_2, Todo_3, Todo_4, Todo_5]
```

典型的更新场景：

| 场景 | 第一轮能处理多少 | 是否进入第二轮 |
|------|:--------------:|:-------------:|
| 追加一个 Todo 到末尾 | 5/5 | ❌（不需要） |
| 删除第一个 Todo | 0/4 | ✅（类型变化） |
| 在中间插入一个 Todo | 3/4 | ✅（key 不匹配） |
| 只更新了部分 props | 5/5 | ❌（全部复用） |

据 React 团队统计，绝大多数列表更新中，**超过 90% 的节点在第一轮就能完成匹配**。因此将"第一轮快速遍历 + 第二轮处理异常"作为默认策略是效率最优的选择。

## 概念与定义

### 第一轮遍历（First Pass）

多节点 Diff 函数 `reconcileChildrenArray` 中的循环过程，从旧的 fiber 链表的第一个节点和新 React Elements 数组的第一个元素开始，逐个比较。

### newIdx 与 oldFiber

```typescript
function reconcileChildrenArray(returnFiber, currentFirstChild, newChildren, lanes) {
  // oldFiber: 从 currentFirstChild 开始遍历的 Fiber 链表
  // newIdx: 新 children 数组的索引
  
  let resultingFirstChild = null;
  let previousNewFiber = null;
  
  let oldFiber = currentFirstChild;
  let nextOldFiber = null;
  let newIdx = 0;
  
  // 第一轮：从左到右逐个比较
  for (; oldFiber !== null && newIdx < newChildren.length; newIdx++) {
    nextOldFiber = oldFiber.sibling;
    
    const newFiber = updateSlot(returnFiber, oldFiber, newChildren[newIdx], lanes);
    if (newFiber === null) {
      // updateSlot 返回 null 表示 key 不匹配 → 跳出第一轮
      break;
    }
    
    // ... 处理复用的节点 ...
    
    oldFiber = nextOldFiber;
  }
  
  // ... 进入第二轮或第三轮 ...
}
```

### updateSlot

比较一个旧 Fiber 和一个新 Element 的 key 和 type 的函数：

```typescript
function updateSlot(returnFiber, oldFiber, newChild, lanes) {
  const key = oldFiber.key;
  
  // 检查 key 是否匹配
  if (newChild !== null && typeof newChild === 'object') {
    switch (newChild.$$typeof) {
      case REACT_ELEMENT_TYPE:
        if (newChild.key === key) {
          // key 匹配，继续比较 type
          if (newChild.type === oldFiber.type) {
            return updateElement(returnFiber, oldFiber, newChild, lanes);
          }
          // key 匹配但 type 不匹配
          return updateElementWithNull(returnFiber, oldFiber, newChild, lanes);
        }
        break;
      // ... 处理 portal, lazy 等类型 ...
    }
  }
  
  // key 不匹配，返回 null 表示无法在此 slot 复用
  return null;
}
```

**`updateSlot` 返回 null 是触发第一轮跳出的直接原因。**

## 核心知识点拆解

### 1. 第一轮遍历的三种出口

**出口 A：正常结束**

当 `oldFiber === null` 或 `newIdx >= newChildren.length` 时退出：

```typescript
// 场景：所有旧节点都匹配了新节点
// newChildren.length = 5, oldFiber 链表节点数 = 5
// 遍历 5 次后 oldFiber === null, newIdx = 5
// 此时 oldFiber === null 且 newIdx === newChildren.length
// 全部处理完毕，无需进入第二三轮
if (oldFiber === null) {
  // 所有旧节点都被复用
  // newChildren 中的剩余节点（如果有）→ 新增
  // 进入第三轮处理剩余的新节点
}
```

**出口 B：新节点先耗尽**

```typescript
// 场景：新节点比旧节点少
// oldFiber 链表节点数 = 5, newChildren.length = 3
// 遍历 3 次后 newIdx = 3 === newChildren.length
// 但 oldFiber 还在（还有 2 个未匹配）
// 进入第三轮：删除 oldFiber 及其剩余的兄弟
if (newIdx === newChildren.length) {
  // 还有剩余的 oldFiber → 全部标记为删除
  deleteRemainingChildren(returnFiber, oldFiber);
  return resultingFirstChild;
}
```

**出口 C：冲突退出（最常见）**

```typescript
// 场景：遇到 key 不匹配或 type 变化，提前跳出
// updateSlot 返回 null → break
// 此时 newIdx 还未遍历完所有新节点
// oldFiber 也还有剩余
// 进入第二轮：通过 Map 查找来匹配剩余节点
```

### 2. 每个匹配节点上的具体操作

当 `updateSlot` 判断 key 和 type 都匹配时，React 会对该 Fiber 执行 `updateElement`（复用已有 DOM + 更新 props）：

```typescript
// 在第一轮循环中
if (newFiber === null) {
  break; // key 不匹配，跳出
}

if (newFiber.alternate === null) {
  // 这是新创建的 fiber，不是复用的
  // 所以 oldFiber 应该被删除
  deleteChild(returnFiber, oldFiber);
}

// 将 newFiber 连接到结果链表
if (previousNewFiber === null) {
  resultingFirstChild = newFiber;
} else {
  previousNewFiber.sibling = newFiber;
}
previousNewFiber = newFiber;

oldFiber = nextOldFiber;
```

这里有一个需要注意的细节：`updateElement` 返回的 Fiber 可能是复用 oldFiber（通过 `useFiber` 创建 workInProgress）也可能是新创建的 Fiber。它是通过检查 `alternate` 是否有值来判断的。

### 3. 第一轮跳出的两种子情况

以下面这个典型场景为例：

```jsx
// 旧列表（Fiber 链表，key 由内到外）
<li key="a">A</li> → <li key="b">B</li> → <li key="c">C</li> → <li key="d">D</li>

// 新列表（React Elements）
<li key="a">A</li> → <li key="b">B</li> → <li key="x">X</li> → <li key="c">C</li> → <li key="d">D</li>
```

第一轮遍历过程：

```
newIdx=0: new.key="a" vs old.key="a" → 匹配 → 复用
newIdx=1: new.key="b" vs old.key="b" → 匹配 → 复用
newIdx=2: new.key="x" vs old.key="c" → 不匹配 → 跳出 
         ↑ 此时 oldFiber 停在 key="c" 的节点上
```

**跳出时状态**：
- 已处理：2 个（key="a" 和 key="b"）
- 旧剩余：`key="c" → key="d"`
- 新剩余：`key="x" → key="c" → key="d"`

**注意**：`updateSlot` 检查的是 key 是否匹配，如果 key 匹配但 type 不同，也会跳出。例如：

```jsx
// 旧列表
<span key="a">A</span> → <li key="b">B</li>

// 新列表
<li key="a">A</li> → <div key="b">B</div>

// 第一轮遍历：
// newIdx=0: new.key="a" vs old.key="a" → key 匹配
//          new.type="li" vs old.type="span" → type 不匹配
//          虽然 key 匹配，但 type 不同 → updateSlot 返回新 fiber 但不复用 old
//          实际也会跳出循环
//          → 注意：这里 updateSlot 不返回 null，但 oldFiber 会被标记删除
```

function FirstPassDemo() {

        <ul style={{ paddingLeft: 20 }}>

                borderRadius: 4,
              }}>
                {item}
                <span style={{ marginLeft: 8, fontSize: 11, color: '#666' }}>

                    ? '(第一轮复用✓)'
                    : '(第二轮处理)'}
                </span>

function getFirstPassCount(scenario, items) {
  switch (scenario) {
    case 'append':
      return 3; // A, B, C 都在第一轮复用
    case 'prepend':
      return 0; // 第一个就不匹配，全部进入第二轮
    case 'delete_first':
      return 0; // 第一个就 type 不匹配
    case 'replace_middle':
      return 2; // A, B 在第一轮复用，到 X 跳出
    default:
      return 0;
  }
}

function FirstPassAnalysis({ scenario, items }) {
  const getOldList = (s) => {
    switch (s) {
      case 'append': return ['Item A', 'Item B', 'Item C'];
      case 'prepend': return ['Item A', 'Item B', 'Item C'];
      case 'delete_first': return ['Item A', 'Item B', 'Item C'];
      case 'replace_middle': return ['Item A', 'Item B', 'Item C'];
      default: return [];
    }
  };

  const oldList = getOldList(scenario);
  
  const lines = [];
  lines.push('┌─────────────────────────────────────────────┐');
  lines.push('│ 第一轮遍历：从左到右逐个比较 key 和 type         │');
  lines.push('└─────────────────────────────────────────────┘');
  
  for (let i = 0; i < Math.max(oldList.length, items.length); i++) {
    const oldKey = oldList[i] ? oldList[i] : '(无)';
    const newKey = items[i] ? items[i] : '(无)';
    
    if (oldList[i] && items[i]) {
      // 位置 i 上都有
      const isMatchingKey = oldList[i] === items[i];
      const isSameType = true; // 简化：假设 type 相同
      
      if (isMatchingKey && isSameType) {
        lines.push(`  [${i}] key=${oldKey} → key=${newKey}  ✅ 匹配（复用）`);
      } else if (isMatchingKey && !isSameType) {
        lines.push(`  [${i}] key=${oldKey} → key=${newKey}  ❌ key 匹配但 type 不同 → 跳出`);
      } else {
        lines.push(`  [${i}] key=${oldKey} → key=${newKey}  ❌ key 不匹配 → 跳出`);
      }
    } else if (oldList[i]) {
      lines.push(`  [${i}] key=${oldKey} → (无新节点)  旧节点剩余 → 进入第三轮`);
    } else {
      lines.push(`  [${i}] (无旧节点) → key=${newKey}  新节点剩余 → 进入第二轮`);
    }
  }
  
  return (
    <div style={{
      marginTop: 16,
      background: '#1e1e1e',
      color: '#d4d4d4',
      padding: 12,
      borderRadius: 8,
      fontFamily: 'monospace',
      fontSize: 12,
      lineHeight: 1.6,
    }}>
      {lines.map((line, i) => (
        <div key={i}>{line}</div>
      ))}
    </div>
  );
}

export default FirstPassDemo;

## 底层原理（含源码分析）

### 1. reconcileChildrenArray 的完整流程

```typescript
// 源码位置：packages/react-reconciler/src/ReactChildFiber.js

function reconcileChildrenArray(
  returnFiber: Fiber,
  currentFirstChild: Fiber | null,
  newChildren: Array<any>,
  lanes: Lanes,
): Fiber | null {
  let resultingFirstChild: Fiber | null = null;
  let previousNewFiber: Fiber | null = null;
  
  let oldFiber = currentFirstChild;
  let nextOldFiber = null;
  let newIdx = 0;
  
  // ============ 第一轮：从左到右比较 ============
  for (; oldFiber !== null && newIdx < newChildren.length; newIdx++) {
    nextOldFiber = oldFiber.sibling;
    
    // updateSlot 尝试在旧 fiber 链表的当前位置匹配新元素
    const newFiber = updateSlot(returnFiber, oldFiber, newChildren[newIdx], lanes);
    
    if (newFiber === null) {
      // updateSlot 返回 null → key 不匹配 → 跳出第一轮
      if (previousNewFiber === null) {
        resultingFirstChild = null;
      }
      break;
    }
    
    // 处理可复用的节点
    if (previousNewFiber === null) {
      resultingFirstChild = newFiber;
    } else {
      previousNewFiber.sibling = newFiber;
    }
    previousNewFiber = newFiber;
    
    oldFiber = nextOldFiber;
  }
  
  // ============ 第一轮结束，判断出口 ============
  
  // 情况 1：所有旧节点都被处理完毕
  if (oldFiber === null) {
    // 剩余的新节点都是新增的
    for (; newIdx < newChildren.length; newIdx++) {
      const newFiber = createChild(returnFiber, newChildren[newIdx], lanes);
      if (newFiber === null) continue;
      
      if (previousNewFiber === null) {
        resultingFirstChild = newFiber;
      } else {
        previousNewFiber.sibling = newFiber;
      }
      previousNewFiber = newFiber;
    }
    return resultingFirstChild;
  }
  
  // 情况 2：新节点先耗尽
  if (newIdx === newChildren.length) {
    // 删除所有剩余的旧节点
    deleteRemainingChildren(returnFiber, oldFiber);
    return resultingFirstChild;
  }
  
  // ============ 情况 3：双方都还有剩余 → 进入第二轮 ============
  
  // 将剩余的旧 fiber 放入 Map
  const existingChildren = mapRemainingChildren(returnFiber, oldFiber);
  
  // 第二轮：遍历剩余的新节点，在 existingChildren Map 中寻找匹配
  for (; newIdx < newChildren.length; newIdx++) {
    const newFiber = updateFromMap(
      existingChildren,
      returnFiber,
      newIdx,
      newChildren[newIdx],
      lanes,
    );
    
    if (newFiber !== null) {
      if (newFiber.alternate !== null) {
        // 从 Map 中找到了可复用的旧 fiber
        // 判断是否需要移动
        existingChildren.delete(
          newFiber.key === null ? newIdx : newFiber.key
        );
      }
      // ... 连接到结果链表 ...
    }
  }
  
  // 第三轮：删除所有未被复用的旧节点
  existingChildren.forEach(child => deleteChild(returnFiber, child));
  
  return resultingFirstChild;
}
```

### 2. updateSlot 的完整实现

```typescript
function updateSlot(
  returnFiber: Fiber,
  oldFiber: Fiber,
  newChild: any,
  lanes: Lanes,
): Fiber | null {
  // 检查 key 是否匹配
  const key = oldFiber.key;
  
  if (typeof newChild === 'string' || typeof newChild === 'number') {
    // 文本节点
    if (key !== null) {
      return null; // 旧节点有 key 但新的是文本 → 不匹配
    }
    return updateTextNode(returnFiber, oldFiber, '' + newChild, lanes);
  }
  
  if (newChild !== null && typeof newChild === 'object') {
    switch (newChild.$$typeof) {
      case REACT_ELEMENT_TYPE: {
        if (newChild.key === key) {
          // key 匹配
          if (newChild.type === REACT_FRAGMENT_TYPE) {
            // Fragment 的特殊处理
            if (oldFiber.tag === Fragment) {
              return updateFragment(returnFiber, oldFiber, newChild.props.children, lanes);
            }
          } else {
            // 普通元素
            if (oldFiber.elementType === newChild.type) {
              return updateElement(returnFiber, oldFiber, newChild, lanes);
            }
            // key 相同但 type 不同 → 标记删除旧节点，返回新创建的 fiber
            // 注意这里不会返回 null，不跳出第一轮
            return updateElementWithNull(
              returnFiber, oldFiber, newChild, lanes, key,
            );
          }
        }
        break;
      }
      case REACT_PORTAL_TYPE:
        // 类似处理 key
        break;
      case REACT_LAZY_TYPE:
        // lazy 类型
        break;
    }
    
    // key 不匹配，返回 null
    if (newChild.key !== key) {
      return null;
    }
  }
  
  return null;
}
```

### 3. 关于 `updateElementWithNull`

当 key 匹配但 type 不匹配时，`updateSlot` 并不会返回 null，而是调用 `updateElementWithNull`。这个函数会**标记旧 fiber 为删除，并创建一个新的 fiber**：

```typescript
function updateElementWithNull(
  returnFiber: Fiber,
  oldFiber: Fiber,
  newChild: ReactElement,
  lanes: Lanes,
  key: any,
): Fiber {
  // 标记旧 fiber 为删除
  deleteChild(returnFiber, oldFiber);
  
  // 创建新的 fiber
  const created = createFiberFromElement(newChild, returnFiber.mode, lanes);
  created.ref = newChild.ref;
  created.return = returnFiber;
  return created;
}
```

这意味着：**key 相同但 type 不同时，虽然 React 会跳出第一轮遍历，但在跳出之前已经被标记删除并创建了新的 fiber**。这个 fiber 会被连接但不会被遍历复用（因为是新创建的，alternate === null）。

## 高频面试题解析

### Q1：多节点 Diff 第一轮遍历的原理是什么？为什么不直接 map 匹配？

**参考答案：**

第一轮遍历的原理是"从左到右、顺序对应"——认为大多数情况下，列表中的元素位置不会大范围变动。这种策略的依据是**实际 UI 交互的统计规律**：

- **追加**：向列表末尾添加新项
- **前置**：在列表开头插入新项（但不常见，因为大多数列表是"历史的"或"排序后的"）
- **删除**：移除一个或少量项
- **更新**：只改 props 不改列表结构

之所以不直接使用 map（哈希表）匹配，是因为：
1. **大多数情况下不需要** map：如果列表没有变化，线性扫描比哈希查找更快
2. **减少内存开销**：只有在需要进入第二轮时才会创建 map，避免不必要的哈希表构建
3. **更好的局部性**：线性扫描对 CPU 缓存友好，while 循环比 map 查找 + 遍历的开销更低

### Q2：第一轮遍历跳出后，React 如何处理？

**参考答案：**

第一轮遍历跳出后，React 进入**第二轮遍历**，流程如下：

1. 将剩余的旧 fiber（从 `oldFiber` 开始）放入一个 Map 结构中，key 为 fiber 的 key，value 为 fiber 节点
2. 遍历剩余的新 children 数组（从 `newIdx` 开始）
3. 对每个新 child，在 Map 中查找是否有相同 key 的旧 fiber
4. 如果找到且 type 匹配 → 复用（需要判断是否需要移动位置）
5. 如果找到但 type 不匹配 → 标记旧 fiber 删除，创建新的
6. 如果在 Map 中没找到 → 创建新的

第二轮结束后进入第三轮，删除 Map 中所有未被匹配的旧 fiber。

### Q3：第一轮遍历中，什么情况下会直接复用节点而不会退出？

**参考答案：**

直接复用的条件是 `updateSlot` 返回非 null 且 oldFiber 被标记为已复用（alternate 不为 null）：

- 旧 fiber.key === 新 element.key
- 旧 fiber.elementType === 新 element.type
- 或者旧 fiber.elementType === REACT_FRAGMENT_TYPE 且新旧都是 Fragment

在这种情况下，`updateSlot` 调用 `updateElement`，内部通过 `useFiber`（即 `createWorkInProgress`）复用 Fiber 对象，仅更新 `pendingProps`。而且 `updateElement` 返回的 fiber 的 alternate 指向 oldFiber，标记为"已复用"。

如果上述条件不满足但 key 相同，`updateSlot` 返回新 fiber 但不复用旧 fiber（oldFiber 被标记删除），结果为：**旧节点被删除，新节点被创建**。

## 总结与扩展

### 总结

多节点 Diff 的第一轮遍历是一个"先快速扫描，有异常再完整处理"的优化设计。它的三个关键特征：

1. **线性扫描**：从左到右逐对比较，时间复杂度 O(n) 的最佳情况
2. **提前跳出**：遇到不匹配立即 break，减少无效比较
3. **三路出口**：正常结束（全部匹配/追加）、新节点耗尽（删除）、冲突跳出（进入第二轮）

### 扩展：为什么不是双端比较？

Vue 3 的 Diff 采用双端比较（从两端向中间逼近），而 React 选择了单向扫描。这反映了两种不同的设计哲学：

- **React 的考量**：单向扫描 + map 容错的设计，代码更简单，可读性更高。更重要的是，React 的 Fiber 链表是**单向链表**（只有 return、child、sibling），不支持从尾部反向遍历。做双端比较需要额外的数据结构支持
- **Vue 的考量**：Vue 3 维护的是"新旧孩子数组"（普通数组），天然支持双端操作。Vue 3 在列表变化较大时的表现可能略优，但在无变化或追加时两者性能相当

这本质上是"工程复杂度"和"理论最优性"的取舍——React 选择用更简单的策略处理前 90% 的场景，剩余 10% 由 map 查找来兜底。
