## 一句话概括

Vue3 的响应式系统基于 **Proxy** 代理对象实现数据的拦截与劫持，通过 **依赖收集（track）** 与 **触发更新（trigger）** 的核心机制，将数据变化自动同步到视图或副作用函数中。本文将从零开始手写一套完整的简易响应式系统，涵盖 `reactive`、`ref`、`computed`、`watch` 和 `effect` 的核心实现。

## 背景与意义

### 为什么需要手写响应式系统？

Vue 作为前端三大框架之一，其响应式机制是区别于 React 和 Angular 的核心差异化能力。对于前端工程师而言，理解响应式系统的内部原理至少有以下几层意义：

**第一层：面试突围**。Vue3 响应式原理是前端面试的"必考题"，从 `Proxy` 到 `依赖收集`，从 `Reflect` 到 `调度器`，几乎每一家大厂都会围绕这些概念展开深度追问。

**第二层：框架反哺**。当你在业务中遇到诡异的数据更新问题——比如数据变了但视图没更新、computed 不重新计算、watch 不触发——如果你不了解响应式系统内部的 track 和 trigger 机制，排查问题的效率会极低。

**第三层：架构视野**。响应式系统的本质是一个**发布-订阅模式**的优雅实践。`track` 是"订阅"环节，`trigger` 是"发布"环节。这种模式不仅存在于 Vue 中，在 RxJS、MobX、SolidJS 等库中也能看到它的影子。掌握了它，你就掌握了一种通用的数据驱动架构设计思想。

### Vue2 vs Vue3 的响应式演进

| 对比维度 | Vue2 (Object.defineProperty) | Vue3 (Proxy) |
|---------|-------------------------------|--------------|
| 拦截粒度 | 属性级别，需要递归遍历所有 key | 对象级别，懒递归 |
| 新增/删除属性 | 无法拦截，需 `$set`/`$delete` | 天然支持 |
| 数组监听 | 重写 7 个变异方法，有局限 | 天然拦截所有数组操作 |
| 性能 | 大对象递归耗时严重 | 懒递归，按需代理 |
| 兼容性 | IE11 可 | 不支持 IE |

Vue3 的 Proxy 方案从根本上解决了 Vue2 的三大痛点：**新增属性监听**、**数组索引变更监听**、**大对象初始化性能**。这也是理解手写响应式系统的出发点。

## 概念与定义

在进入编码之前，先明确几个核心概念：

### Reactive（响应式对象）

通过 `Proxy` 将普通 JavaScript 对象包裹成一个**响应式代理**。当读取属性时触发 `track`（依赖收集），当修改属性时触发 `trigger`（派发更新）。

### Effect（副作用函数）

响应式系统的"消费者"。当响应式数据变化时，与之关联的 `effect` 函数会自动重新执行。在 Vue 中，组件的渲染函数就是一个 `effect`。

### Track（依赖收集）

在响应式对象的 `getter` 中执行。它的工作是将当前正在运行的 `effect` 与目标属性关联起来，存储到一个全局的依赖映射表中。

### Trigger（触发更新）

在响应式对象的 `setter` 中执行。它的工作是找到与目标属性关联的所有 `effect`，依次执行它们。

### Ref（值类型响应式）

`reactive` 只能代理对象，对于基本类型（number、string、boolean）无能为力。`ref` 通过一个 `{ value: ... }` 的包装对象，让基本类型也能享受响应式能力。

### Computed（计算属性）

基于其他响应式数据派生出新值，具有**惰性求值**和**缓存**特性。只有依赖的数据发生变化时才会重新计算。

### Watch（侦听器）

监听一个或多个响应式数据的变化，当变化发生时执行用户定义的回调函数。它允许我们**获取变化前后的值**，并支持深度监听。

这些概念构成了 Vue3 响应式系统的核心骨架。接下来，我们将用不到 200 行代码把它们全部实现。

## 最小示例

在开始手写之前，先看看我们最终要达成的效果：

```javascript
// 创建一个响应式对象
const state = reactive({ count: 0, name: 'Vue3' })

// 副作用：自动追踪依赖
effect(() => {
  console.log(`count 变化了: ${state.count}`)
})

// ref 包装基本类型
const msg = ref('Hello')
effect(() => {
  console.log(`msg: ${msg.value}`)
})

// 计算属性
const doubled = computed(() => state.count * 2)
console.log(doubled.value) // 0

// 侦听器
watch(() => state.count, (newVal, oldVal) => {
  console.log(`watch count: ${oldVal} -> ${newVal}`)
})

// 修改数据 -> 自动触发更新
state.count++  // 输出: count 变化了: 1
              // 输出: watch count: 0 -> 1
console.log(doubled.value) // 2
```

这个最小示例勾勒了我们要手写的全部 API。接下来从最底层的 `track` 和 `trigger` 开始，逐层构建。

## 核心知识点拆解

### 第一层：全局数据结构

响应式系统需要一个全局的数据结构来存储"**属性 → 依赖它的 effects**"的映射关系。

```javascript
// 全局依赖映射表
// targetMap: Map<target, Map<key, Set<effect>>>
const targetMap = new WeakMap()
// 当前正在执行的 effect
let activeEffect = null
// effect 栈，用于处理嵌套 effect
const effectStack = []
```

这里的 `WeakMap` 非常关键：如果 `target` 对象被垃圾回收了，其对应的依赖映射也自动被回收，避免了内存泄漏。

### 第二层：依赖收集函数 track

```javascript
/**
 * track - 依赖收集
 * @param {Object} target - 响应式对象的原始对象
 * @param {string|symbol} key - 访问的属性名
 */
function track(target, key) {
  // 没有活跃的 effect，无需收集
  if (!activeEffect) return

  // 获取 target 对应的 depsMap
  let depsMap = targetMap.get(target)
  if (!depsMap) {
    depsMap = new Map()
    targetMap.set(target, depsMap)
  }

  // 获取 key 对应的 dep (Set<effect>)
  let dep = depsMap.get(key)
  if (!dep) {
    dep = new Set()
    depsMap.set(key, dep)
  }

  // 收集 effect
  if (!dep.has(activeEffect)) {
    dep.add(activeEffect)
    // 反向收集：effect 记住自己属于哪些 dep，用于清理
    activeEffect.deps.push(dep)
  }
}
```

关键设计点：
- `activeEffect` 为空时直接返回，避免不必要的内存占用
- 三层数据结构：`WeakMap<target, Map<key, Set<effect>>>`
- **反向收集**：每个 effect 维护一个 `deps` 数组，存储它被哪些 dep 集合包含，这是实现**分支切换**（branch switching）清理的基础

### 第三层：触发更新函数 trigger

```javascript
/**
 * trigger - 触发更新
 * @param {Object} target - 原始对象
 * @param {string|symbol} key - 被修改的属性名
 */
function trigger(target, key) {
  const depsMap = targetMap.get(target)
  if (!depsMap) return

  const dep = depsMap.get(key)
  if (!dep) return

  // 拷贝一份 effects 列表，避免 set 操作过程中影响原始集合
  const effectsToRun = new Set(dep)

  effectsToRun.forEach(effect => {
    // 避免无限循环：trigger 中正在执行的 effect 不应该再次触发
    if (effect !== activeEffect) {
      // 支持调度器
      if (effect.options?.scheduler) {
        effect.options.scheduler(effect)
      } else {
        effect()
      }
    }
  })
}
```

`trigger` 的设计要点：
- 使用 `new Set(dep)` 拷贝，防止执行 effect 时对 `dep` 的增删影响遍历
- 检查 `effect !== activeEffect` 防止无限递归
- 支持**调度器（scheduler）**，这是 `computed` 和异步更新队列的基础

### 第四层：effect 函数

`effect` 是响应式系统的**执行引擎**。它负责执行传入的函数，并在首次执行时完成依赖收集。

```javascript
/**
 * effect - 创建响应式副作用
 * @param {Function} fn - 要执行的函数
 * @param {Object} options - 配置项
 * @returns {Function} effect 的 runner 函数
 */
function effect(fn, options = {}) {
  // 创建 effect 函数
  const effectFn = () => {
    // 清理旧的依赖关系
    cleanup(effectFn)

    // 将当前 effect 设为活跃
    activeEffect = effectFn
    effectStack.push(effectFn)

    try {
      // 执行 fn，fn 中访问响应式数据会触发 track
      const result = fn()
      return result
    } finally {
      // 恢复上一个 effect
      effectStack.pop()
      activeEffect = effectStack[effectStack.length - 1]
    }
  }

  // 存储配置和反向依赖列表
  effectFn.deps = []
  effectFn.options = options

  // 非 lazy 模式下立即执行
  if (!options.lazy) {
    effectFn()
  }

  return effectFn
}

/**
 * cleanup - 清理旧的依赖关系
 * 这是分支切换的关键：每次执行前清空旧的依赖，执行时重新收集
 */
function cleanup(effectFn) {
  for (let i = 0; i < effectFn.deps.length; i++) {
    const dep = effectFn.deps[i]
    dep.delete(effectFn)
  }
  effectFn.deps.length = 0
}
```

这段代码中有几个精妙的设计：

**1. cleanup + 重新收集**：为什么每次执行前都要 cleanup？考虑一个条件渲染的场景：

```javascript
const state = reactive({ show: true, text: 'hello', temp: 'world' })
effect(() => {
  console.log(state.show ? state.text : state.temp)
})
state.show = false // 切换分支：依赖从 text 变为 temp
```

如果不清理旧的依赖，修改 `state.text` 时也会触发 effect 执行（虽然不再使用 `text`），造成了不必要的性能浪费。cleanup 机制确保了**当前真正使用的依赖**才会被收集。

**2. effectStack 处理嵌套**：Vue 组件中很可能出现嵌套 effect，effectStack 保存了调用链，让 `activeEffect` 可以正确恢复。

### 第五层：reactive 函数

有了 `track` 和 `trigger`，`reactive` 的实现就水到渠成了：

```javascript
/**
 * reactive - 创建响应式对象
 * @param {Object} target - 原始对象
 * @returns {Proxy} 响应式代理对象
 */
function reactive(target) {
  // 使用 Proxy 创建代理
  const proxy = new Proxy(target, {
    get(target, key, receiver) {
      // 依赖收集：谁在读取这个属性
      track(target, key)
      // 使用 Reflect 确保 this 指向正确
      const result = Reflect.get(target, key, receiver)

      // 懒递归：只有真正读取到对象时才递归代理
      if (typeof result === 'object' && result !== null && !isReactive(result)) {
        return reactive(result)
      }
      return result
    },

    set(target, key, value, receiver) {
      // 先判断是新增属性还是修改属性
      const oldValue = target[key]
      const hadKey = Object.prototype.hasOwnProperty.call(target, key)

      // 执行写入
      const result = Reflect.set(target, key, value, receiver)

      // 触发更新
      if (!hadKey) {
        // 新增属性
        trigger(target, key)
      } else if (oldValue !== value && (oldValue === oldValue || value === value)) {
        // 修改属性：排除 NaN !== NaN 的情况
        trigger(target, key)
      }

      return result
    },

    deleteProperty(target, key) {
      const hadKey = Object.prototype.hasOwnProperty.call(target, key)
      const result = Reflect.deleteProperty(target, key)
      if (hadKey && result) {
        trigger(target, key)
      }
      return result
    },

    has(target, key) {
      track(target, key)
      return Reflect.has(target, key)
    },

    ownKeys(target) {
      // ownKeys 没有 key，用 Symbol('iterate') 作为 key
      track(target, Symbol.for('iterate'))
      return Reflect.ownKeys(target)
    }
  })

  // 标记已代理
  reactiveMap.set(target, proxy)
  return proxy
}

// 辅助：判断对象是否已代理
const reactiveMap = new WeakMap()
function isReactive(target) {
  return reactiveMap.has(target)
}
```

**关键设计点详解：**

**1. 使用 Reflect**：`Reflect.get(target, key, receiver)` 中的 `receiver` 参数非常重要。它确保当对象继承时，`this` 指向正确的接收者。如果没有 `receiver`，原型链上的属性访问会丢失响应式。

**2. 懒递归**：Vue2 在初始化时就递归遍历所有属性用 `Object.defineProperty` 转换成 getter/setter，性能开销大。Vue3 只在**真正访问到对象属性**时才递归代理。如果你有一个很大的对象树，但只访问了根属性，深层对象的代理永远不会创建。

**3. 严格区分新增和修改**：新增属性时，Vue2 无法拦截（需要 `$set`），而在 Proxy 中通过 `hasOwnProperty` 的检查天然支持。

**4. 正确处理 NaN**：`NaN !== NaN` 在 JS 中是 true，因此需要额外的 `oldValue === oldValue || value === value` 判断来避免 NaN 导致的错误触发。

### 第六层：ref 函数

`reactive` 无法代理基本类型（如数字、字符串），`ref` 通过包装对象解决了这个问题：

```javascript
/**
 * ref - 创建值类型响应式
 * @param {any} rawValue - 原始值
 * @returns {RefImpl} 具有 value 属性的响应式对象
 */
function ref(rawValue) {
  // 如果已经是 ref，直接返回
  if (isRef(rawValue)) return rawValue

  return new RefImpl(rawValue)
}

class RefImpl {
  constructor(value) {
    this.__v_isRef = true  // 标记为 ref
    this._value = toReactive(value) // 如果值是对象，用 reactive 包装
  }

  get value() {
    // 依赖收集：this._value 是响应式的或被追踪的值
    track(this, 'value')
    return this._value
  }

  set value(newVal) {
    if (newVal !== this._value) {
      this._value = toReactive(newVal)
      trigger(this, 'value')
    }
  }
}

function toReactive(value) {
  return typeof value === 'object' && value !== null
    ? reactive(value)
    : value
}

function isRef(value) {
  return value && value.__v_isRef === true
}
```

`ref` 的核心思路是用一个包含 `value` 属性的对象来包裹基本类型。这样就能复用 `track`/`trigger` 机制。特别地，`ref` 传入对象时，内部会用 `reactive` 代理它——因此 `ref({ count: 1 }).value.count` 也是响应式的。

### 第七层：computed 函数

`computed` 是 Vue 中高频使用的特性，它的核心是**懒计算 + 缓存标记**：

```javascript
/**
 * computed - 计算属性
 * @param {Function|Object} getterOrOptions - getter 函数或 { get, set } 对象
 * @returns {ComputedRefImpl} 计算属性 ref
 */
function computed(getterOrOptions) {
  let getter, setter

  if (typeof getterOrOptions === 'function') {
    getter = getterOrOptions
    setter = () => {
      console.warn('Computed value is read-only')
    }
  } else {
    getter = getterOrOptions.get
    setter = getterOrOptions.set
  }

  return new ComputedRefImpl(getter, setter)
}

class ComputedRefImpl {
  constructor(getter, setter) {
    this._value = undefined
    this._dirty = true // 标记是否需要重新计算
    this._setter = setter

    // 关键：用一个 effect 包裹 getter，自动追踪所有依赖
    this._effect = effect(getter, {
      lazy: true,
      scheduler: () => {
        // 依赖变化时，不立刻重新计算，只标记脏
        if (!this._dirty) {
          this._dirty = true
          // 通知所有依赖 computed 的 effect
          trigger(this, 'value')
        }
      }
    })
  }

  get value() {
    // 只有脏时才重新计算
    if (this._dirty) {
      this._value = this._effect()
      this._dirty = false
    }
    // 收集依赖：当 computed 被其他 effect 使用时
    track(this, 'value')
    return this._value
  }

  set value(newVal) {
    this._setter(newVal)
  }
}
```

`computed` 的设计精妙之处在于 **两个层次的依赖关系**：

> **计算属性的依赖链**：响应式数据 → computed 的 effect → 消费 computed 的 effect

当 `state.count` 变化时：
1. computed 内部的 `_effect` 被标记为 `_dirty = true`
2. 消费 computed 的 effect 通过 `trigger(this, 'value')` 被唤醒
3. 重新读取 `computed.value` 时，`_dirty` 为 true，重新计算

这种**二阶段传播机制**避免了不必要的级联更新，是 Vue 性能优化的核心手段之一。

### 第八层：watch 函数

`watch` 的实现比想象中复杂，因为它需要**获取新旧值**并支持 **deep** 和 **immediate** 选项：

```javascript
/**
 * watch - 侦听器
 * @param {Function|reactive} source - 侦听来源
 * @param {Function} cb - 回调函数
 * @param {Object} options - 选项 { deep, immediate }
 * @returns {Function} 取消侦听的函数
 */
function watch(source, cb, options = {}) {
  let getter
  let oldValue

  if (typeof source === 'function') {
    // 函数形式：watch(() => state.count, cb)
    getter = source
  } else if (isReactive(source)) {
    // reactive 对象：watch(state, cb)，默认深度监听
    getter = () => traverse(source)
    options.deep = true
  } else {
    getter = () => {}
  }

  // 调度器：在每轮调度中调用回调
  const scheduler = () => {
    const newValue = effectFn()
    if (hasChanged(newValue, oldValue)) {
      const old = oldValue
      oldValue = JSON.parse(JSON.stringify(newValue))
      cb(newValue, old)
    }
  }

  // 创建 lazy effect
  const effectFn = effect(
    // getter 返回的值就是 effect 的返回值
    () => getter(),
    {
      lazy: true,
      scheduler
    }
  )

  if (options.immediate) {
    // immediate: 立刻执行一次回调
    scheduler()
  } else {
    // 首次手动调用 effectFn 以收集依赖并获取初始值
    oldValue = effectFn()
  }

  // 返回取消侦听函数
  return () => {
    cleanup(effectFn)
    targetMap.delete(source)
  }
}

/**
 * traverse - 深度遍历对象以收集所有属性的依赖
 * 这是 deep: true 的核心实现
 */
function traverse(value, seen = new Set()) {
  if (typeof value !== 'object' || value === null || seen.has(value)) {
    return value
  }
  seen.add(value)

  if (Array.isArray(value)) {
    for (let i = 0; i < value.length; i++) {
      traverse(value[i], seen)
    }
  } else {
    for (const key of Object.keys(value)) {
      traverse(value[key], seen)
    }
  }

  return value
}

function hasChanged(a, b) {
  return !Object.is(a, b)
}
```

`watch` 的实现理解起来需要一些弯弯绕绕：

**流程拆解**：
1. 创建 `lazy: true` 的 effect，getter 是我们的侦听源
2. 首次调用 `effectFn()` 计算初始值，同时完成依赖收集
3. 当依赖变化时，`scheduler` 被调用，在其中读取新值并与旧值比较
4. 值变化时执行回调，传递新旧值

**deep 的实现**：`traverse` 函数的意义是**访问对象的所有嵌套属性**，触发每一个属性的 `getter`，让它们都收集到当前的 effect。这样任意子属性变化都会触发 watch。

**immediate 选项**：本质上是在首次就调用 scheduler，获得初始值并立即执行一次回调。

```javascript
// ===== 全局数据结构 =====
const targetMap = new WeakMap()
let activeEffect = null
const effectStack = []
const reactiveMap = new WeakMap()

// ===== 依赖收集 =====
function track(target, key) {
  if (!activeEffect) return
  let depsMap = targetMap.get(target)
  if (!depsMap) {
    depsMap = new Map()
    targetMap.set(target, depsMap)
  }
  let dep = depsMap.get(key)
  if (!dep) {
    dep = new Set()
    depsMap.set(key, dep)
  }
  if (!dep.has(activeEffect)) {
    dep.add(activeEffect)
    activeEffect.deps.push(dep)
  }
}

// ===== 触发更新 =====
function trigger(target, key) {
  const depsMap = targetMap.get(target)
  if (!depsMap) return
  const dep = depsMap.get(key)
  if (!dep) return
  const effectsToRun = new Set(dep)
  effectsToRun.forEach(effect => {
    if (effect !== activeEffect) {
      if (effect.options?.scheduler) {
        effect.options.scheduler(effect)
      } else {
        effect()
      }
    }
  })
}

// ===== 清理依赖 =====
function cleanup(effectFn) {
  for (let i = 0; i < effectFn.deps.length; i++) {
    effectFn.deps[i].delete(effectFn)
  }
  effectFn.deps.length = 0
}

// ===== effect =====
function effect(fn, options = {}) {
  const effectFn = () => {
    cleanup(effectFn)
    activeEffect = effectFn
    effectStack.push(effectFn)
    try {
      return fn()
    } finally {
      effectStack.pop()
      activeEffect = effectStack[effectStack.length - 1]
    }
  }
  effectFn.deps = []
  effectFn.options = options
  if (!options.lazy) effectFn()
  return effectFn
}

// ===== reactive =====
function reactive(target) {
  if (reactiveMap.has(target)) return reactiveMap.get(target)

  const proxy = new Proxy(target, {
    get(target, key, receiver) {
      track(target, key)
      const result = Reflect.get(target, key, receiver)
      if (typeof result === 'object' && result !== null && !reactiveMap.has(result)) {
        return reactive(result)
      }
      return result
    },
    set(target, key, value, receiver) {
      const oldValue = target[key]
      const hadKey = Object.prototype.hasOwnProperty.call(target, key)
      const result = Reflect.set(target, key, value, receiver)
      if (!hadKey) {
        trigger(target, key)
      } else if (oldValue !== value && (oldValue === oldValue || value === value)) {
        trigger(target, key)
      }
      return result
    },
    deleteProperty(target, key) {
      const hadKey = Object.prototype.hasOwnProperty.call(target, key)
      const result = Reflect.deleteProperty(target, key)
      if (hadKey && result) trigger(target, key)
      return result
    },
    has(target, key) {
      track(target, key)
      return Reflect.has(target, key)
    },
    ownKeys(target) {
      track(target, Symbol.for('iterate'))
      return Reflect.ownKeys(target)
    }
  })

  reactiveMap.set(target, proxy)
  return proxy
}

function isReactive(target) {
  return reactiveMap.has(target)
}

// ===== ref =====
function ref(rawValue) {
  if (isRef(rawValue)) return rawValue
  return new RefImpl(rawValue)
}

class RefImpl {
  constructor(value) {
    this.__v_isRef = true
    this._value = toReactive(value)
  }
  get value() {
    track(this, 'value')
    return this._value
  }
  set value(newVal) {
    if (newVal !== this._value) {
      this._value = toReactive(newVal)
      trigger(this, 'value')
    }
  }
}

function toReactive(value) {
  return typeof value === 'object' && value !== null ? reactive(value) : value
}

function isRef(value) {
  return value && value.__v_isRef === true
}

// ===== computed =====
function computed(getterOrOptions) {
  let getter, setter
  if (typeof getterOrOptions === 'function') {
    getter = getterOrOptions
    setter = () => console.warn('Computed is read-only')
  } else {
    getter = getterOrOptions.get
    setter = getterOrOptions.set
  }
  return new ComputedRefImpl(getter, setter)
}

class ComputedRefImpl {
  constructor(getter, setter) {
    this._dirty = true
    this._setter = setter
    this._effect = effect(getter, {
      lazy: true,
      scheduler: () => {
        if (!this._dirty) {
          this._dirty = true
          trigger(this, 'value')
        }
      }
    })
  }
  get value() {
    if (this._dirty) {
      this._value = this._effect()
      this._dirty = false
    }
    track(this, 'value')
    return this._value
  }
  set value(newVal) {
    this._setter(newVal)
  }
}

// ===== watch =====
function watch(source, cb, options = {}) {
  let getter
  let oldValue

  if (typeof source === 'function') {
    getter = source
  } else if (isReactive(source)) {
    getter = () => traverse(source)
    options.deep = true
  } else {
    getter = () => {}
  }

  const scheduler = () => {
    const newValue = effectFn()
    if (hasChanged(newValue, oldValue)) {
      const old = oldValue
      oldValue = JSON.parse(JSON.stringify(newValue))
      cb(newValue, old)
    }
  }

  const effectFn = effect(() => getter(), {
    lazy: true,
    scheduler
  })

  if (options.immediate) {
    scheduler()
  } else {
    oldValue = effectFn()
  }

  return () => cleanup(effectFn)
}

function traverse(value, seen = new Set()) {
  if (typeof value !== 'object' || value === null || seen.has(value)) return value
  seen.add(value)
  if (Array.isArray(value)) {
    for (let i = 0; i < value.length; i++) traverse(value[i], seen)
  } else {
    for (const key of Object.keys(value)) traverse(value[key], seen)
  }
  return value
}

function hasChanged(a, b) {
  return !Object.is(a, b)
}
```

## 实战案例

### 案例：简易状态管理器（mini-store）

基于我们的响应式系统，可以快速实现一个简易的全局状态管理器：

```javascript
function createStore(options) {
  const { state, mutations, actions, getters } = options

  const store = reactive(state)
  const computedGetters = {}

  if (getters) {
    for (const [key, fn] of Object.entries(getters)) {
      computedGetters[key] = computed(() => fn(store))
    }
  }

  return {
    state: store,
    getters: computedGetters,
    commit(type, payload) {
      if (mutations[type]) {
        mutations[type](store, payload)
      }
    },
    dispatch(type, payload) {
      if (actions[type]) {
        actions[type]({ state: store, commit: this.commit }, payload)
      }
    }
  }
}

// 使用示例
const store = createStore({
  state: { count: 0, todos: [] },
  mutations: {
    increment(state) { state.count++ },
    addTodo(state, todo) { state.todos.push(todo) }
  },
  actions: {
    asyncAdd({ commit }, amount) {
      setTimeout(() => {
        for (let i = 0; i < amount; i++) commit('increment')
      }, 100)
    }
  },
  getters: {
    doubleCount: (state) => state.count * 2,
    todoCount: (state) => state.todos.length
  }
})

// 在任何地方使用
effect(() => {
  console.log(`count: ${store.state.count}, double: ${store.getters.doubleCount.value}`)
})

store.commit('increment') // 触发 effect 输出
```

这个例子展示了响应式系统的能力边界：一旦理解了 `reactive` + `effect` 的组合，像 Pinia/Vuex 这样的状态管理库的核心实现就不再神秘。

### 案例：响应式 Canvas 绘制

```javascript
function reactiveCanvas(canvasId) {
  const canvas = document.getElementById(canvasId)
  const ctx = canvas.getContext('2d')

  const scene = reactive({
    x: 100,
    y: 100,
    radius: 50,
    color: '#ff6600',
    rotation: 0
  })

  function draw() {
    ctx.clearRect(0, 0, canvas.width, canvas.height)
    ctx.save()
    ctx.translate(scene.x, scene.y)
    ctx.rotate(scene.rotation)
    ctx.fillStyle = scene.color
    ctx.beginPath()
    ctx.arc(0, 0, scene.radius, 0, Math.PI * 2)
    ctx.fill()
    ctx.restore()
  }

  // 任何数据变化自动重绘
  effect(draw)

  // 动画循环
  setInterval(() => {
    scene.rotation += 0.02
    scene.x += Math.sin(Date.now() / 1000) * 2
  }, 16)

  return scene
}
```

## 底层原理

### Proxy 与 Reflect 的搭配哲学

Proxy 是对象的"代理层"，几乎所有对对象的操作都可以被 Proxy 拦截。但为什么一定要配合 Reflect？

```javascript
const parent = reactive({ name: 'parent' })
const child = reactive(Object.create(parent))
child.name = 'child'  // 问题：这里会触发哪个对象的 set？
```

如果 getter 直接使用 `target[key]` 而不是 `Reflect.get(target, key, receiver)`，当 `child.name` 被访问时，由于原型链查找，实际访问的是 `parent.name`，但 `receiver` 是 `child`。`Reflect` 将 `receiver` 正确地传递给了后续的 `get` 操作，这是 Vue3 在原型链场景下正确追踪依赖的保证。

### 分支切换与 cleanup

这是响应式系统中最容易忽略但又至关重要的机制。看这个场景：

```javascript
const state = reactive({ ok: true, a: 1, b: 2 })
effect(() => {
  console.log(state.ok ? state.a : state.b)
})
```

当 `state.ok = true` 时，effect 依赖 `ok` 和 `a`。如果 `state.ok = false`，分支切换后 effect 依赖 `ok` 和 `b`。**如果没有 cleanup，修改 `a` 依然会触发 effect，造成了不必要的更新开销。**

cleanup 机制的本质是：**每次执行前清空旧的依赖集合，执行过程中重新收集**。这保证了 effect 只关联当前真正使用的属性。

### 调度器与异步更新

Vue 的响应式系统没有止步于"数据变化就立即更新"。实际的 Vue 采用**异步队列**来批量处理更新：

```javascript
// Vue 内部的异步更新队列（伪代码）
const queue = new Set()
let isFlushing = false

function queueJob(effect) {
  queue.add(effect)
  if (!isFlushing) {
    isFlushing = true
    Promise.resolve().then(() => {
      queue.forEach(fn => fn())
      queue.clear()
      isFlushing = false
    })
  }
}

// 使用调度器实现异步批量更新
effect(fn, {
  scheduler(effect) {
    queueJob(effect)  // 不立即执行，而是加入队列
  }
})
```

这就是为什么连续修改三次 `count`，Vue 组件的渲染函数只执行一次的原因。我们的手写代码中通过 `effect.options.scheduler` 留出了这个扩展点。

## 高频面试题解析

### Q1：Vue3 为什么用 Proxy 替代 Object.defineProperty？

**核心答案**：Proxy 能拦截对象的**所有操作**，包括属性的新增、删除、数组索引修改、数组长度变化等。而 Object.defineProperty 只能拦截已有属性的 get/set，新增和删除需要额外的 `$set` 和 `$delete`，数组需要重写变异方法。

**扩展回答**：从性能角度看，Object.defineProperty 在初始化时需要递归遍历整个对象树，每个属性都定义 getter/setter。而 Proxy 是懒递归的——只在访问到深层对象时才代理。对于一个深度 10 层、每层 1000 属性的巨大配置对象，Vue2 初始化就要产生 10^30 个 getter/setter（理论上是递归全量），而 Vue3 如果只访问了根属性，深层代理根本不会创建。

### Q2：computed 和 watch 的区别是什么？

| 维度 | computed | watch |
|------|----------|-------|
| 本质 | 派生数据（声明式） | 副作用（命令式） |
| 缓存 | 有，依赖不变不重算 | 无缓存 |
| 返回值 | 新值 | 不处理返回值 |
| 异步 | 不支持 | 支持（异步操作） |
| 使用场景 | 模板中的计算逻辑 | 数据变化后的 IO 操作 |

### Q3：手写 reactive 时，为什么要用 WeakMap 而不是 Map？

WeakMap 的 key 是**弱引用**，当 target 对象不再被任何引用指向时，会被垃圾回收器自动回收，对应的依赖映射表也一并被回收。如果使用 Map，target 即使被置为 null，依赖映射依然存在于 Map 中，造成**内存泄漏**。

```javascript
let target = { count: 1 }
let proxy = reactive(target)

// 如果使用 Map: targetMap 中依然持有 target 的引用，无法 GC
// 如果使用 WeakMap: target = null 后，依赖映射可被 GC
target = null
// WeakMap 中的条目会被垃圾回收
```

### Q4：Vue3 的 effect 调度器是如何实现异步批量更新的？

Vue 使用一个**微任务队列**（microtask）来批量处理所有 pending 的 effect。当响应式数据在同一个 tick 内被多次修改时，effect 不会被立即执行，而是被加入到一个队列中。当前宏任务执行完毕后，微任务开始执行，队列中的 effect 被**去重**后依次执行。这避免了中间状态的重复渲染。

核心代码就在 Vue 的 `scheduler.ts` 中：使用 `Promise.resolve().then(flushJobs)` 将批量刷新推迟到下一个微任务。

### Q5：响应式系统的循环依赖问题如何处理？

```javascript
const obj = reactive({ foo: 1 })
effect(() => {
  obj.foo = obj.foo + 1  // 无限递归！
})
```

这道题的考察点在于：getter 中 track，setter 中 trigger。当 effect 执行时，先读 `obj.foo`（track）然后写 `obj.foo`（trigger），trigger 会再次执行同一个 effect，导致栈溢出。

Vue 的处理方式是在 trigger 中加判断：**如果即将执行的 effect 正是当前正在执行的 activeEffect，则跳过它**。我们手写代码中的 `if (effect !== activeEffect)` 就是为此设计的。

## 总结与扩展

通过不到 200 行的代码，我们从零构建了一个完整可用的 Vue3 响应式系统。回顾一下我们实现的核心模块：

| 模块 | 核心职责 | 关键技术 |
|------|---------|---------|
| `track` | 建立"属性→effect"的映射 | WeakMap + Map + Set 三层结构 |
| `trigger` | 根据属性找到所有 effect 并执行 | effect 去重 + 调度器扩展 |
| `effect` | 执行函数并自动收集依赖 | cleanup + effectStack |
| `reactive` | 代理对象，拦截 get/set | Proxy + Reflect + 懒递归 |
| `ref` | 包装基本类型为响应式 | 带 value 属性的类 |
| `computed` | 派生+缓存 | 脏标记 + 惰性计算 + 二阶段传播 |
| `watch` | 监听数据变化 | lazy effect + scheduler + traverse |

当然，真实的 Vue3 响应式系统远比我们的手写版本复杂。以下是我们的代码与真实 Vue3 的主要差距：

1. **嵌套 reactive 的缓存优化**：真实 Vue3 用 `reactiveMap` 缓存已代理的对象，还会处理 `readonly`、`shallowReactive` 等变体
2. **Ref 解包**：`ref` 在 reactive 对象中会自动解包（`ref.value` → `ref`），我们有简化
3. **`toRef` / `toRefs`**：将 reactive 的某个属性转为 ref，批量解构时保持响应性
4. **数组的特殊处理**：真实 Vue3 对数组的 `length` 变化、`splice` 等有更精细的依赖管理
5. **有限递归深度**：`traverse` 深度遍历在真实 Vue 中有最大深度限制，防止循环引用导致栈溢出
6. **多端适配**：调度器背后是自定义调度器 `queueFlush`，支持与 nextTick 的联动

但掌握了本文的核心思想，再去阅读 Vue3 源码中的 `packages/reactivity` 目录（约 2000 行），你会发现每一行代码都似曾相识。因为**大道至简，核心机制从不复杂**——复杂的是边界情况的处理、性能的极致优化和多平台兼容。

**延伸思考**：如果你理解了本文的内容，不妨进一步探索：
- **SolidJS** 的响应式系统（类似 Vue3 但无 Proxy，靠编译时分析）
- **Svelte 5** 的 `$state` runes（编译时注入响应式）
- **MobX** 的 `observable` 和 `autorun`（类 Vue2 但用 Proxy 增强）

你会发现，响应式编程的思想是相通的。掌握了本质，面对任何新框架都能快速切入核心。
