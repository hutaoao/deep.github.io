## 一句话概括

ES Module 不是 CJS 的语法糖——它换了赛道：**静态结构**让打包工具有上帝视角做 Tree Shaking，**Live Binding** 让跨模块变量变更实时可见，**浏览器原生** `<script type="module">` 让零打包开发成为现实。

## 核心知识点

### 1. 语法全景

```js
// ── 命名导出 ──
export const PI = 3.14;
export function add(a, b) { return a + b; }
export { PI, add };

// ── 默认导出（一个模块唯一）──
export default function main() {}

// ── 重导出（聚合模块 / barrel export）──
export { add as sum } from './math.js';
export { default as React } from 'react';

// ── 导入 ──
import { add, PI } from './math.js';
import * as Math from './math.js';         // 命名空间
import def from './module.js';             // 默认导入（名字随意）
import def, { add } from './module.js';    // 混合

// ── 动态导入 ──
const { heavy } = await import('./heavy.js');
```

### 2. Live Binding——ESM 和 CJS 的根本差异

```js
// counter.mjs
export let count = 0;
export const inc = () => { count++; };

// main.mjs
import { count, inc } from './counter.mjs';
console.log(count); // 0
inc();
console.log(count); // 1  ← 实时更新！CJS 永远做不到

// CJS 对比：
// const { count, inc } = require('./counter');
// inc();
// console.log(count); // 0  ← 永远是导出那一刻的原始值快照
```

**原理：** `import { count }` 不是把值拷过来，而是建立一个指向原模块 `count` 变量的**只读引用**（地址映射）。原模块改了 → 所有 import 端立刻读到新值。

⚠️ **import 端不能赋值：** `count = 5` 会报错——ESM 导入的绑定是不可变的。

### 3. 静态结构 → Tree Shaking

```js
// utils.js
export const used = () => 'used';
export const unused = () => 'unused'; // 没被任何 import 引用

// app.js
import { used } from './utils.js';
// 打包后 unused 的代码直接被删除

// CJS 为什么不行？
// require 可以出现在任何地方——if / 循环 / 表达式里：
if (flag) { const x = require('./x'); }  // 打包工具完全不敢删
```

Tree Shaking 的前提是**编译时画出完整的依赖图**，确定哪些导出被用了、哪些是 dead code。CJS 的动态 `require` 让静态分析无从下手。

### 4. 动态 import——按需加载的基石

```js
// 静态 import 只能出现在模块顶层
// import { heavy } from './heavy.js'; // ❌ 不能放 if 里

// 动态 import 随时可用
const { render } = document.querySelector('#chart').checked
  ? await import('./chart.js')
  : await import('./fallback.js');

// 路由懒加载——现代框架的标配
const Profile = () => import('./pages/Profile.vue');    // Vue Router
const LazyPage = React.lazy(() => import('./Page.tsx')); // React
```

动态 `import()` 返回 Promise。静态 import 确定依赖拓扑，动态 import 实现按需加载——两者互补而非互斥。

### 5. ESM vs CJS 速查

| | CommonJS | ES Module |
|---|---|---|
| 加载时机 | 运行时同步 | 编译时解析，执行时取值 |
| 导出本质 | 值拷贝 | Live Binding（引用） |
| Tree Shaking | ❌ | ✅ |
| 循环依赖 | 拿到不完整对象 | 引用在，值可能 undefined |
| 顶层 await | ❌ | ✅ (ES2022) |
| 浏览器原生 | ❌ | ✅ `<script type="module">` |
| Node.js | 默认 | `.mjs` 或 `"type": "module"` |

## 其实你每天都在用

- **Vite 开发服务器**：直接走浏览器原生 ESM，100+ 模块按需 fetch，秒启动不打包
- **路由懒加载**：`() => import('./Home.vue')` 拆 chunk，首屏体积砍半
- **npm 包的 `"module"` / `"exports"` 字段**：打包工具优先走 ESM 入口获取 Tree Shaking 能力
- **`import type { User } from './types'`**：TS 编译后直接抹掉，不影响运行时产物
- **`import.meta.url`**：ESM 里替代 `__dirname`/`__filename` 的标准方式

## 常见误解

- **❌ 误区：「import 和 require 都同步」** 浏览器 `<script type="module">` 默认**异步 + defer**，Node ESM 也是异步。真正同步阻塞的是 CJS 的 `require`（`readFileSync` + 立即执行）。这个差异决定了 ESM 可以直接在浏览器跑，CJS 必须打包。

- **❌ 误区：「export default 就是比命名导出好」** 对**使用者**方便（命名随意），对**维护者**是灾难——IDE 重构跟踪不到、Tree Shaking 效果更差。社区共识：除非模块只有一个主功能（如组件），优先命名导出。

- **❌ 误区：「ESM 能 import CJS，所以互操作没问题」** 这条桥是**单向**的：ESM → CJS ✅（Node 做了一层包装），CJS → ESM ❌——`require` 处理不了 ESM 的异步加载和 live binding。混用项目注意方向。

- **❌ 误区：「Node 里 CJS 和 ESM 随便切，改个后缀就行」** `.mjs` = ESM，`.cjs` = CJS，`.js` 看 `"type"`。但更要命的是：CJS 文件 `require` 不了 ESM 文件——新旧代码过渡期踩坑重灾区。

## 一句话总结

静态分析让死代码无处藏身，Live Binding 让循环依赖不再断裂，浏览器原生让模块化回归语言本身——ESM 的意义在于：它不只是 Node.js 的一个功能，它是 JavaScript 的一部分。
