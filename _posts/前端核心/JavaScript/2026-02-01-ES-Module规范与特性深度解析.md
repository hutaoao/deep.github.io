## 一句话概括

ES Module 是 JavaScript 语言层面的官方模块系统，相比 CommonJS 的三板斧——**静态结构、实时绑定、异步加载**——让它天然支持 Tree Shaking、循环依赖，以及浏览器直接运行。

## 核心知识点

### 1. 基础语法：export 和 import

```javascript
// —— 导出 ——
// 命名导出（可多个）
export const PI = 3.14;
export function add(a, b) { return a + b; }
// 统一导出
export { add, PI };
// 默认导出（一个模块只能有一个）
export default function sum(a, b) { return a + b; }

// —— 导入 ——
// 命名导入
import { add, PI } from './math.js';
// 重命名
import { add as plus } from './math.js';
// 全量导入
import * as math from './math.js';
// 默认导入（名字随便起）
import sum from './math.js';
// 混合
import sum, { add, PI } from './math.js';
```

### 2. 动态绑定（Live Binding）— 对比 CJS 的关键差异

```javascript
// ES Module
// counter.js
export let count = 0;
export function inc() { count++; }

// main.js
import { count, inc } from './counter.js';
console.log(count); // 0
inc();
console.log(count); // 1 ← 实时绑定！

// ====== 对比 CommonJS ======
// counter.cjs
let count = 0;
module.exports = { count, inc: () => count++ };

// main.cjs
const { count, inc } = require('./counter.cjs');
inc();
console.log(count); // 0 ← 值拷贝，不会变
```

**面试核心：** ESM 导出的是**引用**（live binding），CJS 导出的是**值拷贝**。这是理解 Tree Shaking 和循环依赖行为的关键。

### 3. 静态结构 → Tree Shaking

```javascript
// utils.js
export function used() { return 'used'; }
export function unused() { return 'unused'; }

// app.js —— 只引入 used
import { used } from './utils.js';
console.log(used());

// 打包结果（Rollup / Webpack 处理后）自动删除 unused
```

**原理：** `import` 必须在顶层、不能放在条件语句或循环中。这意味着打包工具不需要执行代码就能确定依赖关系图，安全地删除未使用的导出。

### 4. 动态 import — 按需加载

```javascript
// 静态 import（编译时确定，不支持条件）
// import { heavy } from './heavy.js';  // 不能放在 if 里

// 动态 import（返回 Promise，可以在任何地方调用）
if (needHeavy) {
  const { heavy } = await import('./heavy.js');
  heavy();
}

// React 路由懒加载就是基于这玩意
const LazyPage = React.lazy(() => import('./Page.js'));
```

### 5. CJS vs ESM 速查

| 特性 | CommonJS | ES Module |
|------|----------|-----------|
| 加载时机 | 运行时 | 编译时（静态分析） |
| 加载方式 | 同步 | 异步 |
| 导出方式 | 值拷贝 | 实时绑定（引用） |
| Tree Shaking | ❌ | ✅ |
| 循环依赖 | 可能拿不完整对象 | 不会丢引用，但值可能未初始化 |
| 顶层 await | ❌ | ✅（ES2022） |
| 浏览器原生 | ❌（需打包） | ✅ |

## 其实你每天都在用

- **`import React from 'react'`** — 走的就是 ESM，只是被 Babel/tsc 转成了 CJS 或保留为 ESM
- **Vite 开发服务器** — 直接用浏览器原生 `<script type="module">` 加载 ESM，不打包，秒开
- **`import()` 动态导入** — Vue/React Router 的路由懒加载全靠它把大页面拆成 chunk
- **npm 包的 `"exports"` 字段** — 告诉打包工具不同入口对应不同模块格式
- **Node.js .mjs 文件 / `"type": "module"`** — 让 Node 跑原生 ESM 代码，不再需要转译

## 常见误解

- **❌ 误区：「ESM 和 CJS 只是语法不同，底层一样」** 大错。`import` 是静态的、编译时解析的；`require` 是动态的、运行时的。这决定了 ESM 能做 Tree Shaking、CJS 能做条件 require——它们的设计理念完全不同。

- **❌ 误区：「import 是同步的，所以比 require 慢」** 恰恰相反。浏览器中 `<script type="module">` 的 import 默认是**异步+延迟执行**的，相当于自动加上了 `defer`。CJS 的 require 才是真正同步阻塞的（读文件+执行）。

- **❌ 误区：「export default 和 export const 没区别」** 有。默认导出没有名字绑定，导入方可以随便起名——这会导致 IDE 重构时追踪困难。社区倾向：能用命名导出就不用默认导出，除非模块只有一个主功能。

- **❌ 误区：「ESM 在 Node.js 里可以直接 require CJS 模块，所以互操作没问题」** 箭头是单向的。ESM 可以 `import` CJS 模块（Node 通过包装实现），但 CJS 不能 `require` ESM 模块。混用时要特别注意方向。

## 一句话总结

ES Module 不是 CJS 的语法糖——静态分析、实时绑定、异步加载三个特性一组合，直接定义了现代前端工具链的天花板。
