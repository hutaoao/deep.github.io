## 一句话概括

ES Module 不是 CJS 的语法糖——它是换了赛道：**静态结构**让打包工具有了上帝视角做 Tree Shaking，**Live Binding** 让跨模块变量变更实时可见，**浏览器原生 `<script type="module">`** 让不打包开发成为现实。

## 核心知识点

### 1. 语法全景 —— 比你印象中丰富

```javascript
// ── 命名导出 ──
export const PI = 3.14;
export function add(a, b) { return a + b; }
export { PI, add };

// ── 默认导出（一个模块只能一个）──
export default function main() {}

// ── 重导出（聚合模块常用）──
export { add as sum } from './math.js';
export { default as React } from 'react';

// ── 导入 ──
import { add, PI } from './math.js';
import * as Math from './math.js';         // 命名空间
import def from './module.js';             // 默认导入（名字随便取）
import def, { add } from './module.js';    // 混合

// ── 动态导入（返回 Promise）──
const { heavy } = await import('./heavy.js');
```

### 2. Live Binding — ESM 和 CJS 的最根本差异

```javascript
// counter.mjs
export let count = 0;
export const inc = () => { count++; };

// main.mjs
import { count, inc } from './counter.mjs';
console.log(count); // 0
inc();
console.log(count); // 1 ← 实时更新！CJS 做不到

// 对比 CJS：
// const { count, inc } = require('./counter');
// inc();
// console.log(count); // 0 ← 永远是导出那一刻的快照
```

**原理：** ESM 的 `import` 不是把值拷过来，而是建立了一个指向原模块变量的**地址映射**。`import { count }` 相当于创建了一个到 `counter.mjs` 里 `count` 变量的只读引用——那边改了，这边立刻读到新值。

⚠️ 注意：**import 端不能修改 live binding 的值**，`count = 5` 会报错——它是只读引用。

### 3. 静态结构 → Tree Shaking

```javascript
// utils.js
export const used = () => 'used';
export const unused = () => 'unused'; // 没被 import，打包时会删除

// app.js
import { used } from './utils.js';
// 打包产物里没有 unused 的代码

// 为什么 CJS 做不到？
// require 可以放在 if 里、循环里、任何表达式里：
if (flag) { const x = require('./x'); }  // 打包工具不敢删任何东西
```

**Tree Shaking 的前提是"静态可知"——** 编译时就要画出完整的依赖图，确定哪些导出被用了、哪些是 dead code。CJS 的条件 require 让这种分析无从下手。

### 4. 动态 import — 真正的按需加载

```javascript
// 静态 import 只能出现在模块顶层，不能放条件或循环
// import { heavy } from './heavy.js'; // ❌ 不能这样写

// 动态 import 随时随地
const { render } = document.querySelector('#chart').checked
  ? await import('./chart.js')
  : await import('./fallback.js');

// 路由懒加载——现代框架的基石
const Profile = () => import('./pages/Profile.vue');  // Vue Router
const LazyPage = React.lazy(() => import('./Page.tsx')); // React
```

动态 `import()` 返回一个 Promise，可以和 `async/await`、`.then()` 无缝配合。**和静态 import 的关系：** 静态 import 确定依赖拓扑，动态 import 实现按需加载——两者互补，不是互斥。

### 5. ESM vs CJS 对比速查

| | CommonJS | ES Module |
|---|---|---|
| 加载时机 | 运行时同步 | 编译时解析，执行时取值 |
| 导出方式 | 值拷贝 | Live Binding（引用） |
| Tree Shaking | ❌ | ✅ |
| 循环依赖 | 拿到不完整的对象 | 引用在，值可能是 undefined |
| 顶层 await | ❌ | ✅ (ES2022) |
| 浏览器原生 | ❌ | ✅ `<script type="module">` |
| Node.js | 默认 | `.mjs` 或 `"type": "module"` |

## 其实你每天都在用

- **Vite 开发服务器**：直接走浏览器原生 `<script type="module">`，不打包、秒启动——100+ 个模块也是浏览器自己按需 fetch
- **路由懒加载**：`() => import('./Home.vue')` 把应用拆成独立 chunk，首屏体积减少 50%+
- **npm 包的 `"module"` / `"exports"` 字段**：打包工具优先走 ESM 入口获得 Tree Shaking 能力
- **`import type { User } from './types'`**：TypeScript 编译后直接抹掉，不影响运行时
- **`import.meta.url`**：当前模块的文件 URL，是 ESM 替代 `__dirname`/`__filename` 的标准方式

## 常见误解

- **❌ 误区：「import 和 require 都是同步加载」** 恰恰相反。浏览器 `<script type="module">` 默认是**异步 + defer** 执行。Node 中 ESM 也是异步加载。真正同步阻塞的是 CJS 的 `require`（`readFileSync` + 执行）。这个差异决定了 ESM 可以直接跑在浏览器，而 CJS 必须打包。

- **❌ 误区：「export default 比命名导出更方便」** 对**使用者**方便（随便命名），但对**维护者**是灾难——IDE 重构跟踪不到、Tree Shaking 效果更差。社区共识：除非模块只有一个主功能（如 React 组件），否则优先命名导出。

- **❌ 误区：「ESM 可以 import CJS，所以互操作无痛」** 这条桥是**单向**的。ESM → CJS 可以（Node 做了一层包装），但 CJS → ESM **不行**——`require` 不知道怎么处理 ESM 的异步加载和 live binding。混用项目注意依赖方向别反。

- **❌ 误区：「Node 里 CJS 和 ESM 随便切」** `.mjs` 表示 ESM，`.cjs` 表示 CJS，`.js` 看 `package.json` 的 `"type"` 字段。但更坑的是同个项目里 CJS 和 ESM 文件之间的调用关系——CJS 用 `require` 加载不了 ESM 文件，这个限制经常在新旧代码过渡期踩坑。

## 一句话总结

静态分析让死代码无处藏身，Live Binding 让循环依赖不再断裂，浏览器原生让模块化回归语言本身——这就是 ESM 作为"语言标准"的真正意义：它不是 Node.js 的一个功能，它是 JavaScript 的一部分。
