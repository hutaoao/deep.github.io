## 一句话概括

闭包是一个**"记住它诞生时作用域"的函数**——当函数被创建时，它会在内部保存一个对周围词法环境的引用（`[[Environment]]`），使得即使函数在定义作用域之外执行，依然能访问定义时存在的变量。这是 JavaScript 最核心也最常被面试追问的概念。

## 背景与意义

### 为什么闭包是 JavaScript 的灵魂？

JavaScript 在设计之初就采用了**词法作用域**（静态作用域）的模型——函数的作用域在定义时就确定了，而不是在执行时。这意味着一个函数可以"打包"它的环境带走到处用。

**如果没有闭包：**
```javascript
// ❌ 没有闭包的语言（比如纯 C）
// 函数返回值只能是"值"，不能是"函数+环境"
// 每次都要手动传递上下文
function onButtonClick(context) {
  // 需要手动传递 context
  console.log(context.userId);
}
```

**闭包让这变得自然：**
```javascript
// ✅ JavaScript 的闭包
function createClickHandler(userId) {
  return function() {
    console.log(userId);  // ★ 就是闭包——userId 被记住了
  };
}
const handler = createClickHandler(42);
handler();  // 42 —— 即使 createClickHandler 已经执行完毕
```

## 概念与定义

### 标准的定义

> MDN 定义：**闭包是指能够访问自由变量的函数。**
>
> 自由变量 = 既不在函数参数也不在函数内部声明的变量，而是来自外部作用域的变量。

```javascript
function outer() {
  const a = 1;  // outer 的局部变量

  function inner() {
    // a 对于 inner 来说是自由变量（不在它的参数中，也不在它内部声明）
    console.log(a);
  }

  return inner;
}

const fn = outer();  // ★ 此时 outer 已经执行完毕，正常来说 a 应该被销毁
fn();  // 1 —— a 还活着！因为 inner（闭包）还引用着它
```

### 形成闭包的三个条件

1. **函数嵌套**：一个函数定义在另一个函数内部
2. **引用外部变量**：内部函数使用外部函数的变量
3. **内部函数被"带出去"**：被返回、赋值给全局变量、作为参数传递——只要它的引用存活

```javascript
// 条件 1 ✅ + 条件 2 ✅，但条件 3 ❌ → 不是闭包
function demo1() {
  const x = 1;
  function inner() { console.log(x); } // 嵌套 + 引用外部变量
  inner(); // 直接调用，inner 不在外部存活 → 不是闭包
}

// 条件 1 ✅ + 条件 2 ✅ + 条件 3 ✅ → 是闭包
function demo2() {
  const x = 1;
  return function() { console.log(x); };
}
const closure = demo2(); // closure 带着环境存活 → 是闭包 ✓
```

**💡 重要区分**：技术上，只要内部函数引用了外部变量，即使不被返回，也会形成"短暂闭包"。但在面试中，通常只把"外部存活"的情况称为需要关注的闭包。

## 最小示例

```javascript
// 最简单的闭包
function createGreeting(name) {
  return function() {
    return `Hello, ${name}!`;
  };

const greetAlice = createGreeting('Alice');
const greetBob = createGreeting('Bob');

console.log(greetAlice());  // "Hello, Alice!"
console.log(greetBob());    // "Hello, Bob!"
// ★ 两个独立的作用域副本，互不干扰
```

```javascript
function setupButton(buttonId, message) {

setupButton('btn1', '按钮一被点击');

```

## 核心知识点拆解

### 知识点 1：词法环境（Lexical Environment）—— 闭包的"地基"

JavaScript 引擎在执行代码时，会为每个作用域创建一个**词法环境**。

```javascript
// 词法环境的内部结构
LexicalEnvironment = {
  EnvironmentRecord: {    // 环境记录：存储当前作用域的变量和函数
    a: 1,
    inner: <function>
  },
  outer: <外部词法环境的引用>  // 指向父级作用域
}
```

**闭包的诞生过程：**

```javascript
// 全局环境
const globalEnv = {
  environmentRecord: { outer: fn, global: "G" },
  outer: null  // 全局环境没有外部引用
};

function outer() {
  // outer 执行时创建环境
  const outerEnv = {
    environmentRecord: { a: 1, inner: fn },
    outer: globalEnv  // 指向全局
  };

  function inner() {
    // inner 创建时：记住 outerEnv
    // inner.[[Environment]] = outerEnv
    console.log(a);
  }

  return inner;
}

// 当 inner 被返回并在全局调用时：
// 1. 创建 inner 的执行环境，其 outer 指向 inner.[[Environment]] = outerEnv
// 2. 查找 a → inner 环境没有 → 去 outer 环境（outerEnv） → 找到 a = 1
```

**🎯 面试回答模板**：闭包的底层原理是每个函数都有一个 `[[Environment]]` 内部属性，指向函数创建时的词法环境。当函数被执行时，它的执行上下文的外部作用域引用就指向这个 `[[Environment]]`，从而可以访问定义时的变量。

### 知识点 2：闭包与作用域链的交互

```javascript
const globalVar = "global";

function outer(x) {
  const outerVar = "outer";

  function middle(y) {
    const middleVar = "middle";

    function inner(z) {
      console.log(globalVar, outerVar, middleVar, x, y, z);
      // 输出: global outer middle a b c
      // ★ 它能访问所有祖先作用域中的变量
    }

    return inner;
  }

  return middle;
}

// 作用域链的长度：
// inner → middle → outer → global
// 每层闭包都持有上一层环境的引用
const fn = outer('a')('b');
fn('c');  // 一层层穿透闭包访问变量
```

**作用域链查找规则**：
1. 先在当前函数的环境记录中查找
2. 没找到则通过 `outer` 引用去父级环境查找
3. 重复直到全局环境
4. 全局环境也没找到 → `ReferenceError`

### 知识点 3：闭包与函数作用域的重要辨析

```javascript
// 场景 A：内部函数创建在外部函数中 → 闭包
function outer() {
  let count = 0;
  setInterval(function() {
    console.log(++count);  // count 被闭包捕获
  }, 1000);
}

// 场景 B：同一函数的不同实例 → 独立的闭包
function makeCounter() {
  let count = 0;
  return {
    increment: () => ++count,
    decrement: () => --count,
    getCount: () => count
  };
}

const counter1 = makeCounter();
const counter2 = makeCounter();

counter1.increment(); // 1
counter1.increment(); // 2
counter2.increment(); // 1 （每个实例有独立的闭包环境）
```

### 知识点 4：闭包经典陷阱——var 与 for 循环

**这是面试中 80% 的人会答不完全的题目：**

```javascript
// ❌ 问题代码
for (var i = 0; i < 5; i++) {
  setTimeout(() => console.log(i), 100);
}
// 输出: 5 5 5 5 5
// 而不是期望的: 0 1 2 3 4
```

**为什么会这样？**

```
循环执行过程：
第 1 次迭代：i = 0 → 创建回调 → 回调引用 i（尚未读取）
第 2 次迭代：i = 1 → 创建回调 → 回调引用 i
...
第 5 次迭代：i = 4 → 创建回调 → 回调引用 i
循环结束：  i = 5 → 循环条件不满足，退出

100ms 后所有回调执行：
→ 所有的回调都读取 i（此时 i = 5）
→ 因为 var i 是全局/函数作用域，所有回调引用同一个 i
```

**根源**：`var` 声明的变量没有**块级作用域**，整个循环共享同一个 `i`。闭包捕获的是变量**引用**（不是快照值），所以当回调执行时读取的是 `i` 的最终值。

```javascript
// ✅ 解决方案 1：let（推荐）
for (let i = 0; i < 5; i++) {
  setTimeout(() => console.log(i), 100);
  // let 每次迭代都创建新的绑定
}

// ✅ 解决方案 2：IIFE（立即执行函数）
for (var i = 0; i < 5; i++) {
  ((j) => {
    setTimeout(() => console.log(j), 100);
  })(i);
}

// ✅ 解决方案 3：闭包函数
for (var i = 0; i < 5; i++) {
  setTimeout(console.log.bind(null, i), 100);
}
```

### 知识点 5：闭包在函数式编程中的应用

```javascript
// 柯里化（Currying）
const add = a => b => a + b;
// 等价于:
function add(a) {
  return function(b) {  // 闭包捕获 a
    return a + b;
  };
}

const add5 = add(5);  // 闭包记住 a = 5
console.log(add5(3)); // 8
console.log(add5(10)); // 15

// 偏函数应用
function fetchAPI(baseUrl) {
  return function(endpoint, params) {
    // 闭包捕获 baseUrl
    return fetch(`${baseUrl}${endpoint}`, params);
  };
}

const githubAPI = fetchAPI('https://api.github.com');
const myAPI = fetchAPI('https://myapi.com');

// 直接使用带 base URL 的函数
githubAPI('/users/octocat');
myAPI('/users/me');
```

## 实战案例

### 案例一：防抖函数

```javascript
function debounce(fn, delay) {
  let timer = null;  // ★ 闭包变量：保存定时器引用

  return function (...args) {
    const context = this;

    if (timer) {
      clearTimeout(timer);  // 清除上次的定时器
    }

    timer = setTimeout(() => {
      fn.apply(context, args);
      timer = null;  // ★ 执行完毕后重置
    }, delay);
  };
}

// 使用
const debouncedSearch = debounce((keyword) => {
  console.log('搜索:', keyword);
  // 发送 API 请求
}, 300);

input.addEventListener('input', (e) => {
  debouncedSearch(e.target.value);  // 每次输入都调用，但只有停止输入 300ms 后才触发
});
```

### 案例二：惰性单例模式

```javascript
const getSingleton = (function() {
  let instance = null;  // ★ 闭包中的私有变量

  return function() {
    if (!instance) {
      instance = {
        id: Math.random(),
        createdAt: new Date()
      };
    }
    return instance;
  };
})();

console.log(getSingleton().id);  // 第一次创建
console.log(getSingleton().id);  // 返回同一个实例（两次 id 相同）
```

### 案例三：函数记忆（Memoization）

```javascript
function memoize(fn) {
  const cache = {};  // ★ 闭包保存缓存

  return function(...args) {
    const key = JSON.stringify(args);
    if (cache[key] !== undefined) {
      console.log('从缓存中获取:', key);
      return cache[key];
    }

    console.log('计算新值:', key);
    const result = fn(...args);
    cache[key] = result;
    return result;
  };
}

// 使用
const expensiveCalc = memoize((n) => {
  console.log('执行了一次昂贵计算');
  return n * 2;
});

console.log(expensiveCalc(5)); // 执行计算 → 10
console.log(expensiveCalc(5)); // 从缓存获取 → 10（第二次不计算）
```

### 案例四：闭包在 React Hooks 中的体现

```javascript
// React useEffect cleanup 中的闭包
function TimerComponent() {
  const [count, setCount] = useState(0);
  
  useEffect(() => {
    const timer = setInterval(() => {
      console.log('count:', count);  // ★ 闭包：捕获了 count
    }, 1000);

    return () => {
      clearInterval(timer);  // 闭包清除定时器
    };
  }, []);  // 依赖数组为空 → 只有初始 count = 0 被捕获
  // 这就是"闭包陷阱"的来源——旧值被闭包捕获
}
```

## 底层原理

### V8 引擎如何处理闭包

V8 采用**Ignition（解释器）+ TurboFan（优化编译器）** 双模式，对闭包做了专门优化：

```javascript
// 阶段 1：解析（Parser）
function outer() {
  const a = 1;
  const b = 2;  // b 没有被闭包引用
  const c = 3;  // c 没有被闭包引用
  
  return function inner() {
    return a;  // ★ 只引用 a
  };
}
```

**V8 的闭包优化**：

```
闭包构建阶段：
┌─────────────────────────────────────────┐
│ V8 对闭包对象（Context）做精确追踪        │
│                                          │
│ Context for outer():                     │
│   ├── a: 1          ← 被闭包引用 → 保留  │
│   ├── b: 2          ← 未被引用 → 不放入  │
│   └── c: 3          ← 未被引用 → 不放入  │
│                                          │
│ ★ 只有被闭包引用的变量会进入闭包上下文     │
│ └── 这就是"未使用变量会被 GC 回收"的原理  │
└─────────────────────────────────────────┘
```

```javascript
// V8 对闭包的预编译优化
function outer() {
  const a = 1;
  const b = 2;  // 未在闭包中使用 → 编译器标记为"可以优化"
  
  return function inner() {
    console.log(a);
  };
}

// 执行时的上下文链
/*
inner.[[Context]] = {
  // a 在闭包上下文中
  a: 1,
  // b 不会出现在这里 ← V8 优化
  // outer 的完整上下文不会全部保留
}
*/
```

### 闭包与内存泄漏

```javascript
// ❌ 内存泄漏场景
function createLeak() {
  const largeData = new Array(1000000).fill('leak');
  
  return function() {
    // 只使用了一小部分数据，但整个 largeData 被闭包持有
    console.log(largeData[0]);
  };
}

const leakedFn = createLeak();
// largeData (1MB) 无法被 GC 回收，因为 leakedFn 还引用着它
```

```javascript
// ✅ 避免泄漏方案 1：只引用需要的部分
function createFixed() {
  const largeData = new Array(1000000).fill('leak');
  const firstValue = largeData[0];  // 只取出需要的值
  
  return function() {
    console.log(firstValue);  // 只引用 firstValue
  };
  // largeData 可以被 GC 回收（因为闭包只引用了 firstValue）
}
```

```javascript
// ✅ 避免泄漏方案 2：使用后置空
function createManager() {
  let data = loadBigData();  // 加载大量数据
  
  return {
    getData: () => data,
    destroy: () => {
      data = null;  // ★ 手动释放闭包中的数据
    }
  };
}

const manager = createManager();
// ... 使用数据
manager.destroy();  // 主动释放，让 GC 有机会回收
```

```javascript
// ✅ 避免泄漏方案 3：事件监听器及时移除
function setupButton(id) {
  const button = document.getElementById(id);
  const handler = () => {
    // 闭包中的操作
    console.log('clicked', id);
  };
  
  button.addEventListener('click', handler);
  
  // ★ 返回清理函数
  return function cleanup() {
    button.removeEventListener('click', handler);
  };
}

const cleanup = setupButton('myBtn');
// 组件销毁时：
cleanup();
```

### 闭包与 Context 对象的生命周期

```
函数调用 → 创建 Context 对象
    │
    ├─ 函数内没定义内部函数，或内部函数不引用外部变量
    │    └─ 函数返回后 Context 被 GC 回收
    │
    ├─ 函数定义了内部函数并引用外部变量
    │    └─ 内部函数在外部存活 → Context 保留
    │         → 直到内部函数也被 GC 回收
    │
    └─ 函数定义了内部函数并引用外部变量
         └─ 内部函数不被返回 → 函数返回后 Context 仍可能被回收
```

## 高频面试题解析

### Q1：什么是闭包？请用你自己的话解释。

**参考答案：**
闭包是函数创建时"打包"了它周围词法环境的组合。具体来说：
1. 每个函数都有一个 `[[Environment]]` 内部属性，在创建时被设置为当前词法环境
2. 函数执行时，用这个 `[[Environment]]` 来构建作用域链
3. 所以即使函数在定义之外执行，仍然能访问定义时存在的变量

**一个面试官会满意的回答**：
"闭包是指能访问外部函数变量的函数。它的底层原理是每个函数在创建时都会带上一个 `[[Environment]]` 引用，指向创建时的词法环境。当函数执行时，通过作用域链可以沿着这个引用链访问到外层变量。即使外层函数执行完毕，只要闭包还存在，那些被引用的变量就不会被 GC 回收。"

### Q2：for 循环中用 var 为什么闭包会出问题，如何解决？

**参考答案：**
原因是 `var` 没有块级作用域，整个循环共享同一个变量 `i`。闭包捕获的是变量的**引用**（不是值），所以所有回调都引用了同一个 `i`，执行时 `i` 已经是循环结束后的最终值。

**三种解法：**
1. `let`：块级作用域，每次迭代创建新的绑定
2. IIFE：立即执行函数创建新作用域，传入当前 `i`
3. `bind`：`setTimeout(console.log.bind(null, i), 100)`

### Q3：闭包有哪些实际应用场景？

**参考答案：**
1. **防抖/节流**：保存定时器状态
2. **模块模式**：实现私有变量和公开接口
3. **柯里化**：预设参数
4. **函数记忆**：缓存计算结果
5. **单例模式**：保存实例引用
6. **React Hooks**：useEffect 的 cleanup 函数
7. **高阶组件**：保存上下文

### Q4：闭包一定导致内存泄漏吗？

**参考答案：**
**不一定。** 闭包本身不是内存泄漏，它只是按设计保留了变量的引用。内存泄漏是指不再需要的数据仍然被引用而无法回收。

**真正导致泄漏的场景：**
- 闭包持有了一个大对象，但只用了其中一小部分
- 闭包被全局变量引用，永远不会被释放
- DOM 事件监听器绑定的闭包未被移除

**预防方法：**
- 只保留闭包真正需要的变量
- 使用完后手动置空
- 组件销毁时移除事件监听

### Q5：闭包作用域链的查找效率如何？

**参考答案：**
作用域链查找沿着 `[[Environment]]` 链向外部逐层查找。每多一层闭包，变量访问就多一层间接性。V8 做了优化：
- 使用 Inline Cache 缓存属性查找结果
- 只将闭包真正引用的变量放入 Context 对象
- 多层嵌套的闭包可以按需访问
