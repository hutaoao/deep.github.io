## 一句话概括

模块 A 引用 B、B 又引用 A 就叫循环依赖。CommonJS 和 ES Module 对它处理完全不同：CJS 可能给你一个**半成品模块**，ESM 利用**实时绑定**虽然值可能未初始化，但至少不会丢引用。

## 核心知识点

### 1. CommonJS 循环依赖 — 返回不完整的 exports

```javascript
// a.js
console.log('a 开始');
exports.doneA = false;
const b = require('./b');
exports.doneA = true;
console.log('a 结束');

// b.js
console.log('b 开始');
const a = require('./a');
console.log('b 中 a.doneA =', a.doneA); // false！a 还没执行完
console.log('b 结束');

// main.js
require('./a');

// 输出顺序：
// a 开始 → b 开始 → b 中 a.doneA = false → b 结束 → a 结束
```

**根因：** a.js 执行到 `require('./b')` 时跳转到 b.js，b.js 又 `require('./a')`，此时 a.js 的 `module.exports` 已被**提前缓存**（但 `doneA` 还是初值 `false`），所以拿到的是半成品。

### 2. ES Module 循环依赖 — 实时绑定不丢引用

```javascript
// a.mjs
import { b } from './b.mjs';
export const a = 'A';
console.log('a 中 b =', b); // 'B' ← 能拿到！

// b.mjs
import { a } from './a.mjs';
export const b = 'B';
console.log('b 中 a =', a); // undefined ← 引用在，但值还没到
```

**关键差异：** ESM 在**编译阶段**就建立了变量绑定关系（分配了内存地址），执行阶段只是往这个地址里填值。所以循环依赖时引用不会断，但如果你在值填进去之前就去读，拿到的是 `undefined`。

### 3. 三种解决方案

```javascript
// 方案一：延迟 require / 延迟使用（CJS）
// a.js
exports.foo = () => {
  const b = require('./b');  // 放到函数里，运行时再执行
  return b.bar();
};

// 方案二：提取公共模块（最干净）
// shared.js — 把两个模块都依赖的东西提出来
export const shared = '共用逻辑';

// 方案三：依赖注入（最灵活）
// 通过构造函数或配置把依赖注入，而不是硬编码 import
class ServiceA {
  constructor(serviceB) { this.serviceB = serviceB; }
}
```

### 4. 检测循环依赖的工具

```bash
# madge：生成模块依赖图，自动标出循环
npx madge --circular src/

# ESLint 规则
// .eslintrc.js
rules: { 'import/no-cycle': ['error', { maxDepth: Infinity }] }
```

## 其实你每天都在用

- **Vuex / Redux 的模块间引用**：store 模块 A 的 action 里调用了模块 B 的 action，模块 B 又用到了模块 A 的 state——典型的循环引用
- **React 组件互相引用**：`Parent.tsx` import `Child.tsx`，但 `Child.tsx` 也需要引用 `Parent` 的类型定义——用 `import type` 解
- **utils/tools 互相引用**：`format.ts` 引用了 `validate.ts` 里的校验，`validate.ts` 里又引用了 `format.ts` 里的格式化
- **Node.js 中间件链**：多个中间件文件互相依赖对方的处理结果，容易出现环
- **TypeScript 类型文件**：`types/user.ts` 和 `types/post.ts` 互相引用对方的类型，而这种用 `import type` 解决（TS 3.8+）

## 常见误解

- **❌ 误区：「循环依赖会导致死循环」** 不会。无论是 CJS 的缓存机制还是 ESM 的静态解析，都不会让模块重复执行。问题不是死循环，而是**拿到的数据不完整**。

- **❌ 误区：「ESM 完全解决了循环依赖问题」** 没有完全解决。ESM 确实不会给你空对象，但如果你的代码在模块顶层就直接访问循环依赖方导出的变量，仍然会拿到 `undefined`——因为对方的代码还没执行到赋值那一行。用函数包装延迟访问才是正解。

- **❌ 误区：「用 Webpack 打包就不怕循环依赖了」** Webpack 打包只是把模块系统重新实现了一遍（不管是 CJS 还是 ESM 都统一处理），但循环依赖的根本问题——执行顺序——依然存在。打包工具能帮你检测，但不能帮你消除逻辑问题。

- **❌ 误区：「只要代码跑得通就没问题」** 有些循环依赖依赖的是巧合——恰好某个值先被赋值了。但换一个打包工具（Rollup → Webpack）或者调整了导入顺序，行为可能就变了。这也是为什么 ESLint 的 `import/no-cycle` 规则建议开 error 级别。

## 一句话总结

循环依赖不是 bug，是架构信号——它告诉你模块间的职责分界线已经模糊了，该提取公共模块了。
