## 一句话概括

作用域（Scope）决定了变量和函数的**可访问范围**，JavaScript 采用**词法作用域**（也叫静态作用域，即作用域在代码编写时确定而非执行时），核心类型包括全局作用域、函数作用域和块级作用域（ES6 新增的 `let/const`）。作用域链则是多个嵌套作用域按**从内到外**顺序链接的链式结构，是引擎查找变量的唯一路径。

## 背景与意义

### 为什么需要作用域？

如果没有作用域，所有变量都是全局的：

```javascript
// ❌ 假设没有作用域：所有变量都是全局的
function login() {
  userId = 42;  // 如果不小心覆盖了全局变量...
}

function sendMessage() {
  userId = 100; // 把 login 的 userId 改掉了！
}

login();
sendMessage();
// login 的 userId 被覆盖了！
```

作用域解决的核心问题：
1. **变量隔离**：不同代码块中的同名变量互不干扰
2. **访问控制**：限制变量的可见范围（public / private 等价物）
3. **生命周期**：控制变量何时创建、何时销毁
4. **命名空间**：避免全局命名冲突

### 面试高频信号

作用域是 JavaScript 的基础，面试中出现频率极高：
- "JavaScript 有哪些作用域类型？"（基础）
- "let、const、var 的区别？"（基础必考）
- "作用域链查找规则是什么？"（进阶）
- "词法作用域和动态作用域的区别？"（深度）

## 概念与定义

### 三种作用域类型

```javascript
// 1. 全局作用域
// 在任何函数或块之外声明的变量
const globalVar = "任何人都能访问";
var anotherGlobal = "var 声明也在全局作用域";

// 2. 函数作用域
// 在函数内部声明的变量
function demo() {
  var functionScoped = "只在函数内可访问";
  console.log(functionScoped); // ✅ 函数内可访问
}
console.log(functionScoped); // ❌ ReferenceError

// 3. 块级作用域（ES6）
// 用 let/const 在 {} 内声明的变量
{
  let blockScoped = "只在块内可访问";
  const alsoBlockScoped = "同上";
  var notBlockScoped = "var 没有块级作用域！";
}
console.log(notBlockScoped); // ✅ "var 没有块级作用域！"
console.log(blockScoped);    // ❌ ReferenceError
```

### 词法作用域（静态作用域）

JavaScript 的作用域在**代码编写（解析）阶段**就确定了，而不是在函数执行时：

```javascript
const x = "全局";

function outer() {
  const x = "外部";

  function inner() {
    console.log(x);
  }

  inner();  // 输出: "外部"
  // ★ 无论 inner 在哪里被调用，它都记住"出生时"的作用域
}

outer();
```

```javascript
// 对比：如果是"动态作用域"会怎样？
const x = "全局";

function outer() {
  const x = "外部";
  inner();  // 动态作用域：输出 "外部"
}

function inner() {
  console.log(x);
  // JavaScript 是静态作用域：输出 "全局"
  // 因为 inner 定义时，x 是全局环境的变量
}

outer(); // 实际输出: "全局"
```

**🎯 面试金句**：静态作用域的变量查找取决于函数**定义的位置**而非**调用的位置**。

## 最小示例

```javascript
const globalVar = "G";  // 全局作用域

function outer() {
  const outerVar = "O";  // outer 函数作用域
  
  function inner() {
    const innerVar = "I";  // inner 函数作用域
    console.log(globalVar, outerVar, innerVar);  // G O I
  }
  
  return inner;
}

const fn = outer();
fn();  // G O I —— 作用域链查找
```

## 核心知识点拆解

### 知识点 1：var、let、const 的作用域差异

```javascript
// ★ var 的作用域行为
if (true) {
  var a = 1;
  let b = 2;
  const c = 3;
}

console.log(a);  // 1  ← var 没有块级作用域，变量提升到函数/全局作用域
console.log(b);  // ReferenceError
console.log(c);  // ReferenceError
```

```javascript
// ★ var 在函数内的表现
function demo() {
  if (true) {
    var x = 10;
  }
  console.log(x);  // 10 —— var 在整个函数作用域内可见
}
```

| 特性 | var | let | const |
|------|-----|-----|-------|
| **作用域** | 函数作用域 | 块级作用域 | 块级作用域 |
| **提升（Hoisting）** | ✅ 声明提升，初始化为 `undefined` | ✅ 声明提升，但**暂存死区** | ✅ 声明提升，但**暂存死区** |
| **重复声明** | ✅ 允许 | ❌ 报错 | ❌ 报错 |
| **重新赋值** | ✅ 允许 | ✅ 允许 | ❌ 不允许 |
| **全局对象绑定** | ✅ 成为 `window` 属性 | ❌ | ❌ |

**暂存死区（Temporal Dead Zone，TDZ）**：

```javascript
console.log(a); // undefined（var 被提升并初始化为 undefined）
var a = 1;

// ==== 分隔 ====

console.log(b); // ReferenceError: Cannot access 'b' before initialization
let b = 2;

// TDZ 解释：
// let b; 被提升，但在赋值前处于"暂存死区"
// 在此期间访问 b → ReferenceError
// 必须执行到 let b = 2 这一行才能访问
```

### 知识点 2：作用域链的查找规则

```javascript
const globalVar = "全局";

function fn1() {
  const localVar = "fn1 局部";
  
  function fn2() {
    const anotherVar = "fn2 局部";
    
    function fn3() {
      console.log(globalVar);    // 在 fn3 找不到 → fn2 → fn1 → 全局 → 找到
      console.log(localVar);     // 在 fn3 找不到 → fn2 → fn1 → 找到
      console.log(anotherVar);   // 在 fn3 找到 → 直接使用
      console.log(undefinedVar); // 全部找不到 → ReferenceError
    }
    
    fn3();
  }
  
  fn2();
}

fn1();
```

**查找路径图示**：

```
fn3() 的作用域链：
  fn3.[[Environment]] → fn2.[[Environment]] → fn1.[[Environment]] → Global
       ↓                      ↓                      ↓                    ↓
  { anotherVar }         { localVar }           { fn1, fn2 }       { globalVar }

变量查找：从最内层开始，逐层向外，直到找到或到达全局环境
```

### 知识点 3：函数作用域的"提升"（Hoisting）

```javascript
// ★ 变量提升
console.log(x);  // undefined（不是 ReferenceError）
var x = 5;

// 等价于：
var x;           // 声明被提升
console.log(x);  // undefined
x = 5;           // 赋值在原位置

// ★ 函数声明提升
foo();  // "foo" —— 函数声明整个被提升
function foo() {
  console.log("foo");
}

// ★ 函数表达式不会提升
bar();  // TypeError: bar is not a function（不是 ReferenceError！）
var bar = function() {
  console.log("bar");
};

// 等价于：
var bar;         // 声明提升为 undefined
bar();           // 试图调用 undefined → TypeError
bar = function() { console.log("bar"); };
```

**💡 最佳实践**：始终在作用域的**顶部**声明变量，用 `const` 优先、`let` 次之、`var` 尽量不用。

### 知识点 4：作用域链与执行上下文的关系

**执行上下文（Execution Context）** 是引擎执行函数时的运行时环境，**作用域链**是执行上下文的一个组成部分。

```
执行上下文结构：
ExecutionContext = {
  ThisBinding: <this 的值>,
  LexicalEnvironment: {     ← 存储 let/const 声明的变量
    EnvironmentRecord: { ... },
    outer: <父级词法环境>
  },
  VariableEnvironment: {     ← 存储 var 声明的变量（与词法环境分离）
    EnvironmentRecord: { ... },
    outer: <父级变量环境>
  }
}
```

```javascript
function demo() {
  var a = 1;  // 进入 VariableEnvironment
  let b = 2;  // 进入 LexicalEnvironment
  const c = 3; // 进入 LexicalEnvironment
  
  if (true) {
    let b = 20;  // 新块 → 新的 LexicalEnvironment
    const c = 30; // 新块 → 新的 LexicalEnvironment
    var a = 10;  // ★ var 跳过块作用域，修改外部 VariableEnvironment 的 a
  }
  
  console.log(a); // 10（被 if 块中的 var a 修改了）
  console.log(b); // 2（外层的 let b 没被影响）
  console.log(c); // 3（外层的 const c 没被影响）
}
```

### 知识点 5：严格模式对作用域的影响

```javascript
// 严格模式下的一些作用域行为变化
"use strict";

// 1. 禁止未声明变量的隐式全局
function demo() {
  x = 42;  // ReferenceError: x is not defined（非严格模式下会创建全局变量）
}

// 2. eval 不污染外部作用域
var a = 1;
eval("var a = 2");
console.log(a);  // 2（非严格模式下 a 被污染）

"use strict";
var b = 1;
eval("var b = 2");
console.log(b);  // 1（严格模式下 eval 有自己的作用域）

// 3. with 语句被禁止
with (document) {
  body.style.background = "red";  // SyntaxError
}
```

## 实战案例

### 案例一：作用域链在模块模式中的应用

```javascript
// 利用函数作用域实现模块封装
const myModule = (function() {
  // 私有变量（只能在模块内部访问）
  let privateCounter = 0;
  const privateData = [];

  // 私有函数
  function logChange(action) {
    console.log(`[${new Date().toISOString()}] ${action}: ${privateCounter}`);
  }

  // 公开接口
  return {
    increment() {
      privateCounter++;
      logChange('increment');
    },
    
    decrement() {
      privateCounter--;
      logChange('decrement');
    },
    
    getCount() {
      return privateCounter;
    },
    
    addData(item) {
      privateData.push(item);
      return privateData.length;
    }
  };
})();

myModule.increment();  // 通过公开接口访问，不能直接修改 privateCounter
myModule.increment();
console.log(myModule.getCount());  // 2
// console.log(privateCounter);    // ❌ ReferenceError
```

### 案例二：块级作用域在循环中的不同行为

```javascript
// ★ var：所有点击都输出 5
const buttons = document.querySelectorAll('button');

for (var i = 0; i < 5; i++) {
  buttons[i].onclick = function() {
    console.log(i);  // 始终输出 5
  };
}

// ★ let：每个按钮输出正确的索引
for (let j = 0; j < 5; j++) {
  buttons[j].onclick = function() {
    console.log(j);  // 0, 1, 2, 3, 4
  };
  // let j 每次迭代都创建新的绑定
}

// ★ 用 IIFE 模拟块作用域
for (var k = 0; k < 5; k++) {
  (function(index) {
    buttons[index].onclick = function() {
      console.log(index);  // 0, 1, 2, 3, 4
    };
  })(k);
}
```

### 案例三：catch 语句的块级作用域

```javascript
try {
  throw new Error('something wrong');
} catch (error) {
  console.log(error.message);  // "something wrong"
  // error 只在 catch 块内可见
}

console.log(error);  // ReferenceError: error is not defined

// ES10 允许 catch 省略参数（如果不需要捕获的 error）
try {
  throw new Error('ignored');
} catch {
  // 不需要 error 参数时更简洁
  console.log('发生了一个错误');
}
```

## 底层原理

### 词法环境的创建过程

```
JavaScript 引擎执行代码时的作用域创建流程：

1. 全局代码执行前
   → 创建全局执行上下文
   → 创建全局词法环境（outer = null）

2. 遇到函数定义
   → 函数对象创建时，设置 [[Environment]] = 当前词法环境

3. 函数被调用时
   → 创建新的执行上下文
   → 创建新的词法环境，outer = 函数.[[Environment]]
   → 将参数、内部声明的变量加入环境记录

4. 块级作用域（遇到 { 且使用 let/const）
   → 创建新的词法环境
   → outer = 外部词法环境
   → 块结束时，词法环境被销毁
```

### var 与 let/const 在编译阶段的不同处理

```javascript
// ★ var 的处理：只有"变量环境"
function demoVar() {
  // 编译阶段：
  // 扫描所有 var 声明，提升到函数顶部
  // 初始化为 undefined
  
  console.log(x);  // undefined（已声明，未赋值）
  var x = 10;
  
  // 等价于编译后：
  // var x;          ← 提升
  // console.log(x);  ← undefined
  // x = 10;         ← 赋值在原来位置
}

// ★ let/const 的处理："词法环境" + "暂存死区"
function demoLet() {
  // 编译阶段：
  // 扫描所有 let/const，提升但不初始化
  
  console.log(y);  // ReferenceError（暂存死区：已提升但未初始化）
  let y = 20;
  
  // 在 let y = 20 这一行之前，y 处于 TDZ
  // 引擎在解析阶段知道 y 存在，但强制禁止在初始化前访问
}
```

## 高频面试题解析

### Q1：JavaScript 有哪些作用域类型？let/const/var 的作用域有什么区别？

**参考答案：**
三种作用域：
1. **全局作用域**：代码最外层，全局声明的变量任何地方可访问
2. **函数作用域**：函数内部声明的变量只在函数内可访问
3. **块级作用域**（ES6+）：`let`/`const` 在 `{}` 内具有块级作用域

`var` 是**函数作用域**，`let`/`const` 是**块级作用域**。`var` 允许重复声明和重新赋值，存在变量提升（初始化为 `undefined`）。`let` 和 `const` 也有提升但存在**暂存死区**，赋值前访问会报错。`const` 声明时必须赋值且不能重新赋值。

### Q2：什么是作用域链？变量查找规则是什么？

**参考答案：**
作用域链是嵌套作用域按**从内到外**顺序链接的链式结构。当访问一个变量时，JavaScript 引擎会：
1. 在当前作用域查找
2. 没找到则去父级作用域查找
3. 沿作用域链逐层向上，直到全局作用域
4. 都没找到 → `ReferenceError`

### Q3：词法作用域和动态作用域有什么区别？

**参考答案：**
- **词法作用域**（JavaScript 采用）：作用域在代码编写时确定，变量查找取决于函数的**定义位置**
- **动态作用域**：作用域在函数执行时确定，变量查找取决于函数的**调用位置**

举例：如果在函数 A 内定义了函数 B，但函数 B 在函数 C 中被调用——词法作用域中 B 看到的是 A 的变量；动态作用域中 B 看到的是 C 的变量。

### Q4：什么是暂存死区（TDZ）？

**参考答案：**
`let` 和 `const` 在声明语句执行之前，变量处于"暂存死区"。在此期间访问变量会抛出 `ReferenceError`。这是因为 `let`/`const` 的声明被提升了，但没有初始化（不像 `var` 初始化为 `undefined`），引擎从词法角度知道这些变量存在但不允许访问。

## 总结与扩展

### 核心要点

| 概念 | 要点 |
|------|------|
| **全局作用域** | 最外层，任何地方可访问 |
| **函数作用域** | var 的作用域单位 |
| **块级作用域** | let/const 在 `{}` 内的作用域 |
| **词法作用域** | 作用域在编写时确定，不是执行时 |
| **作用域链** | 从内到外的链式查找路径 |
| **变量提升** | 声明被提升到作用域顶部，赋值在原位置 |
| **暂存死区** | let/const 在声明前不可访问 |
