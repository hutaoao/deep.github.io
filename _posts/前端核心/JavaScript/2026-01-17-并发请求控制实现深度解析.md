## 一句话概括

并发控制的核心是一个**滑动窗口 + 等待队列**：同时最多跑 N 个任务，跑完一个就从队列取出下一个，超出的任务在 Promise 上 `await` 等待——队列空了或被唤醒时才继续。

---

## 核心知识点

### 1. 基础并发控制器 — 30 行搞定

```js
class ConcurrencyPool {
  constructor(limit) {
    this.limit = limit;
    this.running = 0;
    this.queue = [];
  }

  run(task) {
    return new Promise((resolve, reject) => {
      const exec = async () => {
        this.running++;
        try {
          resolve(await task());
        } catch (e) {
          reject(e);
        } finally {
          this.running--;
          if (this.queue.length > 0) {
            this.queue.shift()(); // 取出下一个等待者，唤醒它
          }
        }
      };

      if (this.running >= this.limit) {
        this.queue.push(exec); // 排队
      } else {
        exec(); // 直接跑
      }
    });
  }
}
```

核心机制：`this.running` 计数 → 超过 `limit` 就入队 → 任务完成时 `this.running--` 并 `shift()` 出队列。

### 2. 队列等待的 Promise 魔法

```js
// 排队的本质：创造一个 Promise，把 resolve 存起来
// 等轮到它的时候调用 resolve() → Promise 完成 → await 继续执行
if (this.running >= this.limit) {
  await new Promise(resolve => this.queue.push(resolve));
}
// 执行到这里说明轮到自己了
```

这是一个值得记住的模式——**用 Promise 的 resolve 做"信号量"**。

### 3. 批量执行 + 保持结果顺序

```js
async function runAll(tasks, limit = 3) {
  const pool = new ConcurrencyPool(limit);
  return Promise.all(tasks.map(task => pool.run(task)));
}

// 使用
const urls = ['/api/a', '/api/b', '/api/c', '/api/d', '/api/e'];
const results = await runAll(
  urls.map(url => () => fetch(url).then(r => r.json())),
  3
);
```

`Promise.all` 保证结果的顺序和 `tasks` 输入一致，而并发控制保证同时不超过 3 个在跑。

### 4. 失败重试 — 指数退避

```js
async function runWithRetry(task, { retries = 3, baseDelay = 1000 } = {}) {
  for (let i = 0; i <= retries; i++) {
    try {
      return await task();
    } catch (e) {
      if (i === retries) throw e; // 最后一次还失败就抛出去
      const delay = baseDelay * Math.pow(2, i); // 1s → 2s → 4s
      console.log(`重试 ${i + 1}/${retries}，${delay}ms 后`);
      await new Promise(r => setTimeout(r, delay));
    }
  }
}

// 结合并发池
const task = async () => {
  const res = await fetch('/api/data');
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json();
};
await pool.run(() => runWithRetry(task, { retries: 2 }));
```

指数退避避免重试风暴——服务端已经吃力时，越等越久是礼貌。

### 5. 带进度的完整调度器

```js
class TaskScheduler {
  constructor(limit) {
    this.pool = new ConcurrencyPool(limit);
    this.completed = 0;
    this.total = 0;
  }

  async addAll(tasks, onProgress) {
    this.total = tasks.length;
    this.completed = 0;

    return Promise.all(tasks.map(async (task, i) => {
      const ret = await this.pool.run(task);
      this.completed++;
      onProgress?.({ completed: this.completed, total: this.total, index: i });
      return ret;
    }));
  }
}

// 使用：10 个任务，并发 3 个，显示进度
const scheduler = new TaskScheduler(3);
const tasks = urls.map((url, i) => () =>
  fetch(url).then(r => r.json()).then(d => ({ i, d }))
);
const results = await scheduler.addAll(tasks, ({ completed, total }) => {
  console.log(`进度：${completed}/${total}`);
});
```

## 「其实你每天都在用」

**1. 浏览器对同域名的并发限制**

Chrome 默认对同一域名只开 6 个 TCP 连接。你的 `Promise.all` 虽然一次发了 100 个 fetch，但底层只有 6 个在跑。本文的调度器让你**主动控制**而不是被动等浏览器安排。

**2. 大文件分片上传**

```js
const chunks = splitFile(file, 5 * 1024 * 1024); // 每片 5MB
const scheduler = new TaskScheduler(3);
await scheduler.addAll(
  chunks.map((chunk, i) => () => uploadChunk(chunk, i, chunks.length)),
  ({ completed, total }) => updateProgressBar(completed / total)
);
```

**3. npm 包 `p-limit` / `async-pool`**

底层逻辑和本文的实现一模一样——`pLimit(3)` 就是一个并发池。

**4. Webpack / Vite 的并行打包**

这些工具内部都有并发控制——不会同时开几百个线程，而是根据 CPU 核心数限制并行任务。

**5. 数据库连接池**

`mysql2` 的 `connectionLimit: 10` 就是这样的并发控制——同时最多 10 个查询在跑，剩下的排队。

## 常见误解（FAQ）

**❌ 误区 1：「Promise.all 本身就能控制并发」**

`Promise.all` 只是"等所有人完成"，不做任何并发限制。你传 100 个 Promise，它就同时启动 100 个。控制并发数要自己实现。

**❌ 误区 2：「await 在 for 循环里就是并发控制」**

```js
for (const url of urls) {
  await fetch(url); // 这是串行！一个完成才开始下一个
}
```
这是**串行**不是并发控制。并发控制是"同时跑 N 个，完成一个补一个"。

**❌ 误区 3：「并发数设得越大越快」**

过犹不及。浏览器同域名 6 连接限制意味着设 10 和 6 在浏览器端没区别。服务端角度，瞬间几百个请求可能触发限流或打挂服务。通常 3-6 是安全区间。

**❌ 误区 4：「失败直接 reject 就好」**

生产环境中，网络波动、服务端临时 503 都是常见现象，直接失败而不重试会让用户体验很差。配合指数退避重试 + 区分可重试错误（5xx、网络错误）和不可重试错误（4xx 客户端错误），才是工程级方案。

## 一句话总结

**并发控制不是限制速度，是给异步操作加一个"礼貌"的阀门——不让浏览器和服务端同时承受所有任务，而是排着队、有限度地干活，又快又稳。**
