## 一句话概括

React 的优先级调度机制基于 Lane（车道）优先级模型，将更新任务划分为 32 个不同优先级的车道，通过位运算实现高效的优先级比较、合并与抢占，同时利用饥饿问题处理机制确保低优先级任务最终能被执行。

## 背景与意义

### 为什么需要优先级？

在没有优先级的概念之前，React 中的状态更新都是"平等的"。无论是用户点击按钮触发的紧急更新，还是在异步请求回调中触发的数据更新，它们都会被**按顺序执行**，不能被中断或重新排列。

这在简单场景下没有问题，但在复杂交互中会引发糟糕的体验：

- 用户输入文本时，由于同时在处理一个大型列表的渲染更新，输入出现明显的延迟
- 点击按钮后，由于有多个低优先级的更新在排队，用户无法立即看到反馈
- 低优先级的动画效果阻塞了高优先级的表单提交

**优先级调度的目标**：让紧急的更新（如用户输入）先执行，延迟不紧急的更新（如数据加载），保证用户始终获得流畅的交互反馈。

### 从 ExpirationTime 到 Lane

React 17 及之前使用 **ExpirationTime（过期时间）** 模型来管理优先级。其核心思想是：为每个更新计算一个过期时间（毫秒级绝对值），过期时间越早，优先级越高。

ExpirationTime 模型的缺点：
1. 使用 32 位整型的最大精度限制（只能表示约 24.8 天）
2. 优先级粒度有限，无法精确表达"多个并发更新之间的依赖关系"
3. 位运算表达力不足，难以实现复杂的优先级遮蔽（suspension）和降级

React 18 引入的 **Lane（车道）优先级模型** 彻底重构了这一机制。

## 概念与定义

### Lane（车道）

可以理解为一条"优先级通道"。React 使用 32 位二进制数表示 32 条车道，其中：
- **高优先级的车道位于低位**（数值更小）
- **低优先级的车道位于高位**（数值更大）

```typescript
// Lane 的定义（简化）
type Lane = number;
type Lanes = number;  // 一组 lane 的组合

// 示例：位图表示
const SyncLane: Lane = 0b0000000000000000000000000000001;  // 第0位 - 同步优先级
const InputContinuousLane: Lane = 0b0000000000000000000000000000100;  // 第2位 - 用户输入
const DefaultLane: Lane = 0b0000000000000000000000000010000;  // 第4位 - 默认优先级
const TransitionLane: Lane = 0b0000000001000000000000000000000;  // 第10位 - 过渡式更新
```

### 任务优先级层级

React 的 Lane 模型定义了一个完整的优先级阶梯：

| 优先级级别 | 对应 Lane | 典型场景 |
|-----------|----------|---------|
| SyncLane | Lane 0 | 同步更新、首屏渲染、createRoot |
| InputContinuousLane | Lane 2 | 用户连续输入（input、scroll） |
| DefaultLane | Lane 4 | 普通状态更新（useState/setState） |
| TransitionLanes | Lane 10~21 | useTransition、useDeferredValue |
| RetryLane | Lane 22 | Suspense 重试 |
| IdleLane | Lane 30 | SSR hydrate、过期缓存更新 |

### 饥饿问题（Starvation）

当高优先级更新不断插入时，低优先级更新可能永远得不到执行——这就是饥饿问题。React 通过**自动优先级提升**来解决：每个低优先级任务在等待一段时间后，会自动提升优先级，直到被执行。

## 最小示例

下面的示例直观展示优先级调度的效果——高优先级的输入更新不会被低优先级的列表渲染阻塞：

```jsx
import React, { useState, useTransition } from 'react';

function PriorityDemo() {
  const [query, setQuery] = useState('');
  const [searchResults, setSearchResults] = useState([]);
  const [isPending, startTransition] = useTransition();

  const handleChange = (e) => {
    // 高优先级更新：立即更新输入框的值
    setQuery(e.target.value);

    // 低优先级更新：搜索结果的过滤和渲染
    startTransition(() => {
      const results = [];
      for (let i = 0; i < 50000; i++) {
        if (i.toString().includes(e.target.value)) {
          results.push({ id: i, label: `Item #${i}` });
        }
      }
      setSearchResults(results);
    });
  };

  return (
    <div>
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="输入搜索内容..."
        style={{ width: 300, padding: 8, fontSize: 16 }}
      />
      {isPending && <p>⏳ 搜索更新中...</p>}
      <ul style={{ opacity: isPending ? 0.6 : 1 }}>
        {searchResults.map(item => (
          <li key={item.id}>{item.label}</li>
        ))}
      </ul>
    </div>
  );
}

export default PriorityDemo;
```

当输入变化时，`setQuery` 对应的更新以高优先级进入 SyncLane，立即执行；而 `startTransition` 包裹的更新进入 TransitionLane，放在空闲时执行。用户的输入框保持流畅，不会被 SearchResult 的渲染阻塞。

## 核心知识点拆解

### 1. Lane 的位运算操作

Lane 模型选择 32 位整型不是随意的——JavaScript 的位运算符只能处理 32 位有符号整数，这刚好符合我们需要。

React 源码中大量使用位运算来实现高效的优先级操作：

```typescript
// packages/react-reconciler/src/ReactFiberLane.js

// 判断 subset 是否是 superset 的子集
function isSubsetOfLanes(set: Lanes, subset: Lanes): boolean {
  return (set & subset) === subset;
}

// 合并两个 Lanes
function mergeLanes(a: Lanes, b: Lanes): Lanes {
  return a | b;  // 按位或
}

// 取交集（最高优先级）
function intersectLanes(a: Lanes, b: Lanes): Lanes {
  return a & b;  // 按位与
}

// 找到非零的最低 lane（最高优先级的车道）
function getHighestPriorityLane(lanes: Lanes): Lane {
  return lanes & -lanes;  // 经典技巧：取最右边的 1
}

// 从 lanes 中移除某个 lane
function removeLane(set: Lanes, lane: Lane): Lanes {
  return set & ~lane;  // 按位取反后与
}
```

`lanes & -lanes` 是计算机领域中著名的位运算技巧（互补位运算），用于提取二进制数中最右边的 1。在 React 上下文中，它等价于：**找到当前所有待处理更新中优先级最高的那个**。

### 2. 优先级继承与批处理

当一个组件触发了高优先级更新，在其 Fiber 和整个父链上，React 会通过 `childLanes` 和 `subtreeFlags` 将优先级信息向上传播：

```typescript
function markUpdateLaneFromFiberToRoot(
  sourceFiber: Fiber,
  lane: Lane,
): FiberRoot {
  // 将 lane 标记在当前 fiber 上
  sourceFiber.lanes = mergeLanes(sourceFiber.lanes, lane);
  
  let alternate = sourceFiber.alternate;
  if (alternate !== null) {
    alternate.lanes = mergeLanes(alternate.lanes, lane);
  }
  
  // 沿父链向上传播
  let node = sourceFiber.return;
  while (node !== null) {
    node.childLanes = mergeLanes(node.childLanes, lane);
    if (alternate !== null) {
      alternate.childLanes = mergeLanes(alternate.childLanes, lane);
    }
    node = node.return;
  }
  
  // 最终到达 fiberRoot
  // ...
}
```

这种继承机制确保了一个子树中的优先级更新可以正确地被其父节点感知，从而在 Reconciliation 过程中做正确的跳过决策。

### 3. 优先级降级与饥饿处理

React 在每次渲染循环的开始阶段，会检查所有 pending 的 Lanes，对等待时间过长的任务执行"优先级提升"：

```typescript
// 简化版饥饿处理逻辑
function markStarvedLanesAsExpired(
  root: FiberRoot,
  currentTime: number
): void {
  const pendingLanes = root.pendingLanes;
  const expirationTimes = root.expirationTimes;
  
  let lanes = pendingLanes;
  while (lanes > 0) {
    const lane = getHighestPriorityLane(lanes);
    const expirationTime = expirationTimes[lane];
    
    if (expirationTime === NoTimestamp) {
      // 首次记录，设置过期时间
      expirationTimes[lane] = computeExpirationTime(lane, currentTime);
    } else if (expirationTime <= currentTime) {
      // 已过期，需要提升优先级
      root.expiredLanes |= lane;  // 将其标记为已过期
    }
    
    // 移除已处理的车道位
    lanes &= ~lane;
  }
}
```

当一个 Lane 被标记为 `expiredLanes` 后，它将以同步优先级立刻被执行，不再受时间切片的限制。这就解决了饥饿问题——低优先级更新不会永远等待。

## 实战案例（含完整代码）

### 案例：可中断的 tab 切换

构建一个包含多个数据密集型 tab 页面的组件，演示优先级调度在 tab 切换中的应用：

```jsx
import React, { useState, useTransition, Suspense, lazy } from 'react';

// 假设有三个数据密集的 tab 组件
const DashboardTab = lazy(() => delayForDemo(import('./DashboardTab')));
const AnalyticsTab = lazy(() => delayForDemo(import('./AnalyticsTab')));
const ReportTab = lazy(() => delayForDemo(import('./ReportTab')));

// 模拟数据加载延迟
function delayForDemo(promise) {
  return new Promise(resolve => {
    setTimeout(resolve, 2000);
  }).then(() => promise);
}

const TAB_COMPONENTS = {
  dashboard: DashboardTab,
  analytics: AnalyticsTab,
  report: ReportTab,
};

const TABS = [
  { key: 'dashboard', label: '📊 仪表盘' },
  { key: 'analytics', label: '📈 分析' },
  { key: 'report', label: '📋 报表' },
];

function TabSwitcher() {
  const [activeTab, setActiveTab] = useState('dashboard');
  const [isTransitionPending, startTransition] = useTransition();

  const switchTab = (tabKey) => {
    // tab 的 UI 切换是高优先级
    // 但数据加载和渲染是低优先级
    setActiveTab(tabKey);  // 高优先级：高亮 tab 按钮

    startTransition(() => {
      // 低优先级：加载 tab 内容
      setTabTransition(tabKey => tabKey);
    });
  };

  const [transitionTab, setTabTransition] = useState(tabKey => tabKey);
  const TabComponent = TAB_COMPONENTS[activeTab];

  return (
    <div style={{ fontFamily: 'system-ui, sans-serif' }}>
      <div style={{ display: 'flex', gap: 4, marginBottom: 16 }}>
        {TABS.map(tab => (
          <button
            key={tab.key}
            onClick={() => switchTab(tab.key)}
            style={{
              padding: '8px 20px',
              border: activeTab === tab.key ? '2px solid #1a73e8' : '1px solid #ccc',
              borderRadius: 6,
              background: activeTab === tab.key ? '#e8f0fe' : '#fff',
              cursor: 'pointer',
              fontWeight: activeTab === tab.key ? 'bold' : 'normal',
              transition: 'all 0.15s',
            }}
          >
            {tab.label}
          </button>
        ))}
      </div>

      <div style={{
        opacity: isTransitionPending ? 0.6 : 1,
        transition: 'opacity 0.2s',
        border: '1px solid #eee',
        borderRadius: 8,
        padding: 20,
        minHeight: 300,
      }}>
        <Suspense fallback={<div>⏳ 加载中...</div>}>
          {TabComponent && <TabComponent />}
        </Suspense>
      </div>
    </div>
  );
}
```

在这个案例中：
- **Tab 按钮高亮**是高优先级（SyncLane）：用户点击后立即获得视觉反馈
- **Tab 内容加载和渲染**是低优先级（TransitionLane）：即使数据量很大也不会阻塞按钮切换
- 用户在标签切换过程中，如果快速点击多个 tab，前一个 tab 的渲染会被中断，React 优先处理最新的 tab

没有 `useTransition` 时，点击 tab 会导致 100ms+ 的冻结；有了优先级调度，即使最复杂的 tab 页面也能保证 tab 切换的即时反馈。

## 底层原理（含源码分析）

### 1. FiberRoot 中的 Lane 状态管理

在 FiberRoot（渲染根节点）中，React 通过以下关键字段管理所有的 Lane 状态：

```typescript
// 源码位置: packages/react-reconciler/src/ReactFiberRoot.js
function FiberRootNode(containerInfo, tag, hydrate, identifierPrefix) {
  // 等待处理的所有 lanes
  this.pendingLanes = NoLanes;             // 0
  // 已经过期的 lanes（饥饿提升）
  this.expiredLanes = NoLanes;             // 0
  // 当前渲染中的 lanes
  this.renderLanes = NoLanes;              // 0
  // 已完成但尚未提交的 lanes
  this.finishedLanes = NoLanes;            // 0
  // 各 lane 的过期时间戳
  this.expirationTimes = new Array(31);     // 每个 lane 的绝对过期时间
  
  // 等待被挂起的 lanes
  this.pingedLanes = NoLanes;
  // 被 Suspense 挂起的 lanes
  this.suspendedLanes = NoLanes;
  // 可以 ping 的 lanes
  this.pingCache = null;
}
```

### 2. Lane 与批处理的交互

React 18 的自动批处理（Automatic Batching）也基于 Lane 模型实现：

### 3. getNextLanes 的调度决策

每次渲染前，React 会调用 `getNextLanes` 决定"本次应该渲染哪些 Lanes"：

```typescript
// 源码简化
function getNextLanes(root: FiberRoot, wipLanes: Lanes): Lanes {
  const pendingLanes = root.pendingLanes;
  if (pendingLanes === NoLanes) return NoLanes;

  let nextLanes = NoLanes;
  let suspendedLanes = root.suspendedLanes;
  let pingedLanes = root.pingedLanes;

  // 排除被 Suspense 挂起但未被 ping 的 lanes
  const nonIdlePendingLanes = pendingLanes & ~RetryLane;
  
  // 优先处理已过期的 lanes（饥饿任务）
  if (root.expiredLanes !== NoLanes) {
    nextLanes = root.expiredLanes;
    return nextLanes;
  }

  // 找到优先级最高的非挂起 lanes
  if ((nonIdlePendingLanes & ~suspendedLanes) !== NoLanes) {
    nextLanes = getHighestPriorityLanes(nonIdlePendingLanes);
    return nextLanes;
  }

  // 如果所有非空闲 lanes 都被挂起，尝试 pingedLanes 中的任务
  if (pingedLanes !== NoLanes) {
    nextLanes = getHighestPriorityLanes(pingedLanes);
    return nextLanes;
  }

  return nextLanes;
}
```

关键决策逻辑：
1. **过期优先**：标记为 expired 的 lane 无条件第一时间执行
2. **优先级排序**：从 pendingLanes 中选最高优先级的非挂起 lane
3. **挂起恢复**：如果所有非空闲 lane 都被 Suspense 挂起，但用户通过 fallback 交互后 ping 了某个 lane，则恢复该 lane
4. **Idle 兜底**：如果没有其他待处理 lanes，尝试 IdleLane

## 高频面试题解析

### Q1：React 18 的 Lane 模型相比 ExpirationTime 有什么优势？

**参考答案：**

1. **表达力更强**：32 位位图可以表达组合优先级，而 ExpirationTime 只能表达单维度的"谁更紧急"。例如 React 可以精确表示"这批更新包含 DefaultLane 和 TransitionLane 的混合"
2. **位运算效率高**：优先级合并（|）、交集（&）、取最高优先级（&-）等操作都是单条 CPU 指令，无需数值排序
3. **解决精度问题**：ExpirationTime 在 32 位整型下只能表示约 25 天的时间窗口，而 Lane 模型没有此限制
4. **更好地支持并发**：Lane 模型能精确追踪每个更新的生命周期状态（pending/suspended/pinged/expired），这是实现 Suspense 和并发模式的基础

### Q2：什么是优先级饥饿？React 如何解决？

**参考答案：**

优先级饥饿是指低优先级任务因高优先级任务不断插队而永不得执行的问题。

React 通过在每次渲染周期开始时调用 `markStarvedLanesAsExpired` 来解决：系统为每个 Lane 记录一个"绝对过期时间"，当超过这个时间后，该 Lane 被标记为 `expiredLanes`。在 `getNextLanes` 中，expiredLanes 拥有最高优先级——这意味着即使它原本是一个低优先级的 TransitionLane，一旦过期也会以 SyncLane 的优先级立刻执行。

这个机制确保了"低优先级任务最终一定会被执行"，只是可能被高优先级任务推迟，而不是无限推迟。

### Q3：useTransition 标注的更新优先级低于用户点击事件触发的更新，如何保证用户点击被优先处理？

**参考答案：**

当用户同时有 Transition 更新和 Sync 更新 pending 时，React 的调度流程如下：

1. 两个更新分别进入不同的 Lane——点击进入 DefaultLane，transition 进入 TransitionLanes
2. `getNextLanes` 比较优先级时，发现 SyncLane/DefaultLane 高于 TransitionLanes
3. 工作循环开始时，只有 SyncLane 被选中作为 `renderLanes`
4. Transition 相关的 Fiber 在其 `childLanes` 中标记了 TransitionLanes，但当前渲染不包含这些 lanes，所以遍历时会安全跳过这些子树
5. Sync 更新渲染完毕并 commit 后，再下一轮调度才会处理 TransitionLane
6. 如果在 Transition 渲染过程中又有新的 Sync 更新插入，Transition 会被中断，高优先级更新先执行

这种机制确保了"用户交互始终优先于被动更新"的体验理念。

## 总结与扩展

### 总结

React 的优先级调度机制是 Fiber 架构的心脏。Lane 模型用 32 个二进制位巧妙地实现了：

- **多级优先级**：从同步到空闲，覆盖所有更新场景
- **优先级组合**：通过位运算优雅处理复杂的优先级关系
- **饥饿保护**：过期时间机制保证低优先级任务不会永远等待
- **并发基础**：Suspense、useTransition、useDeferredValue 都依赖于 Lane 模型提供的语义

### 扩展：与现实世界的类比

可以把 React 的 Lane 模型想象成机场的登机口：

- **优先级高的乘客**（头等舱、商务舱、要客）可以随时插队，走专属通道
- **普通乘客**默认排队，但如果等待时间过长（饥饿），地勤人员会将其升级为"需要尽快处理的旅客"
- **中转旅客**（Suspense）可以暂时退出队列，等完成中转手续后再回来（ping 机制）
- 机场的系统能识别每位旅客的"状态位"，从而决定谁该先登机

这种"可升星、可插队、可暂离"的多层排队策略，正是 React 优先级调度的精髓。
