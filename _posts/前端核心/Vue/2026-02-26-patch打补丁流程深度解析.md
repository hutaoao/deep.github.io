## 一句话概括

Vue3 的 `patch` 函数是虚拟 DOM 核心中的核心——它通过**同层比较（Same-Level Comparison）**策略，递归地对比新旧两棵 VNode 树的差异，以最小化的 DOM 操作将变更高效地应用到真实 DOM 上，是驱动 Vue 响应式更新的"心脏泵血"机制。

## 背景与意义

### 为什么需要 patch？

在 Vue 中，当我们修改响应式数据时，组件会重新执行渲染函数生成新的虚拟 DOM 树（VNode Tree）。如果每次数据变化都直接重建整个真实 DOM，代价将极其高昂——浏览器需要重排（Reflow）和重绘（Repaint），性能无法接受。

Vue 的策略是：**用 JavaScript 对象（VNode）描述 DOM 结构，通过 patch 算法比较新旧两棵 VNode 树的差异，仅在需要时操作真实 DOM**。这个"差异比较 + 最小化更新"的过程就是 **patch（打补丁）**。

### Vue2 vs Vue3 的 patch 演进

| 维度 | Vue2 | Vue3 |
|------|------|------|
| 底层 Diff 算法 | 双端比较（4 指针） | 最长递增子序列（LIS）+ 双端混合 |
| 静态节点处理 | 无特殊优化 | `patchFlag` + `hoisted` 静态提升 |
| Fragment 支持 | 不支持 | 通过 `processFragment` 原生支持 |
| 事件缓存 | 无 | `cacheHandlers` 避免重复创建回调 |
| TreeShaking | 部分 | 全面，patch 逻辑模块化拆分 |

Vue3 的 patch 在编译期就打入了大量**优化标记（PatchFlags）**，使运行时可以跳过大量不必要的比较，性能相比 Vue2 提升 1.3~2 倍。

## 概念与定义

### 核心数据结构：VNode

VNode（Virtual Node）是虚拟 DOM 的基本单元，Vue3 中它的核心字段包括：

```typescript
interface VNode {
  __v_isVNode: true  // 标记为 VNode
  type: string | Component | Fragment | Text | Comment | Static
  props: (VNodeProps & ExtraProps) | null
  children: string | VNode[] | null
  shapeFlag: number       // 节点类型标志位
  patchFlag: number       // 动态内容标记
  dynamicProps: string[] | null  // 动态属性列表
  dynamicChildren: VNode[] | null // 动态子节点
  key: string | number | null
  el: HostElement | null  // 关联的真实 DOM 元素
  anchor: HostElement | null  // Fragment 定位锚点
  component: ComponentInternalInstance | null // 组件实例
  suspense: SuspenseBoundary | null
}
```

#### shapeFlag（形状标志）

一个按位掩码（bitmask），标识节点类型：

```typescript
enum ShapeFlags {
  ELEMENT = 1,           // 0001 — HTML 元素
  FUNCTIONAL_COMPONENT = 2, // 0010 — 函数式组件
  STATEFUL_COMPONENT = 4,   // 0100 — 有状态组件
  TEXT_CHILDREN = 8,        // 1000 — 文本子节点
  ARRAY_CHILDREN = 16,      // 10000 — 数组子节点
  SLOTS_CHILDREN = 32,      // 100000 — 插槽子节点
  TELEPORT = 64,
  SUSPENSE = 128,
  COMPONENT_SHOULD_KEEP_ALIVE = 256,
  COMPONENT_KEPT_ALIVE = 512,
}
```

#### patchFlag（补丁标记）

编译期注入的优化标记，告诉运行时"这个节点只需要处理这些变化"：

```typescript
enum PatchFlags {
  TEXT = 1,            // 动态文本节点
  CLASS = 2,           // 动态 class
  STYLE = 4,           // 动态 style
  PROPS = 8,           // 除 class/style 外的动态属性
  FULL_PROPS = 16,     // 所有属性都可能变（key 不固定）
  HYDRATE_EVENT = 32,
  STABLE_FRAGMENT = 64,    // Fragment 子节点顺序稳定
  KEYED_FRAGMENT = 128,    // Fragment 子节点带 key
  UNKEYED_FRAGMENT = 256,  // Fragment 子节点无 key
  NEED_PATCH = 512,        // 非 props 变更（如事件监听器）
  DYNAMIC_SLOTS = 1024,
  DEV_ROOT_FRAGMENT = 2048,
  STATIC = -1,             // 静态节点，全跳过
  BAIL = -2,               // 退出优化模式（退出 indicator）
}
```

### 核心函数签名

```typescript
const patch: (
  n1: VNode | null,      // 旧 VNode（null 表示挂载新节点）
  n2: VNode,              // 新 VNode
  container: HostElement, // 容器 DOM 元素
  anchor?: HostElement | null,  // 插入位置的锚点
  parentComponent?: ComponentInternalInstance | null,
  parentSuspense?: SuspenseBoundary | null,
  isSVG?: boolean,
  slotScopeIds?: string[],
  optimized?: boolean    // 是否启用优化模式
) => void
```

- **n1 === null** → 执行挂载（mount）：基于 n2 创建新的 DOM 节点
- **n1 !== null** → 执行打补丁（patch）：比较 n1 与 n2 差异并更新已有 DOM
- **n1 === n2** → 跳过（sameVnode 已处理引用相等的情况）

## 最小示例

### 一个简单的计数器示意的 patch 流程

{% raw %}
```vue
<template>
  <div class="counter">
    <h1>{{ title }}</h1>
    <button @click="count++">Count: {{ count }}</button>
  </div>
</template>

<script setup>
import { ref } from 'vue'
const count = ref(0)
const title = 'My Counter'
</script>
```
{% endraw %}

**首次渲染（mount）：**

```
render() 生成 VNode 树
    ↓
patch(null, n2, container)  // n1 = null, 执行 mount
    ↓
processElement(n1, n2, ...)  →  type = 'div'
    ↓
mountElement(n2, container)
    ↓
  → document.createElement('div')
  → patchProps(null, n2.props, el)  // 设置 class="counter"
  → mountChildren(n2.children, el)  // 递归挂载子节点
      → h1 文本节点: title 是静态文本，直接 createTextNode
      → button: 再次进入 processElement
          → mountElement
          → 创建 button DOM
          → patchProps(null, { onClick }, el)  // 绑定事件
          → mountChildren → 文本节点 "Count: 0"
    ↓
container.appendChild(el)
```

**点击按钮后（update）：**

{% raw %}
```
count.value 变化触发组件更新
    ↓
组件重新执行 render()
    ↓
生成新的 VNode 树 n2
    ↓
patch(n1, n2, container)  // n1 是旧的 VNode 树
    ↓
  n1.type === n2.type && n1.key === n2.key  // 同节点
    ↓
processElement(n1, n2, ...)
    ↓
patchProps(n1.props, n2.props, el)  // 无属性变化，跳过
    ↓
patchChildren(n1, n2, el)
    ↓
    - 检测到 h1 是静态节点（patchFlag = -1），跳过
    - 检测到 button 文本"Count: {{ count }}"有 patchFlag = 1（TEXT）
    ↓
patchTextContent(buttonEl, 'Count: 1')  // 精准更新文本内容
    ↓     ← 仅更新了一条文本节点！没有 DOM 重建
```
{% endraw %}

这就是 patch 的魔力——**只更新变化的文本节点，div 和 button 的 DOM 元素被完整复用**。

## 核心知识点拆解

### 1. 同层比较策略

Vue3 的 Diff 算法采用**同层比较（Same-Level Comparison）**，这是由两棵树的 BFS（广度优先遍历）性质决定的。

```
         div                     div            ← 同级
       /    \                  /    \
     h1    button    →      h1    span         ← 同级
     /        \             /        \
   "A"     "Count:0"     "A"     "Count:1"     ← 同级
```

**为什么不做跨层比较？**

> 跨层移动 DOM 的概率极低。在 Web 应用中，90% 以上的 DOM 变更都发生在同一层级内（属性修改、文本修改、同层子节点增删）。**放弃跨层比对 = 用 O(1) 复杂度换取 O(n³) 的风险规避**——这是基于 Web 场景统计的工程权衡。

Vue3 的处理方式：如果新旧节点的层级不同，**直接 unmount 整棵树 + mount 新树**，不做深度复用。

```typescript
// 伪代码：同层比较的判断
const patch = (n1, n2, container) => {
  // 如果节点类型不同（跨类型），直接卸载旧节点，挂载新节点
  if (n1 && !isSameVNodeType(n1, n2)) {
    unmount(n1)
    n1 = null  // 标记为全新挂载
  }
  // ...
}

function isSameVNodeType(n1: VNode, n2: VNode): boolean {
  return n1.type === n2.type && n1.key === n2.key
}
```

### 2. 节点类型判断（多路分支）

Vue3 的 `patch` 函数使用 `switch-case` + `ShapeFlags` 掩码判断节点类型，分发到不同的处理函数：

```typescript
// 简化自 Vue3 源码 runtime-core/src/renderer.ts
const patch = (n1, n2, container, anchor, ...) => {
  if (n1 === n2) return  // 引用相等，跳过

  const { type, ref, shapeFlag } = n2

  // 先处理类型（type 优先于 shapeFlag）
  switch (type) {
    case Text:         processText(n1, n2, container, anchor); break
    case Comment:      processCommentNode(n1, n2, container, anchor); break
    case Static:       processStaticNode(n1, n2, container, anchor); break
    case Fragment:     processFragment(n1, n2, container, anchor, ...); break
    default:
      if (shapeFlag & ShapeFlags.ELEMENT) {
        processElement(n1, n2, container, anchor, ...)
      } else if (shapeFlag & ShapeFlags.COMPONENT) {
        processComponent(n1, n2, container, anchor, ...)
      } else if (shapeFlag & ShapeFlags.TELEPORT) {
        // Teleport 有自己独立的 patch 逻辑
        type.process(n1, n2, container, anchor, ...)
      } else if (shapeFlag & ShapeFlags.SUSPENSE) {
        type.process(n1, n2, container, anchor, ...)
      }
  }

  // 处理 ref
  if (ref != null && parentComponent) {
    setRef(ref, n1 && n1.ref, parentSuspense, n2)
  }
}
```

**关键点：** `type` 判断优先级高于 `shapeFlag`——Fragment、Text 等内置类型走 `type` 分支，HTML 标签和组件走 `shapeFlag` 分支。

### 3. processElement 处理函数

`processElement` 是元素节点的处理入口，负责挂载和更新：

```typescript
const processElement = (n1, n2, container, anchor, ...) => {
  if (n1 === null) {
    // 首次挂载 → 创建 DOM 元素
    mountElement(n2, container, anchor, ...)
  } else {
    // 更新 → 打补丁
    patchElement(n1, n2, ...)
  }
}

const mountElement = (vnode, container, anchor, ...) => {
  // 1. 创建 DOM 元素
  const el = vnode.el = document.createElement(vnode.type)

  // 2. 处理 props
  if (vnode.props) {
    for (const key in vnode.props) {
      patchProp(el, key, null, vnode.props[key])
    }
  }

  // 3. 处理子节点
  const { shapeFlag, children } = vnode
  if (shapeFlag & ShapeFlags.TEXT_CHILDREN) {
    el.textContent = children  // 文本子节点：直接设置 textContent
  } else if (shapeFlag & ShapeFlags.ARRAY_CHILDREN) {
    mountChildren(children, el, ...)  // 数组子节点：递归 mount
  }

  // 4. 插入到容器
  hostInsert(el, container, anchor)
}
```

**mountElement 的关键细节：**
- 子节点是**文本**时直接 `el.textContent = children`，性能极佳——不需要递归创建 DOM
- 子节点是**数组**时递归调用 `mountChildren`，深入下去创建子树
- `anchor` 参数控制插入位置（`insertBefore` vs `appendChild`）

`patchElement` 是更新入口：

```typescript
const patchElement = (n1, n2, ...) => {
  const el = n2.el = n1.el  // 复用 DOM 元素

  // 处理 props
  const { props: oldProps, patchFlag: oldPatchFlag } = n1
  const { props: newProps, patchFlag: newPatchFlag, dynamicProps } = n2

  // patchFlag 优化：编译期已知哪些属性是动态的
  if (newPatchFlag > 0) {
    // 只处理标记的动态属性
    if (newPatchFlag & PatchFlags.CLASS) {
      if (oldProps.class !== newProps.class) {
        hostPatchProp(el, 'class', null, newProps.class, ...)
      }
    }
    if (newPatchFlag & PatchFlags.STYLE) {
      hostPatchProp(el, 'style', oldProps.style, newProps.style, ...)
    }
    if (newPatchFlag & PatchFlags.PROPS) {
      // 只更新 dynamicProps 中列出的属性
      patchDynamicProps(el, oldProps, newProps, dynamicProps, ...)
    }
    if (newPatchFlag & PatchFlags.TEXT) {
      if (n1.children !== n2.children) {
        hostSetElementText(el, n2.children)
      }
    }
  } else {
    // 无 patchFlag → 全量对比 props
    patchProps(n1.props, n2.props, el, ...)
  }

  // 处理子节点
  if (newPatchFlag > 0 && newPatchFlag & PatchFlags.TEXT) {
    // 文本子节点已在上面的 TEXT 分支更新，跳过 patchChildren
  } else if (dynamicChildren) {
    // 仅对动态子节点做 patchBlockChildren
    patchBlockChildren(n1.dynamicChildren, n2.dynamicChildren, el, ...)
  } else {
    // 全量子节点 Diff
    patchChildren(n1, n2, el, ...)
  }
}
```

**这是 Vue3 性能的核心秘密：** 编译期标记出哪部分内容是动态的，运行时只 diff 这些动态部分。

### 4. processText 处理函数

文本节点是最简单的节点类型，不需要子 Diff：

```typescript
const processText = (n1, n2, container, anchor) => {
  if (n1 === null) {
    // 挂载文本节点
    n2.el = document.createTextNode(n2.children as string)
    hostInsert(n2.el, container, anchor)
  } else {
    // 更新文本
    const el = n2.el = n1.el
    if (n2.children !== n1.children) {
      el.nodeValue = n2.children as string  // 直接修改 nodeValue
    }
  }
}
```

**为什么用 `nodeValue` 而不是 `textContent`？** `nodeValue` 只作用于 Text 节点本身，`textContent` 会操作整个子树。在已知是纯文本节点时，`nodeValue` 是更精确的 API，性能也更好。

### 5. processFragment 处理函数

Fragment（`type === Symbol.for('v-fgt')`）是 Vue3 新增的能力，允许组件返回多个根节点。它的 patch 逻辑需要处理**多个锚点**：

```typescript
const processFragment = (n1, n2, container, anchor, ...) => {
  const fragmentStartAnchor = n2.el = n1 ? n1.el : document.createTextNode('')
  const fragmentEndAnchor = n2.anchor = n1 ? n1.anchor : document.createTextNode('')

  if (n1 === null) {
    // 挂载：插入两个锚点 + 子节点
    hostInsert(fragmentStartAnchor, container, anchor)
    hostInsert(fragmentEndAnchor, container, anchor)
    mountChildren(n2.children as VNode[], container, fragmentEndAnchor, ...)
  } else {
    // 更新：patch 子节点
    patchChildren(n1, n2, container, fragmentEndAnchor, ...)
  }
}
```

**Fragment 的两个锚点**——`el`（起始锚点）和 `anchor`（结束锚点）——用来在 DOM 中标定 Fragment 的范围，确保插入/移动操作的正确性。

### 6. patchProps 更新属性

`patchProps`（Vue3 源码中名为 `patchProp`）负责对比新旧 props 并更新 DOM 属性：

```typescript
// 完整的 patchProps 流程（简化版）
const patchProps = (oldProps, newProps, el, ...) => {
  if (oldProps === newProps) return

  // 新增 & 更新
  for (const key in newProps) {
    const prev = oldProps ? oldProps[key] : undefined
    const next = newProps[key]
    if (prev !== next) {
      hostPatchProp(el, key, prev, next, ...)
    }
  }

  // 删除旧属性中不存在的
  if (oldProps) {
    for (const key in oldProps) {
      if (!(key in newProps)) {
        hostPatchProp(el, key, oldProps[key], null, ...)
      }
    }
  }
}
```

`hostPatchProp` 是平台相关的（DOM 环境 vs SSR 环境），在 DOM 环境中它会：

```typescript
const patchDOMProp = (el, key, value) => {
  switch (key) {
    case 'class':
      el.className = normalizeClass(value) || ''
      break
    case 'style':
      // style 可以传字符串或对象
      if (typeof value === 'string') {
        el.style.cssText = value
      } else if (value && typeof value === 'object') {
        for (const prop in value) {
          el.style[prop] = value[prop]
        }
      }
      break
    default:
      // 事件监听：以 on 开头
      if (/^on[A-Z]/.test(key)) {
        // 采用事件缓存（cacheHandlers）：复用同一个回调引用
        const event = key.slice(2).toLowerCase()
        // 移除旧事件监听
        if (oldValue) el.removeEventListener(event, oldValue)
        // 添加新事件监听
        el.addEventListener(event, value)
      } else if (key === 'ref') {
        // ref 处理
      } else {
        // HTML attribute 或 DOM property
        // 优先 DOM property：el[key] = value
        // 兜底 setAttribute
      }
  }
}
```

**值得一提的优化——事件缓存（`cacheHandlers`）：** 当编译期检测到静态事件处理函数时，会把回调函数缓存起来，避免重新创建，这样 `removeEventListener` / `addEventListener` 也不需要执行了（引用没变）。

### 7. patchChildren 子节点更新

`patchChildren` 是 patch 中最复杂的部分，需要处理多种子节点类型组合：

```typescript
const patchChildren = (n1, n2, container, anchor, ...) => {
  const c1 = n1.children
  const c2 = n2.children
  const { shapeFlag: prevShapeFlag } = n1
  const { shapeFlag: nextShapeFlag } = n2

  // 9 种组合情况
  if (nextShapeFlag & ShapeFlags.TEXT_CHILDREN) {
    // 新子节点是文本
    if (prevShapeFlag & ShapeFlags.ARRAY_CHILDREN) {
      // 旧子节点是数组 → 先卸载，再设置文本
      unmountChildren(c1 as VNode[], ...)
    }
    if (c2 !== c1) {
      // 设置文本
      hostSetElementText(container, c2 as string)
    }
  } else if (nextShapeFlag & ShapeFlags.ARRAY_CHILDREN) {
    // 新子节点是数组
    if (prevShapeFlag & ShapeFlags.TEXT_CHILDREN) {
      // 旧是文本 → 清空容器，挂载新子节点
      hostSetElementText(container, '')
      mountChildren(c2 as VNode[], container, anchor, ...)
    } else if (prevShapeFlag & ShapeFlags.ARRAY_CHILDREN) {
      // 新旧都是数组 → 进入核心 Diff 算法
      patchKeyedChildren(c1 as VNode[], c2 as VNode[], container, anchor, ...)
    } else {
      // 旧无子节点 → 直接挂载
      mountChildren(c2 as VNode[], container, anchor, ...)
    }
  } else {
    // 新无子节点
    if (prevShapeFlag & ShapeFlags.TEXT_CHILDREN) {
      hostSetElementText(container, '')
    } else if (prevShapeFlag & ShapeFlags.ARRAY_CHILDREN) {
      unmountChildren(c1 as VNode[], ...)
    }
  }
}
```

**核心 Diff 算法（`patchKeyedChildren`）** 是 Vue3 最具标志性的部分。它采用**先快后慢**的策略：

```
patchKeyedChildren(c1, c2)
    ① 从头找到相同的前缀序列    → 从左向右 patch
    ② 从尾找到相同的后缀序列    → 从右向左 patch
    ③ 剩余部分执行 LIS 优化 Diff
        - 如果是无 key 子节点 → 简单遍历（O(n²) 退化兜底）
        - 如果是有 key 子节点 → 构建 key→index 映射表 + LIS
```

**最长递增子序列（LIS）算法：**

```typescript
const patchKeyedChildren = (c1, c2, container, ...) => {
  let i = 0
  const l2 = c2.length
  let e1 = c1.length - 1
  let e2 = l2 - 1

  // ① 从左向右同步
  while (i <= e1 && i <= e2 && c1[i].key === c2[i].key) {
    patch(c1[i], c2[i], container, ...)
    i++
  }

  // ② 从右向左同步
  while (i <= e1 && i <= e2 && c1[e1].key === c2[e2].key) {
    patch(c1[e1], c2[e2], container, ...)
    e1--
    e2--
  }

  // ③ 处理新增或删除
  if (i > e1) {
    // 新节点多于旧节点 → 挂载新增节点
    while (i <= e2) {
      patch(null, c2[i], container, anchor, ...)
      i++
    }
  } else if (i > e2) {
    // 旧节点多于新节点 → 卸载多余旧节点
    while (i <= e1) {
      unmount(c1[i], ...)
      i++
    }
  } else {
    // ④ 有 key 子节点的 Diff
    const s1 = i
    const s2 = i

    // 构建 key → index 映射
    const keyToNewIndexMap = new Map()
    for (i = s2; i <= e2; i++) {
      const nextChild = c2[i]
      if (nextChild.key != null) {
        keyToNewIndexMap.set(nextChild.key, i)
      }
    }

    // 遍历旧节点，匹配并打补丁，记录新索引
    let j
    let patched = 0
    const toBePatched = e2 - s2 + 1
    let moved = false
    let maxNewIndexSoFar = 0
    const newIndexToOldIndexMap = new Array(toBePatched).fill(0)

    for (i = s1; i <= e1; i++) {
      const prevChild = c1[i]
      if (patched >= toBePatched) {
        // 所有新节点都已 patch → 卸载剩余旧节点
        unmount(prevChild, ...)
        continue
      }
      const newIndex = keyToNewIndexMap.get(prevChild.key)
      if (newIndex === undefined) {
        // 旧节点不在新节点中 → 卸载
        unmount(prevChild, ...)
      } else {
        // 记录旧节点索引（+1 避免 0 值歧义）
        newIndexToOldIndexMap[newIndex - s2] = i + 1
        // 判断是否需要移动
        if (newIndex >= maxNewIndexSoFar) {
          maxNewIndexSoFar = newIndex
        } else {
          moved = true
        }
        patch(prevChild, c2[newIndex], container, ...)
        patched++
      }
    }

    // 移动和挂载
    const increasingNewIndexSequence = moved
      ? getSequence(newIndexToOldIndexMap)  // 计算 LIS
      : []

    j = increasingNewIndexSequence.length - 1
    for (i = toBePatched - 1; i >= 0; i--) {
      const nextIndex = s2 + i
      const nextChild = c2[nextIndex]
      const anchor = nextIndex + 1 < l2 ? c2[nextIndex + 1].el : parentAnchor

      if (newIndexToOldIndexMap[i] === 0) {
        // 全新节点 → 挂载
        patch(null, nextChild, container, anchor, ...)
      } else if (moved) {
        // 需要移动
        if (j < 0 || i !== increasingNewIndexSequence[j]) {
          // 不在 LIS 中 → 需要移动
          hostInsert(nextChild.el, container, anchor)
        } else {
          // 在 LIS 中 → 保持原位
          j--
        }
      }
    }
  }
}
```

**LIS 为什么能优化移动？** 最长递增子序列代表了"不需要移动就能保持相对顺序"的节点。Vue3 只移动**不在 LIS 中的节点**，而 LIS 中的节点保持原位。这比 Vue2 的双端比较在纯移动场景下更优。

### 8. 静态节点跳过优化（PatchFlags.STATIC）

Vue3 在编译期会把完全静态的子树（没有响应式绑定）**提升（hoist）** 到渲染函数外部，在 patch 时直接跳过。这是 Vue3 相比 Vue2 最大的性能优势之一：

```typescript
// BEFORE（无静态提升）
function render() {
  return h('div', [
    h('span', { class: 'static' }, 'Static Text'),  // 每次渲染都创建
    h('span', null, ctx.dynamicText)                 // 动态内容
  ])
}

// AFTER（Vue3 编译后）
const _hoisted_1 = h('span', { class: 'static' }, 'Static Text')
//   ↑ 静态 VNode 在渲染函数外部创建，永久复用

function render() {
  return h('div', [
    _hoisted_1,                                    // 直接复用，patch 时跳过
    h('span', null, ctx.dynamicText)               // 正常响应式
  ])
}
```

**静态提升的好处：**
1. **VNode 对象复用**：同一静态子树在每次渲染时使用同一个 VNode 引用
2. **patch 跳过**：`n1 === n2` 时直接 `return`，连进入 `patch` 函数都不需要
3. **内存分配减少**：避免 GC 压力

### 9. 按位掩码（Bitmask）的高效判断

Vue3 大量使用按位运算来判断节点类型和更新类型：

```typescript
// 是否为组件
shapeFlag & ShapeFlags.COMPONENT  // 检查第1位或第2位

// 是否为元素
shapeFlag & ShapeFlags.ELEMENT     // 检查第0位

// 组合判断
shapeFlag & (ShapeFlags.ELEMENT | ShapeFlags.COMPONENT)

// patchFlag 组合
patchFlag & PatchFlags.PROPS       // 是否有 PROPS 标记
patchFlag & (PatchFlags.PROPS | PatchFlags.STYLE | PatchFlags.CLASS)
```

位运算的特性——**单一 CPU 指令完成，无需分支预测**——使 Vue3 的类型判断比 Vue2 的 `typeof` / switch-case 快一个数量级。

### 案例 1：列表拖拽重排

{% raw %}
```vue
<template>
  <div>
    <TransitionGroup name="list">
      <div v-for="item in items" :key="item.id">
        {{ item.text }}
      </div>
    </TransitionGroup>
  </div>
</template>

<script setup>
const items = ref([
  { id: 1, text: 'A' },
  { id: 2, text: 'B' },
  { id: 3, text: 'C' },
  { id: 4, text: 'D' },
])

function shuffle() {
  items.value = [
    items.value[2],   // C
    items.value[0],   // A
    items.value[3],   // D
    items.value[1],   // B
  ]
}
</script>
```
{% endraw %}

**patch 过程跟踪：**

```
执行 shuffle() 后，新顺序：C, A, D, B
    ① look-ahead: C (idx=0) vs A (idx=0) → key 不同，退出
    ② look-behind: B (idx=3) vs B (idx=3) → key 相同，patch
    ③ look-behind: D (idx=2) vs D (idx=2) → key 相同，patch
    ④ 剩余: [C, A] vs [C, A]
        → 构建 key→index: { C: 0, A: 1 }
        → 遍历旧 [C, A]:
            C → newIndex=0, newIndexToOldIndexMap = [1, 0]
            A → newIndex=1, newIndexToOldIndexMap = [1, 2]
        → 计算 LIS: [1, 2] 即 [C, A] 全部在 LIS 中
        ⑤ 结论：不需要移动任何 DOM！
```

**关键洞察：** 虽然视觉上顺序变了（从 A-B-C-D 变成 C-A-D-B），但因为前置和后置匹配已经处理了 B 和 D，剩余部分 C→C、A→A 的对应关系意味着 DOM 引用都正确——最终只有 D 和 B 不动，C 和 A 保持原地，完美复用。

### 案例 2：组件切换与 KeepAlive

```vue
<template>
  <div>
    <KeepAlive>
      <CompA v-if="active === 'A'" />
      <CompB v-else-if="active === 'B'" />
      <CompC v-else />
    </KeepAlive>
  </div>
</template>
```

当 `active` 从 `A` 切换到 `B`：

```
新 VNode tree: type = Fragment（v-if/v-else 生成 Fragment）
    ① processFragment → patchChildren
    ② patchKeyedChildren:
        旧子节点: [CompA]    (key: 'A')
        新子节点: [CompB]    (key: 'B')
    ③ 左匹配失败（key 不同），右匹配失败
    ④ 进入剩余 Diff:
        keyToNewIndexMap: { B: 0 }
        遍历旧节点: CompA 的 key 'A' 不在映射表中 → unmount(CompA)
        遍历新索引: newIndexToOldIndexMap[0] = 0 → 全新节点
        ⑤ mount(CompB)

    KeepAlive 介入: unmount(CompA) 时被 KeepAlive 拦截
        → 不销毁 CompA 的 DOM，而是缓存到 KeepAlive 的 cache
        → CompB mount 时，KeepAlive 会先查 cache
        → 如果 CompB 之前存在，从 cache 恢复，跳过创建
```

**KeepAlive + patch 的协作机制：**
- `unmount` 时 `KeepAlive` 接管卸载逻辑，将组件实例缓存
- `mount` 时 `KeepAlive` 优先从缓存恢复，避免重复创建
- `shapeFlag` 中的 `COMPONENT_SHOULD_KEEP_ALIVE` / `COMPONENT_KEPT_ALIVE` 标志使 patch 走特殊路径

### 案例 3：大规模表格的性能优化

假设一个 1000 行的表格，每行 10 列，其中只有 2 列是动态的：

{% raw %}
```vue
<template>
  <table>
    <tr v-for="row in rows" :key="row.id">
      <td>{{ row.name }}</td>      <!-- 动态文本 -->
      <td>{{ row.status }}</td>     <!-- 动态文本 -->
      <td class="fixed">操作</td>   <!-- 静态 -->
      <td class="fixed">详情</td>   <!-- 静态 -->
    </tr>
  </table>

```
{% endraw %}

编译后，`v-for` 生成的 Fragment 被标记为 `KEYED_FRAGMENT`（`patchFlag = 128`），且每行的 `patchFlag` 为 `TEXT`（`patchFlag = 1`）：

```
patch 过程：
    ① patchElement(table) → patchChildren
    ② patchKeyedChildren(trs)
        → 比较 key 匹配
        → 对每个匹配的 tr patchElement
    ③ patchElement(tr) → patchChildren
        → 检测到：前 2 个 td 动态（patchFlag = 1），后 2 个 td 静态

    ④ patchBlockChildren:
        → 只更新 2 个动态 td 的文本内容
        → 静态 td 完全跳过
```

**性能对比：**

| 操作 | Vue2 | Vue3 | 提升 |
|------|------|------|------|
| 1000 行全量更新 | ~12ms | ~6ms | ~2x |
| 1000 行单列更新 | ~12ms（全量比较） | ~0.5ms（精确命中） | ~24x |
| 10000 行，10 列中 2 列动态 | ~185ms（全量 Diff） | ~15ms（Block Tree） | ~12x |

## 底层原理

### Block Tree（块树）优化

Vue3 最核心的编译优化是 **Block Tree**。它把模板结构分成"块（Block）"，每个块跟踪其内部的动态节点：

```typescript
// 编译后的 render 函数
function render(_ctx, _cache) {
  return (_openBlock(), _createBlock('div', null, [
    _createVNode('h1', null, _ctx.title, 1 /* TEXT */),
    _createVNode('button', { onClick: _ctx.increment }, 'Count: ' + _ctx.count, 1 /* TEXT */),
    _hoisted_1  // 静态节点
  ]))
}
```

每个 `_createBlock` 会调用 `openBlock` 收集动态子节点到一个 `dynamicChildren` 数组中：

```typescript
// openBlock + createBlock 的工作原理
let currentBlock: VNode[] | null = null

function openBlock(disableTracking = false) {
  currentBlock = disableTracking ? null : []
}

function createBlock(type, props, children, patchFlag) {
  const vnode = createVNode(type, props, children, patchFlag)
  vnode.dynamicChildren = currentBlock
  closeBlock()
  // 将 block 自身加入父 block 的 dynamicChildren
  if (currentBlock && !isBlockTreeEnabled) {
    currentBlock.push(vnode)
  }
  return vnode
}
```

这种"**只跟踪动态节点**"的策略意味着：

1. **静态子树在 VNode 级别就被跳过**——整个子树不在 `dynamicChildren` 中
2. **patch 时只遍历 `dynamicChildren`**，而不是全量子节点
3. **即使在嵌套结构中**，也能精准定位变化的位置

### unmount 的完整流程

当节点需要被移除时，`unmount` 会处理各种类型的清理：

```typescript
const unmount = (vnode, parentComponent, doRemove = false) => {
  const { type, props, ref, children, dynamicChildren, shapeFlag } = vnode

  // 1. 触发 beforeUnmount 钩子（组件）
  if (shapeFlag & ShapeFlags.COMPONENT) {
    // 调用组件实例的 beforeUnmount 生命周期
    invokeDirectiveHook(vnode, null, parentComponent, 'beforeUnmount')
  }

  // 2. 如果有 ref，清除 ref
  if (ref != null) {
    setRef(ref, null, parentSuspense, vnode)
  }

  // 3. 递归卸载子节点（通过 dynamicChildren 加速）
  if (dynamicChildren) {
    // 使用 Block Tree 快速卸载
    unmountChildren(dynamicChildren, parentComponent, ...)
  } else if (shapeFlag & ShapeFlags.ARRAY_CHILDREN) {
    unmountChildren(children as VNode[], parentComponent, ...)
  }

  // 4. 卸载 Teleport / Suspense
  if (shapeFlag & ShapeFlags.TELEPORT) {
    type.remove(vnode, ...)
  }
  if (shapeFlag & ShapeFlags.SUSPENSE) {
    // Suspense 内部可能有异步依赖
  }

  // 5. 触发 unmounted 钩子（组件）
  if (shapeFlag & ShapeFlags.COMPONENT) {
    invokeDirectiveHook(vnode, null, parentComponent, 'unmounted')
  }

  // 6. 移除 DOM
  if (doRemove) {
    hostRemove(vnode.el)
  }
}
```

**关键设计：** `unmount` 遵循与 `patch` 相同的模式——通过 `dynamicChildren` 实现 O(dynamic) 而非 O(all) 的递归。

### PatchFlags 的编译注入机制

PatchFlags 不是运行时计算出来的，而是 Vue 编译器在编译模板时**静态分析**注入的：

```typescript
// 编译阶段（compiler-core/src/transforms/transformElement.ts）
function transformElement(node, context) {
  return () => {
    // ... 编译后
    if (hasDynamicTextChild(node)) {
      patchFlag |= PatchFlags.TEXT
    }
    if (hasDynamicClass(node)) {
      patchFlag |= PatchFlags.CLASS
    }
    if (hasDynamicStyle(node)) {
      patchFlag |= PatchFlags.STYLE
    }
    if (hasOtherDynamicProps(node)) {
      if (hasDynamicKeys(node)) {
        patchFlag |= PatchFlags.FULL_PROPS
      } else {
        patchFlag |= PatchFlags.PROPS
        // 记录具体的动态属性名
        dynamicProps = getDynamicProps(node)
      }
    }
  }
}
```

**编译时 vs 运行时：**

{% raw %}
```
编译时（模板 → render 函数）          运行时（patch 过程）
├── 解析模板 AST                     ├── 接收 VNode 和 PatchFlags
├── 检测动态绑定（{{ }}、:bind、@）   ├── 通过 PatchFlags 判断更新范围
├── 生成 patchFlag、dynamicProps      ├── 只更新有标记的内容
├── 静态提升（hoisted）              ├── 跳过静态子树
├── 事件缓存（cacheHandlers）        ├── 监听器引用不变时跳过
└── Block Tree 构建                  └── 只遍历 dynamicChildren
```
{% endraw %}

这种"**编译时做更多、运行时做更少**"的哲学是 Vue3 性能领先的核心原因。

## 高频面试题解析

### Q1：Vue3 的 patch 和 React 的 reconcile 有什么本质区别？

**答：** 核心差异在于"编译时优化"。

- **React 的 reconcile**：纯运行时的 Diff。render 函数执行后生成 React Element 树，reconcile 算法全量比较新旧两棵树。即使没有任何动态更新，也会遍历整棵子树。
- **Vue3 的 patch**：编译时的 PatchFlags + Block Tree。模板在编译阶段被标记，运行时只需要处理标记为动态的部分。静态子树直接被跳过。

**类比：** React 是"把所有内容都扫一遍，找出差异"；Vue3 是"提前画好重点，只看有变化的部分"。

**性能差异：** 在 90% 静态 + 10% 动态的典型业务场景中，Vue3 的 patch 只需处理 10% 的节点，而 React 需要处理 100%。这是为什么 Vue3 在大型静态页面中性能优势更明显。

### Q2：为什么 Vue3 要把 Fragment 当作一等类型？

**答：** 主要为了支持**多根节点组件**和**更精确的子节点 Diff**。

- 多根节点：`v-for`、`v-if/v-else`、`<template>` 都隐式产生 Fragment
- 精准定位：Fragment 的起止锚点（`el` 和 `anchor`）让 Vue 知道子节点范围
- 优化：`STABLE_FRAGMENT` / `KEYED_FRAGMENT` / `UNKEYED_FRAGMENT` 三种标记允许运行时选择最合适的 Diff 策略

### Q3：key 的作用是什么？什么时候不需要 key？

**答：** `key` 的主要作用是帮助 Diff 算法**识别 VNode 的身份**，判断是"同一个节点的移动"还是"不同节点的替换"。

**不需要 key 的场景：**
1. **列表顺序稳定且不增删**（如静态菜单项）
2. **列表中的所有元素都不依赖 DOM 状态**（没有表单输入、没有 CSS 动画）
3. **使用 `UNKEYED_FRAGMENT` 标记时**（Vue3 内部会采用简单遍历）

但官方推荐：**只要使用 `v-for`，始终提供唯一的 `:key`**。这能让 Diff 从 O(n) 降到 O(1) 的匹配效率。没有 key 时，Vue3 退化为"同位置比较"，可能导致不必要的 DOM 重建。

### Q4：Vue3 的 Diff 比 Vue2 快在哪里？

**答：** 主要有以下几个层面：

| 优化维度 | Vue2 | Vue3 |
|---------|------|------|
| **算法** | 双端比较（4 指针） | 前缀/后缀 + LIS |
| **编译优化** | 无 | PatchFlags + 静态提升 + 事件缓存 |
| **子节点 Diff** | 全量子节点 | Block Tree（只 diff 动态子节点） |
| **静态节点** | 每次比较 | 完全跳过（引用相等） |
| **Props Diff** | 全量对比 | 只对比 dynamicProps 中的属性 |
| **位运算** | 较少 | 大量使用，分支更少 |

**实测数据：** Vue3 Template Explorer 编译以下代码：

{% raw %}
```vue
<div>
  <span>Static</span>
  <span>{{ dynamic }}</span>
</div>
```
{% endraw %}

Vue2 编译结果：每次渲染都会为两个 span 创建 VNode，patch 时比较两个 span 的全部属性。
Vue3 编译结果：第一个 span 被提升为 `_hoisted_1`，第二个 span 被标记 `patchFlag: 1 (TEXT)`。patch 时只比较 `<span>{% raw %}{{ dynamic }}{% endraw %}</span>` 的文本内容。

### Q5：patch 过程中如果有 computed 或 watch 触发会怎样？

**答：** Vue3 的响应式系统与 patch 通过**调度器（Scheduler）** 协作：

1. 响应式数据变化 → 触发组件的 effect（渲染副作用）
2. effect 被推入**异步队列**（`queueJob`）
3. 同一 tick 内的多次变更被合并为**一次更新**
4. 执行 `component.update()` → 运行 `render()` + `patch()`
5. `patch` 过程中如果再次触发响应式变更（如 computed 的 getter），会被标记依赖但**不会立即执行 patch 嵌套**

**核心保障：** Vue 保证**一个组件在同一 tick 内最多更新一次**。watch 的回调在 patch 完成后才执行（`post` effect），computed 在 patch 中被读取时只是做惰性求值。

## 总结与扩展

### 一张图总结 patch 整体流程

```
n1 = null? ──是──→ mountElement / mountComponent / mountFragment...
   │
   否
   │
   ├─ type 不同? ──→ unmount(n1) + mount(n2)
   │
   ├─ type 相同（元素）→ processElement(n1, n2)
   │     ├─ patchProps(n1.props, n2.props, el)
   │     │     ├─ 新 props 遍历 → patchProp (增/改)
   │     │     └─ 旧 props 遍历 → 删除多余属性
   │     │
   │     └─ patchChildren(n1, n2, el)
   │           ├─ 文本↔数组 → 清空/卸载 + 挂载
   │           ├─ 数组↔文本 → 卸载 + 设置文本
   │           ├─ 数组↔数组 → patchKeyedChildren
   │           │     ├─ 前缀匹配
   │           │     ├─ 后缀匹配
   │           │     ├─ 剩余: 新增/删除
   │           │     └─ 剩余: 移动(hash map + LIS)
   │           └─ 无↔有 → 卸载/挂载
   │
   ├─ type 相同（组件）→ processComponent(n1, n2)
   │     ├─ 更新 props → updateComponent
   │     ├─ 触发 beforeUpdate 钩子
   │     └─ 递归 patch → 子树更新
   │
   ├─ type 相同（Fragment）→ processFragment
   └─ type 相同（Text）→ processText → 更新 nodeValue
```

### 可扩展的思考方向

1. **SSR Hydration**：`patch` 在客户端激活（Hydrate）时，不再是"创建 DOM"，而是"遍历已有 DOM + 绑定事件"，这是 `hydrate` 函数做的事。

2. **自定义渲染器（Custom Renderer）**：`patch` 底层通过 `hostInsert` / `hostCreateElement` / `hostPatchProp` 等接口抽象了平台差异。你可以用 `createRenderer` 创建 Canvas、WebGL、Native 等任意平台的渲染器——`patch` 的逻辑完全复用，只需替换宿主 API。

3. **跨框架对比**：Solid.js 放弃了 VNode Diff，改为"编译时直接确定更新代码"（无运行时 Diff），Svelte 也是如此。Vue3 的折中方案是"编译时优化 + 运行时 Diff 兜底"，这在动态列表等场景中比纯编译方案更灵活。

4. **性能边界**：当列表规模超过 10 万条时，即使 LIS 优化 + Block Tree 也会感受到性能瓶颈。此时需要考虑虚拟滚动（Virtual Scrolling）——即从源头减少需要 Diff 的节点数量。

### 一句话再概括

Vue3 的 `patch` 不是一个单纯的 Diff 算法，而是一套**编译+运行时的协同系统**：编译阶段通过静态分析标记动态节点（PatchFlags）、提升静态子树、缓存事件处理函数；运行时基于这些标记，以 Block Tree 为骨架，只追踪和处理真正变化的部分，做到"不多做一次比较，不比对不变的内容"。

理解了 patch，也就理解了 Vue3 响应式渲染引擎的灵魂。
