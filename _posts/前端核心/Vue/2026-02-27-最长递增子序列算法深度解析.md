## 一句话概括

**最长递增子序列（Longest Increasing Subsequence, LIS）** 是在一个无序序列中找到一个最长的、元素严格递增的子序列（不要求连续）的算法。在前端领域，Vue 3 的虚拟 DOM Diff 算法巧妙地将 LIS 应用于移动最小 DOM 元素的决策——通过找出「不需要移动」的最长稳定子序列，从而将需要移动的元素数量降到理论最低，实现 O(n log n) 时间复杂度的极致优化。

## 背景与意义

### 一个老问题的前世今生

最长递增子序列问题并非新概念。早在 1970 年代，计算机科学家就已开始研究这类序列模式匹配问题。但真正让它走入前端开发者视野的，是 Vue 3 在 2020 年发布的 Composition API 中，对虚拟 DOM Diff 算法的全面重写。

在 Vue 2 时代，虚拟 DOM 的 Diff 算法使用的是**双端比较（dual-side comparison）** 策略，通过四个指针在两个数组的头尾同时进行比较和移动。这种方式简单直观，但在大量元素的**重新排序**场景下表现不佳——它只能处理部分模式下的移动优化，无法做到全局最优。

Vue 3 的作者尤雨溪（Evan You）在重写 Diff 算法时，引入了一个灵感来源于 **patience diff** 算法的思路：当新旧两个子节点列表的「头尾相同部分」被移除后，剩下的**乱序中间部分**，可以通过求解最长递增子序列来得到最少需要移动的节点数量。

### 为什么前端开发者需要理解 LIS？

很多前端工程师把「虚拟 DOM Diff」当作黑盒——会用 Vue/React 就够了，不需要了解底层。但当我深入了解 LIS 在 Vue 3 中的应用后，我深刻意识到：

1. **性能优化有天花板：** 不理解 Diff 算法，你就无法在遇到列表重排性能瓶颈时做出正确的优化决策
2. **算法思维是核心竞争力：** 将经典的算法问题与工程实践结合的能力，是区分普通工程师和高级工程师的关键
3. **面试高频区：** LIS 的贪心 + 二分实现是前端面试中的「必刷题」级别，Vue 3 Diff 源码分析更是高频考点

## 概念与定义

### 形式的定义

给定一个序列 `a[1..n]`，最长递增子序列问题要求找到一个最长的子序列 `a[i1], a[i2], ..., a[ik]`，满足：

1. `i1 < i2 < ... < ik`（索引严格递增，即**子序列不要求连续**）
2. `a[i1] < a[i2] < ... < a[ik]`（值严格递增）

返回值通常是**长度**或**具体子序列**。

### 关键区分：子串 vs 子序列

| 概念 | 要求 | 示例（序列为 [3, 5, 7, 1, 2, 8]） |
|------|------|----------------------------------|
| 子串（Substring） | 连续 | `[3, 5, 7]`、`[1, 2, 8]` |
| 子序列（Subsequence） | 不要求连续，仅索引递增 | `[3, 5, 7, 8]`、`[3, 5, 8]` |
| **最长递增子序列（LIS）** | 子序列 + 严格递增 | `[3, 5, 7, 8]`（长度 4） |

### 严格递增 vs 非严格递增

Vue 3 Diff 中需要使用**严格递增**（即 `a[i] < a[j]`，而不是 `a[i] <= a[j]`），因为 Vue 使用 **key 值**作为节点的唯一标识，同一个 key 不会重复出现，因此「严格」与「非严格」在 Vue 的场景下没有区别。但在一般的 LIS 问题中，是否需要允许相等元素，取决于具体场景。

## 最小示例

### 从一道经典算法题开始

```
输入: [10, 9, 2, 5, 3, 7, 101, 18]
输出: 4
解释: 最长递增子序列是 [2, 3, 7, 101]，长度为 4
```

等等——你可能注意到了，`[2, 5, 7, 101]` 也是长度为 4 的有效解。最长递增子序列**不一定唯一**，但长度是唯一的。

### 可视化求解过程

以序列 `[3, 5, 7, 1, 2, 8]` 为例，手动推演：

```
扫描位置 1: 3  → 当前递增序列: [3]
扫描位置 2: 5  → 比 3 大，追加: [3, 5]
扫描位置 3: 7  → 比 5 大，追加: [3, 5, 7]
扫描位置 4: 1  → 比 3 小，替换: [1, 5, 7]
扫描位置 5: 2  → 比 1 大、比 5 小，替换: [1, 2, 7]
扫描位置 6: 8  → 比 7 大，追加: [1, 2, 7, 8]
```

最终长度 = 4。

你可能会困惑：为什么「替换」操作不会破坏最终结果？这个问题非常关键——后面在「底层原理」部分我会详细解释。

## 核心知识点拆解

### 方案一：动态规划（O(n²)）

这是最直观的解法，也是理解 LIS 问题的入门钥匙。

**思路：**
- 定义 `dp[i]` 表示以第 `i` 个元素结尾的 LIS 长度
- 对于每个 `i`，遍历所有 `j < i`，如果 `a[j] < a[i]`，则 `dp[i] = max(dp[i], dp[j] + 1)`

**代码实现：**

```typescript
function lengthOfLIS(nums: number[]): number {
  const n = nums.length;
  const dp = new Array(n).fill(1);
  let maxLen = 1;

  for (let i = 1; i < n; i++) {
    for (let j = 0; j < i; j++) {
      if (nums[j] < nums[i]) {
        dp[i] = Math.max(dp[i], dp[j] + 1);
      }
    }
    maxLen = Math.max(maxLen, dp[i]);
  }

  return maxLen;
}
```

**复杂度分析：**
- 时间复杂度：O(n²) —— 两层循环，n 较大时（如 10⁵）不可接受
- 空间复杂度：O(n)

### 方案二：贪心 + 二分查找（O(n log n)）

这是 Vue 3 实际使用的思路，也是面试考查的重点。

**核心思想：**
- 维护一个数组 `tails`，其中 `tails[i]` 表示长度为 `i+1` 的递增子序列的**最小末尾值**
- `tails` 本身一定是严格递增的（这是一个关键的数学性质）
- 对于每个新元素 `num`，在 `tails` 中二分查找第一个 `>= num` 的位置，然后替换之；如果 `num` 比所有值都大，则追加到末尾

**为什么用「替换」而不是「追加」？**
替换操作并不会「破坏」已有长度的可能性——它只是为后续元素创造了**更有利的条件**。长度更小的末尾值意味着后续更多的数字能接上去形成更长的递增序列。这就是贪心策略的精髓：**每一步都保留当前最优的「潜力」**。

**代码实现：**

```typescript
function lengthOfLIS(nums: number[]): number {
  const tails: number[] = [];

  for (const num of nums) {
    let left = 0, right = tails.length;

    while (left < right) {
      const mid = (left + right) >> 1;
      if (tails[mid] < num) {
        left = mid + 1;
      } else {
        right = mid;
      }
    }

    if (left === tails.length) {
      tails.push(num);
    } else {
      tails[left] = num;
    }
  }

  return tails.length;
}
```

**复杂度分析：**
- 时间复杂度：O(n log n) —— 每个元素做一次二分查找
- 空间复杂度：O(n)

### 方案三：重构具体 LIS 序列

Vue 3 中 `getSequence` 函数不仅需要 LIS 的长度，还需要**具体的子序列**（哪些元素的索引不需要移动）。仅仅用方案二的 `tails` 数组是不够的——`tails` 中存储的**不是真正的 LIS**，只是「最优末尾」。

正确重构 LIS 需要**记录每个元素在 LIS 中的前驱位置**，最后回溯重建。

**优化版实现（带回溯）：**

```typescript
function getLIS(nums: number[]): number[] {
  const n = nums.length;
  // tails[i] = 长度为 i+1 的 LIS 的最小末尾值在原数组中的索引
  const tails: number[] = [];
  // 记录前驱索引，用于回溯重建
  const prev: number[] = new Array(n).fill(-1);

  for (let i = 0; i < n; i++) {
    let left = 0, right = tails.length;

    while (left < right) {
      const mid = (left + right) >> 1;
      if (nums[tails[mid]] < nums[i]) {
        left = mid + 1;
      } else {
        right = mid;
      }
    }

    // 记录前驱：如果 left > 0，它的前驱就是 tails[left - 1]
    if (left > 0) {
      prev[i] = tails[left - 1];
    }

    if (left === tails.length) {
      tails.push(i);
    } else {
      tails[left] = i;
    }
  }

  // 回溯重建 LIS
  const lis: number[] = [];
  let idx = tails[tails.length - 1];
  while (idx !== -1) {
    lis.unshift(nums[idx]);
    idx = prev[idx];
  }

  return lis;
}
```

## 实战案例

### 从 Vue 3 Diff 看 LIS 的应用

要理解 Vue 3 为什么需要 LIS，我们需要先理解它的 Diff 策略的整体流程。

#### 第一步：同步头部和尾部

Vue 3 Diff 的第一步是**从前向后**和**从后向前**同时比较新旧子节点列表，把相同的前缀和后缀直接「消化掉」：

```
旧列表: [A, B, C, D, E, F, G]
新列表: [A, B, E, D, C, F, G]
         ↑  ↑              ↑  ↑
         公共前缀          公共后缀
```

经过处理后，剩下的「乱序部分」为：
```
旧: [C, D, E]
新: [E, D, C]
```

#### 第二步：建立 key → 索引映射

Vue 3 会在新列表的剩余部分上建立一个 `key → index` 的映射表，然后遍历旧列表的剩余部分，生成一个**在新列表中的位置数组**：

```
旧列表（乱序部分）: [C, D, E]
旧元素在新列表中的位置:
  C → index 2
  D → index 1
  E → index 0
位置数组: [2, 1, 0]
```

> 如果某个旧元素的 key 在新列表中不存在，就标记为「已删除」。

#### 第三步：求解 LIS

对位置数组 `[2, 1, 0]` 求解最长递增子序列：

```
位置数组: [2, 1, 0]
最长递增子序列长度: 1（可以是 [2]、[1] 或 [0]）
```

这意味着：**最多只有一个元素可以保持不动**。如果我们选择保持 `C`（索引 2）不动，那么 `D` 和 `E` 都需要移动；如果我们选择保持 `E`（索引 0）不动，那么 `C` 和 `D` 都需要移动。

关键洞察：**LIS 中的元素就是「不需要移动」的元素**。LIS 越长，需要移动的元素越少，DOM 操作次数越少。

### 一个更直观的例子

```
旧列表: [1, 2, 3, 4, 5]
新列表: [3, 4, 5, 1, 2]

去除公共前缀/后缀后:
  没有公共前缀（3 ≠ 1）
  没有公共后缀（2 ≠ 5）
  
剩余部分（全部）:
  旧: [1, 2, 3, 4, 5]
  新: [3, 4, 5, 1, 2]

位置数组: [2, 3, 4, 0, 1]
           ↑  ↑  ↑  ↑  ↑
          1  2  3  4  5 在新列表中的位置

对 [2, 3, 4, 0, 1] 求解 LIS:
  最长递增子序列: [2, 3, 4]（对应元素 1, 2, 3？注意这里的 2,3,4 是位置值）

等等，这里需要非常小心——
位置数组 [2, 3, 4, 0, 1] 中:
  索引 0（值 2）→ 旧元素 1，在新列表位置 2
  索引 1（值 3）→ 旧元素 2，在新列表位置 3
  索引 2（值 4）→ 旧元素 3，在新列表位置 4
  索引 3（值 0）→ 旧元素 4，在新列表位置 0
  索引 4（值 1）→ 旧元素 5，在新列表位置 1

LIS: [2, 3, 4]（位置值）→ 对应旧元素的索引 [0, 1, 2] → 旧元素 [1, 2, 3]
```

结论：旧元素 `1, 2, 3` 可以保持不动，只需要把 `4` 和 `5` 移动到正确位置。相比于全部卸载再重新创建，这是一个巨大的性能提升。

#### 第四步：DOM 移动

有了 LIS 之后，Vue 3 会从新列表的尾部向前遍历，使用一个**「尾指针」（lastIndex）**来判定哪些元素在 LIS 中、哪些需要移动：

```typescript
// 伪代码示意
for (let i = newChildren.length - 1; i >= 0; i--) {
  if (lis.includes(newIndex)) {
    // 这个节点不需要移动
    lastIndex = newIndex;
  } else {
    // 这个节点需要插入到 lastIndex 对应的节点之前
    parent.insertBefore(node, refNode);
  }
}
```

这种**从后向前遍历**的策略非常巧妙：它保证了当我们插入一个元素时，它后面的所有元素都已经被放置到了正确的位置，因此插入的 `refNode` 总是可用的。

## 底层原理

### 贪心 + 二分的数学本质

理解 LIS 贪心算法的关键在于证明 `tails` 数组的两个重要性质：

**性质一：`tails` 数组是严格递增的**

证明：假设 `tails[i] >= tails[j]` 且 `i < j`。根据定义，`tails[j]` 是长度为 `j+1` 的递增子序列的最小末尾。这个长度 `j+1` 的子序列中，前 `j` 个元素构成的长度为 `j` 的子序列的末尾值一定小于 `tails[j]`。而 `tails[i]` 是长度为 `i+1` 的最小末尾，且 `i < j`，所以 `tails[i] <= 前 j 个子序列的第 i+1 个元素 < ... < tails[j]`。因此 `tails[i] < tails[j]`，与原假设矛盾。证毕。

正是这个性质保证了我们可以对 `tails` 使用二分查找。

**性质二：`tails[i]` 存储的是「当前已扫描元素中，长度为 i+1 的递增子序列的最小可能的末尾值」**

这意味着我们一直在用**更优的可能**替换**次优的末尾**。举个例子：
- 扫描到 3：`tails = [3]`
- 扫描到 1：`tails = [1]` —— 我们用 1 替换了 3，因为长度为 1 的 LIS 末尾是 1 比 3 更好（后继可以接更多元素）

这种「贪心替换」是算法的精髓——**我们始终保留最优的「种子」**。

### 时间复杂度 O(n log n) 的深入分析

```
扫描 n 个元素     → O(n)
每个元素二分查找   → O(log n)（在最长为 n 的 tails 数组中查找）
总体              → O(n log n)
```

这个复杂度在理论上已经是最优了——基于比较的排序算法的下界是 Ω(n log n)，而 LIS 问题可以通过排序规约证明其下界也是 Ω(n log n)。

### 为什么 tails 存储的是索引而不是值？

Vue 3 的 `getSequence` 函数中，`tails` 存储的是**元素在原数组中的索引**而不是值。这是因为：
1. Vue 3 需要最终输出**具体的 LIS（索引列表）**，而不仅仅是长度
2. 存储索引使得回溯重建成为可能
3. 比较时通过 `nums[tails[mid]]` 间接取值

### patience diff 的关联

Vue 3 的 Diff 算法很大程度上参考了 **patience diff**——一种用于文本比较的算法。Patience diff 的核心思想是：

1. 找到两个文件中「唯一匹配」的行的最长递增子序列
2. 这些匹配行被视为「已对齐」的不变内容
3. 剩下的内容用传统方式比较

Vue 3 将这个思路迁移到了虚拟 DOM Diff：
- 「唯一匹配的 key」对应 patience diff 中的「唯一匹配的行」
- LIS 求解对应 patience diff 中的对齐策略
- 未匹配的元素对应需要移动/创建/删除的节点

这个迁移是一次非常精彩的**跨领域降维打击**——将文本比较算法应用于 UI Diff。

### Vue 3 getSequence 源码逐行分析

以下是 Vue 3 源码中 `getSequence` 的核心实现（基于 `packages/runtime-core/src/renderer.ts`）：

```typescript
function getSequence(arr: number[]): number[] {
  const p = arr.slice() // 前驱数组，用于回溯
  const result = [0]    // result 就是 tails（存储索引），初始包含第一个元素的索引
  let i, j, u, v, c
  const len = arr.length

  for (i = 0; i < len; i++) {
    const arrI = arr[i]
    if (arrI !== 0) {  // 值为 0 表示新节点（需要创建），不参与 LIS 计算
      j = result[result.length - 1]
      if (arr[j] < arrI) {
        // 当前元素 > 当前 LIS 末尾 → 可以扩展 LIS
        p[i] = j       // 记录前驱
        result.push(i) // 追加到 result
        continue
      }

      // 二分查找：在 result 中找到第一个值 >= arrI 的位置
      u = 0
      v = result.length - 1
      while (u < v) {
        c = (u + v) >> 1
        if (arr[result[c]] < arrI) {
          u = c + 1
        } else {
          v = c
        }
      }

      // 替换：用更小的末尾值替换原来的值
      if (arrI < arr[result[u]]) {
        if (u > 0) {
          p[i] = result[u - 1]  // 记录前驱
        }
        result[u] = i
      }
    }
  }

  // 回溯重建 LIS 索引序列
  u = result.length
  v = result[u - 1]
  while (u-- > 0) {
    result[u] = v
    v = p[v]
  }

  return result
}
```

**关键代码解读：**

1. **`p` 数组（前驱）：** `p[i]` 表示在最终 LIS 中，索引 `i` 的前一个元素在原始数组中的索引。这保证了回溯的正确性。

2. **`result` 数组：** 这就是 `tails`（存储索引），它始终是递增的（指 `arr[result[i]]` 递增），但 `result[i]` 本身不一定递增。

3. **值为 0 的跳过：** Vue 3 中 `arr` 数组的 0 表示「新节点」，这些节点没有对应的旧节点，需要创建而不是移动。所以它们不参与 LIS 计算。

4. **`arr[j] < arrI` 分支：** 贪心策略——能扩展 LIS 就尽量扩展。这里的 `j` 是 `result` 末尾元素在原始数组中的索引。

5. **替换分支：** 当不能扩展时，用更小的值替换。注意这里比较的是 `arr[result[c]] < arrI`——用值比较来决定在 `tails` 中的位置。

6. **回溯部分：** 这是最容易被忽视但也是最精妙的部分。`result` 中存储的是 LIS 对应的索引，但经过替换操作后，`result` 直接存储的索引**并不是**真正的 LIS 索引序列。需要从 `result` 的最后一个元素开始，通过 `p` 数组向前回溯，才能得到真正的 LIS。

## 高频面试题解析

### 面试题 1：LIS 算法的基础实现

> **问题：** 请实现一个函数，求给定数组的最长递增子序列的长度。

**考察点：** O(n²) 的 DP 实现 + O(n log n) 的贪心+二分优化。面试官通常要求先讲 DP 思路，再过度到贪心优化。

**回答要点：**
- 先明确「子序列不要求连续」这个前提
- DP 思路：`dp[i]` 定义、状态转移方程、初始化
- 优化思路：`tails` 数组的维护、二分查找替换操作

### 面试题 2：Vue 3 为什么用 LIS 优化 Diff？

> **问题：** Vue 3 的 Diff 算法中，最长递增子序列有什么作用？为什么要用它？

**考察点：** 对 Vue 3 Diff 整体流程的理解 + LIS 的工程意义

**回答要点：**
- Diff 的基本流程：去头尾 → 建立 key 映射 → 生成位置数组 → LIS → DOM 操作
- LIS 的工程价值：找到「不需要移动」的最长子序列，最小化 DOM 操作
- O(n log n) 的高效性保证了大列表更新的性能
- 与 Vue 2 双端比较策略的对比

### 面试题 3：能否更高效？

> **问题：** LIS 问题能否做到 O(n) 时间复杂度？

**考察点：** 对算法复杂度理论的理解

**回答要点：**
- 不能。LIS 可以规约到排序问题，基于比较的算法下界是 Ω(n log n)
- 如果输入有特殊性质（如值是 1..n 的排列），有一些特定优化但不改变理论下界

### 面试题 4：Patience Diff 和 Vue 3 Diff 的关系

> **问题：** 讲讲 patience diff 算法，它和 Vue 3 的 Diff 有什么关系？

**考察点：** 知识广度 + 跨领域联想能力

**回答要点：**
- Patience diff 是文本比较算法，寻找唯一匹配行的 LIS 来对齐
- Vue 3 迁移思路：key → 唯一行，位置数组 → 行号数组
- 核心思想一致：找到「稳」的部分，只处理「变」的部分

## 总结与扩展

### 核心要点回顾

1. **LIS 定义：** 不要求连续的、严格递增的最长索引递增子序列
2. **O(n²) DP：** 理解问题的入门钥匙，面试时先讲
3. **O(n log n) 贪心+二分：** Vue 3 使用的高效实现，面试重点是 `tails` 数组的维护逻辑和二分查找的边界处理
4. **Vue 3 Diff 中的 LIS：** 位置数组 → LIS → 找出不动元素 → 最小化 DOM 移动
5. **`getSequence` 源码：** tails 数组 + 前驱数组 p + 回溯重建 LIS

### 扩展思考

- **React 的 Fiber Diff 为什么不使用 LIS？** React 的 Fiber 架构使用另一种 Diff 策略（基于链表和优先级），与 Vue 3 的静态分析策略走向了不同的技术路线。每种方案都有其适用场景。

- **LIS 在其它领域的应用：** 除了文本 diff 和虚拟 DOM，LIS 还在生物信息学（基因序列比对）、金融分析（寻找股票的最长上升期）等领域有广泛应用。

- **如果 key 不稳定的情况：** Vue 3 的 LIS 优化严重依赖于稳定的 key。如果你在渲染列表时使用数组索引作为 key，LIS 优化的效果将大打折扣。

- **性能实测：** 在实际项目中，对于含有大量节点的列表（如 1000+ 项），Vue 3 的 LIS 优化可以将完全乱序重排的 DOM 操作次数减少 50%-70%。特别是在拖拽排序、列表动态重组等场景下，性能提升尤为明显。
