## 一句话概括

Vue3 的 Diff 算法是在 Vue2 双端比较基础之上重构的**高效虚拟 DOM 对比算法**，通过「预处理 + 最长递增子序列」策略，将传统 O(n³) 的树对比问题压缩为 O(n) 级别的同级节点对比，在保持正确性的前提下大幅减少了 DOM 操作次数。

## 背景与意义

### 为什么需要 Diff 算法？

前端框架的核心使命之一，是将**状态变化高效地映射为视图更新**。在 Vue 中，每个组件都有对应的 `render` 函数，当响应式数据发生变化时，Vue 会重新执行 `render` 得到新的 Virtual DOM（虚拟 DOM）树。但直接销毁旧树、重建新树会带来巨大的性能开销——真实 DOM 的创建和插入是浏览器中最昂贵的操作之一。

**Diff 算法的价值**就在于：以最小的代价计算出新旧两棵虚拟 DOM 树之间的差异，然后只针对变化的部分进行真实 DOM 操作。

### Vue2 Diff 的局限

Vue2 的 Diff 算法采用的是**双端比较（Four-Way Comparison）**策略。它的核心思想是：同时从新旧两个子节点列表的头部和尾部向中间比较，每次比较尝试四种匹配方式：

1. 新头 vs 旧头
2. 新尾 vs 旧尾
3. 新尾 vs 旧头
4. 新头 vs 旧尾

如果四种匹配都不成功，再用旧节点的 key 建立哈希表，根据新节点的 key 去查找。这种策略在大多数场景下表现不错，但在**列表尾部新增元素**等场景下，Vue2 的 key 查找逻辑会引入不必要的遍历。

### Vue3 为何重构 Diff

Vue3 的作者尤雨溪在直播和多个公开场合提到过，Vue3 的 Diff 算法进行了**彻底的重新设计**，主要基于两个考量：

1. **编译优化**：Vue3 引入了 Block Tree + PatchFlags 机制，静态节点在编译阶段就被标记，Diff 时直接跳过，运行时只需关注动态节点。
2. **算法精简**：移除了 Vue2 中复杂的四路双端比较，改用**预处理 + 最长递增子序列**的简洁策略。虽然最坏情况下的复杂度依然是 O(n²)，但实际场景中几乎都接近 O(n)。

> 一个常被误解的事实是：Vue3 的 Diff 算法**不是传统的双端比较**，而是"先预处理再找最长递增子序列"的混合策略。所谓"双端比较"在 Vue3 中仅指预处理阶段的头尾对比。

## 概念与定义

在深入代码之前，先明确几个核心概念：

### 虚拟 DOM（VNode）

```typescript
interface VNode {
  type: string | Symbol | Component // 节点类型
  props: Record<string, any> | null  // 属性
  children: VNode[] | string | null  // 子节点
  key: string | number | null        // 唯一标识（Diff 的关键）
  el: Node | null                    // 对应的真实 DOM
  shapeFlag: number                  // 节点形状标记
  patchFlag: number                  // 动态标记（Vue3 编译优化核心）
  dynamicChildren: VNode[] | null    // 动态子节点（Block Tree）
}
```

### Key 的作用

`key` 是 Diff 算法的灵魂。它给每个 VNode 一个**稳定的身份标识**，让算法能判断出"同一个节点"在不同渲染中的位置变化。没有 key，Diff 只能按位置匹配（即"就地复用"），这在列表排序、插入等场景中会导致严重的 DOM 复用错误。

### PatchFlag（补丁标记）

Vue3 编译时会给动态节点打上 `patchFlag`，标记哪些属性可能会变化：

```typescript
export const enum PatchFlags {
  TEXT = 1,         // 动态文本节点
  CLASS = 1 << 1,   // 动态 class
  STYLE = 1 << 2,   // 动态 style
  PROPS = 1 << 3,   // 动态属性
  FULL_PROPS = 1 << 4, // 有动态 key 的属性
  NEED_PATCH = 1 << 5, // 需要完整 diff
  CHILDREN = 1 << 6,   // 动态子节点
  ...
}
```

有了 `patchFlag`，Vue3 可以在运行时**按需 patch**。例如，一个节点只打了 `TEXT` 标记，Diff 时只对比文本内容，完全跳过 props、children 等无关检查。

## 最小示例

先从一个最简单但能覆盖核心流程的场景开始。

假设我们有如下新旧两个子节点列表（数组中的字母代表节点的 key）：

**旧列表（旧 VNode 数组）：** `[A, B, C, D, E, F]`

**新列表（新 VNode 数组）：** `[A, B, D, E, C, G]`

肉眼观察变化：
- A、B 顺序没变
- C 从第 3 个位置移动到了第 5 个
- D、E 整体前移了（因为 C 移走了）
- F 被删除
- G 是新增

Vue3 Diff 要做的，就是精确地识别出这些操作，尽可能复用已有的 DOM 节点。

## 核心知识点拆解

Vue3 的 Diff 算法整体分为**五个阶段**，我们逐一拆解。

### 第一阶段：同步头部节点

从新旧列表的头部（索引 0）开始，逐一对比。如果 `key` 相同（或没有 key 时 type 相同），则直接 patch 并同时向后移动指针，直到遇到第一个不匹配的节点。

```
初始状态：
旧: [A, B, C, D, E, F]     i=0 (旧头)
新: [A, B, D, E, C, G]     j=0 (新头)

第1步：A === A → patch，i=1, j=1
第2步：B === B → patch，i=2, j=2
第3步：C !== D  → 停止头部同步
```

结果：
```
旧头指针 i: 2（指向 C）
新头指针 j: 2（指向 D）
```

### 第二阶段：同步尾部节点

从新旧列表的尾部（`旧尾 = len-1`，`新尾 = len-1`）开始，逐一对比。同样，key 相同则 patch 并向**前**移动指针，直到遇到第一个不匹配的节点。

```
初始：
旧尾 e1: 5（指向 F）
新尾 e2: 5（指向 G）

第1步：F !== G → 停止尾部同步

等等，我们需要同时从尾部往前对比：
```

正确的尾部同步应该是：

```
旧尾指针 e1: 5（指向 F）
新尾指针 e2: 5（指向 G）

第1步：F !== G → 停止

但如果列表是：
旧: [A, B, C, D, E, F]
新: [X, B, C, D, E, F]

则尾部同步会前进：
F === F → patch, e1=4, e2=4
E === E → patch, e1=3, e2=3
D !== X → 停止
```

在我们的示例中：

```
旧: [A, B, C, D, E, F]    e1=5
新: [A, B, D, E, C, G]    e2=5

第1步：F !== G → 停止头部尾部同步

尾部指针没有移动，e1=5, e2=5
```

### 第三阶段：判断是否纯新增或纯删除

经过前两个阶段的预处理，如果旧列表已经完全遍历完（`i > e1`），说明新列表中剩余的都是**新增节点**。反之，如果新列表已经遍历完（`j > e2`），说明旧列表中剩余的都是**需要删除的节点**。

在我们的示例中，`i=2, e1=5, j=2, e2=5`，既不满足 `i > e1` 也不满足 `j > e2`，所以既有删除也有新增和移动。进入第四阶段。

### 第四阶段：构造 key 索引映射表

当新旧列表都有剩余节点时，Vue3 会**用新列表剩余部分的 key 构建哈希表**（Map），然后遍历旧列表的剩余部分，标记哪些节点可以复用、哪些需要删除。

```
剩余区间：
旧剩余: [C, D, E, F]    (i=2 ~ e1=5)
新剩余: [D, E, C, G]    (j=2 ~ e2=5)
```

**步骤 1：构建新剩余节点的 key → 索引 Map**

```typescript
const keyToNewIndexMap = new Map()
keyToNewIndexMap.set('D', 0)  // 相对于 j 的偏移量
keyToNewIndexMap.set('E', 1)
keyToNewIndexMap.set('C', 2)
keyToNewIndexMap.set('G', 3)
```

**步骤 2：遍历旧剩余节点，打标记**

创建一个 `newIndexToOldIndexMap` 数组，长度等于新剩余节点数，初始化为 0。这个数组用于**建立新旧节点之间的索引映射关系**。

遍历旧剩余节点：

1. **C**：在 Map 中找到新索引 2，`newIndexToOldIndexMap[2] = 2`（旧索引为 2，+1 以区分默认值 0）
2. **D**：在 Map 中找到新索引 0，`newIndexToOldIndexMap[0] = 3`
3. **E**：在 Map 中找到新索引 1，`newIndexToOldIndexMap[1] = 4`
4. **F**：不在 Map 中 → 标记为**删除**

```
newIndexToOldIndexMap = [3, 4, 2, 0]
                         D  E  C  G未匹配
```

### 第五阶段：最长递增子序列（LIS）与最小移动

这是 Vue3 Diff 最精妙的部分。通过最长递增子序列算法，找出**那些不需要移动的节点**，剩下的节点只需围绕它们进行插入和移动。

**最长递增子序列**（Longest Increasing Subsequence，LIS）是一个经典的动态规划问题。给定一个序列，找到其中最长的、严格递增的子序列（不要求连续）。

在我们的例子中，`newIndexToOldIndexMap = [3, 4, 2, 0]`，我们需要找到这个序列的最长递增子序列对应的**索引**。

```
原始序列: [3, 4, 2, 0]
递增子序列: [3, 4]  (值 3→4)
对应索引: [0, 1]    (在原序列中的位置)
```

这意味着：**D 和 E 这两个节点在旧列表中的相对顺序与新列表一致，不需要移动。** 而 C（索引 2，值 2）打破了递增关系，需要移动。

现在，Vue3 从新剩余节点的尾部向前遍历（`k = 新剩余长度 - 1`），同时维护 `<unmount>` 指针指向 LIS 的末尾：

```
新剩余: [D, E, C, G]    逆序遍历

k=3: G → newIndexToOldIndexMap[3] === 0 → 纯新增节点，mount
k=2: C → 不在 LIS 中（索引 2 !== LIS[1]）→ 需要移动，将 C 插入到 D 前面
k=1: E → 在 LIS 中（索引 1 === LIS[1]）→ 不动，LIS 指针前移
k=0: D → 在 LIS 中（索引 0 === LIS[0]）→ 不动，LIS 指针前移
```

最终确定的 DOM 操作：
1. 删除 F
2. 新增 G（插入到最后）
3. 将 C 移动到 D 前面

一共仅 **3 次 DOM 操作**，相比全量重建的 6 次操作，减少了 50%。

### 完整模拟演示

让我们用一个更复杂的例子，从头到尾模拟一次完整的 Vue3 Diff 过程。

**场景：** 一个待办事项列表

```
旧列表（已排序）:  [买牛奶, 写代码, 跑步, 读书, 睡觉]  （key: A, B, C, D, E）
新列表（重新排序）: [写代码, 读书, 打游戏, 买牛奶, 睡觉, 看电影]  （key: B, D, F, A, E, G）
```

**第1步：头部同步**

```
i=j=0: A(买牛奶) vs B(写代码) → 不匹配，停止
头部指针：i=0, j=0
```

**第2步：尾部同步**

```
旧尾 e1=4(E), 新尾 e2=5(G)
E(睡觉) vs G(看电影) → 不匹配，停止
尾部指针：e1=4, e2=5
```

**第3步：判断非纯增删**

```
i(0) <= e1(4) 且 j(0) <= e2(5) → 进入复杂 diff
```

**第4步：构建 key 索引映射**

```
旧剩余: [A(0), B(1), C(2), D(3), E(4)]   → 索引 0~4
新剩余: [B(0), D(1), F(2), A(3), E(4), G(5)] → 相对于 j 偏移 0~5

keyToNewIndexMap: B→0, D→1, F→2, A→3, E→4, G→5

遍历旧剩余：
A：新索引 3 → newIndexToOldIndexMap[3] = 0+1 = 1
B：新索引 0 → newIndexToOldIndexMap[0] = 1+1 = 2
C：不在 Map → 删除
D：新索引 1 → newIndexToOldIndexMap[1] = 3+1 = 4
E：新索引 4 → newIndexToOldIndexMap[4] = 4+1 = 5

newIndexToOldIndexMap = [2, 4, 0, 1, 5, 0]
                        B  D  F  A  E  G
```

**第5步：LIS 计算**

```
序列: [2, 4, 0, 1, 5, 0]
LIS: [2, 4, 5]（值 2→4→5）
对应索引: [0, 1, 4]（B, D, E 不需要移动）

逆序遍历新剩余，组装 DOM 操作：
k=5: G → newIndexToOldIndexMap[5]=0 → mount（新增）
k=4: E → LIS[2]=4? 是 ✓ → 不动，LIS指针←
k=3: A → LIS[1]=4? 否 → 移动到下一个锚点后面（即 E 之前）
k=2: F → newIndexToOldIndexMap[2]=0 → mount（新增）
k=1: D → LIS[1]=4? 是... 等等，D 的 LIS 索引是 1

// 重新正确走一遍 LIS 匹配
LIS 索引数组（在 newIndexToOldIndexMap 中的位置）:
LIS = [0, 1, 4]  // 对应 B, D, E
LIS 指针 = len(LIS) - 1 = 2

k=5: G(新) → 取值 0 → mount
k=4: E → LIS[2]=4? 是 → 不动，LIS指针→1
k=3: A → LIS[1]=1? 否 → 移动到下一个"已处理"的节点之前
k=2: F → 取值 0 → mount
k=1: D → LIS[1]=1? 是 → 不动，LIS指针→0
k=0: B → LIS[0]=0? 是 → 不动，LIS指针→-1
```

**最终 DOM 操作：**

| 操作 | 节点 | 位置 |
|------|------|------|
| 删除 | C(跑步) | — |
| mount | F(打游戏) | 插入到 A 前面 |
| mount | G(看电影) | 插入到末尾 |
| 移动 | A(买牛奶) | 移动到 E 前面 |
| 保持 | B(写代码) | 不变 |
| 保持 | D(读书) | 不变 |
| 保持 | E(睡觉) | 不变 |

共 5 次操作（2 删除/新增 + 1 移动），而最坏情况需要 7 次（全量重建）。

## 实战案例

### 案例 1：列表尾部插入

```
旧: [A, B, C]
新: [A, B, C, D, E]
```

Vue3 的处理：
1. 头部同步：A、B、C 完全匹配，`i=3, j=3`
2. `i > e1 (3 > 2)` → 纯新增，直接 mount D 和 E

**DOM 操作：** 2 次 mount，0 次移动/删除。

Vue2 的处理：
1. 双端比较，匹配到 A、B、C 之后，旧列表已空
2. 剩余新节点做新增

实际结果相同，但 Vue3 的头部同步更简洁直接。

### 案例 2：列表头部插入

```
旧: [A, B, C]
新: [D, E, A, B, C]
```

Vue3 的处理：
1. 头部同步：D vs A → 不匹配，停止，`i=0, j=0`
2. 尾部同步：C vs C → 匹配，`e1=1, e2=3`
   B vs B → 匹配，`e1=0, e2=2`
   A vs A → 匹配，`e1=-1, e2=1`
3. `j <= e2 (0 <= 1) 且 i > e1 (0 > -1)` → 纯新增，在旧头部位置 mount D 和 E

**DOM 操作：** 2 次 mount（在旧 A 之前插入），0 次移动/删除。

### 案例 3：列表重新排序（拖拽场景）

```
旧: [A, B, C, D, E]
新: [A, C, E, B, D]
```

Vue3 Diff：
1. 头部同步：A 匹配，`i=1, j=1`
2. 尾部同步：E vs D → 不严格匹配。但仔细看旧尾 E 与新尾 D：
   旧尾 e1=4(E)，新尾 e2=4(D)
   E vs D → 不匹配
   D 在旧列表中，但不是尾部。尝试继续：
   e1=4(E), e2=4(D) → 不匹配，停止
   
   不过等等，让我们重新检查尾部同步。
   
   旧尾：指向 E，新尾：指向 D
   D 在新列表中是最后一个，在旧列表中是第三个。
   
   E vs D → 不匹配，尾部同步停止。e1=4, e2=4

3. 进入复杂阶段，剩余区间：
   ```
   旧剩余: [B, C, D, E] (索引 1~4)
   新剩余: [C, E, B, D] (索引 1~4)
   ```

4. 构建映射：
   ```
   keyToNewIndexMap: C→0, E→1, B→2, D→3
   
   遍历旧剩余：
   B(旧idx=1) → 新idx=2 → map[2] = 1+1 = 2
   C(旧idx=2) → 新idx=0 → map[0] = 2+1 = 3
   D(旧idx=3) → 新idx=3 → map[3] = 3+1 = 4
   E(旧idx=4) → 新idx=1 → map[1] = 4+1 = 5
   
   newIndexToOldIndexMap = [3, 5, 2, 4]
                            C  E  B  D
   ```

5. LIS：序列 `[3, 5, 2, 4]`，LIS 为 `[3, 5]`（C→E），对应的索引 `[0, 1]`

6. 逆序遍历新剩余，决定操作：
   ```
   LIS 索引位置: [0, 1]，LIS 指针 = 1
   
   k=3: D → LIS[1]=1? 否 → 移动 D
   k=2: B → LIS[1]=1? 否 → 移动 B
   k=1: E → LIS[1]=1? 是 → 不动，LIS指针→0
   k=0: C → LIS[0]=0? 是 → 不动，LIS指针→-1
   ```

**DOM 操作：** 2 次移动（B、D），其余保持不动。对比全量 5 次操作，效率提升 60%。

## 底层原理

### patchChildren 源码解读

Vue3 的核心 Diff 实现在 `packages/runtime-core/src/renderer.ts` 的 `patchChildren` 和 `patchUnkeyedChildren`/`patchKeyedChildren` 函数中。

关键源码流程（简化）：

```typescript
const patchKeyedChildren = (
  c1: VNode[],  // 旧子节点
  c2: VNode[],  // 新子节点
  container: RendererElement,
  parentAnchor: RendererNode | null,
  parentComponent: ComponentInternalInstance | null,
  parentSuspense: SuspenseBoundary | null,
  isSVG: boolean,
  slotScopeIds: string[] | null,
  optimized: boolean
) => {
  let i = 0               // 头部指针
  const l2 = c2.length     // 新列表长度
  let e1 = c1.length - 1   // 旧列表尾部指针
  let e2 = l2 - 1          // 新列表尾部指针

  // 阶段1: 头部同步 (从左到右)
  while (i <= e1 && i <= e2) {
    const n1 = c1[i]
    const n2 = c2[i]
    if (isSameVNodeType(n1, n2)) {
      patch(n1, n2, container, ...)
    } else {
      break
    }
    i++
  }

  // 阶段2: 尾部同步 (从右到左)
  while (i <= e1 && i <= e2) {
    const n1 = c1[e1]
    const n2 = c2[e2]
    if (isSameVNodeType(n1, n2)) {
      patch(n1, n2, container, ...)
    } else {
      break
    }
    e1--
    e2--
  }

  // 阶段3: 纯新增或纯删除
  if (i > e1) {
    // 旧列表已遍历完 → 新增剩余新节点
    if (i <= e2) {
      const nextPos = e2 + 1
      const anchor = nextPos < l2 ? c2[nextPos].el : parentAnchor
      while (i <= e2) {
        patch(null, c2[i], container, anchor, ...)
        i++
      }
    }
  } else if (i > e2) {
    // 新列表已遍历完 → 删除剩余旧节点
    while (i <= e1) {
      unmount(c1[i], ...)
      i++
    }
  } else {
    // 阶段4: 复杂情况 — 索引映射 + LIS
    const s1 = i  // 旧剩余起始
    const s2 = i  // 新剩余起始

    // 4a. 构建新剩余节点 key → index 映射
    const keyToNewIndexMap = new Map()
    for (i = s2; i <= e2; i++) {
      const nextChild = c2[i]
      if (nextChild.key != null) {
        keyToNewIndexMap.set(nextChild.key, i)
      }
    }

    // 4b. 遍历旧剩余，标记复用/删除
    let j
    let patched = 0
    const toBePatched = e2 - s2 + 1
    let moved = false
    let maxNewIndexSoFar = 0
    const newIndexToOldIndexMap = new Array(toBePatched).fill(0)

    for (i = s1; i <= e1; i++) {
      const prevChild = c1[i]
      if (patched >= toBePatched) {
        // 旧节点多于新节点 → 多余的删除
        unmount(prevChild, ...)
        continue
      }
      let newIndex = prevChild.key != null
        ? keyToNewIndexMap.get(prevChild.key)
        : findIdxInOld(/* ... */)
      
      if (newIndex === undefined) {
        unmount(prevChild, ...)  // 不在新列表中 → 删除
      } else {
        newIndexToOldIndexMap[newIndex - s2] = i + 1
        if (newIndex >= maxNewIndexSoFar) {
          maxNewIndexSoFar = newIndex
        } else {
          moved = true  // 出现了非递增 → 需要移动
        }
        patch(prevChild, c2[newIndex], container, ...)
        patched++
      }
    }

    // 4c. 最长递增子序列 + 移动/新增
    const increasingNewIndexSequence = moved
      ? getSequence(newIndexToOldIndexMap)
      : []

    j = increasingNewIndexSequence.length - 1

    for (i = toBePatched - 1; i >= 0; i--) {
      const nextIndex = s2 + i
      const nextChild = c2[nextIndex]
      const anchor = nextIndex + 1 < l2 ? c2[nextIndex + 1].el : parentAnchor

      if (newIndexToOldIndexMap[i] === 0) {
        // 值=0 → 纯新增
        patch(null, nextChild, container, anchor, ...)
      } else if (moved) {
        if (j < 0 || i !== increasingNewIndexSequence[j]) {
          // 不在 LIS 中 → 需要移动
          hostInsert(nextChild.el!, container, anchor)
        } else {
          j--  // 在 LIS 中 → 不动
        }
      }
    }
  }
}
```

### 最长递增子序列的实现

Vue3 中的 `getSequence` 函数是**耐心排序（Patience Sort）**的一种变体，采用贪心 + 二分查找，时间复杂度为 O(n log n)：

```typescript
function getSequence(arr: number[]): number[] {
  const p = arr.slice()  // 前驱记录，用于回溯
  const result = [0]     // 存储序列末尾的最小值索引
  let i, j, u, v, c

  const len = arr.length
  for (i = 0; i < len; i++) {
    const arrI = arr[i]
    if (arrI === 0) continue  // 跳过未匹配（新增节点）
    
    j = result[result.length - 1]
    if (arr[j] < arrI) {
      p[i] = j  // 记录前驱
      result.push(i)
      continue
    }

    // 二分查找：找到第一个 >= arrI 的位置
    u = 0
    v = result.length - 1
    while (u < v) {
      c = (u + v) >> 1
      if (arr[result[c]] < arrI) {
        u = c + 1
      } else {
        v = c
      }
    }

    if (arrI < arr[result[u]]) {
      if (u > 0) {
        p[i] = result[u - 1]  // 记录前驱
      }
      result[u] = i  // 替换
    }
  }

  // 回溯构建完整序列
  u = result.length
  v = result[u - 1]
  while (u-- > 0) {
    result[u] = v
    v = p[v]
  }

  return result
}
```

这段代码的巧妙之处在于它返回的是**索引数组**而非值数组。`result` 中存储的是 `newIndexToOldIndexMap` 中的位置索引，对应到新剩余节点中哪些节点可以保持不动，不需要移动。

### Vue2 vs Vue3 Diff 的本质差异

| 维度 | Vue2 | Vue3 |
|------|------|------|
| **核心策略** | 四路双端比较 + key 查找 | 预处理 + 最长递增子序列 |
| **头尾同步** | 双端比较（四路） | 双向同步（两路） |
| **移动判定** | 通过 key 查找决定 | LIS 决定最小移动集 |
| **最坏复杂度** | O(n²) | O(n²)（但实际接近 O(n)） |
| **编译优化** | 无 | PatchFlag + Block Tree |
| **key 不传时** | 按索引复用 | 按索引复用（同上） |
| **静态节点** | 仍然参与 diff | 编译时跳过，完全不参与 |

**Vue3 并非在所有场景都比 Vue2 快**。在极少数全部节点都移动且没有 key 的场景中，两者的复杂度接近。但得益于编译优化和 PatchFlags，Vue3 在日常开发中的绝大多数情况下都优于 Vue2。

### Block Tree 的加持

Vue3 Diff 另一个不可忽视的优化是 **Block Tree（区块树）**。在编译时，Vue3 会将模板的静态结构分析成层层嵌套的"区块"（Block），每个 Block 只追踪其内部的动态节点（通过 `patchFlag` 标记）：

{% raw %}
```html
<!-- 模板 -->
<div>
  <span>静态文本</span>
  <span>{{ dynamic }}</span>
  <ul>
    <li v-for="item in list">{{ item }}</li>
  </ul>
</div>
```
{% endraw %}

编译后，外层 div 是一个 Block，ul 是子 Block。当 `dynamic` 变化时，只有对应的 `<span>` 参与 Diff，外层的静态 `<span>` 被完全跳过。同理，`v-for` 列表变化时只影响其所在的 Block。

这意味着：**Vue3 的 Diff 粒度可以精确到一个节点，而不是整棵树**。这是 Vue2 完全不具备的能力。

## 高频面试题解析

### Q1：Vue3 为什么不用双端比较了？双端比较不是更优吗？

**答：** 这是一个常见的误解。Vue2 的双端比较（Four-Way Comparison）在每次循环中尝试四种匹配，理论上覆盖了更多场景。但 Vue3 的作者通过大量实际场景的 benchmark 发现，**Vue2 的双端比较在三路或四路不匹配时，退化为 key 查找，实际效果与 Vue3 的预处理 + LIS 策略相似甚至更差**。

Vue3 的策略更"直白"：
1. 先偷懒地从两边往中间收拢——因为大多数场景的变化发生在两端（尾部追加、头部插入）；
2. 中间剩下的部分，用 LIS 找到"不必移动"的节点集合，只操作需要动的。

这样代码更少、逻辑更清晰，而且 LIS 保证移动次数是最优的——这是双端比较做不到的。

### Q2：什么情况下应该使用 key？key 用 index 有什么问题？

**答：** 只要使用 `v-for`，就应该传 `key`。如果列表是**纯展示且不会重新排序**，index 作 key 不会出问题。但一旦涉及：

1. **列表排序**：index 不变但内容变了，Vue 会复用错误的 DOM
2. **列表中间插入/删除**：插入点之后的节点 index 全部变化，导致所有后续节点被卸载重建
3. **有状态的子组件**：组件内部状态（如 input 输入值）会被错误保留或丢失

{% raw %}
```html
<!-- 错误示范 -->
<li v-for="(item, index) in list" :key="index">{{ item }}</li>

<!-- 正确示范 -->
<li v-for="item in list" :key="item.id">{{ item.name }}</li>
```
{% endraw %}

用 index 作 key 在列表变化时，Diff 算法会认为所有节点都变了（因为每个位置的 key 都变了），导致全量卸载重建，性能反而更差。

### Q3：最长递增子序列为什么能减少 DOM 移动？

**答：** LIS 找到的是"在新列表顺序中，同时保持了旧列表相对顺序"的那部分节点。这些节点**在新旧列表之间的相对位置是自然正确的**，不需要移动。

举个例子：旧列表 `[A, B, C, D]`，新列表 `[B, D, A, C]`。LIS 找到 B→D（或 A→C），说明 B 和 D 的相对顺序在新旧列表中是一致的，它们不需要动。剩下的 A 和 C 需要移动来匹配新的顺序。

算法的妙处在于：**保留那些"不需要动的"，只移动剩下的**，这样总移动次数在所有可行方案中是最少的。这就是 LIS 的最小移动保证。

### Q4：没有 key 时 Vue3 的 Diff 怎么工作？

**答：** 当 `v-for` 没有传 key 时，Vue3 会走 `patchUnkeyedChildren` 路径——按索引逐一对比：

```typescript
const patchUnkeyedChildren = (c1, c2, ...) => {
  const oldLength = c1.length
  const newLength = c2.length
  const commonLength = Math.min(oldLength, newLength)
  
  // 按索引逐一 patch
  for (let i = 0; i < commonLength; i++) {
    patch(c1[i], c2[i], ...)
  }
  
  // 旧长新短 → 删除多余
  if (oldLength > newLength) {
    unmountChildren(c1, commonLength, ...)
  } 
  // 新长旧短 → 新增多余
  else {
    mountChildren(c2, commonLength, ...)
  }
}
```

这本质上就是**就地复用**——位置相同的节点直接 patch。优点是快（O(n)），缺点是当列表重排序时会复用到错误的 DOM。所以始终建议传 key。

### Q5：Vue3 Diff 和 React Diff 有什么异同？

**答：** 

相同点：
- 都基于同层对比（Tree Diff 降级为 List Diff）
- 都使用 key 作为节点标识
- 都用 `while`/`for` 循环遍历

不同点：

| 维度 | Vue3 | React |
|------|------|-------|
| **对比方向** | 双向（头→尾 + 尾→头） | 单向（从左到右） |
| **复用策略** | LIS 最小移动 | 仅检查 key 位置变化 |
| **移动判定** | 新索引是否递减 | key 是否匹配 |
| **编译优化** | 有（PatchFlag + Block Tree） | 无（全靠运行时） |
| **渲染触发** | 响应式系统精细触发 | 全量 diff + fiber 时间切片 |

React 的 Diff 策略更朴素：遇到 key 不同的节点就标记删除和重建。React 不追求"最小移动"，因为 React 团队认为 DOM 移动和 DOM 删除+重建的性能差异在实际中可以忽略，而 LIS 的 O(n log n) 计算也有开销。

Vue3 的 LIS 策略本质上是在"移动"和"计算"之间做权衡：用一点计算时间换取更少的 DOM 操作，这在 DOM 操作昂贵、JS 计算便宜的现代浏览器中是划算的。

## 总结与扩展

### 核心要点回顾

1. **Vue3 Diff ≠ 双端比较**：预处理阶段用的是双向同步，核心是 LIS 最小移动策略
2. **五个阶段**：头同步 → 尾同步 → 纯增删判断 → 索引映射 → LIS 移动
3. **LIS 的价值**：保证最小 DOM 移动次数，这是双端比较无法达到的
4. **编译优化是 Vue3 的真正底牌**：PatchFlag 和 Block Tree 让大多数节点根本不参与 Diff
5. **始终使用 key**：即使 Vue3 性能更强，不传 key 仍会走就地复用路径

### 延伸思考

Vue3 的 Diff 算法到目前（2026 年）已经非常成熟。展望未来，可能的方向包括：

- **编译时进一步前移**：将更多的 Diff 逻辑从运行时移到编译时，甚至部分模板可以在编译阶段就计算出精确的 DOM 更新指令
- **SSR 流式水合**：Vue3 的 Vapor Mode（无虚拟 DOM 模式）正在研发中，它将完全绕过虚拟 DOM Diff，直接在编译时将模板编译为命令式 DOM 操作
- **同质化趋势**：随着 Solid.js 等"细粒度响应式"框架的兴起，Vue 也可能在部分场景中探索不依赖虚拟 DOM 的方案
