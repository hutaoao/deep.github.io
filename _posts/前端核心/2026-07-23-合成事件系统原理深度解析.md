## 一句话概括

React 合成事件（SyntheticEvent）是 React 对浏览器原生事件的一层封装和抽象，通过事件委托在根容器上统一监听、通过事件池复用事件对象、通过 Fiber 树模拟事件传播路径，实现跨浏览器一致性、内存优化和 Fiber 架构下的生命周期管理。

## 背景与意义

### 为什么需要合成事件？

在传统的 DOM 事件模型中，每个元素上的事件监听都是独立注册的：

```javascript
// 传统方式：在 100 个元素上注册 click 事件
document.querySelectorAll('li').forEach(el => {
  el.addEventListener('click', handler);
  // 100 个元素 = 100 个事件监听器
});
```

问题显而易见：
1. **内存浪费**：每个监听器都占用内存，大列表时尤其严重
2. **跨浏览器差异**：不同浏览器的事件对象属性和行为不一致（比如 `e.preventDefault()` 在 IE 中只能通过 `returnValue = false` 实现）
3. **难以与 Fiber 集成**：原生事件的回调执行时机无法与 React 的渲染循环（render-commit 周期）协调

合成事件系统一次性解决了这三个问题。

### 事件委托的演进

React 在版本演进中改变过事件委托的方式：

| 版本 | 委托位置 | 说明 |
|-----|---------|------|
| React 16 | `document` | 统一在 document 上委托 |
| React 17+ | `rootNode` | 委托到 React 根容器（`createRoot` 的 container） |
| React 18+ | `rootNode` | 保持不变，支持多个 React 根共存 |

React 17 将委托点从 document 改为 rootNode 的主要原因是**隔离性**——如果页面中有多个 React 应用或多个非 React 代码片段，document 级别的事件委托可能导致事件顺序混乱和冲突。

## 概念与定义

### SyntheticEvent

React 对浏览器原生事件（`MouseEvent`、`TouchEvent`、`KeyboardEvent` 等）的跨浏览器封装：

```typescript
// 源码简化：packages/react-dom/src/events/SyntheticEvent.js

class SyntheticEvent {
  nativeEvent: Event;       // 原始浏览器事件
  type: string;             // 事件类型，如 'click'
  target: EventTarget;      // 触发事件的 DOM 元素
  currentTarget: EventTarget; // 当前正在处理事件的元素
  _reactName: string;       // React 内部使用的事件名
  _targetInst: Fiber;       // 触发事件的 Fiber 节点
  
  isPropagationStopped(): boolean;
  isDefaultPrevented(): boolean;
  preventDefault(): void;
  stopPropagation(): void;
  persist(): void;          // 从事件池中取出此事件
}
```

### 事件池（Event Pool）

React 15/16 中，合成事件对象被创建后不会直接丢弃，而是放入一个"池子"中复用。当事件处理函数执行完毕后，事件对象的所有属性被置为 null，对象本身回到池中等待下一次使用。

```typescript
// React 16 的事件池
const EventPool = {
  pool: [],
  
  acquire(): SyntheticEvent {
    if (this.pool.length > 0) {
      return this.pool.pop();
    }
    return new SyntheticEvent();
  },
  
  release(event: SyntheticEvent): void {
    // 清空所有属性
    event.destructor();
    // 放回池子
    this.pool.push(event);
  }
};
```

> **注意**：React 17 已经移除了事件池机制！这是 React 团队在不断优化中做出的取舍。现在我们来深入分析。

### 事件委托入口

```typescript
// React 17+ 事件委托的核心入口
const rootNode = document.getElementById('root');
ReactDOM.createRoot(rootNode);

// 内部：React 在 rootNode 上注册了几乎所有事件的监听
// rootNode.addEventListener('click', dispatchEvent, { capture: false });
// rootNode.addEventListener('keydown', dispatchEvent, { capture: false });
// ... 数十种事件
```

## 核心知识点拆解

### 1. 事件委托的三大概念

**委托监听（Delegation）**：React 不在每个 DOM 元素上单独注册事件处理函数，而是在根容器上注册"一个"监听器，通过事件冒泡捕获所有子元素的事件。

**事件分发（Dispatching）**：当根容器收到事件后，React 根据 `event.target` 找到对应的 Fiber 节点，然后从该节点开始，沿着 Fiber 父链向上遍历，依次触发绑定了该事件名的回调。

**事件处理（Handling）**：React 为每个事件名维护了一个"插件"（Event Plugin），负责：
- 从原生事件创建合成事件对象
- 管理事件处理函数的执行顺序
- 在事件处理完成后清理事件对象

### 2. 事件池的存废

在 React 16 及之前：

```jsx
function handleClick(e) {
  // e 是 SyntheticEvent
  setTimeout(() => {
    console.log(e.type); // ❌ 报错：e.type 已被置为 null
  }, 100);
}

// 解决方案：
function handleClick(e) {
  e.persist(); // 从事件池中取出，不再复用
  setTimeout(() => {
    console.log(e.type); // ✅ 'click'
  }, 100);
}
```

**React 17 移除了事件池**，原因是：

1. **增加复杂性**：开发者经常忘记 `e.persist()`，导致异步代码中事件对象为空
2. **性能收益微小**：现代浏览器的 GC 已足够高效，事件对象池的分配与回收不再是性能瓶颈
3. **对内存占用有反作用**：池中保留的事件对象虽然被清空了属性，但对象本身仍占用内存

所以从 React 17 开始，每次创建新的事件对象，处理完成后自然释放。`e.persist()` 现在是一个空操作（保留只为向后兼容）。

### 3. 事件体系的三层架构

React 事件系统有三层：

```
第一层：原生事件（Native Event）
        └── 浏览器捕获/冒泡机制，将事件传递到 rootNode
第二层：事件插件（Event Plugin）
        └── SimpleEventPlugin（click/keydown 等传统事件）
        └── EnterLeaveEventPlugin（mouseEnter/mouseLeave 等）
        └── ChangeEventPlugin（onChange）
        └── SelectEventPlugin（onSelect）
        └── BeforeInputEventPlugin（composition events）
第三层：调度与执行（Dispatch & Execution）
        └── 创建合成事件
        └── 遍历 Fiber 链，收集回调
        └── 按顺序执行回调
```

### 4. 事件路径的 Fiber 遍历

当事件到达 rootNode 后，React 需要确定"哪些组件在哪些 Fiber 上注册了这个事件"。这个过程称为**事件派发路径采集**：

```typescript
// 源码简化：packages/react-dom-bindings/src/events/EventListener.js

function createEventListenerWrapperWithPriority(
  targetContainer: EventTarget,
  domEventName: string,
  eventSystemFlags: number,
): Function {
  const eventPriority = getEventPriority(domEventName);
  
  // 根据事件类型分配不同的优先级
  let listenerWrapper;
  switch (eventPriority) {
    case DiscreteEvent:
      // click, keydown, focus, blur 等需离散处理的事件
      listenerWrapper = dispatchDiscreteEvent.bind(
        null, domEventName, eventSystemFlags, targetContainer,
      );
      break;
    case ContinuousEvent:
      // mousemove, scroll, touchmove 等连续事件
      listenerWrapper = dispatchEvent.bind(
        null, domEventName, eventSystemFlags, targetContainer,
      );
      break;
    default:
      // 其他事件
      listenerWrapper = dispatchEvent.bind(
        null, domEventName, eventSystemFlags, targetContainer,
      );
  }
  
  return listenerWrapper;
}
```

在 Fiber 树中找到注册了事件回调的路径：

```typescript
function getListeners(
  fiber: Fiber,
  reactEventName: string,
): Function[] {
  const listeners = [];
  
  let node = fiber;
  while (node !== null) {
    // 检查当前 Fiber 节点是否绑定了该事件
    const instance = node.stateNode;
    if (instance !== null) {
      const props = node.memoizedProps;
      if (props !== null && props[reactEventName] !== undefined) {
        // 找到绑定的回调！
        listeners.push({
          instance,
          listener: props[reactEventName],
          currentTarget: instance,
        });
      }
    }
    
    // 继续沿 Fiber 父链向上
    node = node.return;
  }
  
  return listeners;
}
```

### 案例：合成事件中的 stopPropagation 行为

一个展示原生事件和合成事件交互的复杂示例：

```jsx
import React, { useState, useRef, useEffect } from 'react';

function EventPropagationDemo() {
  const [logs, setLogs] = useState([]);
  const nativeBtnRef = useRef(null);
  const nativeDivRef = useRef(null);
  const logRef = useRef(null);

  const addLog = (msg) => {
    setLogs(prev => [...prev.slice(-49), msg]);
  };

  // 注册原生事件（对比用）
  useEffect(() => {
    const nativeBtn = nativeBtnRef.current;
    const nativeDiv = nativeDivRef.current;
    if (!nativeBtn || !nativeDiv) return;

    const handleNativeBtn = (e) => {
      addLog(`[原生] 按钮 click (e.target=${e.target.id || e.target.tagName})`);
    };
    const handleNativeDiv = (e) => {
      addLog(`[原生] 容器 click (e.target=${e.target.id || e.target.tagName})`);
    };

    nativeBtn.addEventListener('click', handleNativeBtn);
    nativeDiv.addEventListener('click', handleNativeDiv);

    addLog('⚠️ 注册了原生事件监听（用于对比）');
    addLog('');

    return () => {
      nativeBtn.removeEventListener('click', handleNativeBtn);
      nativeDiv.removeEventListener('click', handleNativeDiv);
    };
  }, []);

  // React 合成事件
  const handleReactBtn = (e) => {
    addLog(`[React] 按钮 click (target=${e.target.tagName})`);
    // 不阻止传播
  };

  const handleReactDiv = (e) => {
    addLog(`[React] 容器 click (target=${e.target.tagName})`);
  };

  const handleBtnWithStop = (e) => {
    addLog(`[React] 按钮 click + stopPropagation`);
    e.stopPropagation();
    // 阻止后，父容器的合成事件不会触发
    // 但父容器的原生事件呢？
    addLog('  → e.stopPropagation() 已调用');
  };

  const handleBtnWithNativeStop = (e) => {
    addLog(`[React] 按钮 click + nativeEvent.stopPropagation`);
    e.nativeEvent.stopPropagation();
    // 阻止了原生事件的继续冒泡
    addLog('  → e.nativeEvent.stopPropagation() 已调用');
    addLog('  → 父容器的原生事件和 React 事件都不会收到');
  };

  const clearLogs = () => setLogs([]);

  // 自动滚动到底部
  useEffect(() => {
    if (logRef.current) {
      logRef.current.scrollTop = logRef.current.scrollHeight;
    }
  }, [logs]);

  return (
    <div style={{ fontFamily: 'system-ui, sans-serif', maxWidth: 600 }}>
      <h3>🎯 合成事件 vs 原生事件传播对比</h3>

      {/* React 合成事件容器 */}
      <div
        onClick={handleReactDiv}
        style={{
          border: '2px solid #1a73e8',
          borderRadius: 8,
          padding: 20,
          marginBottom: 16,
        }}
      >
        <p style={{ fontSize: 13, color: '#555', margin: '0 0 8px 0' }}>
          React 事件容器（点击按钮测试传播路径）
        </p>

        <div ref={nativeDivRef} style={{ display: 'flex', gap: 8, flexWrap: 'wrap' }}>
          <button onClick={handleReactBtn} style={demoBtnStyle}>
            React 按钮（不阻止）
          </button>
          <button onClick={handleBtnWithStop} style={demoBtnStyle}>
            React 按钮（stopPropagation）
          </button>
          <button
            ref={nativeBtnRef}
            onClick={handleBtnWithNativeStop}
            style={demoBtnStyle}
          >
            React 按钮（native stopPropagation）
          </button>
        </div>

        <p style={{ fontSize: 11, color: '#999', marginTop: 8 }}>
          注意：每个按钮上同时绑定了原生 click 事件和 React onClick
        </p>
      </div>

      <button onClick={clearLogs} style={{
        padding: '4px 12px',
        border: '1px solid #e74c3c',
        borderRadius: 6,
        background: '#fff',
        color: '#e74c3c',
        cursor: 'pointer',
        fontSize: 12,
        marginBottom: 8,
      }}>
        清空日志
      </button>

      <div ref={logRef} style={{
        background: '#1e1e1e',
        color: '#d4d4d4',
        padding: 12,
        borderRadius: 8,
        fontFamily: 'monospace',
        fontSize: 11,
        maxHeight: 350,
        overflow: 'auto',
        lineHeight: 1.6,
      }}>
        {logs.length === 0 && (
          <div style={{ color: '#666' }}>点击按钮查看事件传播路径...</div>
        )}
        {logs.map((log, i) => (
          <div key={i} style={{
            color: log.startsWith('[') ? '#ffcc00' : '#d4d4d4',
            fontWeight: log.startsWith('[') ? 'bold' : 'normal',
          }}>
            {log}
          </div>
        ))}
      </div>

      <div style={{ marginTop: 12, background: '#f5f5f5', padding: 12, borderRadius: 8, fontSize: 12 }}>
        <strong>📌 关键观察：</strong>
        <ul style={{ margin: '4px 0 0 0', paddingLeft: 20 }}>
          <li>
            <strong>e.stopPropagation()</strong>：仅阻止 React 合成事件链的继续传播，
            不阻止父容器的原生事件监听
          </li>
          <li>
            <strong>e.nativeEvent.stopPropagation()</strong>：阻止原生事件冒泡，
            父容器的原生事件和 React 事件都不会触发
          </li>
          <li>
            <strong>React 事件实际注册在根容器上</strong>，所以"阻止事件传播"实际上阻止的是

          </li>
        </ul>
      </div>
    </div>
  );
}
```

## 底层原理（含源码分析）

### 1. 事件插件的注册机制

事件插件是 React 实现"不同事件名 → 不同处理逻辑"的抽象层：

```typescript
// 源码简化：packages/react-dom-bindings/src/events/ReactDOMEventListener.js

// 注册监听器到根容器
function listenToEvent(
  domEventName: string,
  targetContainer: EventTarget,
): void {
  // 检查是否已注册
  if (targetContainer._reactListeners?.has(domEventName)) {
    return;
  }
  
  // 建立 targetContainer._reactListeners 集合
  if (!targetContainer._reactListeners) {
    targetContainer._reactListeners = new Set();
  }
  targetContainer._reactListeners.add(domEventName);
  
  // 创建包装后的监听器
  const listener = createEventListenerWrapperWithPriority(
    targetContainer,
    domEventName,
    0,
  );
  
  // 注册到 DOM（默认在冒泡阶段）
  targetContainer.addEventListener(domEventName, listener, false);
  
  // 某些特殊事件在捕获阶段处理
  // capture phase events: onScroll, onFocus, onBlur, etc.
}
```

### 2. dispatchEvent 的执行链路

当原生事件触发时，执行流程如下：

```typescript
function dispatchEvent(
  domEventName: string,
  eventSystemFlags: number,
  targetContainer: EventTarget,
  nativeEvent: Event,
): void {
  // 1. 找到触发事件的 Fiber 节点
  const nativeEventTarget = getEventTarget(nativeEvent);
  const targetInst = getClosestInstanceFromNode(nativeEventTarget);
  
  // 2. 创建合成事件对象
  // 根据事件类型选择不同的事件插件
  const SyntheticEventCtor = getEventClassForType(domEventName);
  const reactEvent = new SyntheticEventCtor(
    domEventName,
    targetInst,
    nativeEvent,
    targetContainer,
  );
  
  // 3. 批量开始（进入事务环境）
  batchedUpdates(() => {
    // 4. 处理事件
    // 沿着 Fiber 链向上查找注册了回调的节点
    // 按捕获 → target → 冒泡的顺序执行
    runEventsInBatch(reactEvent, domEventName, targetInst);
  });
  
  // 5. 批量结束（React 16 中会在此处释放事件对象到池中）
}
```

### 3. runEventsInBatch 与捕获/冒泡模拟

合成事件的传播顺序也是"捕获 → target → 冒泡"，但这是 React 通过 Fiber 遍历模拟出来的：

```typescript
function runEventsInBatch(
  dispatchQueue: DispatchQueue[],
  event: ReactSyntheticEvent,
  targetFiber: Fiber,
): void {
  // 1. 收集捕获阶段的监听器（从根向下到 target）
  const captureListeners = [];
  let fiber = targetFiber;
  const capturePath = [];
  
  // 向上遍历到根，构建捕获路径
  while (fiber !== null) {
    capturePath.unshift(fiber);
    fiber = fiber.return;
  }
  
  // 捕获阶段：从根开始（根 → 父 → target）
  for (let i = 0; i < capturePath.length; i++) {
    const props = capturePath[i].memoizedProps;
    const captureCallback = props[`on${event.type}Capture`];
    if (captureCallback) {
      event.currentTarget = capturePath[i].stateNode;
      captureCallback(event);
      if (event.isPropagationStopped()) break;
    }
  }
  
  // 2. target 阶段
  event.currentTarget = targetFiber.stateNode;
  const targetCallback = targetFiber.memoizedProps[`on${event.type}`];
  if (targetCallback) {
    targetCallback(event);
  }
  
  // 3. 冒泡阶段：从 target 向上到根
  if (!event.isPropagationStopped()) {
    fiber = targetFiber.return;
    while (fiber !== null) {
      const props = fiber.memoizedProps;
      const bubbleCallback = props[`on${event.type}`];
      if (bubbleCallback) {
        event.currentTarget = fiber.stateNode;
        bubbleCallback(event);
        if (event.isPropagationStopped()) break;
      }
      fiber = fiber.return;
    }
  }
}
```

**注意**：上面的 `onXxxPropagationStopped` 是通过 `event.isPropagationStopped()` 标记来控制的。当用户调用 `e.stopPropagation()` 时，React 会设置 `_isPropagationStopped = true`，之后 Fiber 链上注册的其他回调就不会执行。

## 高频面试题解析

### Q1：React 合成事件和原生事件有什么区别？为什么会存在合成事件？

**参考答案：**

**主要区别：**

| 维度 | 合成事件（SyntheticEvent） | 原生事件（Native Event） |
|------|-------------------------|------------------------|
| 注册方式 | 通过 JSX 属性（onClick） | addEventListener 或 DOM 属性 |
| 跨浏览器 | 统一所有浏览器行为 | 不同浏览器有差异 |
| 内存效率 | 事件委托，减少监听器数量 | 每个元素单独注册 |
| 事件对象 | 跨浏览器包装对象 | 浏览器原生 Event |
| 对象回收 | React 17+ 自动释放，16 有对象池 | 由浏览器管理 |
| stopPropagation | 阻止 React 事件链 | 阻止 DOM 事件冒泡 |

**存在合成事件的原因：**
1. **跨浏览器一致性**：抹平 IE、Chrome、Firefox、Safari 等浏览器的事件行为差异
2. **性能优化**：通过事件委托减少内存中事件监听器的数量
3. **与 Fiber 架构集成**：合成事件能正确映射到 Fiber 节点，在 Concurrent 模式下正确处理事件优先级
4. **提供统一的编程接口**：`e.preventDefault()`、`e.stopPropagation()` 在所有浏览器中行为一致

### Q2：e.stopPropagation() 和 e.nativeEvent.stopPropagation() 有什么区别？

**参考答案：**

```javascript
// 两者有本质区别：
e.stopPropagation();                 // 阻止 React 事件链（合成事件层面）
e.nativeEvent.stopPropagation();     // 阻止 DOM 事件冒泡（原生事件层面）
```

**具体区别：**

1. `e.stopPropagation()` 阻止的是 React 沿 Fiber 树的回调遍历。调用后，父组件的 React 事件处理函数不会执行，但**靠 root 更近的原生事件监听器仍可能触发**
2. `e.nativeEvent.stopPropagation()` 阻止的是浏览器原生事件的传播。调用后，即使是绑定在 root 容器上的 React 事件分发也不会触发（因为事件已经无法冒泡到 root 容器）

**关键场景**：如果页面中既有 React 代码又有非 React 代码：

```jsx
// 在一个 React 组件内
<button onClick={(e) => {
  e.nativeEvent.stopPropagation();
  // 确保不会冒泡到 document 上绑定的非 React 监听器
}}>
```

### Q3：React 17 为什么移除了事件对象池（Event Pooling）？

**参考答案：**

React 17 移除事件池的主要原因是：

1. **简化心智模型**：事件池在 React 16 中导致了一个常见的陷阱——在异步代码中访问合成事件属性会得到 null：
   ```jsx
   function handleClick(e) {
     fetch(url).then(() => {
       console.log(e.type); // React 16: null! 需要 e.persist()
     });
   }
   ```
   开发者经常忘记调用 `e.persist()` 导致难以排查的 Bug

2. **性能收益不再显著**：事件池的设计最初是为了减少垃圾回收压力，但现代引擎（V8、SpiderMonkey）的 GC 和对象分配已经非常高效。池化带来的性能优势微乎其微

3. **避免内存浪费**：池化需要维护一个后备的对象列表，即使是未使用的事件对象也占用内存

4. **调试困难**：在 React DevTools 中，池化的事件对象难以调试，因为它的属性在异步调用时已被清空

从 React 17 开始，每次事件处理都创建全新的 SyntheticEvent 对象，处理完毕后直接丢弃等待 GC 回收。这意味着 `e.persist()` 变成了空操作，只是为了向后兼容保留。

## 总结与扩展

### 总结

React 合成事件系统是连接"浏览器原生事件"和"React Fiber 渲染"的桥梁：

- **事件委托**：在根容器上集中注册，减少监听器数量
- **事件对象包装**：提供跨浏览器一致的 API
- **Fiber 路线遍历**：模拟捕获-冒泡的三阶段传播模型
- **事件优先级**：为 Concurrent 模式提供离散/连续事件分类

### 扩展：即将到来的 React Forget

React 团队正在开发的新编译器（React Forget）对事件系统的影响值得关注。如果编译时能够确定事件回调的依赖关系和执行时机，React 可能进一步优化事件分发过程——比如跳过不需要重新渲染的组件的回调收集步骤。但无论如何，合成事件作为"跨浏览器一致性和性能优化的统一封装层"的核心价值不会改变。
