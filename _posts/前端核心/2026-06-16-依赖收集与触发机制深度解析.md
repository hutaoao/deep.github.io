## 一句话概括

本文深入解析 Vue 3 响应式系统的核心——依赖收集（track）与触发更新（trigger）机制的完整实现原理，从数据结构到源码实现，从基础概念到高阶场景，帮助读者彻底理解 Vue 3 响应式的底层运作逻辑。

## 背景与意义

### 为什么响应式如此重要？

现代前端框架的核心使命之一，就是"让数据变化自动反映到视图上"。在 jQuery 时代，开发者需要手动操作 DOM 来同步数据变化；而在 Vue/React 等现代框架中，这种同步被抽象成了响应式系统。

Vue 3 的响应式系统基于 Proxy 实现，相比于 Vue 2 的 `Object.defineProperty`，它在能力上有了质的飞跃——支持数组索引监听、属性动态增删、Map/Set 等原生集合类型。然而，真正让这套系统高效运转的核心引擎，是**依赖收集（track）**与**触发更新（trigger）**机制。

### 理解依赖收集的意义

很多开发者在使用 Vue 3 时，对 `ref`、`reactive`、`computed`、`watch` 等 API 已经驾轻就熟，但当遇到性能问题、内存泄漏、或需要编写高复杂度的自定义逻辑时，往往感到力不从心。原因就在于对底层机制的理解不够深入。

具体来说，理解依赖收集与触发机制可以帮你解决以下实际问题：

1. **性能优化**：知道何时会触发重新渲染，避免不必要的计算
2. **调试复杂问题**：当视图没有按预期更新时，能快速定位是收集错了还是触发失败了
3. **避免内存泄漏**：理解依赖清理机制，在组件销毁时正确释放资源
4. **编写自定义响应式逻辑**：能够基于 `effect` 和 `track/trigger` 构建自己的响应式工具
5. **深入源码阅读**：这是阅读 Vue 3 源码的基石，几乎全部响应式相关 API 都建立在这套机制之上

依赖收集机制本身是通用且优雅的：**"谁用到了我的数据，我就记住谁；当我的数据变化时，通知所有用到我的人"**。这个看似简单的模式，内部却涉及多个精巧的数据结构和算法设计。

## 概念与定义

在深入代码之前，我们先梳理清楚核心概念。

### 响应式对象（Reactive Object）

被 Vue 3 的 `reactive()` 或 `ref()` 包装后的对象。它们通过 Proxy 拦截了属性的读取（get）和设置（set）操作，从而在读取时收集依赖，在设置时触发更新。

### targetMap

Vue 3 全局维护的一个 `WeakMap`，用于存储所有响应式对象与其依赖之间的关系。它的结构如下：

```
targetMap (WeakMap)
 ├── target1 (Object) → depsMap (Map)
 │    ├── key1 (string) → deps (Set)
 │    │    ├── effectFn1
 │    │    └── effectFn2
 │    └── key2 (string) → deps (Set)
 │         └── effectFn3
 └── target2 (Object) → depsMap (Map)
      └── ...
```

- **targetMap**：外层 WeakMap，key 是原始响应式对象，value 是对应的 depsMap
- **depsMap**：Map 结构，key 是对象的属性名，value 是该属性对应的依赖集合
- **deps**：Set 结构，存储该属性上所有依赖的 effect 函数

使用 `WeakMap` 的巧妙之处在于：当响应式对象不再被引用时，可以自动被垃圾回收，对应的依赖关系也随之消失，无需手动清理。

### effect（副作用函数）

`effect` 是 Vue 3 响应式系统的核心函数。简单来说，它接收一个函数并执行，在这个过程中如果读取了响应式数据，就会自动建立依赖关系——将当前 effect 注册到被读取数据的依赖列表中。

当数据变化时，所有依赖它的 effect 函数都会被重新执行。

从类型上看，每个 effect 对应一个 `ReactiveEffect` 实例，它包含以下关键信息：

| 属性 | 类型 | 说明 |
|------|------|------|
| `fn` | Function | 用户传入的原始函数 |
| `deps` | Set[] | 该 effect 依赖的所有 Dep 集合 |
| `active` | boolean | 当前 effect 是否活跃（未停止） |
| `scheduler` | Function | 自定义调度器（computed/watch 使用） |

### activeEffect

一个全局变量（实际上是一个栈结构），指向**当前正在运行的 effect 函数**。当响应式对象的 getter 被触发时，通过 `activeEffect` 可以知道"谁正在读取这个值"，从而建立依赖关系。

### track（依赖收集）

当响应式对象的某个属性被读取时，`track` 函数会被调用。它的职责是：

1. 从 `targetMap` 中找到该对象对应的 `depsMap`
2. 从 `depsMap` 中找到该属性对应的 `deps`（依赖集合）
3. 将当前的 `activeEffect` 添加到 `deps` 中
4. 将 `deps` 也反向记录到 `activeEffect.deps` 中（用于后续清理）

### trigger（触发更新）

当响应式对象的某个属性被修改时，`trigger` 函数会被调用。它的职责是：

1. 从 `targetMap` 中找到该对象和属性对应的所有 effect
2. 取出所有依赖的 effect 并重新执行
3. 如果有 `scheduler`，则调用调度器决定执行时机

## 最小示例

让我们从一个最简单的例子开始，亲手感受一下依赖收集与触发更新的运作流程。

```javascript
// 引入 Vue 3 响应式系统
const { reactive, effect } = require('vue')

// 1. 创建一个响应式对象
const state = reactive({ count: 0, name: 'Vue' })

// 2. 创建一个 effect（副作用）
effect(() => {
  console.log('effect 执行:', state.count)
})
// 输出: effect 执行: 0

// 3. 修改响应式数据
state.count++
// 输出: effect 执行: 1

state.count = 10
// 输出: effect 执行: 10
```

这个例子虽然简单，但背后发生了完整的依赖收集和触发流程：

**首次执行 effect 时**：

1. 将当前 effect 设置到全局 `activeEffect`
2. 执行用户传入的函数 `() => console.log(state.count)`
3. 函数中读取了 `state.count`，触发 Proxy 的 get 拦截
4. get 拦截中调用 `track(state, 'count')`
5. `track` 将 `activeEffect` 存入 `state.count` 对应的依赖集合
6. 函数执行完毕，恢复 `activeEffect`

**修改 state.count 时**：

1. 触发 Proxy 的 set 拦截
2. set 拦截中调用 `trigger(state, 'count')`
3. `trigger` 从 `targetMap` 中取出 `state.count` 的所有依赖 effect
4. 依次执行这些 effect 函数
5. effect 执行时再次读取 `state.count`，重新建立依赖

再来看一个多个属性依赖的例子：

```javascript
const user = reactive({ name: 'Alice', age: 25, score: 90 })

// effect 1: 依赖 name 和 age
effect(() => {
  console.log(`用户信息: ${user.name}, ${user.age}岁`)
})

// effect 2: 依赖 score
effect(() => {
  if (user.score >= 60) {
    console.log(`${user.name} 及格了`)
  } else {
    console.log(`${user.name} 不及格`)
  }
})
```

这里 effect 2 同时依赖了 `user.name` 和 `user.score`。当 `user.name` 变化时，两个 effect 都会被触发。

## 核心知识点拆解

### 一、targetMap 数据结构深度解析

`targetMap` 是整个依赖收集系统的"中央仓库"。理解它的设计是理解一切的基础。

```typescript
// Vue 3 源码中的核心数据结构
type KeyToDepMap = Map<any, Dep>
type Dep = Set<ReactiveEffect>
type TargetMap = WeakMap<object, KeyToDepMap>
```

为什么设计成三层嵌套？

**第一层：WeakMap<object, KeyToDepMap>**

- 使用 `WeakMap` 而非普通 `Map` 是因为 key 是对象引用
- 当响应式对象被垃圾回收时，整个条目自动移除，避免内存泄漏
- 只存储对象的元数据，不产生额外的强引用

**第二层：Map<any, Dep>（depsMap）**

- 对象的每个属性对应一个依赖集合
- 使用 `Map` 是因为属性名可以是字符串或 Symbol
- `depsMap.size` 可以反映该对象被依赖的属性数量

**第三层：Set<ReactiveEffect>（dep）**

- 使用 `Set` 确保同一个 effect 不会被重复收集
- 天然的幂等性：多次 `track` 同一个 effect 只有一次生效

这种三层嵌套结构的精妙之处在于：

> 响应式对象 → 对象的属性 → 属性上的依赖

每一层都是在**缩小匹配范围**。当触发更新时，通过 `target` 和 `key` 可以 O(1) 时间复杂度定位到需要调度的所有 effect。

```javascript
// 手动演练 targetMap 的构建过程
const targetMap = new WeakMap()

function track(target, key) {
  let depsMap = targetMap.get(target)
  if (!depsMap) {
    depsMap = new Map()
    targetMap.set(target, depsMap)
  }
  
  let deps = depsMap.get(key)
  if (!deps) {
    deps = new Set()
    depsMap.set(key, deps)
  }
  
  // 将当前 effect 加入依赖
  deps.add(activeEffect)
  // 反向记录（用于清理）
  activeEffect.deps.push(deps)
}
```

### 二、track 依赖收集的实现细节

`track` 的完整实现比上面的简化版复杂得多，需要考虑各种边界情况：

```typescript
// Vue 3 源码 packages/reactivity/src/effect.ts
export function track(target: object, type: TrackOpTypes, key: unknown) {
  // 1. 如果当前没有活跃的 effect，直接返回
  if (!shouldTrack || activeEffect === undefined) {
    return
  }
  
  // 2. 查找或创建 depsMap
  let depsMap = targetMap.get(target)
  if (!depsMap) {
    targetMap.set(target, (depsMap = new Map()))
  }
  
  // 3. 查找或创建 dep (Set)
  let dep = depsMap.get(key)
  if (!dep) {
    depsMap.set(key, (dep = createDep()))
  }
  
  // 4. 收集依赖
  trackEffects(dep)
}

export function trackEffects(dep: Dep) {
  // 将当前 activeEffect 添加到 dep 集合
  dep.add(activeEffect!)
  
  // 反向记录：让 effect 知道自己被哪些 dep 引用
  // 这样在清理时可以找到对应的 dep 并移除自己
  activeEffect!.deps.push(dep)
}
```

关键设计要点：

- **`shouldTrack` 全局标志位**：控制是否允许收集依赖，在特定场景（如暂停追踪）下关闭
- **`activeEffect === undefined` 短路**：如果没有正在执行的 effect，track 没有任何意义，直接返回
- **惰性创建**：只在有依赖时创建 `depsMap` 和 `dep`，避免无谓的内存占用
- **`trackEffects` 抽离**：方便在 `track` 之外的其他场景复用（如手动触发依赖收集）

### 三、trigger 触发更新的实现细节

当响应式对象的属性被修改时，`trigger` 负责"叫醒"所有相关的 effect。

```typescript
// Vue 3 源码 packages/reactivity/src/effect.ts
export function trigger(
  target: object,
  type: TriggerOpTypes,
  key?: unknown,
  newValue?: unknown,
  oldValue?: unknown,
  oldTarget?: Map<unknown, unknown> | Set<unknown>
) {
  // 1. 获取 depsMap
  const depsMap = targetMap.get(target)
  if (!depsMap) {
    // 从未被 track 过的对象，无需触发
    return
  }
  
  // 2. 收集所有需要运行的 effects
  const effects = new Set<ReactiveEffect>()
  
  // 当 key 不为 undefined 时，获取该 key 的依赖
  if (key !== undefined) {
    addEffect(depsMap.get(key))
  }
  
  // 3. 处理特殊类型（新增属性、数组长度变化等）
  // 新增属性（TriggerOpTypes.ADD）
  // 需要触发所有依赖（for...in 迭代）
  if (type === TriggerOpTypes.ADD) {
    // 触发数组迭代器的依赖
    if (isArray(target)) {
      addEffect(depsMap.get('length'))
    }
    // 触发 ITERATE_KEY 的依赖（Map/Set/对象迭代）
    addEffect(depsMap.get(ITERATE_KEY))
  }
  
  // 4. 依次执行 effects
  const run = (effect: ReactiveEffect) => {
    if (effect.scheduler) {
      // 有调度器则调用调度器（computed/watch 场景）
      effect.scheduler()
    } else {
      // 直接执行 effect
      effect.run()
    }
  }
  
  effects.forEach(run)
}
```

核心观察：

- **`effects` 使用 Set**：防止同一个 effect 被重复执行（同一个 effect 可能因为不同 key 的变化被多次加入）
- **ITERATE_KEY**：Vue 3 内部使用一个 Symbol 类型的 `ITERATE_KEY` 来收集"所有 key 的遍历依赖"。当对象新增属性时，任何遍历该对象的 effect 都应该重新执行
- **调度器分支**：`scheduler` 的存在使得 `computed` 和 `watch` 可以延迟 effect 的执行，而非同步立刻运行

### 四、effect 函数的完整实现

`effect` 是整个系统的入口。它创建了一个 `ReactiveEffect` 实例并立即执行。

```typescript
// Vue 3 源码简化版
function effect<T = any>(
  fn: () => T,
  options?: ReactiveEffectOptions
): ReactiveEffectRunner {
  // 1. 创建 ReactiveEffect 实例
  const _effect = new ReactiveEffect(fn)
  
  // 2. 应用选项
  if (options) {
    extend(_effect, options)
    if (options.scope) {
      recordEffectScope(_effect, options.scope)
    }
  }
  
  // 3. 立即执行（触发依赖收集）
  if (!options || !options.lazy) {
    _effect.run()
  }
  
  // 4. 返回 runner（可再次手动调用的 effect）
  const runner = _effect.run.bind(_effect) as ReactiveEffectRunner
  runner.effect = _effect
  return runner
}
```

`ReactiveEffect` 类的核心实现：

```typescript
class ReactiveEffect {
  active = true      // 是否活跃
  deps: Dep[] = []   // 记录该 effect 依赖了哪些 dep
  parent: ReactiveEffect | undefined = undefined  // 嵌套 effect 的父级
  
  constructor(
    public fn: () => T,
    public scheduler: EffectScheduler | null = null
  ) {}
  
  run() {
    if (!this.active) {
      // 不活跃的 effect 直接执行但不收集依赖
      return this.fn()
    }
    
    // 1. 清理旧依赖（分支切换的关键）
    cleanupEffect(this)
    
    // 2. 保存当前 activeEffect 作为父级（嵌套处理）
    const parent = activeEffect
    activeEffect = this
    
    // 3. 执行用户函数
    const result = this.fn()
    
    // 4. 恢复上一个 activeEffect
    activeEffect = parent
    
    return result
  }
  
  stop() {
    // 停止 effect：清理依赖并标记为非活跃
    cleanupEffect(this)
    this.active = false
  }
}
```

### 五、activeEffect 全局变量与栈管理

`activeEffect` 是一个简单的全局变量，但它配合 `parent` 指针实际上构成了一个隐式的"调用栈"。

```javascript
// 全局变量
let activeEffect: ReactiveEffect | undefined
```

为什么需要 `parent`？

考虑嵌套 effect 的场景：

```javascript
effect(() => {
  // outer: activeEffect = effect1
  console.log(state.outer)
  
  effect(() => {
    // inner: activeEffect = effect2
    console.log(state.inner)
  })
  // 这里 activeEffect 应该恢复为 effect1
  
  // 但注意：此时读取 state.other 时，
  // activeEffect 应该还是 effect1，不是 effect2
  console.log(state.other)
})
```

在 Vue 3.2 之前，Vue 使用一个**栈数组** `effectStack` 来管理嵌套 effect：

```javascript
const effectStack: ReactiveEffect[] = []
```

每次 `run()` 时 push，结束后 pop。Vue 3.2+ 采用了更高效的方案——使用 `parent` 指针形成链表，因为大多数场景嵌套深度很浅（通常不超过 2-3 层），用链表比数组更节省内存和操作成本。

```javascript
// Vue 3.2+ 的优化方案（链表替代数组）
const parent = activeEffect
activeEffect = this
// ...执行函数...
activeEffect = parent
```

### 六、依赖清理机制

这是理解响应式系统健壮性的关键。当 effect 重新执行时，它需要先**清理所有旧的依赖关系**，然后再根据新的执行结果重新收集。

```javascript
function cleanupEffect(effect: ReactiveEffect) {
  const { deps } = effect
  if (deps.length) {
    // 遍历旧依赖，从每个 dep 集合中移除当前 effect
    for (let i = 0; i < deps.length; i++) {
      deps[i].delete(effect)
    }
    // 清空 deps 数组
    deps.length = 0
  }
}
```

为什么需要清理？来看一个分支切换的经典场景：

```html
<template>
  <div>{{ user.isVIP ? user.specialContent : user.normalContent }}</div>
</template>
```

```javascript
const user = reactive({
  isVIP: true,
  specialContent: 'VIP 专属内容',
  normalContent: '普通内容'
})

effect(() => {
  console.log(user.isVIP ? user.specialContent : user.normalContent)
})
```

**初始状态（isVIP = true）**：effect 依赖了 `user.isVIP` 和 `user.specialContent`

**切换到 isVIP = false**：effect 重新执行，因为条件变化，改为读取 `user.normalContent`

- **清理前**：effect 的 `deps` = [dep(isVIP), dep(specialContent)]
- **清理后**：effect 的 `deps` = []（从所有旧依赖中移除自己）
- **重新收集**：读取 `isVIP` → 加入 dep(isVIP)，读取 `normalContent` → 加入 dep(normalContent)

**如果不清理**，effect 仍然在 dep(specialContent) 中。当 `specialContent` 变化时，effect 被触发重新执行，但输出结果并没有变化（因为 isVIP 是 false），这就是**不必要的重新计算**。

### 七、嵌套 effect 处理

嵌套 effect 是一个非常重要的边界场景，computed、watch、组件渲染等都构建在嵌套之上。

```javascript
const state = reactive({ count: 0, multiplier: 2 })

// 外层 effect（模拟组件渲染）
effect(() => {
  console.log('外层 effect 开始')
  
  // 内层 effect（模拟 computed 的计算逻辑）
  const double = effect(() => {
    console.log('  内层 computed 计算:', state.count * state.multiplier)
    return state.count * state.multiplier
  }, { lazy: true })
  
  console.log('  使用 computed 值:', double())
  console.log('外层 effect 结束')
})
```

当 `state.count` 变化时：
1. `trigger` 被调用
2. 遍历依赖集合，同时包含外层和内层 effect
3. 先执行外层 effect → 清理旧依赖 → 运行时又创建了新的内层 effect
4. 内层 effect 执行清理 → 重新收集依赖

关键问题：**如何避免无限递归**？

```javascript
const state = reactive({ count: 0 })

// 这会死循环吗？
effect(() => {
  state.count++
})
```

答案是：**会**！执行 effect → `count++` 触发 set → trigger 重新执行 effect → `count++` 再次触发 → ...

Vue 3 通过以下机制处理：

```typescript
// trigger 中的保护逻辑
const effects = new Set<ReactiveEffect>()
// ... 收集依赖 ...

// 将当前正在运行的 effect 从待执行列表中剔除
effects.forEach(effect => {
  if (effect !== activeEffect) {
    run(effect)
  }
})
```

但更好的做法是使用 `watch` 或 `computed` 来处理派生状态：

```javascript
// 正确的做法：使用 computed 避免手动触发响应式循环
const double = computed(() => state.count * 2)
```

### 八、分支切换与依赖重新收集

分支切换是依赖清理的最典型应用场景。在 Vue 组件中，模板中常常有条件渲染：

```vue
<template>
  <div>
    <p v-if="tabs.active === 'profile'">{{ user.profile }}</p>
    <p v-else>{{ user.settings }}</p>
  </div>
</template>

<script setup>
const tabs = reactive({ active: 'profile' })
const user = reactive({
  profile: '个人资料',
  settings: '系统设置'
})
</script>
```

当 `tabs.active` 从 `'profile'` 切换到 `'settings'` 时：

1. 渲染 effect 被触发重新执行
2. 清理阶段：从 `user.profile` 的依赖列表中移除该 effect
3. 重新收集：`user.settings` 的依赖列表中加入该 effect
4. 后续 `user.profile` 变化时，不会重新渲染组件（节省性能）
5. 后续 `user.settings` 变化时，会重新渲染组件

这个过程可以用一张图表示：

```
    初始收集                切换后
  ┌─────────────┐       ┌─────────────┐
  │ tabs.active │       │ tabs.active │
  │              │       │              │
  │ user.profile │  →→  │ user.settings│
  └─────────────┘       └─────────────┘
        ↓                     ↓
    清理 user.profile    收集 user.settings
    的旧依赖              的新依赖
```

这个机制的巧妙之处在于：**依赖关系是动态的、实时的**。Vue 不需要像某些框架那样声明依赖的确定性列表，而是通过执行代码时的 get 操作自动发现依赖，实现了"按需收集、按需触发"。

### 案例二：自定义响应式工具

结合 `track` 和 `trigger`，可以构建更灵活的响应式工具：

```javascript
import { reactive, track, trigger, effect } from './reactivity'

/**
 * 一个带有过期时间的响应式缓存
 */
function cacheWithExpiry(fetchFn, ttl = 5000) {
  const state = reactive({
    data: null,
    timestamp: 0,
    loading: false,
    error: null
  })
  
  async function getData() {
    const now = Date.now()
    
    // 缓存未过期，直接返回
    if (state.data && (now - state.timestamp) < ttl) {
      return state.data
    }
    
    state.loading = true
    
    try {
      const result = await fetchFn()
      state.data = result
      state.timestamp = now
      state.loading = false
      state.error = null
      return result
    } catch (err) {
      state.error = err.message
      state.loading = false
      throw err
    }
  }
  
  function invalidate() {
    state.timestamp = 0
  }
  
  return { state, getData, invalidate }
}

// 使用
const cache = cacheWithExpiry(() => fetch('/api/user').then(r => r.json()))

effect(() => {
  // 自动读取响应式状态
  if (cache.state.loading) {
    console.log('加载中...')
  } else if (cache.state.error) {
    console.log('错误:', cache.state.error)
  } else if (cache.state.data) {
    console.log('用户数据:', cache.state.data)
  }
})
```

### 案例三：手动控制依赖收集

```javascript
import { effect, pauseTracking, enableTracking } from 'vue'

const data = reactive({ items: [] })
```


## 底层原理

### 源码级分析：Proxy + track/trigger 的整合

Vue 3 的 `reactive` 函数创建 Proxy 时，在 getter 中调用 `track`，在 setter 中调用 `trigger`：

```typescript
// packages/reactivity/src/reactive.ts 简化版
function createReactiveObject(
  target: Target,
  isReadonly: boolean,
  baseHandlers: ProxyHandler<any>
) {
  const proxy = new Proxy(target, baseHandlers)
  return proxy
}
```

`baseHandlers` 中的 getter 实现：

```typescript
// packages/reactivity/src/baseHandlers.ts
const get = createGetter()

function createGetter(isReadonly = false, shallow = false) {
  return function get(target: Target, key: string | symbol, receiver: object) {
    // 1. 获取原始值
    const res = Reflect.get(target, key, receiver)
    
    // 2. 如果不是只读的，进行依赖收集
    if (!isReadonly) {
      track(target, TrackOpTypes.GET, key)
    }
    
    // 3. 如果是对象，递归包装（惰性深度响应式）
    if (isObject(res)) {
      return isReadonly ? readonly(res) : reactive(res)
    }
    
    return res
  }
}
```

setter 实现：

```typescript
// packages/reactivity/src/baseHandlers.ts
const set = createSetter()

function createSetter(shallow = false) {
  return function set(
    target: Target,
    key: string | symbol,
    value: unknown,
    receiver: object
  ) {
    // 1. 获取旧值
    const oldValue = (target as any)[key]
    
    // 2. 判断是否有旧属性（用于确定操作类型）
    const hadKey = hasOwn(target, key)
    
    // 3. 设置新值
    const result = Reflect.set(target, key, value, receiver)
    
    // 4. 触发更新（跳过原型链上的属性）
    if (target === toRaw(receiver)) {
      if (!hadKey) {
        // 新增属性
        trigger(target, TriggerOpTypes.ADD, key, value)
      } else if (hasChanged(value, oldValue)) {
        // 修改已有属性
        trigger(target, TriggerOpTypes.SET, key, value, oldValue)
      }
    }
    
    return result
  }
}
```

注意：Vue 3 区分了 `TriggerOpTypes.SET`（修改）和 `TriggerOpTypes.ADD`（新增），因为对于数组来说，`push` 操作需要同时触发 `length` 属性的依赖。

### ref 的实现依赖 track/trigger

`ref` 是 Vue 3 最常用的响应式 API 之一。它的实现同样基于 `track` 和 `trigger`，只是包装形式不同。

```typescript
// packages/reactivity/src/ref.ts
class RefImpl<T> {
  private _value: T
  public dep?: Dep = undefined  // 依赖集合
  private _rawValue: T
  
  constructor(value: T, public readonly __v_isShallow: boolean) {
    this._rawValue = toRaw(value)
    this._value = toReactive(value) // 如果 value 是对象，包装为 reactive
  }
  
  get value() {
    // 读取时收集依赖
    trackRefValue(this)
    return this._value
  }
  
  set value(newVal) {
    newVal = toRaw(newVal)
    if (hasChanged(newVal, this._rawValue)) {
      this._rawValue = newVal
      this._value = toReactive(newVal)
      // 修改时触发更新
      triggerRefValue(this)
    }
  }
}
```

可以看到，`ref` 的 getter 和 setter 本质上是在对 `value` 这一固定的 key 做 `track` 和 `trigger`，而不是像 `reactive` 那样拦截任意属性。

### computed 的实现

`computed` 是另一个基于 effect 的高级 API，它的实现巧妙利用了 `lazy` 和 `scheduler`：

```typescript
// packages/reactivity/src/computed.ts
class ComputedRefImpl<T> {
  public dep?: Dep
  private _value!: T
  private _dirty = true  // 脏标记：是否需要重新计算
  
  public readonly effect: ReactiveEffect<T>
  
  constructor(getter: ComputedGetter<T>) {
    this.effect = new ReactiveEffect(getter, () => {
      // scheduler：当依赖变化时，标记为脏
      if (!this._dirty) {
        this._dirty = true
        // 并触发 computed 自身的依赖
        triggerRefValue(this)
      }
    })
  }
  
  get value() {
    // 收集依赖
    trackRefValue(this)
    
    // 如果是脏的，重新计算
    if (this._dirty) {
      this._dirty = false
      this._value = this.effect.run()!
    }
    
    return this._value
  }
}
```

`computed` 的核心机制是**惰性求值**：

1. `_dirty = true` 时，读取 `value` 才会重新计算（lazy）
2. 依赖变化时，通过 `scheduler` 设置 `_dirty = true`，但不立即重新计算
3. 下次读取 `value` 时，发现是脏的，重新计算并返回新值
4. 同时通过脏标志的 `trigger` 通知外部 effect 更新

这种设计避免了不必要的计算：如果一个 computed 的值在变化后没人读取，就不会执行计算函数。

### 完整的 effect 调度流程

现在，我们可以画出一条完整的数据变化链路：

```
state.count = 42
        │
        ▼
  Proxy set 拦截
        │
        ▼
  trigger(target, 'count')
        │
        ├──► 从 targetMap 获取 depsMap
        │         │
        │         ▼
        │   获取 deps = [effect1, effect2, ...]
        │
        └──► 遍历执行 effects
                  │
          ┌───────┴───────┐
          ▼               ▼
    有 scheduler?     无 scheduler?
          │               │
          ▼               ▼
    scheduler()       effect.run()
          │               │
          │               ├──► cleanupEffect()
          │               ├──► track 重新收集
          │               ├──► fn() 执行
          │               └──► 恢复 activeEffect
          │
    常见场景:          常见场景:
    - computed         - render effect
    - watch (async)    - watch (sync)
```

## 高频面试题解析

### 面试题 1：Vue 3 的响应式为什么比 Vue 2 快？

**考点**：响应式实现原理、Proxy 与 Object.defineProperty 的差异

**参考答案**：

1. **全面拦截**：Proxy 可以拦截数组索引和属性动态增删，无需像 Vue 2 那样用 `$set` / `$delete` 或重写数组方法
2. **惰性深度响应式**：Vue 3 在 getter 中递归创建 reactive 包装，只有在实际读取深层属性时才进行包装，减少了初始化开销
3. **依赖收集优化**：Vue 3 使用 `WeakMap + Map + Set` 结构，O(1) 级别的查询效率，且 `WeakMap` 自动 GC 减少了内存管理成本
4. **effect 改用链表替代数组**：Vue 3.2+ 使用 `parent` 指针替代 `effectStack` 数组，减少了内存分配

### 面试题 2：为什么 Vue 3 的响应式数据可以解构而 Vue 2 不行？

**考点**：Proxy 与 getter/setter 的差异

**参考答案**：

Vue 2 使用 `Object.defineProperty` 将属性转换为 getter/setter。当解构时：

```javascript
const { count } = this.data
// count 只是一个普通值，丢失了 getter 引用
count++ // 不会触发视图更新
```

Vue 3 使用 `Proxy` 对整个对象进行代理，解构后的变量虽然脱离了对原始对象的引用，但如果解构的是 `ref`：

```javascript
let { count } = useStore()
// 如果 count 是 ref，解构后仍然保留 .value 的 getter/setter
```

但需要注意：`reactive` 对象的解构仍然会丢失响应式，因为解构出的每个属性是一个固定值，不再是 Proxy 的一部分。推荐使用 `toRefs` 将 `reactive` 转换为普通对象（每个属性都是 ref）。

### 面试题 3：watch 和 effect 的区别是什么？

**考点**：懒执行、依赖追踪范围、scheduler 调度

**参考答案**：

| 特性 | `effect` | `watch` |
|------|----------|---------|
| 立即执行 | 是（默认） | 否（除非 flush: 'sync'） |
| 依赖范围 | 自动收集所有读取的响应式数据 | 显式指定要监听的源 |
| 新旧值 | 无 | 提供 `(newVal, oldVal)` |
| 调度 | 同步执行 | 可配置 `flush: 'pre'\|'post'\|'sync'` |
| 清理 | 无 | `onCleanup` 处理竞态 |

```javascript
// effect - 自动收集，立即执行
effect(() => {
  console.log(state.count) // 自动追踪
})

// watch - 显式指定，惰性执行
watch(
  () => state.count, // 显式指定源
  (newVal, oldVal) => {
    console.log(newVal, oldVal)
  }
)
```

底层来看，`watch` 内部创建了一个 `effect`，但配置了 `scheduler` 来控制执行时机，并记录了上次执行的值以提供新/旧值比较。

### 面试题 4：什么是副作用函数？Vue 中哪些地方使用了 effect？

**参考答案**：

副作用函数是指**执行过程中对外部环境产生可观察影响的函数**。在 Vue 3 中，以下场景都使用了 effect：

1. **组件渲染**：每个组件实例有一个 `render effect`，依赖变化时重新执行 render 函数
2. **computed**：使用 `lazy effect` + `scheduler` 实现惰性求值
3. **watch**：使用 `effect` + `scheduler` 实现异步执行
4. **watchEffect**：等于是 `effect` 加上 `flush: 'pre'` 的调度
5. **自定义 `useEffect`**：用户可以通过 `effect` API 构建自己的响应式逻辑

### 面试题 5：targetMap 为什么要用 WeakMap 而不是 Map？

**考量点**：内存管理、GC 机制

**参考答案**：

`WeakMap` 的 key 是**弱引用**，不会阻止垃圾回收。具体好处：

```javascript
let obj = { name: 'test' }
const vm = reactive(obj)

// 当 obj 不再被外部引用时：
obj = null

// 如果使用 WeakMap: vm 对应的条目可以被 GC 自动回收
// 如果使用 Map: 必须手动清除，否则内存泄漏
// targetMap.delete(obj) // 需要手动做这件事
```

在复杂的 SPA（单页应用）中，组件频繁创建和销毁。如果使用普通 `Map` 且忘记手动清理，未被 GC 的响应式对象和它们的依赖关系会持续占用内存。使用 `WeakMap` 让响应式对象的依赖关系随对象本身一同被回收，无需任何额外操作。

### 面试题 6：Vue 3 如何避免 effect 死循环？

**考点**：trigger 中的保护机制、`activeEffect` 比对

**参考答案**：

```javascript
// 会导致死循环的写法
effect(() => {
  state.count++ // 读取 → track → 修改 → trigger → 再次执行 → ...
})
```

Vue 3 在 `trigger` 中做了如下保护：

```typescript
// 执行 effect 时排除当前正在运行的 effect 本身
const run = (effect) => {
  if (effect !== activeEffect) {
    // 不执行当前活跃的 effect
    if (effect.scheduler) {
      effect.scheduler()
    } else {
      effect.run()
    }
  }
}
```

但更好的做法是从设计上避免这种模式。应该使用 `computed` 处理派生值：

```javascript
// 正确做法
const double = computed(() => state.count * 2)

// 而非
effect(() => {
  state.double = state.count * 2
})
```

## 总结与扩展

### 本文要点回顾

1. **targetMap（WeakMap）** 是 Vue 3 依赖收集的中心仓库，存储了所有响应式对象到其属性依赖的映射关系
2. **track（依赖收集）** 在响应式对象的 getter 中触发，将当前正在执行的 `activeEffect` 添加到对应属性的依赖集合中
3. **trigger（触发更新）** 在响应式对象的 setter 中触发，从 `targetMap` 中取出所有相关 effect 并执行
4. **activeEffect** 全局变量追踪当前正在执行的 effect，通过 `parent` 指针形成隐式栈来处理嵌套 effect
5. **依赖清理机制** 在 effect 重新执行前先清理旧的依赖关系，确保依赖关系始终反映最新状态
6. **分支切换** 是依赖清理的最典型应用，使 Vue 能够按需收集、动态跟踪依赖
7. **computed / watch** 等高级 API 都构建在 effect + scheduler 的组合之上

### 与其他知识点的关联

- **组件渲染机制**：每个 Vue 组件的 `render` 函数都是一个 effect，依赖了模板中读取的所有响应式数据
- **虚拟 DOM diff**：响应式变化触发 render effect 重新执行，产生新的 VNode，然后进行 diff 更新真实 DOM
- **异步更新队列**：Vue 3 使用微任务调度机制（Promise.then），将同一 tick 内的多次响应式更新合并为一次渲染
- **`onUnmounted` 生命周期**：组件卸载时会自动停止 render effect，触发 `stop()` 清理所有依赖
