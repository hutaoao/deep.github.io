## 一句话概括

React 多节点 Diff 的第二轮处理通过"剩余节点 Map + lastPlacedIndex 算法"来解决位置移动问题——将未处理的旧 Fiber 放入哈希表，遍历剩余新 Element 时在哈希表中查找匹配节点，并通过 lastPlacedIndex 判断是否需要移动 DOM 位置，未被复用的旧节点在第三轮中批量标记删除。

## 背景与意义

### 为什么需要第二轮？

第一轮遍历虽然高效地处理了"无位置变化"的常见场景，但现实中有很多操作会导致列表乱序：

- **拖拽排序**：用户把列表中第 5 项拖到第 2 项
- **前端筛选/排序**：点击表头排序，列表顺序全部变化
- **数据合并**：从两个不同的 API 接口合并数据导致元素位置大改
- **新增+移动混合**：在列表中插入几项的同时调整已有项的顺序

这些场景无法在第一轮处理，必须进入第二轮。

### 第二轮的核心挑战

第二轮的挑战在于：**在不知道新老列表关系的情况下，以最少的 DOM 移动操作完成更新**。

例如：
```
旧列表：A(0) → B(1) → C(2) → D(3) → E(4)
新列表：A   →   E   →   D   →   C   →   B
```

如果简单地"逐个删除重建"，需要 4 次删除 + 4 次创建。但通过位置复用 + 移动，可以降到：
- 复用全部 5 个 DOM 节点
- 最多 4 次移动（`insertBefore` 或 `appendChild`）

lastPlacedIndex 算法就是为了找到最少移动次数的策略。

## 概念与定义

### lastPlacedIndex（放置索引）

React 用一个简单但有效的算法判断旧节点是否需要移动：维护一个 `lastPlacedIndex` 变量，记录当前已处理的所有新节点中，对应旧节点在原链表中的最大索引位置。如果新节点对应的旧节点在原列表中的索引比 `lastPlacedIndex` 小，说明该节点需要被移动到前面。

```typescript
// lastPlacedIndex 算法的核心逻辑
let lastPlacedIndex = 0;

for (; newIdx < newChildren.length; newIdx++) {
  const newFiber = updateFromMap(existingChildren, returnFiber, newIdx, newChildren[newIdx], lanes);
  
  const oldIndex = newFiber.alternate?.index ?? Infinity;
  
  if (oldIndex < lastPlacedIndex) {
    // 旧节点在原列表中排在 lastPlacedIndex 对应的节点之前
    // 但在新列表中它排在后面 → 需要移动
    newFiber.flags |= Placement;
  } else {
    // 旧节点在原列表中的位置与顺序一致 → 无需移动
    lastPlacedIndex = oldIndex;
  }
}
```

### existingChildren Map

第一轮跳出后，将旧 fiber 链表中剩余的节点构建成一个 `Map<key, fiber>`：

```typescript
function mapRemainingChildren(
  returnFiber: Fiber,
  currentFirstChild: Fiber,
): Map<string | number, Fiber> {
  const existingChildren = new Map();
  
  let existingChild = currentFirstChild;
  while (existingChild !== null) {
    if (existingChild.key !== null) {
      existingChildren.set(existingChild.key, existingChild);
    } else {
      existingChildren.set(existingChild.index, existingChild);
    }
    existingChild = existingChild.sibling;
  }
  
  return existingChildren;
}
```

### updateFromMap

在 Map 中查找匹配的旧 fiber，然后执行复用或创建新 fiber：

```typescript
function updateFromMap(
  existingChildren: Map<string | number, Fiber>,
  returnFiber: Fiber,
  newIdx: number,
  newChild: any,
  lanes: Lanes,
): Fiber | null {
  // 处理文本节点等简单类型
  if (typeof newChild === 'string' || typeof newChild === 'number') {
    // 文本节点以 index 为 key
    const matchedFiber = existingChildren.get(newIdx) || null;
    return updateTextNode(returnFiber, matchedFiber, '' + newChild, lanes);
  }
  
  if (typeof newChild === 'object' && newChild !== null) {
    switch (newChild.$$typeof) {
      case REACT_ELEMENT_TYPE: {
        const key = newChild.key;
        let matchedFiber = null;
        
        if (key !== null) {
          // 以 key 查找
          matchedFiber = existingChildren.get(key);
        } else {
          // 没有 key 时以 index 查找
          matchedFiber = existingChildren.get(newIdx);
        }
        
        if (matchedFiber !== null) {
          // 找到匹配的旧 fiber
          return updateElement(returnFiber, matchedFiber, newChild, lanes);
        } else {
          // 没有匹配，创建新节点
          return createFiberFromElement(newChild, returnFiber.mode, lanes);
        }
      }
      // ... portal, lazy 等类似处理
    }
  }
  
  return null;
}
```

### Placement Flag

`Placement` 是 Fiber 节点上的一个 flags 标记，表示该节点在 commit 阶段需要执行插入或移动操作——本质上是 `parentNode.insertBefore(fiber.stateNode, beforeChild)`。

## 核心知识点拆解

### 1. lastPlacedIndex 算法的全过程

用一个完整的示例来理解：

```
旧链表（索引号即 index）：
A(0) → B(1) → C(2) → D(3) → E(4)

第一轮：A(0) 匹配 A → 复用，跳出
剩余旧节点：B(1) → C(2) → D(3) → E(4)
新节点剩余：E → D → C → B

构建 Map：
{ B: B(1), C: C(2), D: D(3), E: E(4) }

第二轮遍历：
lastPlacedIndex = 0
├─ newIdx=1: 处理 E(4)
│  ├─ Map中找到 E(4), oldIndex=4
│  ├─ oldIndex (4) >= lastPlacedIndex (0) → 无需移动
│  └─ lastPlacedIndex = 4
├─ newIdx=2: 处理 D(3)
│  ├─ Map中找到 D(3), oldIndex=3
│  ├─ oldIndex (3) < lastPlacedIndex (4) → 需要移动！
│  └─ newFiber.flags |= Placement
├─ newIdx=3: 处理 C(2)
│  ├─ Map中找到 C(2), oldIndex=2
│  ├─ oldIndex (2) < lastPlacedIndex (4) → 需要移动！
│  └─ newFiber.flags |= Placement
└─ newIdx=4: 处理 B(1)
   ├─ Map中找到 B(1), oldIndex=1
   ├─ oldIndex (1) < lastPlacedIndex (4) → 需要移动！
   └─ newFiber.flags |= Placement

第三轮：Map 中所有节点都已匹配，无删除

最终：复用全部 5 个 DOM 节点
      移动次数：3（D, C, B 各一次）
      DOM 操作总数：3
```

### 2. 最优性的证明

为什么 lastPlacedIndex 算法能保证"最少的 DOM 操作"？

这个算法等价于**寻找新旧列表之间的最长递增子序列（LIS）**。在旧的索引序列中，无需移动的节点就是 LIS 的成员。

```
旧索引序列（按新列表顺序）：[4, 3, 2, 1]
最长递增子序列（LIS）：[4]（长度 1）
                 （或者 [3] 但长度为 1）
无需移动的数量 = LIS 长度 = 1
需移动的数量 = 总节点数 - LIS 长度 = 4 - 1 = 3

这与算法结果一致：只有 E(4) 不需要移动。
```

如果 React 使用真正的 LIS 算法（如 Vue 3），相同场景会得到相同的 3 次移动。但 React 的 lastPlacedIndex 是一个**贪心算法**，在大多数情况下接近最优，且实现更简单。

**需要注意**：lastPlacedIndex 并不总是最优。当有大量插入和删除时，它可能比理论最小移动次数多一次操作。但相比复杂性增加的收益微乎其微。

### 3. 无 key 列表的处理

当列表没有 key 时，React 使用索引（index）作为默认 key。这会导致一些微妙的问题：

```jsx
// 没有 key 的列表
{items.map((item, index) => (
  <ListItem data={item} />
))}
```

在 `updateFromMap` 中，无 key 节点会以索引查找 Map：

```typescript
// 无 key 的查找逻辑
if (key !== null) {
  matchedFiber = existingChildren.get(key);  // 使用指定 key
} else {
  matchedFiber = existingChildren.get(newIdx);  // 使用新列表中的索引
}
```

这意味着当列表长度不变时，无 key 的列表在 Map 中用索引查找时可能找到错误的节点：

```
旧列表：A(0) → B(1) → C(2)
新列表：B(0) → C(1) → A(2)

无 key 时的 Map：{ 0: A, 1: B, 2: C }

第二轮：
- newIdx=0: 查找 key=0 → 找到 A → 复用 A → 但实际应该是 B！
- newIdx=1: 查找 key=1 → 找到 B → 复用 B → 但实际应该是 C！
- newIdx=2: 查找 key=2 → 找到 C → 复用 C → 但实际应该是 A！

结果：内容错位
```

这就是为什么 React 官方**强烈建议在动态列表中指定 key**。

```jsx
function SecondPassDemo() {
  // 预设几种列表变化场景
  const scenarios = [
    {

      label: '反转顺序',

function getFirstPassCount(scenario, newItems) {
  // 计算第一轮匹配了多少个
  const oldItems = scenario.oldItems;
  let count = 0;
  for (let i = 0; i < Math.min(oldItems.length, newItems.length); i++) {
    if (oldItems[i] === newItems[i]) {
      count++;
    } else {
      break;
    }
  }
  return count;
}

function getMoveInfo(scenario, newItems, item, newIndex) {
  const oldIndex = scenario.oldItems.indexOf(item);
  if (oldIndex === -1) return '新增节点 ✨';
  if (oldIndex === newIndex) return '位置不变 ✅';
  if (oldIndex < newIndex) return `右移 (${oldIndex} → ${newIndex}) → 标记 Placement`;
  return `左移 (${oldIndex} → ${newIndex}) → 标记 Placement`;
}

function SecondPassAnalysis({ scenario, showNew }) {
  if (!showNew) {
    return (
      <div style={{ marginTop: 12, color: '#999', fontSize: 13 }}>
        切换到新列表以查看 Diff 分析
      </div>
    );
  }

  const oldItems = scenario.oldItems;
  const newItems = scenario.newItems;

  const firstPassCount = getFirstPassCount(scenario, newItems);
  const remainingOld = oldItems.slice(firstPassCount);
  const remainingNew = newItems.slice(firstPassCount);

  // 模拟 lastPlacedIndex 算法
  const moveAnalysis = [];
  let lastPlacedIndex = 0;

  for (let i = 0; i < remainingNew.length; i++) {
    const item = remainingNew[i];
    const oldIndex = remainingOld.indexOf(item);
    const originalIndex = oldItems.indexOf(item);
    
    if (oldIndex === -1) {
      moveAnalysis.push({ item, oldIndex: -1, lastPlacedIndex, needMove: false, isNew: true });
    } else {
      const needMove = originalIndex < lastPlacedIndex;
      if (needMove) {
        moveAnalysis.push({ item, oldIndex: originalIndex, lastPlacedIndex, needMove: true });
      } else {
        lastPlacedIndex = originalIndex;
        moveAnalysis.push({ item, oldIndex: originalIndex, lastPlacedIndex, needMove: false });
      }
    }
  }

  return (
    <div style={{ marginTop: 16 }}>
      <div style={{
        background: '#1e1e1e',
        color: '#d4d4d4',
        padding: 12,
        borderRadius: 8,
        fontFamily: 'monospace',
        fontSize: 12,
        lineHeight: 1.8,
      }}>
        <div># Diff 分析报告</div>
        <div># 第一轮匹配：{firstPassCount} 个</div>
        <div># 进入第二轮：剩余 {remainingOld.length} 个旧节点 → {remainingNew.length} 个新节点</div>
        <br/>
        <div># 第二轮 lastPlacedIndex 计算过程：</div>
        {moveAnalysis.map((m, i) => (
          <div key={i} style={{ marginLeft: 12 }}>
            {m.isNew
              ? `newIdx=${i}: 处理 "${m.item}" → ⭐ 新增节点`
              : m.needMove
                ? `newIdx=${i}: 处理 "${m.item}" oldIndex=${m.oldIndex} < lastPlacedIndex=${m.lastPlacedIndex} → 🔄 需要移动`
                : `newIdx=${i}: 处理 "${m.item}" oldIndex=${m.oldIndex} >= lastPlacedIndex=${m.lastPlacedIndex} → ✅ 无需移动`}
          </div>
        ))}
        <br/>
        <div># 第三轮：删除旧节点</div>
        {(() => {
          const matchedItems = new Set(moveAnalysis.filter(m => !m.isNew).map(m => m.item));
          const deleted = remainingOld.filter(item => !matchedItems.has(item));
          return deleted.length === 0
            ? '  (无待删除节点)'
            : deleted.map(item => `  ❌ 删除 "${item}"`).join('\n');
        })()}
      </div>
    </div>
  );
}

export default SecondPassDemo;
```

```jsx
// 模拟 React 的 Diff 分析
function analyzeDiff(oldItems, newItems, addLog) {
  // 计算第一轮匹配
  let firstPassIdx = 0;
  while (firstPassIdx < Math.min(oldItems.length, newItems.length)) {
    if (oldItems[firstPassIdx].id === newItems[firstPassIdx].id) {
      firstPassIdx++;
    } else {
      break;
    }
  }

  addLog(`第一轮匹配: ${firstPassIdx} 个`);
  const remainingNew = newItems.slice(firstPassIdx);

  if (firstPassIdx === oldItems.length) {
    addLog(`新列表有 ${remainingNew.length} 个新增项`);
    return;
  }

  if (firstPassIdx === newItems.length) {
    addLog(`旧列表有 ${oldItems.length - firstPassIdx} 个待删除`);
    return;
  }

  // 第二轮：lastPlacedIndex
  const remainingOld = oldItems.slice(firstPassIdx);
  const oldMap = new Map(remainingOld.map((item, i) => [item.id, i]));

  let lastPlacedIndex = 0;
  let moveCount = 0;
  let newCount = 0;

  addLog(`剩余的旧节点: [${remainingOld.map(i => i.id).join(', ')}]`);
  addLog(`剩余的新节点: [${remainingNew.map(i => i.id).join(', ')}]`);

  for (const newItem of remainingNew) {
    const oldIdx = oldMap.get(newItem.id);
    if (oldIdx === undefined) {
      newCount++;
      addLog(`  "${newItem.text}" → 新增`);
    } else {
      const oldOriginalIndex = firstPassIdx + oldIdx;
      if (oldOriginalIndex < lastPlacedIndex) {
        moveCount++;
        addLog(`  "${newItem.text}" → oldIdx=${oldOriginalIndex} < last=${lastPlacedIndex} → 需移动`);
      } else {
        addLog(`  "${newItem.text}" → oldIdx=${oldOriginalIndex} >= last=${lastPlacedIndex} → 不变`);
        lastPlacedIndex = oldOriginalIndex;
      }
    }
  }

  // 第三轮：删除
  const newIds = new Set(remainingNew.map(i => i.id));
  const deletedItems = remainingOld.filter(item => !newIds.has(item.id));
  if (deletedItems.length > 0) {
    addLog(`待删除: [${deletedItems.map(i => i.id).join(', ')}]`);
  } else {
    addLog(`无待删除项`);
  }
}

export default DragSortBoard;
```

### 2. updateFromMap 的完整源码

```typescript
function updateFromMap(
  existingChildren: Map<string | number, Fiber>,
  returnFiber: Fiber,
  newIdx: number,
  newChild: any,
  lanes: Lanes,
): Fiber | null {
  if (typeof newChild === 'string' || typeof newChild === 'number') {
    // 文本节点
    const matchedFiber = existingChildren.get(newIdx) || null;
    return updateTextNode(returnFiber, matchedFiber, '' + newChild, lanes);
  }

  if (typeof newChild === 'object' && newChild !== null) {
    switch (newChild.$$typeof) {
      case REACT_ELEMENT_TYPE: {
        const key = newChild.key;
        let matchedFiber = null;

        if (key !== null) {
          // 有 key → 用 key 查找
          matchedFiber = existingChildren.get(key);
        } else {
          // 无 key → 用索引查找
          matchedFiber = existingChildren.get(newIdx);
        }

        if (matchedFiber !== null) {
          // 找到匹配的旧 fiber
          if (matchedFiber.elementType === newChild.type) {
            // type 也匹配 → 复用
            const newFiber = updateElement(returnFiber, matchedFiber, newChild, lanes);
            // 重要：从 Map 中移除，说明已找到匹配
            // 后续第三轮不会删除它
            existingChildren.delete(
              key !== null ? key : newIdx
            );
            return newFiber;
          } else {
            // key 匹配但 type 不匹配
            // 标记旧 fiber 删除，创建新的
            deleteChild(returnFiber, matchedFiber);
            existingChildren.delete(
              key !== null ? key : newIdx
            );
          }
        }

        // 没找到匹配 → 创建新节点
        const created = createFiberFromElement(newChild, returnFiber.mode, lanes);
        created.ref = newChild.ref;
        created.return = returnFiber;
        return created;
      }

      case REACT_PORTAL_TYPE: {
        // 类似 REACT_ELEMENT_TYPE 的处理
        // ...
        break;
      }

      case REACT_LAZY_TYPE: {
        // Lazy 组件的解析
        // ...
        break;
      }
    }
  }

  // 处理 null/undefined/false（条件渲染为 false 的情况）
  if (newChild === null || newChild === false || newChild === true) {
    return null;
  }

  // 处理数组（Fragment 子节点嵌套）
  if (isArray(newChild)) {
    return updateFragmentMap(
      existingChildren,
      returnFiber,
      newIdx,
      newChild,
      lanes,
    );
  }

  throwOnInvalidObjectType(returnFiber, newChild);
  return null;
}
```

## 高频面试题解析

### Q1：React 如何判断一个旧节点需要被移动？lastPlacedIndex 算法的原理是什么？

**参考答案：**

lastPlacedIndex 是 React 多节点 Diff 中判断节点是否需要 DOM 移动的核心算法。

**原理**：维护一个变量 `lastPlacedIndex`，初始值为 0。在第二轮遍历中，对于每个新节点：

1. 在 Map 中找到对应的旧节点，获取它在**原始列表**中的索引 `oldIndex`
2. 如果 `oldIndex >= lastPlacedIndex`：说明该节点在旧列表中的位置**不小于**之前已处理节点在旧列表中的位置，顺序一致 → 无需移动，更新 `lastPlacedIndex = oldIndex`
3. 如果 `oldIndex < lastPlacedIndex`：说明该节点在旧列表中的位置**小于**之前已处理节点，顺序错位 → 需要移动

**为什么能保证最小移动？**
这个算法的本质是找出旧节点索引按新顺序排列后的"最长递增子序列"（LIS）。递增子序列中的元素不需要移动，其余元素需要移动到正确位置。React 的贪心版本虽然不是理论上最优的 LIS（Vue 3 实现了精确的 LIS），但在常见场景中已足够接近最优。

### Q2：为什么 React 在第二轮中使用 Map 查找而不是再次线性比较？

**参考答案：**

当第一轮跳出时，新旧节点的"位置对应关系"已经不再成立。剩余的旧节点（B→C→D→E）和剩余的新节点（E→D→C→B）之间的关系是**乱序**的。

如果保持线性遍历，需要 O(n²) 的时间复杂度来找出匹配关系——每个新节点都需要遍历所有剩余的旧节点。使用 Map 可以将查找降到 O(1)，让第二轮的复杂度为 O(n)：

- **Map 构建**：O(k)，k 为剩余旧节点数
- **第二轮遍历**：O(m)，m 为剩余新节点数
- **第三轮清理**：O(k - matched)，被匹配的节点从 Map 移除，剩余的直接删除

整体时间复杂度为 O(n)，与列表长度线性相关。

### Q3：什么情况下多节点 Diff 会失败或产生错误的 DOM 状态？

**参考答案：**

多节点 Diff 在以下情况下可能产生不符合预期的结果（主要集中在**无 key 或错误 key**时）：

1. **无 key 的列表排序变化**：
   ```jsx
   // 旧：['A', 'B', 'C'] → 新：['B', 'C', 'A']
   // 无 key 时，React 用索引查找，导致 A、B、C 全部复用错误
   ```

2. **相同的非唯一 key**：
   ```jsx
   // key 不唯一时，Map 中后面的值会覆盖前面的
   // 导致某些节点被错误地复用，某些节点被错误地删除
   ```

3. **key 类型混乱**（混合了字符串和数字 key）：
   ```jsx
   // Map 中 '1' 和 1 是不同的 key，可能导致查找失败
   ```

4. **Fragment 的 key 问题**：React 没有强制 `<React.Fragment>` 必须有 key，但 Fragment 内部的 key 传播逻辑可能带来意外行为

React 在开发模式下会通过 `console.error` 提示这些错误（如 Key Warning），但在生产环境下它们只是静默地产生错误的 DOM 结果。**所以始终为动态列表中的元素设置稳定、唯一的 key 是最佳实践**。

## 总结与扩展

### 总结

多节点 Diff 的第二轮处理是 React 面对"列表变化"的核心优化器：

- **Map 查找**：以 O(1) 时间完成剩余节点的匹配
- **lastPlacedIndex**：以 O(n) 时间判断哪些节点需要移动
- **Placement flag**：将移动操作延迟到 commit 阶段执行
- **第三轮清理**：删除所有未被复用的旧节点

### 扩展：React Diff 与 Vue 3 Diff 的对比

| 方面 | React | Vue 3 |
|-----|-------|-------|
| 第一轮策略 | 单向扫描（从左到右） | 双端扫描（从两端向中间） |
| 第二轮策略 | Map + lastPlacedIndex | 最长递增子序列（LIS） |
| 移动判断 | 贪心（非精确 LIS） | 精确 LIS |
| 数据结构 | 单向链表（Fiber） | 数组 |
| key 强制要求 | 警告但不强制 | 警告但不强制 |

Vue 3 的精确 LIS 算法在某些极端场景（如大量元素重排）中可能产生比 React 更少的 DOM 操作（约少 1-2 次）。但在日常使用中，两者的 Diff 性能差异可以忽略不计，真正的性能瓶颈通常在组件渲染而非 DOM 操作本身。
