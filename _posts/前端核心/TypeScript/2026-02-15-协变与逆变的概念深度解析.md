> **一句话概括：** 协变（Covariance）与逆变（Contravariance）是类型系统中描述"父子类型关系如何在复合类型中传递"的核心规则，掌握它们才能真正理解 TypeScript 的类型安全边界。

## 背景与意义

你写了一段 TypeScript 代码，编译通过了，运行时却炸了？或者反过来——你明明觉得某个赋值是合理的，TS 编译器却拦住了你？

这两种场景的背后，大概率站着协变与逆变这对概念。

在日常业务开发中，协变与逆变直接影响着以下场景：

- **函数类型兼容性**：回调函数的参数声明是否必须完全匹配？
- **泛型类型安全**：`Array<Dog>` 能否赋值给 `Array<Animal>`？
- **Vue/React 事件处理**：事件处理函数签名的类型推断。
- **Redux 等状态管理**：Action 分发器的类型安全。

很多开发者学完基础类型后卡在高级类型理解上，根本原因就是没有建立协变/逆变的心智模型。一旦理解了这对概念，整个类型系统的设计意图就豁然开朗了。

## 概念与定义

### 先理解赋值兼容性

我们先建立最基础的概念：**类型兼容性**。

```typescript
// 定义类型层级
class Animal {
  name: string = '';
  eat() {}
}

class Dog extends Animal {
  breed: string = '';
  bark() {}
}

// 子类型可以赋值给父类型（安全）
const a: Animal = new Dog(); // OK - Dog is a subtype of Animal
```

这是最直观的规则：如果 `Dog` 是 `Animal` 的子类型，那么任何需要 `Animal` 的地方，我们都可以放心地给一个 `Dog`。

但在复合类型中，事情就复杂了。

### 什么是变型（Variance）

"变型"描述的是：**当类型构造器应用到泛型参数上时，子类型关系的传递方向**。

四种变型规则：

| 变型类型 | 方向 | 直观含义 |
|----------|------|----------|
| **不变（Invariant）** | 必须完全相等 | `T<A>` 和 `T<B>` 无关 |
| **协变（Covariant）** | 跟随原方向 | `T<A>` 是 `T<B>` 的子类型，如果 A 是 B 的子类型 |
| **逆变（Contravariant）** | 反转方向 | `T<A>` 是 `T<B>` 的子类型，如果 B 是 A 的子类型 |
| **双变（Bivariant）** | 两个方向都行 | 双向兼容，实际上不安全 |

### 协变（Covariance）

**定义**：如果 `A` 是 `B` 的子类型，那么 `T<A>` 也是 `T<B>` 的子类型。子类型关系沿**相同方向**传递。

```typescript
// 协变示例：数组（TypeScript 中数组是协变的）
const dogs: Dog[] = [new Dog(), new Dog()];
const animals: Animal[] = dogs; // OK - 协变
```

### 逆变（Contravariance）

**定义**：如果 `A` 是 `B` 的子类型，那么 `T<B>` 是 `T<A>` 的子类型。子类型关系沿**相反方向**传递。

```typescript
// 逆变示例：函数参数
type Handler<T> = (arg: T) => void;

const animalHandler: Handler<Animal> = (a: Animal) => a.eat();
const dogHandler: Handler<Dog> = animalHandler; // 不安全？实际上 TS 默认行为在这里...
```

这里的关键问题是：当参数类型是 `Dog` 的回调，能不能用参数类型是 `Animal` 的回调替代？

答案要从调用方视角来理解。我们会在"核心知识点拆解"中详细展开。

## 最小示例

```typescript
class Animal {
  name = 'animal';
}
class Dog extends Animal {
  breed = 'golden';
}

// ---------- 协变演示 ----------
interface Producer<T> {
  produce(): T;
}

const dogProducer: Producer<Dog> = { produce: () => new Dog() };
const animalProducer: Producer<Animal> = dogProducer; // OK - 协变
// 返回值位置：Dog → Animal 是安全的

// ---------- 逆变演示 ----------
interface Consumer<T> {
  consume(value: T): void;
}

const animalConsumer: Consumer<Animal> = {
  consume: (a: Animal) => console.log(a.name),
};
const dogConsumer: Consumer<Dog> = animalConsumer; // OK - 逆变
// 参数位置：Animal → Dog 是安全的（可以处理更具体的 Dog 参数）

// ---------- TypeScript 严格模式的函数兼容性 ----------
type FnAnimal = (x: Animal) => void;
type FnDog = (x: Dog) => void;

let fAnimal: FnAnimal = (a: Animal) => a.eat();
let fDog: FnDog = fAnimal; // strict 下 OK，因为参数逆变
```

## 核心知识点拆解

### 1. 为什么函数参数必须逆变？

假设函数类型 `(x: Dog) => void` 对参数类型是协变的：

```typescript
type Handler = (x: Dog) => void;

const badHandler: Handler = (x: Animal) => {
  x.eat(); // 安全，Animal 有 eat
  // 但我们不能在这里调用 x.bark()
  // 因为 x 的类型是 Animal，不是 Dog
};
```

从**调用方**角度看：

```typescript
function doSomething(callback: (dog: Dog) => void) {
  const d = new Dog();
  callback(d); // 这里传入的是 Dog
}
```

- 如果 callback 实际是 `(a: Animal) => void` → 安全！Dog 拥有 Animal 所有属性
- 如果 callback 实际是 `(d: Dog) => void` → 也安全
- 如果 callback 实际是 `(p: Puppy) => void`（Puppy 是 Dog 的子类型）→ **不安全**！因为 callback 可能访问 Puppy 独有的属性

所以：**函数参数接受更宽泛的类型（父类型），调用时传入更具体的类型（子类型），这样最安全**。这就是逆变——参数类型向父类型方向兼容。

### 2. TypeScript 的默认行为：双变

从 TypeScript 2.6 之前，函数参数默认是**双变（Bivariant）**的：

```typescript
// 在没有 strictFunctionTypes 的情况下
let f1: (x: Dog) => void = (x: Animal) => {};  // OK - 逆变
let f2: (x: Animal) => void = (x: Dog) => {};  // 也 OK - 双变允许了协变
```

为什么 TS 允许双变？因为 JavaScript 中回调非常常见，严格逆变可能导致大量类型错误。例如：

```typescript
// DOM 事件处理常见模式
const buttons = document.querySelectorAll('button');
buttons.forEach((btn) => btn.click());
// forEach 回调签名为 (value: T, index: number, array: T[]) => void
// 如果严格逆变，很多简化写法会报错
```

**`--strictFunctionTypes` 标志启用后**：方法参数仍然双变（为了兼容性），但**函数参数变为逆变**。

```typescript
interface Comparer<T> {
  compare(a: T, b: T): number;
  // 方法声明：参数仍然双变
}

type ComparerFn<T> = (a: T, b: T) => number;
// 函数类型声明：参数严格逆变
```

### 3. 返回值位置：协变

```typescript
interface Factory<T> {
  make(): T;
}

const dogFactory: Factory<Dog> = { make: () => new Dog() };
const animalFactory: Factory<Animal> = dogFactory; // 安全！

// 但反过来就不行
const anotherFactory: Factory<Dog> = animalFactory; // 类型错误！
// 因为 make() 实际返回 Animal，但调用者期待 Dog
```

**返回值是"生产者"**。如果函数说它返回 `Animal`，调用方只能使用 Animal 的成员。如果实际返回的是 `Dog`，那么额外的成员也不会引起问题。但如果函数说返回 `Dog` 却实际只返回了 `Animal`，调用方尝试使用 Dog 特有的 `breed` 时就会崩溃。

### 4. 不变（Invariant）的应用

可变容器必须是不变的：

```typescript
// mutable 容器需要不变性
interface Box<T> {
  value: T;
}

const dogBox: Box<Dog> = { value: new Dog() };
const animalBox: Box<Animal> = dogBox; // 如果允许：
animalBox.value = new Cat(); // Cat 也是 Animal... 但 dogBox.value 现在是 Cat 了！
```

所以 TypeScript 中，**可写属性**和**可变容器**天然需要不变性保证。这也是为什么 `Array<T>` 虽然是协变的但实际有安全隐患——这是 TS 为了兼容 JS 生态做的权衡。

## 实战案例

### 案例1：事件总线的类型安全

```typescript
type EventHandler<E> = (event: E) => void;

interface EventBus {
  on<E>(eventType: string, handler: EventHandler<E>): void;
  emit<E>(eventType: string, event: E): void;
}

// 在复杂的 UI 框架中应用
interface UIEvent {
  timestamp: number;
  target: HTMLElement;
}

interface ClickEvent extends UIEvent {
  x: number;
  y: number;
}

const eventBus: EventBus = createEventBus();

// 注册一个处理 UIEvent 的处理器
eventBus.on('click', (e: UIEvent) => {
  console.log(`Event at ${e.timestamp}`);
});

// 实际 emit ClickEvent（更具体的事件）
eventBus.emit('click', {
  timestamp: Date.now(),
  target: document.body,
  x: 100,
  y: 200,
});
// 逆变保证了：可以传入更具体的 ClickEvent 给期望 UIEvent 的处理器
```

### 案例2：类型安全的 Redux Reducer

```typescript
interface Action {
  type: string;
}

interface IncrementAction extends Action {
  type: 'INCREMENT';
  payload: number;
}

interface DecrementAction extends Action {
  type: 'DECREMENT';
  payload: number;
}

type CounterAction = IncrementAction | DecrementAction;

// Reducer 需要处理所有 Action 类型
type Reducer<S, A extends Action> = (state: S, action: A) => S;

// 一个只处理 IncrementAction 的 reducer
// 在逆变规则下可以被用在期望 CounterAction 的位置吗？
const incrementReducer: Reducer<number, IncrementAction> = (state, action) =>
  state + action.payload;

// 在 TypeScript 中，如果 strictFunctionTypes 开启：
// incrementReducer 不能直接赋值给 Reducer<number, CounterAction>
// 因为参数位置逆变，而 IncrementAction 是 CounterAction 的子类型
// 子类型逆变意味着：需要更具体参数的反而不兼容更宽泛的函数签名
// 所以 Redux 通常用联合类型 + switch-case 来保证类型安全
```

### 案例3：泛型工具类型的变型行为

```typescript
// 深入理解 Pick、Record、Readonly 等工具类型的变型

// Pick 是协变的（提取属性，只读操作）
type PickedDog = Pick<Dog, 'name'>; // { name: string }
// Pick<Dog, K> 是 Pick<Animal, K> 的子类型？是的，因为 Pig 是协变的

// Record 是不变的
// Record<K, Dog> 和 Record<K, Animal> 之间无子类型关系
// 因为 Record 可以写入

// Readonly 是协变的（只读不可写入）
const readonlyDogs: ReadonlyArray<Dog> = [new Dog()];
const readonlyAnimals: ReadonlyArray<Animal> = readonlyDogs; // OK
```

## 底层原理

### 从代数类型系统看变型

在代数类型理论中，泛型类型构造器的变型由其**出现位置**决定：

```
T<A> = A 出现在返回值位置 → 协变（"产出"）
T<A> = A 出现在参数位置   → 逆变（"消耗"）
T<A> = A 出现在两个位置   → 不变
```

用更形式化的语言（借用函数式编程的记号）：

```
如果 A <: B（A 是 B 的子类型），则：
  |- covariant(T<A>) <: covariant(T<B>)    // 协变：方向相同
  |- contravariant(T<B>) <: contravariant(T<A>)  // 逆变：方向相反
```

### TypeScript 内部是如何处理的

TypeScript 编译器在类型检查时，会递归计算每个位置的**变型深度（Variance Depth）**：

```
type DeepReadonly<T> = {
  readonly [K in keyof T]: DeepReadonly<T[K]>;
};

// 对于嵌套类型，TS 会追踪每一层的变型标记
declare const a: DeepReadonly<Animal>;
declare const d: DeepReadonly<Dog>;
a = d; // OK - 因为 DeepReadonly 中所有位置都是协变的
```

### 何时 TypeScript 选择了不安全

TypeScript 在某些场景牺牲了类型安全以换取实用性：

1. **数组的协变性**：`Array<Dog>` 可赋值给 `Array<Animal>`，但这在 push 操作时"不安全"
2. **方法的双变性**：`interface` 中的方法声明参数是双变的（历史原因）
3. **`strictNullChecks` 关闭时的 null 处理**：各种隐式兼容

这些"不安全"的决策反映了 TypeScript 的设计哲学：**实用性优先于纯粹性**。

## 高频面试题解析

### Q1: 什么是协变和逆变？请用代码举例说明。

**解析**：这道题考察对概念的基本理解。关键在于能清晰地区分两个方向。

```typescript
// 协变：子类型关系沿相同方向传递
// Producer<T> 中的 T 在返回值位置，是协变的
interface Producer<T> {
  get(): T;
}

// 逆变：子类型关系沿相反方向传递
// Consumer<T> 中的 T 在参数位置，是逆变的
interface Consumer<T> {
  set(value: T): void;
}

class Vehicle {}
class Car extends Vehicle {}

const carGetter: Producer<Car> = { get: () => new Car() };
const vehicleGetter: Producer<Vehicle> = carGetter; // 协变 OK

const vehicleSetter: Consumer<Vehicle> = { set: (v: Vehicle) => {} };
const carSetter: Consumer<Car> = vehicleSetter; // 逆变 OK
```

### Q2: TypeScript 为什么对函数参数默认使用双变？

**解析**：这道题考察对 TS 设计哲学的深入理解。

**核心原因**：TypeScript 最优先的目标是渐进类型（Gradual Typing），让 JS 项目平滑迁移。如果默认严格逆变：

1. 大量现有的 JS 回调模式（如 `forEach` 中省略参数）会报错
2. 事件系统的常见用法（如 EventEmitter）需要大量显式类型标注
3. 第三方库的声明文件需要大规模重写

从 TS 2.6 以后，`--strictFunctionTypes` 对函数类型启用严格逆变，但接口方法仍保持双变——这是向后兼容的精巧平衡。

### Q3: 函数类型兼容性中，参数和返回值的检查规则有何不同？

**解析**：考察对"返回值协变、参数逆变"的理解深度。

```typescript
type Fn = (x: Dog) => Dog;

// 返回值：可以返回更具体的类型（协变）
const fn1: Fn = (x) => new Dog();    // OK
const fn2: Fn = (x) => new Animal(); // ❌ 不 OK，返回的是父类型

// 参数：可以接受更宽泛的类型（逆变，strict 模式下）
const fn3: Fn = (x: Animal) => new Dog();  // OK
const fn4: Fn = (x: Puppy) => new Dog();   // ❌ 不 OK，子类型参数

// 总结
// 返回值：子 → 父 不兼容，父 → 子 兼容（协变）
// 参数（strict）：父 → 子 兼容，子 → 父 不兼容（逆变）
```

### Q4: 如何让一个泛型类型参数强制为不变（Invariant）？

**解析**：考察对实际类型约束的运用能力。

```typescript
// 让 T 同时出现在参数和返回值位置
type Invariant<T> = {
  read: () => T;
  write: (value: T) => void;
};

// 或者使用 TypeScript 4.7 引入的方差注解（Variance Annotations）
// 尽管 TS 本身不支持显式声明方差标记，但可以通过结构定义实现

// 实用场景：类型安全的 Store
class Store<T> {
  private value: T;

  constructor(value: T) {
    this.value = value;
  }

  get(): T {
    return this.value;
  }

  set(value: T): void {
    this.value = value;
  }
}

// Store<Dog> 和 Store<Animal> 之间不能互相赋值
const dogStore: Store<Dog> = new Store(new Dog());
// const animalStore: Store<Animal> = dogStore; // ❌ 不变，类型错误
```

### Q5: 协变和逆变在前端框架中有什么实际应用？

**解析**：考察将理论应用到实际的综能力。

```typescript
// Vue 的事件发射器实际上是逆变的

  emit<K extends keyof Events>(event: K, payload: Events[K]): void;

type MyEvents = {
  click: MouseEvent;

// 当你定义更具体的事件类型时，
// 传递处理函数可以利用逆变规则

// React 事件处理也是类似的
// onChange: (e: React.ChangeEvent<HTMLInputElement>) => void
// 你可以传入一个处理更宽泛事件的函数：
// onChange: (e: React.ChangeEvent<HTMLElement>) => void  // 逆变安全
```

## 总结与扩展

### 核心记忆法则

- **"生产者"（产出值）→ 协变**：如果产出更具体的值，没问题
- **"消费者"（接收值）→ 逆变**：如果消费更一般的值，没问题
- **"既是生产者又是消费者"→ 不变**：两者兼顾时必须严格匹配

### 一句话记忆

> 你给我什么，我按更宽的取（返回值）；我要你给我什么，你得给更细的（参数）。

### 进阶方向

1. **Flux 架构和不变性**：Redux 的 reducer 签名与变型规则的关系

3. **其他语言的变型机制**：Java 的 `? extends` / `? super` wildcard、C# 的 `in` / `out` 标记、Scala 的 `+` / `-` 方差注解
4. **TypeScript 的 Variance Annotations**：TS 4.7 引入的内部方差计算优化，了解 TS 编译器的类型检查策略
