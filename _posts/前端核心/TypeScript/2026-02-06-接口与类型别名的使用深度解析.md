## 一句话概括

`interface` 能声明合并（给第三方库打补丁），`type` 能表达联合/交叉/元组（interface 做不到）。这不是喜好选择题——库的公开类型用 interface 留扩展点，组件 Props 用 type 写联合 variant，大多数场景两者都行。

## 核心知识点

### 1. 对象定义 —— 几乎等价

```typescript
// 两种写法效果一样
interface User { name: string; age: number }
type User = { name: string; age: number }

// 都能：
// - 被 class implements
// - 定义函数签名
// - extends / & 扩展
class Admin implements User { name = ''; age = 0 }
```

### 2. 声明合并 — interface 独有的超能力

```typescript
// 同名 interface 自动合并
interface User { name: string }
interface User { age: number }
// 最终类型：{ name: string; age: number }

const u: User = { name: 'Alice', age: 30 }; // ✅

// 🔑 这才是真正的价值：给第三方库扩展类型
declare interface Window {
  __INITIAL_STATE__: AppState; // 不覆盖 Window 原有属性
}

// declare interface Array<T> {
//   last(): T | undefined;   // 给原生 Array 加方法类型
// }

// type 重复定义直接报错
// type User = { age: number } // ❌ Duplicate identifier
```

⚠️ 非函数成员类型冲突会报错。同名函数成员自动合并为重载。

### 3. type 能干 interface 干不了的

```typescript
// 1. 原始类型别名
type ID = string;

// 2. 联合类型 —— type 的王牌
type Status = 'pending' | 'success' | 'error';

// 3. 元组
type Point = [number, number];

// 4. 交叉类型
type AdminUser = User & { role: 'admin'; permissions: string[] };

// 5. 从值提取类型
const config = { host: 'localhost', port: 3000 };
type Config = typeof config;

// 6. 映射类型 / 条件类型（只能用 type 定义）
type Readonly<T> = { readonly [K in keyof T]: T[K] };
type ReturnOf<T> = T extends (...args: any[]) => infer R ? R : never;
```

### 4. 扩展方式的关键差异

```typescript
interface A { a: string }
interface B { b: number }
interface C extends A, B { c: boolean }  // interface extends 多继承

type T = A & B & { c: boolean }          // type 用 & 交叉

// 可以互相扩展
interface D extends A { d: Date }        // interface extends type ✅
type E = B & { e: Date }                 // type & interface ✅

// ⚠️ 关键差异：同名属性冲突
interface X { a: number }
// interface Y extends X { a: string }  // ❌ 编译直接报错——好！
type Z = X & { a: string }              // ✅ 编译通过——但 a 的类型是 never（坏！）
```

**这个差异很重要：** `extends` 遇到冲突立刻报错帮你发现；`&` 把冲突属性静默合并成 `never`——编译通过但那个属性没法用。

### 5. 选型决策矩阵

| 场景 | interface | type |
|---|---|---|
| 定义对象/类形状 | ✅ 推荐 | ✅ |
| 需要声明合并（扩库） | ✅ 唯一选择 | ❌ |
| 联合/交叉/元组 | ❌ | ✅ |
| 映射/条件类型 | ❌ | ✅ |
| 库的公开 API 类型 | ✅ 推荐 | ❌ |
| React Props（带 variant 联合） | ❌ | ✅ |
| 日常业务对象定义 | ✅ 团队统一就行 | ✅ |

## 其实你每天都在用

- **给 window 打补丁**：`declare interface Window { appConfig: Config }` —— 只有 interface 能在不破坏原有类型的前提下扩展
- **组件 Props 带 variant**：`type ButtonProps = { variant: 'primary' | 'secondary' } & HTMLAttributes<HTMLButtonElement>` —— 联合 + 交叉一把梭
- **开源库 .d.ts 全是 interface**：Vue、React 的类型定义大量使用 interface，给你留声明合并的扩展点
- **API 响应泛型**：`interface ApiRes<T> { code: number; data: T; message: string }` —— 对象形状用 interface 语义更贴切
- **后端枚举约束**：`type Role = 'admin' | 'editor' | 'viewer'` —— 这种字面量联合只能用 type

## 常见误解

- **❌ 误区：「interface 是 type 的子集，表达能力弱」** 恰恰相反：就对象定义而言两者几乎等价；type 在联合/映射/条件类型上强于 interface；但 interface 有声明合并——这是 type 没有的能力。两者是**大部分重叠 + 各自独占**的关系，不是子集。

- **❌ 误区：「type 不适合大型项目因为不能 extends」** type 用 `&` 实现组合，表达能力甚至更强。只是 `extends` 遇到冲突报错更早（好），`&` 可能静默产生 `never`（隐患）。

- **❌ 误区：「React 官方用 interface，所以我也用」** React 类型定义源码既有 interface 也有 type。TS 官方文档的态度：能说清楚的用哪个都行，社区无硬性标准。

- **❌ 误区：「extends 和 & 遇到冲突都报错」** 不一样。`extends` 直接报红——帮你发现。`&` 把冲突属性合并成 `never`——编译过了但运行时废了。这是很多人踩坑后站 `extends` 的原因。

## 一句话总结

一个简单决策法则：需要被外部扩展（库的公开类型）→ `interface`；涉及联合、元组、映射类型 → `type`；只定义对象形状 → 团队约定优先。成熟项目的 .d.ts 里，两者混用才是常态。
