## 一句话概括

`interface` 和 `type` 都能描述对象形状，但 interface 支持**声明合并**（declaration merging），type 能定义**联合/交叉/原始类型别名**——选型原则不是喜好，而是场景。

## 核心知识点

### 1. 基础对比：语法不同，但对象定义能力几乎一样

```typescript
// interface 写法
interface User {
  name: string;
  age: number;
}

// type 写法
type User = {
  name: string;
  age: number;
};

// 两者都可以被 class 实现、都可以扩展
class Admin implements User { name = ''; age = 0; }
```

### 2. 声明合并 — interface 的独门绝技

```typescript
// 同名的 interface 自动合并
interface User {
  name: string;
}
interface User {
  age: number;
}
// 合并后等价于：{ name: string; age: number }

const u: User = { name: 'Alice', age: 30 }; // ✅

// ⚠️ 非函数成员类型冲突会报错
// interface User { name: number; } // ❌ 和 string 冲突

// ⚠️ 同名函数成员会被合并为函数重载（后声明的优先级更高）
```

**type 无法声明合并，重复定义直接报错。**

### 3. type 能干的，interface 干不了

```typescript
// 1. 原始类型别名
type ID = string;

// 2. 联合类型 — type 的杀手锏
type Status = 'pending' | 'success' | 'error';

// 3. 交叉类型
type AdminUser = User & { role: 'admin' };

// 4. 元组
type Point = [number, number];

// 5. 从值提取类型
const config = { host: 'localhost', port: 3000 };
type Config = typeof config;

// 6. 映射类型 / 条件类型
type Readonly<T> = { readonly [K in keyof T]: T[K] };
```

### 4. 扩展方式的差异

```typescript
// interface 用 extends（可多继承）
interface A { a: string }
interface B { b: number }
interface C extends A, B { c: boolean }

// type 用 & 交叉
type A = { a: string };
type B = { b: number };
type C = A & B & { c: boolean };

// 两者可以互相扩展
interface D extends A { d: Date }     // interface extends type ✅
type E = B & { e: Date };              // type & interface ✅
```

### 5. 选型原则速查

| 需求 | 选 interface | 选 type |
|------|:-----------:|:------:|
| 定义对象/类的形状 | ✅ 推荐 | ✅ |
| 需要声明合并（给三方库打补丁） | ✅ 唯一 | ❌ |
| 联合类型、元组、原始类型别名 | ❌ | ✅ |
| 映射类型 / 条件类型 | ❌ | ✅ |
| 公开 API（给人扩展用） | ✅ 推荐 | ❌ |
| React Props | 🟡 都可以 | ✅ 更灵活 |
| 函数类型 | ✅ `interface Fn { (x: number): void }` | ✅ `type Fn = (x: number) => void` |

## 其实你每天都在用

- **给 window 打补丁**：`declare interface Window { __CUSTOM__: string }` — 只有 interface 能做到不覆盖已有 Window 类型
- **组件 Props 用 type**：`type ButtonProps = { variant: 'primary' | 'secondary' } & HTMLAttributes` — 联合+交叉一把梭
- **三方库的 `.d.ts` 文件**：几乎全是 interface，就是为了让用户能通过声明合并扩展类型
- **API 响应类型定义**：`interface ApiResponse<T> { code: number; data: T }` — 对象形状，interface 更语义化
- **枚举值类型约束**：`type Role = 'admin' | 'editor' | 'viewer'` — 这种联合类型只能 type

## 常见误解

- **❌ 误区：「interface 和 type 性能差别大，type 编译慢」** 对于绝大多数项目，两者的编译性能差异**可以忽略不计**。只有极端复杂的递归类型（几百层嵌套）才可能测出微小差异。不要以性能为由选型。

- **❌ 误区：「type 不能用 extends，所以表达能力比 interface 弱」** type 用 `&` 交叉类型实现同样的效果，且 `&` 比 `extends` 更灵活——它能组合联合类型、原始类型等 interface 无法表达的东西。type 的表达能力**强于** interface，只是不支持声明合并。

- **❌ 误区：「interface 只能用 extends，不能交叉」** `interface C extends A, B` 实际上就是在做交叉。区别在于：`extends` 遇到同名属性类型冲突会报错（早暴露问题），`&` 交叉则会悄悄合并为 `never`（可能隐藏 bug）。

- **❌ 误区：「React 官方推荐用 interface 定义 Props」** React 官方类型定义自己用的是 `type`。社区没有统一标准，TS 官方文档的态度是：能描述清楚的用哪个都行。实际选型看团队约定和是否需要声明合并。

## 一句话总结

一个简单的判断：如果这个类型将来**可能被用户扩展**（库的类型定义），用 interface；如果涉及**联合类型、元组、映射类型**，用 type；如果只描述普通对象形状，看团队规范——两者都能胜任。
