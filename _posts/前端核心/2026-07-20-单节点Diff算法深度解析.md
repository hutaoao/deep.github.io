## 一句话概括

React 单节点 Diff 算法遵循"先比较 type（类型），再比较 key（标识）"的二分判断策略，当类型和 key 同时匹配时复用 DOM 节点并更新属性，否则删除旧节点重新创建——以最多一次 DOM 操作的代价完成 Reconciliation。

## 背景与意义

### 为什么需要单节点 Diff？

在 React 组件中，大量场景下同一父元素只有一个子元素：

```jsx
// 条件渲染
<div>
  {isLoggedIn ? <Dashboard /> : <LoginForm />}
</div>

// 元素类型切换
<div>
  {isExpanded ? <DetailedView data={data} /> : <SummaryView data={data} />}
</div>

// 单子元素更新
<Greeting name={user.name} />
```

虽然单节点 Diff 看起来比多节点 Diff "简单"，但它的正确性是整个 Diff 体系的基础——单节点 Diff 每一次都涉及 DOM 节点的创建或销毁，对性能影响直接而显著。

### 与多节点 Diff 的区别

- **单节点 Diff**：同一父元素下至多一个子节点。比较的是"一个新 vnode vs 一个旧 fiber"的关系
- **多节点 Diff**：同一父元素下有多个子节点。比较的是"一组新 vnode vs 一组旧 fiber"的数组关系，涉及位置重排

单节点 Diff 不需要处理"移动"（不移位），只需要判断"复用还是重建"。但在**兄弟节点的影响**上，单节点 Diff 有独特的"删除剩余兄弟"步骤。

### 单节点 Diff 的触发场景

- 条件渲染（`{condition && <Child/>}`）
- 三元表达式（`{cond ? <A/> : <B/>}`）
- 单个子节点的 key 或 type 变化
- 单个子节点的 props 更新

## 概念与定义

### key

唯一的标识符，帮助 React 识别在多次渲染间哪些子元素是"同一个"。对于单节点，key 不是必需的，但如果你使用了 key，React 会用它参与比较。

```jsx
// 两个 key 相同的节点会被视为"同一个节点"
{cond ? <Comp key="main" /> : <Comp key="main" />}
// key 不同则被视为"不同的节点"，必须销毁重建
```

### type

React 元素的类型，可以是：
- 字符串（DOM 元素标签名，如 `'div'`、`'span'`）
- 函数（FunctionComponent）
- 类（ClassComponent）
- React.Fragment

```typescript
type ReactElementType = string | ((props: any) => JSX.Element) | (new (props: any) => Component);
```

### reconcileSingleElement 函数

React 进行单节点 Diff 的核心函数，位于 `ReactChildFiber.js` 中。它的输入是：
- 当前 Fiber（current tree 中对应的旧节点）
- 新的 React Element
- 当前渲染的 Lanes

输出是一个新的 Fiber（workInProgress 中的节点）。

## 核心知识点拆解

### 1. 单节点 Diff 的判断流程树

```
reconcileSingleElement(current, newChild, lanes)
    │
    ├── current === null?
    │   ├── 是 → 没有旧节点 → 直接创建新 Fiber → 返回
    │   └── 否 → 进入判断
    │
    ├── 比较 key 是否相同？
    │   ├── key 不相同 → 标记当前 Fiber 为 Deletion
    │   │   └── 继续删除其兄弟节点（如果有）
    │   │   └── 创建新的 Fiber 返回
    │   └── key 相同 → 进入 type 判断
    │
    ├── 比较 type 是否相同？
    │   ├── type 相同 → 复用当前 Fiber
    │   │   └── 标记兄弟姐妹为 Deletion
    │   │   └── 返回复用的 Fiber（useFiber）
    │   └── type 不同 → 标记当前 Fiber 为 Deletion
    │       └── 继续删除其兄弟节点
    │       └── 创建新的 Fiber 返回
    │
    └── 注意删除兄弟节点 → 沿着 sibling 链表全部标记 Deletion
```

### 2. 四种具体的比较场景

**场景一：无旧节点，有新节点（首次渲染）**

```jsx
// 之前：null
// 之后：<div key="a" />
// 结果：创建新 Fiber，无复用
```

**场景二：旧节点和新节点 key 不同**

```jsx
// 之前：<div key="a" />
// 之后：<span key="b" />
// 
// 判断流程：
// 1. key 不同（'a' !== 'b'）
// 2. 标记旧 div fiber 为 Deletion（后续移除）
// 3. 删除所有兄弟节点的 fiber
// 4. 根据 <span key="b" /> 创建新 Fiber
// 5. 返回新创建的 Fiber
//
// 结果：销毁旧 DOM 节点，创建新 DOM 节点
```

**场景三：旧节点和新节点 key 相同但 type 不同**

```jsx
// 之前：<div key="a" />
// 之后：<span key="a" />
//
// 判断流程：
// 1. key 相同（均为 'a'）
// 2. type 不同（'div' !== 'span'）
// 3. 标记旧 div fiber 为 Deletion
// 4. 根据 <span key="a" /> 创建新 Fiber
// 5. 返回新 Fiber
//
// 结果：即使 key 相同，type 不同也必须销毁重建
```

**场景四：旧节点和新节点 key 相同且 type 相同**

```jsx
// 之前：<div key="a" className="old" />
// 之后：<div key="a" className="new" />
//
// 判断流程：
// 1. key 相同（均为 'a'）
// 2. type 相同（均为 'div'）
// 3. 复用旧 Fiber，更新 pendingProps
// 4. 标记 siblings 为 Deletion
// 5. useFiber 返回复用的 Fiber
//
// 结果：DOM 节点被复用，只更新 className 属性
```

### 3. key 在使用时的实际行为

当父元素是单一子节点时，key 不是必需的。但有趣的是，React 在和旧 fiber 比较时总是先比较 key——**即使你没有显式指定 key，React 也会为元素自动生成 key（默认为 null）**。

```typescript
// React 如何比较 key（简化）
function reconcileSingleElement(returnFiber, currentFirstChild, newChild, lanes) {
  const key = newChild.key;  // 如果未指定，key = null
  
  // 尝试复用已有的第一个 child
  let child = currentFirstChild;
  while (child !== null) {
    // 注意：key 比较是用 ===（严格比较）
    // 无 key 时 child.key 和 newChild.key 都是 null → 相等
    if (child.key === key) {
      // key 匹配，继续比较 type
      if (child.elementType === newChild.type) {
        // type 也匹配，复用
        deleteRemainingChildren(returnFiber, child.sibling);
        const existing = useFiber(child, newChild.props);
        existing.ref = newChild.ref;
        existing.return = returnFiber;
        return existing;
      }
      // type 不匹配，删除所有兄弟并创建新 Fiber
      deleteRemainingChildren(returnFiber, child);
      break;
    } else {
      // key 不匹配，删除当前 child
      deleteChild(returnFiber, child);
    }
    
    child = child.sibling;
  }
  
  // 创建新的 Fiber
  const created = createFiberFromElement(newChild, returnFiber.mode, lanes);
  created.ref = newChild.ref;
  created.return = returnFiber;
  return created;
}
```

## 底层原理（含源码分析）

### 1. reconcileSingleElement 的完整源码

```typescript
// 源码位置：packages/react-reconciler/src/ReactChildFiber.js

function reconcileSingleElement(
  returnFiber: Fiber,
  currentFirstChild: Fiber | null,
  element: ReactElement,
  lanes: Lanes,
): Fiber {
  const key = element.key;
  let child = currentFirstChild;
  
  // 遍历兄弟链表（理论上单节点只有第一个 child，但可能有多余的兄弟）
  while (child !== null) {
    // 第一步：比较 key
    if (child.key === key) {
      // key 匹配！进入 type 比较
      const elementType = element.type;
      
      if (elementType === REACT_FRAGMENT_TYPE) {
        // Fragment 的特殊处理
        if (child.tag === Fragment) {
          deleteRemainingChildren(returnFiber, child.sibling);
          const existing = useFiber(child, element.props.children);
          existing.return = returnFiber;
          return existing;
        }
      } else {
        // 普通节点的 type 比较
        if (child.elementType === elementType || (
          // 检查是否兼容的类型（如 lazy 组件解析前后）
          typeof elementType === 'object' &&
          elementType !== null &&
          elementType.$$typeof === REACT_LAZY_TYPE &&
          child.type === elementType._init
        )) {
          // type 匹配！复用
          deleteRemainingChildren(returnFiber, child.sibling);
          const existing = useFiber(child, element.props);
          existing.ref = element.ref;
          existing.return = returnFiber;
          return existing;
        }
      }
      
      // key 匹配但 type 不匹配
      // 删除所有剩余的节点并创建新的
      deleteRemainingChildren(returnFiber, child);
      break;
    } else {
      // key 不匹配：删除当前 child
      deleteChild(returnFiber, child);
    }
    
    child = child.sibling;
  }
  
  // 没有找到可复用的节点，创建新的
  const created = createFiberFromElement(element, returnFiber.mode, lanes);
  created.ref = element.ref;
  created.return = returnFiber;
  return created;
}
```

### 2. deleteRemainingChildren 的批量删除

当确定了要复用的节点后，React 会把当前节点之后的所有兄弟节点标记删除：

```typescript
function deleteRemainingChildren(
  returnFiber: Fiber,
  currentFirstChild: Fiber | null,
): null {
  if (!shouldTrackSideEffects()) {
    return null;
  }
  
  let childToDelete = currentFirstChild;
  while (childToDelete !== null) {
    deleteChild(returnFiber, childToDelete);
    childToDelete = childToDelete.sibling;
  }
  
  return null;
}

function deleteChild(returnFiber: Fiber, childToDelete: Fiber): void {
  if (!shouldTrackSideEffects()) return;
  
  // 将 childToDelete 加入到 returnFiber 的 deletions 链表中
  const deletions = returnFiber.deletions;
  if (deletions === null) {
    returnFiber.deletions = [childToDelete];
    returnFiber.flags |= ChildDeletion;
  } else {
    deletions.push(childToDelete);
  }
}
```

`deletions` 数组是 Fiber 节点上的字段，记录了当前 fiber 的所有子节点中需要被删除的节点。在 commit 阶段，React 会遍历这些 deletions 并执行 `commitDeletion`。

### 3. useFiber 的复用逻辑

当 key 和 type 都匹配时，React 调用 `useFiber` 来复用 Fiber 节点：

```typescript
function useFiber(fiber: Fiber, pendingProps: mixed): Fiber {
  // 通过 alternate 实现双缓冲复用
  const clone = createWorkInProgress(fiber, pendingProps);
  clone.index = 0;  // 重置 index
  clone.sibling = null;  // 清除兄弟关系
  return clone;
}
```

`useFiber` 本质上就是调用了 `createWorkInProgress`，它通过 alternate 指针复用 Fiber 对象，只更新了 `pendingProps` 等可变字段。这就是"复用"的真正含义——不是复制对象，而是**重置可变状态并更新 props**。

## 高频面试题解析

### Q1：React 在什么情况下会销毁旧 DOM 节点并创建新节点？

**参考答案：**

在以下任一情况发生时，React 会销毁旧节点创建新节点：

1. **key 不同**：如果新旧节点的 key 不同，React 认为它们是不同的实体，即使 type 完全相同也无法复用
2. **type 不同**：即使 key 相同，如果 type（如 'div' → 'span'）不同，也必须销毁重建——因为 DOM 标签类型无法通过更新属性来改变
3. **首次渲染**（current === null）：没有旧节点可复用，自然创建新节点

需要特别注意：**type 变化意味着 DOM 标签变化**，这是浏览器本身的限制——你无法将 `<div>` 的 DOM 元素变成 `<span>`，所以必须销毁重建。

### Q2：为什么不建议在列表中使用 index 作为 key？单节点中有影响吗？

**参考答案：**

**列表中使用 index 作为 key 的主要问题**发生在列表元素的插入、删除、重排场景：

```jsx
// 初始渲染：['a', 'b', 'c']
// key: index → 0, 1, 2
// key: id → 'a', 'b', 'c'

// 删除第一个元素后：
// ['b', 'c']（新渲染）
// key: index → 0('b'), 1('c')  ← 所有 key 都变了
// key: id → 'b', 'c'  ← key 不变，index 没错位

// 使用 index 作为 key 会引发的问题：
// - 旧 index=0（'a'）↔ 新 index=0（'b'）：key 匹配但内容不同 → 复用错误节点
// - 导致输入框的值错误地保留在错误的行上
```

**单节点中不使用 key 不会有此问题**，因为单节点没有"位置移动"或"中间插入"的场景。单节点要么存在要么不存在、要么 A 要么 B，不存在错位问题。所以在单节点中，不指定 key 是安全的。

但是，如果单节点你在条件渲染中手动指定了 key，反而可能造成意想不到的销毁重建——例如前面例子中的 `cond ? <Comp key="one" /> : <Comp key="two" />` 会导致每次切换都销毁重建，即使 type 相同。

### Q3：单节点 Diff 中 deleteRemainingChildren 的作用是什么？为什么需要删除兄弟节点？

**参考答案：**

`deleteRemainingChildren` 的作用是：**在确定了要复用的节点后，清理所有未被使用的兄弟节点**。

虽然听起来是"单节点"场景，但旧 Fiber 树上可能有多个兄弟节点（也就是实际上是一个短列表变成了单节点的情况）：

```jsx
// 之前：多个兄弟
<div>
  <A key="a" />
  <B key="b" />
  <C key="c" />
</div>

// 之后：缩略为单节点
<div>
  <A key="a" />
</div>
```

如果 React 没有在复用 `<A>` 之后删除 `<B>` 和 `<C>` 的兄弟节点，这两个 Fiber 节点会残留在 Fiber 树中。当 commit 阶段执行 DOM 操作时，`<B>` 和 `<C>` 的 DOM 节点不会被移除，导致**页面残留**。

所以，`deleteRemainingChildren` 是这个过程的**必要清理步骤**，确保从多节点到单节点的转换中，多余的节点被正确标记删除。

## 总结与扩展

### 总结

React 单节点 Diff 算法的本质是"最多一次 DOM 操作"的优化：

1. **key 优先比较**：先确定"是不是同一个节点"的身份问题
2. **type 二次确认**：再确定 DOM 型的建立问题
3. **决定结果**：key+type 都匹配→复用只改 props；不匹配→直接销毁重建

### 扩展：React 为什么不尝试复用 type 不同的节点？

这是一个"复杂度和收益"的权衡。理论上，可以实现一种"从旧 DOM 元素创建新类型 DOM 元素时，尽量保留旧 DOM 节点的属性"的策略。但代价是：

- 代码复杂度大幅增加
- 调试和推测性变差
- 实际收益有限（大多数 type 变化是组件级别的，而非 DOM 元素级别的）

React 选择了"如果 type 变化就销毁重建"的简单策略。这是一种**在正确性、简单性和性能之间最合理的权衡**。
