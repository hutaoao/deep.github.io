## 一句话概括

React Fiber 是一套基于**链表结构的可中断协调引擎**，它将不可分割的递归渲染过程拆解为可暂停、可优先级的异步工作单元，使 React 能够在长时间渲染任务中主动让出主线程，从而保证用户交互与动画的流畅性，奠定了 React 并发渲染（Concurrent Rendering）的底层基础。

## 背景与意义

### React 的前 Fiber 时代：Stack Reconciler

在 React 15 及更早版本中，协调（Reconciliation）过程由 Stack Reconciler 实现。其名称来源于**递归调用栈**——当 React 开始渲染时，它会从根节点开始递归地遍历整个虚拟 DOM 树，比较前后两棵树的差异，同步执行所有变更，直到整棵树处理完毕。

React 15 的协调流程可以简化为如下伪代码：

```javascript
function reconcile(parentDOM, element) {
  const newDOM = createDOM(element);      // 创建真实 DOM
  parentDOM.appendChild(newDOM);           // 挂载
  element.children.forEach(child => {
    reconcile(newDOM, child);              // 递归处理子节点
  });
}
```

这段代码直观地展现了 Stack Reconciler 的核心问题：**递归是不可中断的**。一旦开始协调，调用栈就会一路深入直到叶子节点再回溯，整个过程中 JavaScript 主线程被完全占用。当一个组件树的复杂度较高（例如包含数千个节点的大型列表、深层嵌套的表单页面），一次渲染耗时可能达到几十甚至上百毫秒。

### Stack Reconciler 的三大局限

**1. 不可中断（非异步性）**

递归调用栈一旦压入就无法弹出，直到全部执行完毕。这意味着任何交互事件（点击、输入、滚动）都必须排队等待当前渲染任务完成才能响应。从用户感知的维度来看，浏览器每秒产生 60 帧画面，每帧可用时间约 **16.6ms**。如果一次渲染耗时超过这个阈值，页面就会出现明显的卡顿（jank）。

**2. 缺乏优先级调度**

所有更新在 Stack Reconciler 中地位平等——一次鼠标点击触发的状态更新与一次 offscreen 区域的数据加载，在渲染管线上享有完全相同的优先级。但在真实场景中，用户对交互反馈的实时性要求远高于后台数据预加载。Stack Reconciler 无法区分"用户正在输入"和"收到 WebSocket 推送"这两种更新的紧急程度。

**3. 主线程独占策略**

Stack Reconciler 将虚拟 DOM 的 diff、真实 DOM 的创建与插入、生命周期函数的执行全部打包在同一个同步过程中，期间不向浏览器归还控制权。浏览器无法插入帧渲染、事件处理、垃圾回收等工作，导致视觉更新出现空洞（frame drop）。

### 为什么需要 Fiber

2016 年，React 团队在 F8 大会上首次公开 Fiber 架构。核心动机非常明确：**让 React 的渲染过程从「不可中断的同步递归」演变为「可中断的异步协作」**。Fiber 并非单纯优化 Diff 算法——Diff 算法本身（基于 key 和 type 的 O(n) 比较）在 React 15 中已经相当高效。Fiber 要解决的是**调度层面的结构性缺陷**。

换句话说，问题不在于"比两棵树谁更快"，而在于"比到一半能不能停一下去处理用户点击"。Stack Reconciler 体系下不能，Fiber 体系下可以。这就是 Fiber 存在的根本意义。

> 如果不理解"为什么需要 Fiber"，就无法真正理解 Fiber 的设计。Fiber 首先是一个**调度器**的产物，其次才是一个协调器。

## 概念与定义

### 什么是 Fiber 节点

Fiber 是一个普通的 JavaScript 对象，但它背后承载的概念远比对象本身深刻。Fiber 本质上是 **React 对组件实例化过程中的一个"工作单元"（unit of work）的抽象**。

在 Stack Reconciler 时代，React 对组件的处理隐含在递归调用栈中——每个递归帧隐式地代表一个组件的处理状态。Fiber 将这些隐式状态显式化为一个对象，让 React 可以：

1. **随时暂停**一个 Fiber 的处理，将其状态保存在内存中
2. **随时恢复**从上次中断的位置继续处理
3. **优先处理**更紧急的 Fiber，推迟不那么紧急的 Fiber

每一个 React 元素（React Element）在渲染阶段都会对应生成一个 Fiber 节点。Fiber 节点构成了 Fiber 树的节点。

### Fiber 节点的核心属性

Fiber 节点的结构反映了 React 对"如何构建一棵可遍历的树"的完整思考：

```typescript
interface Fiber {
  // ========== 节点标识 ==========
  tag: WorkTag;          // 节点类型标识：FunctionComponent(0)、ClassComponent(1)、HostComponent(5) 等
  key: string | null;    // 用于列表 diff 的唯一标识
  type: any;             // 组件类型：函数组件就是函数本身，类组件就是 class，原生 DOM 就是 'div'
  elementType: any;      // 与开发模式下 type 略有差异的版本

  // ========== 树结构指针 ==========
  child: Fiber | null;       // 指向第一个子 Fiber
  sibling: Fiber | null;     // 指向下一个兄弟 Fiber
  return: Fiber | null;      // 指向父 Fiber（注意不是 parent，是 return）
  index: number;             // 在兄弟节点中的位置索引

  // ========== 双缓冲相关 ==========
  alternate: Fiber | null;   // 指向缓存树中对应的 Fiber 节点

  // ========== 状态与副作用 ==========
  pendingProps: any;         // 待处理的 props
  memoizedProps: any;        // 上一次渲染的 props
  memoizedState: any;        // 上一次渲染的 state（hooks 链表也挂在这里）
  updateQueue: any;          // 更新队列（状态更新、副作用等）
  flags: Flags;              // 副作用标记（Placement、Update、Deletion 等）
  subtreeFlags: Flags;       // 子树的副作用标记（收集优化）
  deletions: Fiber[] | null; // 需要删除的子节点列表

  // ========== 生命周期相关 ==========
  lanes: Lanes;              // 优先级标记
  childLanes: Lanes;         // 子树中的最高优先级
  mode: TypeOfMode;          // 渲染模式（legacy 还是 concurrent）

  // ========== DOM 相关 ==========
  stateNode: any;            // 对应该 Fiber 的真实 DOM 节点或组件实例
}
```

值得特别注意的是 `return` 字段。Fiber 树本质上是一棵**单向链表化的树**，通过 `child -> sibling -> return` 构成一个螺旋遍历结构。之所以叫 `return` 而不是 `parent`，是因为在遍历过程中，`return` 表示"处理完当前节点后应该返回的地方"，语义上更贴近遍历的控制流，而非静态的树结构。

### 为什么用链表代替递归

Stack Reconciler 使用递归调用栈，本质上利用的是**系统调用栈**来存储遍历状态。Fiber 使用链表，本质上是**用堆内存结构显式存储遍历状态**。

二者在计算机科学层面代表了两种不同的模型：

| 特性 | 递归调用栈 | Fiber 链表 |
|------|-----------|------------|
| 存储介质 | 系统栈（有限空间） | 堆空间（几乎无限） |
| 中断能力 | 不可中断 | 每个 Fiber 之间都是中断点 |
| 恢复方式 | 自动恢复（调用栈回溯） | 通过链表指针恢复 |
| 优先级 | 无（后进先出） | 支持优先级重排 |

链表带来的最大优势在于：Fiber 树可以在任意节点处暂停遍历，也可以从任意节点恢复遍历，甚至可以在遍历过程中**插入新的工作单元**。

## 最小示例

让我们通过一个极简的 JSX 结构来理解 Fiber 树的形态：

```jsx
function App() {
  return (
    <div>
      <h1>Hello</h1>
      <p>World</p>
    </div>
  );
}
```

React 在协调阶段构建的 Fiber 树拓扑如下：

```
App（FiberNode, tag: FunctionComponent）
 │
 ├─ child
 │
 ▼
<div>（FiberNode, tag: HostComponent）
 │
 ├─ child
 │
 ▼
<h1>（FiberNode, tag: HostComponent）
 │
 ├─ sibling
 │
 ▼
<p>（FiberNode, tag: HostComponent）
```

关键遍历路径不是简单的树递归，而是**螺旋结构**：

1. **下行**：从 App → child → div → child → h1
2. **兄弟跳转**：h1 → sibling → p
3. **上溯**：p → return → div → return → App

每个节点都是一个**独立的工作单元（unit of work）**，React 的 Scheduler 决定了何时处理哪个单元、何时暂停、何时把控制权交还给浏览器。

## 核心知识点拆解

### 1. 可中断渲染的设计理念

Fiber 最核心的设计理念是**协作式多任务（Cooperative Scheduling）**。它不像操作系统那样可以抢占（preemptive），而是依赖于"每个工作单元主动检查是否应该让出"。

核心算法是 **workLoop**：

```javascript
function workLoop(deadline) {
  let shouldYield = false;
  while (nextUnitOfWork !== null && !shouldYield) {
    nextUnitOfWork = performUnitOfWork(nextUnitOfWork);
    shouldYield = deadline.timeRemaining() < 1;
  }
  if (nextUnitOfWork !== null) {
    requestIdleCallback(workLoop);  // 下次空闲继续
  }
}
```

这个循环的关键在于：
- **workLoop 是协作式的**：每次执行一个 Fiber 单元后就检查 `timeRemaining()`
- **每个 Fiber 节点就是一个检查点**：执行完一个 Fiber 后可以决定是否让出
- **中断不丢失状态**：`nextUnitOfWork` 指针保存了下一个待处理的 Fiber，下次恢复时直接从这里继续

这种设计让 React 能够在渲染过程中"见缝插针"地让出主线程。用户在输入时，浏览器可以优先处理输入事件，等空闲了再继续渲染——用户体验瞬间提升一个台阶。

### 2. Fiber 树的遍历算法

Fiber 树的遍历采用了**深度优先 + 链表回溯**的方式，分为两个阶段：

**阶段一：beginWork（向下探索）**

从根节点出发，沿 `child` 指针向下逐步深入，为每一个 Fiber 执行 beginWork。beginWork 中会进行组件更新、diff 子节点、根据 diff 结果生成新的子 Fiber。如果当前 Fiber 需要执行副作用，会在 flags 中打标记。

处理完当前 Fiber 后，优先进入 child 继续 beginWork。如果没有 child，则进入下一阶段。

**阶段二：completeUnitOfWork（向上回溯）**

当到达叶子节点（没有 child），执行 completeWork——这个阶段主要处理 DOM 节点的创建（初次渲染）或属性更新（更新渲染），并将子节点的 DOM 归约到父节点。

处理完 completeWork 后：
- 如果有 sibling，切换到 sibling 进行 beginWork
- 如果没有 sibling，回到 return 父节点继续 completeWork

整体流程如下：

```
beginWork(A) → beginWork(A.child.div) → beginWork(div.child.h1)
→ completeWork(h1) → 无 sibling → completeWork(div)
→ 处理 p（从 div 的 sibling 开始新的 beginWork）
→ beginWork(p) → completeWork(p) → completeWork(App)
```

这个遍历模式确保每个 Fiber 节点恰好被访问两次：一次 beginWork（进入），一次 completeWork（离开）。类组件在 beginWork 中执行 render，在 completeWork 中处理副作用；原生 DOM 组件在 completeWork 中创建或更新 DOM。

### 3. 时间切片原理

时间切片（Time Slicing）是 Fiber 实现"可中断渲染"的关键机制。它不同于简单的异步，而是一种**基于帧预算的时间管理策略**。

**requestIdleCallback 与 requestAnimationFrame**

浏览器渲染主线程的工作可以抽象为以下循环：

1. 处理输入事件（touch、click、keydown）
2. 执行 requestAnimationFrame 回调
3. 解析样式、计算布局、绘制
4. 如果有空闲时间 → 执行 requestIdleCallback

React 的原始方案依赖 `requestIdleCallback`，利用浏览器每一帧渲染完成后剩余的空闲时间来执行渲染工作。如果空闲时间不够，React 就暂停，等到下一帧的空闲时间继续。

> React 团队发现 `requestIdleCallback` 在 Chrome 上的触发频率不够高（最密大约 50ms 一次），且 Safari/Firefox 兼容性不佳。因此 React 实现了自己的 Scheduler，使用 **`MessageChannel`** + **`setTimeout`** 作为备选方案来实现高性能的时间切片。

**Scheduler 的实现细节**

React Scheduler 的核心思路是：

1. 维护两个任务队列：**未过期任务队列**（timer queue）和**已过期任务队列**（task queue）
2. 使用 `requestAnimationFrame` 感知帧边界，计算出帧剩余时间
3. 在 `MessageChannel` 的 `port.postMessage` 中执行任务切片，利用其**宏任务**特性让浏览器有机会在两次回调之间完成渲染
4. 每次切片执行完后检查 `performance.now()`，超过 5ms 就主动让出

代码层面看 Scheduler 的核心循环：

```javascript
function workLoop(hasTimeRemaining, initialTime) {
  let currentTime = initialTime;
  advanceTimers(currentTime);  // 将到期的定时任务移入 taskQueue
  currentTask = peek(taskQueue);
  while (currentTask !== null) {
    if (currentTask.expirationTime > currentTime && shouldYieldToHost()) {
      break;  // 任务未过期且需要让出主线程 -> 中断
    }
    const callback = currentTask.callback;
    const didCallbackComplete = callback();
    if (didCallbackComplete) {
      // 任务完成，从队列移除
      pop(taskQueue);
    } else {
      // 任务未完成（被中断），下次继续
    }
    advanceTimers(currentTime);
    currentTask = peek(taskQueue);
  }
  // 如果还有剩余任务，安排下一次执行
  if (currentTask !== null) {
    return true;
  }
  // 检查 timerQueue，安排最小堆顶的定时任务
  requestHostTimeout(handleTimeout, ...);
  return false;
}
```

通过这种机制，React 可以将一次耗时 100ms 的渲染任务拆分成大约 20 个 5ms 的时间切片，均匀分布在 20 帧中执行。用户在这 20 帧期间可以流畅地点击、滚动、输入，感受不到渲染任务的存在。

### 4. 任务优先级与 Lane 模型

Fiber 引入了一套完整的**优先级系统**，在不同场景下触发更新时，React 会赋予不同的优先级。

**React 17 及之前的 ExpirationTime 模型**

在 React 17 中，优先级使用 `ExpirationTime`（过期时间）表示。越紧急的任务，过期时间越短。这个模型的问题在于：**精度有限、无法表达多个独立的优先级车道**。

**React 18 的 Lane 模型（车道模型）**

React 18 引入了全新的 Lane 模型。Lane 本质上是**一组 32 位的二进制位掩码**，每一位代表一条"车道"：

```javascript
export const NoLanes = /*               */ 0b0000000000000000000000000000000;
export const NoLane = /*                */ 0b0000000000000000000000000000000;
export const SyncLane = /*              */ 0b0000000000000000000000000000001;
export const InputContinuousLane = /*   */ 0b0000000000000000000000000000010;
export const DefaultLane = /*           */ 0b0000000000000000000000000000100;
export const IdleLane = /*              */ 0b1000000000000000000000000000000;
```

Lane 模型相比 ExpirationTime 的核心优势：

| 能力 | ExpirationTime | Lane 模型 |
|------|---------------|----------|
| 多任务并行 | 难表达 | 位运算天然支持 |
| 优先级批处理 | 需额外逻辑 | 按位合并 |
| 新任务插入 | O(log n) 堆操作 | O(1) 位运算 |
| 区分同优先级任务 | 不能 | 不同 Lane 可以 |
| 饥饿处理 | 过期自然提高 | 通过 Starvation 标记 |

**优先级从高到低**

React 的主要优先级层级（从高到低）：

1. **SyncLane**：同步更新，用户交互直接触发（点击、输入），不经过 Scheduler，立即执行
2. **InputContinuousLane**：连续输入（鼠标移动、滚动），允许稍后处理但不该久等
3. **DefaultLane**：普通的 setState/useState 更新
4. **TransitionLane**：过渡更新（startTransition），可以推迟
5. **RetryLane**：Suspense 重试
6. **IdleLane**：预加载、后台数据同步等非关键操作

**Lane 的批处理策略（Automatic Batching）**

Lane 的另一大优势是天然的批处理能力。当多个更新出现在同一个事件循环中时，React 通过位运算 `|`（按位或）将它们的 Lane 合并：

```javascript
// 同一个事件中触发了两次 setState
function handleClick() {
  setCount(c => c + 1);   // SyncLane
  setFlag(f => !f);       // SyncLane
}
// React 将它们合并为一次渲染传递，lanes 仍是 SyncLane
```

React 18 中所有更新都会自动批处理，不再依赖 `unstable_batchedUpdates`。

### 5. 双缓冲树机制

双缓冲（Double Buffering）是图形学中的一个经典概念，用于避免画面撕裂（tearing）。GPU 在后台缓冲区绘制帧，绘制完成后交换到前台缓冲区显示，用户看到的是完整的画面。

React Fiber 的双缓冲机制与此异曲同工。React 在内存中维护了两棵 Fiber 树：

**current 树**：对应当前屏幕上显示的内容，基于上一次渲染生成。每个 Fiber 节点的 `alternate` 不为空。

**workInProgress 树**：正在构建中的新 Fiber 树，基于本次更新的 diff 结果。构建完成后会通过 `root.current = finishedWork` 完成"树的切换"（类似 GPU 的缓冲区交换）。

**为什么要双缓冲？**

1. **中断安全**：workInProgress 树构建到一半被中断时，current 树完好无损，屏幕上显示的是旧内容。用户不会看到半渲染的残缺 UI。
2. **对象复用**：大部分情况下，workInProgress 的 Fiber 节点复用的是 current 树中对应节点的 `alternate`（通过 `createWorkInProgress` 复用），减少 GC 压力。
3. **差异追踪**：通过比较 current 和 workInProgress，React 可以精确知道哪些节点是新增的、修改的、删除的。

**树的切换时机**

在 commit 阶段（所有 Fiber 的 completeWork 完成后），React 执行：

```javascript
root.current = finishedWork;
```

这条赋值语句将所有 pending Fiber 工作一次性应用到屏幕上。这就是 Fiber 架构能做到"所有 DOM 变更原子提交"的原因——所有副作用在 workInProgress 树构建阶段被标记，在 commit 阶段统一执行。

### 6. 并发渲染（Concurrent Rendering）

并发渲染是 React 18 最受瞩目的新特性，其底层完全依赖 Fiber 架构提供的能力。并发（Concurrency）不等于并行（Parallelism）——JavaScript 仍然是单线程的。并发渲染的意思是：**渲染过程可以被更高优先级的更新"插队"**。

**startTransition**

```jsx
import { startTransition } from 'react';

// 紧急更新：用户输入的搜索结果要立刻显示
setInputValue(e.target.value);

// 过渡更新：搜索结果列表可以延迟渲染
startTransition(() => {
  setSearchQuery(e.target.value);
});
```

这种机制背后的原理是：`startTransition` 范围内的更新被标记为 `TransitionLane`，这些 Lane 的优先级低于 `SyncLane` 和 `InputContinuousLane`。当用户继续输入时，输入的更新（SyncLane）会**打断**正在进行的搜索结果渲染（TransitionLane），React 暂停 workInProgress 树的构建，保存当前进度，转而处理 SyncLane 的更新。

**useDeferredValue**

```jsx
const deferredValue = useDeferredValue(value);
```

`useDeferredValue` 允许一个值"滞后于"另一个值。当 `value` 频繁变化时，`deferredValue` 只在浏览器空闲时更新，且可以被新的 `value` 更新打断。

**Suspense 与数据获取**

并发渲染也允许 Suspense 组件在等待数据时"暂停"渲染，而不阻塞其他 UI 部分。在 Fiber 层面，当遇到 Suspense 边界时，如果子组件正在挂起（抛出 Promise），Fiber 可以在不丢弃当前 workInProgress 树的情况下暂停渲染，等数据到达后继续。

## 实战案例

### 案例一：大型表单的长列表性能优化

**场景**：一个企业级后台管理系统，表单中包含一个可以动态添加行的表格（500+ 行，每行 10 个输入框）。用户在输入时，同时需要对整张表做公式计算和实时校验。

**问题**：在 Stack Reconciler 中，每次输入导致的 setState 都会触发整棵组件树的递归 diff。即使用 `shouldComponentUpdate` 做了部分优化，表单校验的渲染依然可能阻塞用户的下一个按键。

**Fiber 方案**：

```jsx
function LargeForm() {
  const [rows, setRows] = useState(initialRows);
  const [validationMap, setValidationMap] = useState({});

  const handleCellChange = useCallback((rowIndex, colIndex, value) => {
    // 紧急更新：修改当前输入框的值
    setRows(prev => updateCell(prev, rowIndex, colIndex, value));

    // 过渡更新：全表公式计算与校验——优先级低，可以被输入打断
    startTransition(() => {
      const newValidation = recalculateValidation(rows, rowIndex, colIndex);
      setValidationMap(newValidation);
    });
  }, []);

  return (
    <table>
      {rows.map((row, i) => (
        <TableRow key={row.id} data={row} validation={validationMap[i]} />
      ))}
      <StatusBar validationSummary={validationMap} />
    </table>
  );
}
```

**效果**：用户的每次按键输入立即更新当前单元格（SyncLane），全表的公式重算被降级为 TransitionLane。用户在连续快速输入时，校验计算在每一帧的空闲时间中增量执行，不会阻塞输入。即使校验任务被中断 10 次，用户也毫无感知。

### 案例二：搜索联想下拉的并发优化

**场景**：一个搜索输入框，需要在用户输入时展示联想建议列表。请求是异步的，但渲染结果可能很大（如 200+ 条建议）。

**传统问题**：每次输入变化时，setState 触发列表重新渲染。即使采用防抖（debounce），当搜索结果返回时，渲染 200 条建议可能占用主线程 30ms 以上，导致下一次输入字符丢失。

**Fiber 方案**：

{% raw %}
```jsx
function SearchBox() {
  const [query, setQuery] = useState('');
  const [suggestions, setSuggestions] = useState([]);
  const deferredQuery = useDeferredValue(query);

  useEffect(() => {
    if (deferredQuery.length > 0) {
      fetchSuggestions(deferredQuery).then(data => {
        startTransition(() => setSuggestions(data));
      });
    }
  }, [deferredQuery]);

  const isStale = query !== deferredQuery;

  return (
    <div>
      <input value={query} onChange={e => setQuery(e.target.value)} />
      <SuggestionsList
        items={suggestions}
        style={{ opacity: isStale ? 0.5 : 1 }}
      />
    </div>
  );
}
```
{% endraw %}

**核心要点**：
- `useDeferredValue` 让 `query` 的延迟版本 `deferredQuery` 仅在空闲时更新
- 用户快速输入时，`query` 即时更新（输入框不卡顿），但 `deferredQuery` 滞后，搜索请求被节流
- `startTransition` 包裹的列表渲染可以被后续输入打断，即使渲染到一半也能保留进度

### 案例三：可中断的动画序列

**场景**：一个页面上同时运行 CSS 动画和 React 驱动的动态内容更新（如实时数据看板）。

**传统问题**：React 同步渲染导致的 layout thrashing 会破坏 CSS 动画的流畅度。即使动画是通过 CSS transform 触发的（由 GPU 合成），React 的长任务依然可能导致主线程繁忙，延迟 `requestAnimationFrame` 的执行。

**Fiber 方案**：

利用 React 18 的 `flushSync` 与 `startTransition` 的组合策略，精确控制更新优先级：

```jsx
function Dashboard() {
  const [liveData, setLiveData] = useState({});
  const [analytics, setAnalytics] = useState({});

  // 高优先级：保持动画所在的组件轻量
  useEffect(() => {
    const ws = new WebSocket('wss://api.example.com/live');
    ws.onmessage = (event) => {
      const data = JSON.parse(event.data);
      // 动画组件的更新保持高优先级，但不触发全树渲染
      flushSync(() => {
        setLiveData(data.live);
      });
    };
    return () => ws.close();
  }, []);

  // 低优先级：复杂分析报表的重新计算
  useEffect(() => {
    const intervalId = setInterval(() => {
      startTransition(() => {
        setAnalytics(computeHeavyAnalytics(liveData));
      });
    }, 3000);
    return () => clearInterval(intervalId);
  }, [liveData]);
}
```

**效果**：仪表盘中的实时指标更新（可能触发布局重绘）保持高优先级，而复杂的数据分析计算（可能涉及数千行数据的遍历）被降级到每帧空闲时逐步执行。CSS 动画在任何情况下都不会受 React 长任务影响。

## 底层原理（结合 React 源码分析）

### 从入口出发：递归的终结

React 渲染的起点是 `scheduleUpdateOnFiber`。在 Stack Reconciler 中，这里是递归调用的入口。但在 Fiber 架构中，这是一个**调度决策点**：

```javascript
// packages/react-reconciler/src/ScheduleUpdateOnFiber.js
function scheduleUpdateOnFiber(fiber, lane, eventTime) {
  // 1. 标记当前 Fiber 需要更新
  // 2. 向上遍历找到 FiberRoot
  const root = markUpdateLaneFromFiberToRoot(fiber, lane);

  // 3. 决定是否提前进入 commit 阶段
  if (lane === SyncLane) {
    // 同步更新：立即调度，不经过 Scheduler
    performSyncWorkOnRoot(root);
  } else {
    // 异步更新：放入 Scheduler 的任务队列
    scheduleCallback(priorityLevel, performConcurrentWorkOnRoot.bind(null, root));
  }
}
```

这个入口函数揭示了 Fiber 体系的核心分层：

- **Scheduler 层**：决定**何时**执行工作
- **Reconciler 层**：决定**什么**工作（beginWork → completeUnitOfWork）
- **Renderer 层**：决定**如何**更新 DOM（commitRoot）

### beginWork：Diff 的起点

```javascript
// packages/react-reconciler/src/ReactFiberBeginWork.js
function beginWork(current, workInProgress, renderLanes) {
  if (current !== null) {
    // 检查 props 和 context 是否变化
    const oldProps = current.memoizedProps;
    const newProps = workInProgress.pendingProps;

    // 如果没有变化且没有更新队列，可以 bailout
    if (oldProps === newProps && !hasContextChanged()) {
      return bailoutOnAlreadyFinishedWork(current, workInProgress, renderLanes);
    }
  }

  // 根据 tag 分发到不同的处理函数
  switch (workInProgress.tag) {
    case FunctionComponent: {
      return updateFunctionComponent(current, workInProgress, ...);
    }
    case ClassComponent: {
      return updateClassComponent(current, workInProgress, ...);
    }
    case HostComponent: {
      return updateHostComponent(current, workInProgress, ...);
    }
    // ... 其他类型
  }
}
```

**bailout** 是 Fiber 中一个极其重要的优化。如果某个组件的 props 和 state 都没有变化，beginWork 可以直接返回 null（表示无需深入该子树），整棵子树都不需要 diff。这在大型表单或列表中能跳过大量的无效工作。

### completeWork：DOM 的归约

```javascript
// packages/react-reconciler/src/ReactFiberCompleteWork.js
function completeWork(current, workInProgress, renderLanes) {
  switch (workInProgress.tag) {
    case HostComponent: {
      if (current !== null && workInProgress.stateNode != null) {
        // 更新阶段：对比新旧 props，更新 DOM 属性
        updateHostComponent(current, workInProgress, ...);
      } else {
        // 首次渲染：创建 DOM 节点
        const instance = createInstance(type, newProps, ...);
        // 将子节点 DOM 追加到当前节点
        appendAllChildren(instance, workInProgress, ...);
        workInProgress.stateNode = instance;
      }
      return null;
    }
    // ... 其他类型
  }
}
```

completeWork 的核心是 **DOM 归约**。每个 HostComponent 在 completeWork 阶段将自身的子节点的 DOM 递归地挂载到自身。由于 completeWork 的遍历顺序是从叶子节点向上回溯的，所以 DOM 的构建顺序是：先构建最深层子节点，再构建父节点。这保证了在 commit 阶段之前，workInProgress 树中每个节点都已经存储了完整的 DOM 子树。

### commitRoot：生效的瞬间

```javascript
// packages/react-reconciler/src/ReactFiberWorkLoop.js
function commitRoot(root) {
  // 1. 交换双缓冲
  const finishedWork = root.current.alternate;
  root.current = finishedWork;

  // 2. 获取副作用列表
  let nextEffect = finishedWork.firstEffect;

  // 3. before mutation 阶段
  // 执行 getSnapshotBeforeUpdate、调度 useEffect 等

  // 4. mutation 阶段（DOM 操作）
  while (nextEffect !== null) {
    const flags = nextEffect.flags;
    if (flags & Placement) {
      commitPlacement(nextEffect); // 插入或移动 DOM
    } else if (flags & Update) {
      commitUpdate(nextEffect);    // 更新 DOM 属性
    } else if (flags & Deletion) {
      commitDeletion(nextEffect);  // 删除 DOM
    }
    nextEffect = nextEffect.nextEffect;
  }

  // 5. layout 阶段
  // 执行 componentDidMount、componentDidUpdate、useLayoutEffect
}
```

commit 阶段是同步的、不可中断的。因为在 commit 阶段操作的是真实的 DOM，如果中断会导致用户看到残缺的 UI。commit 阶段分为三个子阶段：

- **before mutation**：读取 DOM 状态（getSnapshotBeforeUpdate），调度 useEffect
- **mutation**：执行 DOM 的增删改
- **layout**：执行 useLayoutEffect、componentDidMount/Update

### effectList：性能的关键

在 Stack Reconciler 中，找到哪些节点需要更新需要递归遍历整棵树。Fiber 引入了 **effectList** 来避免这个问题。

在 completeUnitOfWork 阶段，如果有副作用标记的节点会被串成一条单向链表：

```javascript
function completeUnitOfWork(unitOfWork) {
  // 将当前节点加入 effectList
  if (flags > PerformedWork) {
    if (completedWork.return !== null) {
      // 将父节点的 effectList 末尾指向当前节点
      if (completedWork.return.firstEffect === null) {
        completedWork.return.firstEffect = completedWork;
      }
      if (completedWork.return.lastEffect !== null) {
        completedWork.return.lastEffect.nextEffect = completedWork;
      }
      completedWork.return.lastEffect = completedWork;
    }
  }
}
```

commit 阶段只需要遍历 effectList 这条链表，而不需要遍历整棵 Fiber 树。在只有 5 个节点更新的情况下，effectList 就只包含 5 个节点，不论整棵树有 5000 还是 50000 个节点。这是 Fiber 架构在性能层面非常有效的优化手段。

## 高频面试题解析

### Q1：Fiber 架构解决了什么核心问题？

Fiber 解决了 React 15（Stack Reconciler）中**递归渲染不可中断**的问题。具体来说：

- **不可中断 → 可中断**：遍历从递归调用栈改为链表结构，支持在任意节点暂停/恢复
- **无优先级 → 有优先级**：引入 Lane 模型，区分高优先级（用户交互）和低优先级（后台数据、预加载）
- **同步独占 → 并发协作**：结合 Scheduler 的时间切片，在每一帧的空闲时间中执行渲染工作，避免阻塞主线程

**面试加分点**：提到 Fiber 不仅是为了"异步渲染"，更是为了给 React 引入**调度能力（Scheduling）**——让框架拥有"决策什么先做、什么后做"的能力。

### Q2：Fiber 节点中的 return 和 parent 有什么区别？

Fiber 节点中只有 `return` 而没有 `parent`。`return` 语义上表示"处理完当前节点后应该返回的父 Fiber"，强调的是遍历流程中的控制流关系。这个命名体现了 Fiber 的核心设计：它首先是一个**可遍历的工作单元**，其次才是树结构中的节点。不同于传统的树结构使用 `parent` 表示静态的从属关系，`return` 天然适合深度优先遍历中的回溯需求。

### Q3：React 怎样实现时间切片？（手写伪代码）

主要依靠 Scheduler 中的 workLoop：

```javascript
function workLoopConcurrent() {
  while (workInProgress !== null && !shouldYield()) {
    workInProgress = performUnitOfWork(workInProgress);
  }
}

function shouldYield() {
  // 检查是否超过当前帧的时间预算
  return getCurrentTime() - startTime > yieldInterval;
}
```

Scheduler 使用 `MessageChannel` 的宏任务特性来实现切片：每次消息事件处理完一个切片后，通过 `postMessage` 触发下一次消息，给浏览器插入渲染、事件处理的空间。

### Q4：双缓冲树机制是怎么工作的？

React 维护两棵 Fiber 树：
- **current 树**：映射当前屏幕内容，首次渲染后构建
- **workInProgress 树**：基于本次更新构建，完成后通过 `root.current = finishedWork` 切换

这种模式带来三大好处：
1. **中断安全**：workInProgress 构建中断时，current 树不受影响
2. **节点复用**：通过 `createWorkInProgress` 复用 current 中的 Fiber 对象，减少对象创建
3. **原子提交**：所有 DOM 变更统一在 commit 阶段生效

### Q5：什么是 Lane 模型？它比过期时间模型强在哪？

Lane 模型使用 32 位二进制位掩码表示优先级：

- **更高的表达能力**：位运算支持多个 Lane 的合并、比较、移除，天然支持多任务并行
- **更强的批处理能力**：同一事件循环中的多次 setState，对应的 Lane 通过按位或合并，自动触发批量渲染
- **更好的饥饿预防**：可以通过 Lane 的 Starvation 标志检测长时间未被处理的任务
- **效率更高**：位运算的 O(1) 复杂度优于 ExpirationTime 中最小堆的 O(log n)

## 总结与扩展

### 核心要义回顾

Fiber 架构通过三个层面的重新设计，彻底改变了 React 的渲染能力：

| 层面 | 从 | 到 | 关键机制 |
|------|----|----|---------|
| 数据结构 | 递归调用栈 | Fiber 链表 | child/sibling/return 的螺旋遍历 |
| 调度策略 | 同步执行 | 协作式调度 | Scheduler + 时间切片 |
| 优先级 | 无 | Lane 位掩码模型 | 32 位二进制位，支持并发更新 |

### 对前端开发的深远影响

Fiber 架构的意义超越了 React 本身。它代表了前端框架在调度层面的一次范式升级：

1. **从"声明式 UI"到"声明式调度"**：开发者不仅要声明 UI 长什么样，还要声明什么更新是紧急的、什么是可延迟的
2. **从"渲染完整性"到"响应优先"**：渲染不必一次完成，可以分片增量完成，用户交互优先
3. **前端框架进入"调度器"竞争时代**：React Fiber + Scheduler、Vue 3 的 Scheduler、Svelte 等都在调度层面发力
