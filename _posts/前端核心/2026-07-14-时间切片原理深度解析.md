## 一句话概括

时间切片（Time Slicing）是 React Fiber 架构的基石技术，它将长耗时的渲染任务拆解为多个 5ms 左右的小片段，利用浏览器帧空闲时间逐步执行，从而避免阻塞主线程，保持页面流畅响应。

## 背景与意义

### 为什么需要时间切片？

在 React 16 之前的 Stack Reconciler 时代，React 的 Reconciliation（协调）过程是**同步递归**的。一旦开始更新，整个虚拟 DOM 树的 Diff 比较必须一次性完成，中途无法中断。这意味着：

- 如果一个组件树的节点数达到数万个，单次更新可能耗时 50ms 甚至 100ms+
- 浏览器的主线程被占满，无法处理用户输入、动画帧、滚动事件
- 用户会明显感知到页面卡顿、点击无响应

以 60fps 为目标，浏览器每一帧的可用时间约 **16.6ms**。如果渲染任务占用过长，帧率就会下降，出现视觉卡顿（Jank）。

### 时间切片的解决思路

时间切片的核心思想很朴素：**把一个大任务切成很多小任务，在每一帧的空闲时间去执行一小块，既不侵占浏览器处理用户交互的时间，又保证任务最终能完成。**

这只是将 JavaScript 的执行从"一次性大块"变成"多次小块"，但给用户带来的体验提升是巨大的——页面始终保持流畅响应。

## 概念与定义

### 时间切片（Time Slicing）

指将一个大型同步渲染任务分解为多个可中断的微型工作单元（unit of work），每个单元执行完毕后检查是否还剩余帧时间，如果时间耗尽则主动让出主线程。

### requestIdleCallback

浏览器原生提供的 API，允许开发者在浏览器空闲时段注册回调函数。React 的调度器基于此 API 思想来实现任务的时间切片。

```typescript
// 浏览器原生 API 签名
interface IdleDeadline {
  didTimeout: boolean;       // 是否超时（必须执行）
  timeRemaining(): DOMHighResTimeStamp;  // 当前帧剩余空闲时间
}

window.requestIdleCallback(
  callback: (deadline: IdleDeadline) => void,
  options?: { timeout: number }
);
```

### 帧预算（Frame Budget）

每一帧中用于 JavaScript 执行的最大预算时间。React 默认设置为 **5ms**，即每个时间切片执行后检查是否超过 5ms，超过则让出主线程。

### shouldYield

React Scheduler 中用于判断是否需要让出主线程的核心方法。

```typescript
function shouldYield(): boolean {
  const currentTime = getCurrentTime();
  // 如果帧开始后的执行时间超过了 5ms 预算，则让出
  if (currentTime - frameDeadline > 5) {
    return true;
  }
  return false;
}
```

## 最小示例

通过一个简单示例感受时间切片的威力——渲染 10000 个节点：

```jsx
import React, { useState } from 'react';

function HeavyList() {
  const [items] = useState(() => {
    return Array.from({ length: 10000 }, (_, i) => ({
      id: i,
      text: `Item #${i}`,
    }));
  });

  return (
    <div style={{ height: '300px', overflow: 'auto' }}>
      {items.map(item => (
        <div key={item.id}>
          <span>{item.text}</span>
          <span style={{ marginLeft: 8, color: '#666' }}>
            {Math.random().toString(36).slice(2, 8)}
          </span>
        </div>
      ))}
    </div>
  );
}
```

在 Stack Reconciler 下，上述组件首次渲染会一次性遍历 10000 个 Fiber 节点。而在 Fiber + 时间切片下，Scheduler 会将这个遍历过程拆解为多个小片段，每执行约 5ms 就暂停一下，检查是否有更高优先级的用户事件需要处理。

你可以打开 Chrome DevTools Performance 面板来验证：在时间切片模式下，Main 线程的执行不再是单一的"长条"，而是由多个"短条"组成，中间穿插着浏览器处理输入事件的空闲窗口。

## 核心知识点拆解

### 1. 帧的生命周期与时间切片槽位

浏览器一帧的标准流程：

```
输入事件处理 → requestAnimationFrame → 样式计算 → 布局 → 绘制 → 复合 → 空闲时间
                                           ^                         ^
                                           |                         |
                                   主线程繁忙区域               requestIdleCallback 执行时机
```

React 的时间切片利用了**帧末尾的空闲时间段**。Scheduler 通过 `requestAnimationFrame` 在每一帧开始时注册回调，记录帧开始时间戳，从而计算出当前帧的 deadline：

```
帧理想时长: 1000 / 60 ≈ 16.67ms
5ms 切片预算 → 剩余约 11ms 留给浏览器布局和绘制
```

### 2. Scheduler 的工作循环

React Scheduler 维护了一个**任务队列（taskQueue）**，每个任务都关联一个优先级（过期时间）。调度器的核心是 `workLoop`：

```typescript
// 简化版 Scheduler 工作循环
function workLoop(hasTimeRemaining: boolean, initialTime: number) {
  let currentTask = peek(taskQueue);
  
  while (currentTask !== null) {
    // 检查是否有更高优先级任务插入
    if (currentTask.expirationTime > currentTime && shouldYield()) {
      // 时间片用完，让出，等待下一帧继续
      break;
    }
    
    // 执行当前任务的一个切片
    const continuationCallback = currentTask.callback(
      didUserCallbackTimeout(currentTask.expirationTime <= currentTime)
    );
    
    if (typeof continuationCallback === 'function') {
      // 任务未完成，继续执行
      currentTask.callback = continuationCallback;
    } else {
      // 任务完成，从队列移除
      pop(taskQueue);
    }
    
    currentTask = peek(taskQueue);
  }
  
  if (currentTask !== null) {
    // 还有未完成任务，请求下一次空闲回调
    requestHostCallback(flushWork);
  }
}
```

### 3. isInputPending API

Chrome 在 87 版本后引入了 `navigator.scheduling.isInputPending()` API，允许 JavaScript 在执行过程中主动检查是否有 pending 的用户输入事件。React Scheduler 后来也集成了此 API，进一步提升响应灵敏度：

```typescript
function shouldYieldToHost() {
  const timeElapsed = getCurrentTime() - startTime;
  if (timeElapsed < frameInterval) {
    // 帧时间预算内
    return false;
  }
  
  // 检查是否有用户输入等待处理
  if (typeof navigator !== 'undefined' && 
      navigator.scheduling?.isInputPending?.()) {
    return true;
  }
  
  return true; // 超过预算且无输入也要让出
}
```

## 实战案例（含完整代码）

### 案例：可中断的大列表搜索

构建一个带有搜索过滤功能的大列表，展示时间切片如何防止搜索时的界面卡顿：

```jsx
import React, { useState, useDeferredValue, useMemo } from 'react';

// 模拟一个大数据集
function generateLargeDataset(size = 50000) {
  const data = [];
  for (let i = 0; i < size; i++) {
    data.push({
      id: i,
      name: `User_${i}`,
      email: `user${i}@example.com`,
      department: ['工程', '产品', '设计', '市场'][Math.floor(Math.random() * 4)],
      score: Math.floor(Math.random() * 100),
    });
  }
  return data;
}

const allData = generateLargeDataset();

function SearchResult({ query, data }) {
  // 使用 useDeferredValue 标记为可延迟更新
  const deferredQuery = useDeferredValue(query);
  const isStale = query !== deferredQuery;

  const filteredData = useMemo(() => {
    if (!deferredQuery.trim()) return data.slice(0, 200);
    return data
      .filter(item =>
        item.name.toLowerCase().includes(deferredQuery.toLowerCase()) ||
        item.email.toLowerCase().includes(deferredQuery.toLowerCase())
      )
      .slice(0, 200);
  }, [deferredQuery, data]);

  return (
    <div style={{ opacity: isStale ? 0.6 : 1 }}>
      <p>{isStale ? '⏳ 搜索中...' : `找到 ${filteredData.length} 条结果`}</p>
      <div style={{ display: 'flex', flexDirection: 'column', gap: 2 }}>
        {filteredData.map(item => (
          <div key={item.id} style={{
            padding: '4px 8px',
            background: item.score > 80 ? '#e6f7e6' : '#fff',
            borderRadius: 4,
          }}>
            <strong>{item.name}</strong> — {item.email}
            <span style={{ marginLeft: 12, color: '#999' }}>[{item.department}]</span>
          </div>
        ))}
      </div>
    </div>
  );
}

export default function SearchableList() {
  const [query, setQuery] = useState('');

  return (
    <div style={{ maxWidth: 600, margin: '0 auto', padding: 20 }}>
      <h2>🔍 搜索用户（支持时间切片）</h2>
      <input
        type="text"
        value={query}
        onChange={e => setQuery(e.target.value)}
        placeholder="输入用户名或邮箱搜索..."
        style={{
          width: '100%', padding: '8px 12px', fontSize: 16,
          border: '1px solid #ddd', borderRadius: 6,
        }}
      />
      <SearchResult query={query} data={allData} />
    </div>
  );
}
```

**关键点分析：**

- `useDeferredValue` 是 React 18 提供的时间切片友好 API，它将下游渲染标记为低优先级
- 用户在输入时，`query` 状态立刻更新保证输入框流畅，但 `SearchResult` 中的过滤逻辑会延迟执行
- `isStale` 状态让 UI 给出视觉反馈："搜索进行中……"
- 如果用户持续输入，React 会中断上一次未完成的搜索渲染，转而处理最新状态的渲染，避免了无用计算

| 指标 | Stack Reconciler | Fiber + 时间切片 |
|------|------------------|-------------------|

## 底层原理（含源码分析）

### 1. requestHostCallback 与 Message Channel

React Scheduler 并没有直接使用 `requestIdleCallback`，原因是：
- `requestIdleCallback` 的浏览器兼容性在 React 16 时代不够好
- `requestIdleCallback` 的触发频率较低，通常只有 20-50fps

React 采用了一套巧妙的**降级方案**：优先尝试 `setImmediate`（Node/IE），然后是 `MessageChannel`，最后回退到 `setTimeout(fn, 0)`。

```typescript
// 源码位置: packages/scheduler/src/forks/SchedulerHostConfig.default.js

let schedulePerformWorkUntilDeadline;
if (typeof setImmediate === 'function') {
  // IE / Node.js
  schedulePerformWorkUntilDeadline = () => {
    setImmediate(performWorkUntilDeadline);
  };
} else if (typeof MessageChannel !== 'undefined') {
  // 主流浏览器
  const channel = new MessageChannel();
  const port = channel.port2;
  channel.port1.onmessage = performWorkUntilDeadline;
  schedulePerformWorkUntilDeadline = () => {
    port.postMessage(null);
  };
} else {
  // 回退方案
  schedulePerformWorkUntilDeadline = () => {
    setTimeout(performWorkUntilDeadline, 0);
  };
}
```

**为什么 MessageChannel 优于 setTimeout 优先？**

`setTimeout(fn, 0)` 在浏览器中有 4ms 的最小延迟（嵌套层数超过 5 次后），而 `MessageChannel` 的回调可以在当前帧的空闲期立刻执行，延迟接近 0ms。

### 2. performWorkUntilDeadline 的实现

这是时间切片的核心入口：

```typescript
// 源码简化
function performWorkUntilDeadline() {
  if (scheduledHostCallback !== null) {
    const currentTime = getCurrentTime();
    // 计算本帧的 deadline
    deadline = currentTime + yieldInterval;  // yieldInterval 默认为 5ms
    const hasTimeRemaining = true;
    
    try {
      const hasMoreWork = scheduledHostCallback(hasTimeRemaining, currentTime);
      
      if (!hasMoreWork) {
        // 所有任务完成，重置状态
        isMessageLoopRunning = false;
        scheduledHostCallback = null;
      } else {
        // 还有未完成任务，postMessage 触发下一次切片
        port.postMessage(null);
      }
    } catch (error) {
      port.postMessage(null);
      throw error;
    }
  } else {
    isMessageLoopRunning = false;
  }
}
```

关键流程：
1. 每次 `postMessage` 触发一次 `performWorkUntilDeadline`
2. 计算当前切片的 deadline = `currentTime + 5ms`
3. 调用 `scheduledHostCallback`（即 `flushWork`）开始工作循环
4. 如果 `flushWork` 返回 `true`（还有工作未完成），立刻再次 `postMessage` 申请下一次执行
5. 不断重复直到所有工作完成

### 3. 5ms 为何是理想的切片大小？

5ms 这个数字不是随意选取的。其推导逻辑如下：

```
60fps 帧预算: 1000/60 ≈ 16.67ms
预留渲染时间: 约 10-11ms（布局 + 绘制 + 复合）
JavaScript 预算: 16.67 - 10 ≈ 5-6ms
```

这 5ms 的切片大小确保了：
- 每帧至少有 10ms 留给浏览器进行样式计算、布局和绘制
- 即使 FPS 高于 30，每一帧的主线程空闲量也充足
- 如果用户交互密集，React 在不断 yield 的过程中能及时响应输入

### 4. 与 requestIdleCallback 的对比

```typescript
// React 时间切片 vs 原生 requestIdleCallback
// 原生 rIC —— 被动等待浏览器空闲
requestIdleCallback(myWork, { timeout: 3000 });
// 浏览器决定是否触发，频率不可控

// React 方案 —— 主动申请执行
postMessage(null);  // 触发 performWorkUntilDeadline
// 每次切片完成后立即申请下一次，形成一个"主动的微任务链"
```

React 的方案**频率更高，更主动**。原生 rIC 的触发频率约为 20-50fps，而 React 的 MessageChannel 方案可以达到接近 60fps 的执行频率。

## 高频面试题解析

### Q1：React 时间切片为什么没有直接使用 requestIdleCallback？

**参考答案：**

1. **兼容性问题**：requestIdleCallback 在 React 16 发布时（2017年）的浏览器兼容性不足，Safari 至今未支持
2. **触发频率不足**：rIC 的触发频率较低，通常在 20-50fps 之间，而 React 需要更密集的调度颗粒度来保证 60fps 的渲染体验
3. **无法精确控制**：rIC 的回调时机完全由浏览器决定，React 无法主动调度下一个工作单元的时机

React 采用 MessageChannel + requestAnimationFrame 的方案，实现了更主动、更高频的时间切片调度。

### Q2：时间切片会导致 React 更新变慢吗？

**不能简单回答"变慢"或"变快"，需要区分两个维度：**

- **总耗时**：时间切片不会减少 CPU 总消耗。实际上由于上下文切换的开销，总完成时间可能略长于同步执行
- **感知体验**：时间切片将任务拆解，让浏览器每帧都有空闲处理输入事件和渲染，用户的感知流畅度**显著提升**

> 类比：做一顿饭需要 30 分钟，但你可以一次做完所有事（同步）——中间 30 分钟不能接电话。也可以每个步骤之间穿插接电话（时间切片）——总用时变成 33 分钟，但全程可接电话。用户会觉得后者"体验更好"。

### Q3：时间切片在 concurrent 模式下是如何支持 useTransition 和 useDeferredValue 的？

**参考答案：**

`useTransition` 和 `useDeferredValue` 是 React 18 内置的并发特性，它们的工作原理与时间切片深度绑定：

1. 当通过 `startTransition` 包裹状态更新时，该更新被标记为 **Transition 优先级**（低优先级）
2. Scheduler 在时间切片循环中始终优先处理高优先级任务（如输入事件触发的更新）
3. 低优先级的 Transition 更新只在帧空闲时段逐步推进
4. 如果高优先级任务插队，低优先级任务会被**中断**，其 Fiber 树切换回未完成的中间态
5. 高优先级任务完成后，React 会重新从头开始低优先级任务（因为状态已经变了）

这种"中断重启"模式确保了高优先级更新始终优先执行，这也是 Concurrent 模式的核心价值。

## 总结与扩展

### 总结

时间切片是 React Fiber 架构的基石，它通过将渲染任务拆解为 5ms 级别的小切片，利用 MessageChannel 在每一帧的空闲时段逐步执行，实现了：

- **可中断渲染**：大型渲染任务可以在任意切片后暂停
- **优先级抢占**：高优先级任务可以打断低优先级任务的执行
- **持续流畅**：主线程始终保留足够的空闲时间处理用户交互

### 扩展：时间切片与其他框架

Vue 3 的异步渲染策略与 React 有所不同。Vue 3 的虚拟 DOM 更新仍然是同步的（通过 Proxy 响应式系统的细粒度追踪，Vue 在大多数字典中不需要时间切片也能达到高性能）。但在需要的地方，Vue 3 通过 `v-memo` 和 Suspense 实现异步更新策略。

Solid.js 等细粒度响应式框架则完全不需要时间切片——它们直接更新到具体 DOM 节点，不存在大型虚拟 DOM 树遍历的问题。

**时间切片不是万能的**，它解决的是"大型虚拟 DOM Diff"场景下的卡顿问题。如果你的应用组件树很小（<1000 个节点），时间切片带来的好处微乎其微。但如果是中大型应用，时间切片就是 React 体验的守护者。
