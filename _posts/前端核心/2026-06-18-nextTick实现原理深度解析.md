## 一句话概括

`nextTick` 是 Vue 框架中用于在 **下一次 DOM 更新循环结束之后** 执行延迟回调的异步微任务机制，它利用 JavaScript 事件循环（Event Loop）的微任务队列，将回调推迟到当前同步代码执行完毕、DOM 完成批量更新后再执行，是 Vue 响应式系统与异步更新策略的"最后一公里"连接器。

## 背景与意义

### 为什么需要 nextTick？

Vue 的响应式系统是一个经典的"依赖追踪 + 异步更新"模型。当数据发生变化时，Vue 并不会立即更新 DOM，而是将更新操作推入一个异步更新队列，然后在下一次"tick"中批量执行。这种设计带来了显著的性能收益：

```javascript
// 伪代码：如果没有异步更新，每次修改都会触发 DOM 更新
data.count = 1 // 立即触发 DOM 更新 —— 频繁！
data.count = 2 // 再次触发 —— 频繁！
data.count = 3 // 再次触发 —— 频繁！
```

而在 Vue 的实际设计中，以上三次修改只会触发 **一次** DOM 更新。这意味着在同一个 tick 内的所有数据变更都会被合并，只在当前同步代码执行完毕后统一执行一次 DOM 批量更新。

```javascript
// Vue 实际行为：异步批量更新
data.count = 1 // 推入更新队列
data.count = 2 // 合并到同一个更新任务
data.count = 3 // 合并到同一个更新任务
// 同步代码执行完毕 → 批量执行一次 DOM 更新
```

这个异步更新机制带来一个问题：**如果开发者需要在 DOM 更新后操作真实 DOM，应该怎么办？**

```vue
<template>
  <div ref="myDiv">{{ message }}</div>
</template>

<script setup>
import { ref, nextTick } from 'vue'

const message = ref('hello')
const myDiv = ref(null)

message.value = 'world'

// ❌ 此时 DOM 尚未更新，myDiv.value.textContent 仍然是 'hello'
console.log(myDiv.value.textContent) // 'hello'

// ✅ 使用 nextTick 等待 DOM 更新
nextTick(() => {
  console.log(myDiv.value.textContent) // 'world'
})
</script>
```

这就是 `nextTick` 存在的核心意义：**在数据变化后、DOM 更新完成后的恰当时机执行回调**。

### Vue 2 → Vue 3 的演进

Vue 2 中 `nextTick` 位于 `src/core/util/next-tick.js`，使用了一系列降级策略：
- `Promise.then`（微任务）
- `MutationObserver`（微任务）
- `setImmediate`（宏任务，IE 专用）
- `setTimeout`（宏任务，兜底）

Vue 3 重写了 `nextTick`，整体设计更加简洁、可维护，核心逻辑集中在 `packages/runtime-core/src/scheduler.ts` 中。最大的变化是：**Vue 3 仅使用 `Promise` 作为微任务实现**，不再需要复杂的降级策略。这是因为 Vue 3 不再支持 IE 11 及更低版本浏览器，而现代浏览器对 `Promise` 的支持已经非常完

## 概念与定义

### Tick（滴答）与 Event Loop

要理解 `nextTick`，首先要理解 JavaScript 的事件循环机制。JavaScript 是单线程语言，其异步执行依赖事件循环（Event Loop）：

```
┌───────────────────────────────────────┐
│               Event Loop              │
├───────────────────────────────────────┤
│                                       │
│  ┌─────────────────────────────┐      │
│  │         Call Stack          │      │
│  │   (执行同步代码，如栈)        │      │
│  └─────────────────────────────┘      │
│                   ↓                    │
│  ┌─────────────────────────────┐      │
│  │     Microtask Queue         │      │
│  │   (Promise.then,           │      │
│  │    MutationObserver)        │      │
│  └─────────────────────────────┘      │
│                   ↓                    │
│  ┌─────────────────────────────┐      │
│  │    Macrotask Queue          │      │
│  │  (setTimeout, setInterval,  │      │
│  │   I/O, UI渲染)              │      │
│  └─────────────────────────────┘      │
│                                       │
└───────────────────────────────────────┘
```

一个 **tick** 在事件循环中对应一个完整的循环周期：

1. 从宏任务队列取出一个任务执行
2. 执行过程中产生的微任务排入微任务队列
3. 同步代码执行完毕后，清空整个微任务队列
4. 可能触发 UI 渲染
5. 进入下一轮宏任务循环

Vue 的 `nextTick` 正是利用"微任务队列清空"这个时间点，在 DOM 更新完成后执行回调。

### 异步更新队列（Async Update Queue）

Vue 3 中维护了一套异步更新队列，核心概念包括：

| 概念 | 说明 |
|------|------|
| **queue** | 待执行的更新任务（effect/副作用函数）队列 |
| **pendingFlush** | 标记是否已发起刷新任务 |
| **flushIndex** | 当前正在执行的任务索引，用于暂停/恢复 |
| **postFlushCb** | `nextTick` 回调存储队列（后置刷新回调） |
| **preFlushCb** | 前置刷新回调队列 |

### nextTick 的本质

从数据结构上说，`nextTick` 本质上是一个 **微任务包装器（microtask wrapper）**。它将回调函数推入 `pendingPostFlushCbs` 队列，然后在当前微任务中异步执行：

```typescript
// Vue 3 源码中 nextTick 的简化逻辑
const pendingPostFlushCbs: Function[] = []
let flushIndex = 0
let isFlushPending = false
let isFlushScheduled = false

// 核心 nextTick 函数
export function nextTick<T = void>(
  this: T,
  fn?: (this: T) => void
): Promise<void> {
  const p = currentFlushPromise || resolvedPromise
  return fn ? p.then(this ? fn.bind(this) : fn) : p
}
```

## 核心知识点拆解

### 1. queueJob 机制：任务的入队与去重

`queueJob` 是 Vue 3 异步更新系统的入口函数，负责将更新任务（effect）推入队列：

```typescript
// packages/runtime-core/src/scheduler.ts
export function queueJob(job: SchedulerJob) {
  // 核心逻辑：去重 + 入队
  if (
    !queue.length ||
    !queue.includes(
      job,
      isFlushing && job.allowRecurse ? flushIndex + 1 : flushIndex
    )
  ) {
    if (job.id == null) {
      queue.push(job)
    } else {
      // 根据 job.id 保持队列的有序性
      queue.splice(findInsertionIndex(job.id), 0, job)
    }
  }
  
  // 触发刷新调度
  queueFlush()
}
```

**关键设计点：**

1. **去重**：通过 `queue.includes()` 检查同一任务是否已入队，避免重复执行。
2. **有序插入**：如果 job 有 `id` 属性（组件更新 job），按 id 升序插入，保证父组件优先于子组件更新。
3. **递归保护**：当 `job.allowRecurse` 为 true 且正在刷新时，从当前位置之后查找，防止无限递归。

### 2. queueFlush 机制：调度刷新

`queueFlush` 负责启动异步刷新流程：

```typescript
export function queueFlush() {
  if (!isFlushing && !isFlushPending) {
    isFlushPending = true
    // 使用 Promise 创建微任务
    currentFlushPromise = resolvedPromise.then(flushJobs)
  }
}
```

**核心要点：**

- 使用 `resolvedPromise.then()` 将 `flushJobs` 包装为微任务。
- `isFlushPending` 标志位确保在同一 tick 内只调度一次刷新。
- `isFlushing` 标志位标记当前是否正在执行刷新，用于刷新过程中新增任务的递归处理。

### 3. flushJobs 流程：完整的刷新执行

`flushJobs` 是整个异步更新系统的核心执行函数，执行流程如下：

```typescript
export function flushJobs(seen?: CountMap) {
  isFlushPending = false
  isFlushing = true
  
  if (__DEV__) seen = seen || new Map()
  
  // 第一步：先执行所有 pre flush 回调
  flushPreFlushCbs(seen)
  
  // 第二步：循环执行组件更新 job（支持刷新过程中新增 job）
  try {
    for (
      flushIndex = 0;
      flushIndex < queue.length;
      flushIndex++
    ) {
      const job = queue[flushIndex]
      if (job && job.active !== false) {
        if (__DEV__) {
          // 开发环境检测循环更新
          checkRecursiveUpdates(seen!, job)
        }
        // 执行组件的更新 effect
        callWithErrorHandling(job, null, ErrorCodes.SCHEDULER)
      }
    }
  } finally {
    // 第三步：重置状态
    flushIndex = 0
    queue.length = 0
    
    // 第四步：执行后置 post flush 回调（nextTick 回调）
    flushPostFlushCbs(seen)
    
    isFlushing = false
    
    // 第五步：递归检查——刷新过程中可能又产生了新任务
    if (
      queue.length > 0 ||
      pendingPostFlushCbs.length > 0
    ) {
      flushJobs(seen)
    }
  }
}
```

**完整的执行顺序：**

```
同步代码执行
  ↓
Promise.then（微任务调度）
  ↓
flushPreFlushCbs（前置回调，如 watch 的 flush: 'pre'）
  ↓
flushJobs —— 遍历 queue 执行所有组件更新 effect
  ↓                        ↑
  └── 执行过程中的新增 job → 继续处理（循环直到队列清空）
  ↓
flushPostFlushCbs（后置回调，即 nextTick 注册的回调）
  ↓
重置 isFlushing = false
  ↓
递归检查是否有新增任务
```

### 4. nextTick 函数：Promise 化接口

```typescript
// Vue 3 源码
const resolvedPromise = Promise.resolve() as Promise<any>

export function nextTick<T = void>(
  this: T,
  fn?: (this: T) => void
): Promise<void> {
  const p = currentFlushPromise || resolvedPromise
  return fn ? p.then(this ? fn.bind(this) : fn) : p
}
```

**设计要点：**

- 持有对 `currentFlushPromise` 的引用，当存在正在执行的刷新时，`nextTick` 的回调会自动排在当前刷新的 `then` 链之后。
- 如果没有正在进行的刷新，使用 `resolvedPromise` 作为默认微任务。
- 支持 `fn` 存在/不存在两种调用方式：
  - `nextTick(fn)` → 返回 Promise，fn 在 DOM 更新后执行
  - `await nextTick()` → 直接等待 DOM 更新

### 5. flushPostFlushCbs：后置回调的执行

```typescript
export function flushPostFlushCbs(seen?: CountMap) {
  if (pendingPostFlushCbs.length > 0) {
    // 1. 复制并清空原始队列（支持递归添加）
    const deduped = [...new Set(pendingPostFlushCbs)]
    pendingPostFlushCbs.length = 0
    
    // 2. 如果当前已经有后置回调在执行，追加到现有队列
    if (activePostFlushCbs) {
      activePostFlushCbs.push(...deduped)
      return
    }
    
    // 3. 执行所有回调
    activePostFlushCbs = deduped
    if (__DEV__) seen = seen || new Map()
    
    for (
      activePostFlushIndex = 0;
      activePostFlushIndex < activePostFlushCbs.length;
      activePostFlushIndex++
    ) {
      if (__DEV__) {
        checkRecursiveUpdates(seen!, activePostFlushCbs[activePostFlushIndex])
      }
      activePostFlushCbs[activePostFlushIndex]()
    }
    
    // 4. 清理
    activePostFlushCbs = null
    activePostFlushIndex = 0
  }
}
```

### 6. 降级策略的历史演变

Vue 2 中使用的降级策略是一道经典的面试题，虽然 Vue 3 已不再需要：

```javascript
// Vue 2 的 nextTick 降级策略（面试高频）
export function nextTick(cb, ctx) {
  let _resolve
  callbacks.push(() => {
    if (cb) {
      try {
        cb.call(ctx)
      } catch (e) {
        handleError(e, ctx, 'nextTick')
      }
    } else if (_resolve) {
      _resolve(ctx)
    }
  })
  
  if (!pending) {
    pending = true
    timerFunc()
  }
  
  // 非回调模式返回 Promise
  if (!cb && typeof Promise !== 'undefined') {
    return new Promise(resolve => { _resolve = resolve })
  }
}

// Vue 2 的 timerFunc 降级
let timerFunc

// Level 1: Promise（微任务）
if (typeof Promise !== 'undefined' && isNative(Promise)) {
  const p = Promise.resolve()
  timerFunc = () => {
    p.then(flushCallbacks)
    // iOS UIWebView 中 Promise.then 可能被 UI 渲染阻塞
    // 使用空 setTimeout 强制刷新
    if (isIOS) setTimeout(noop)
  }
  isUsingMicroTask = true
}
// Level 2: MutationObserver（微任务，H5 API）
else if (!isIE && typeof MutationObserver !== 'undefined' && (
  isNative(MutationObserver) ||
  MutationObserver.toString() === '[object MutationObserverConstructor]'
)) {
  let counter = 1
  const observer = new MutationObserver(flushCallbacks)
  const textNode = document.createTextNode(String(counter))
  observer.observe(textNode, { characterData: true })
  timerFunc = () => {
    counter = (counter + 1) % 2
    textNode.data = String(counter)
  }
  isUsingMicroTask = true
}
// Level 3: setImmediate（宏任务，Node.js 和 IE Edge）
else if (typeof setImmediate !== 'undefined' && isNative(setImmediate)) {
  timerFunc = () => setImmediate(flushCallbacks)
}
// Level 4: setTimeout（宏任务，兜底）
else {
  timerFunc = () => setTimeout(flushCallbacks, 0)
}
```

**Vue 3 的简化选择：**

Vue 3 仅保留 `Promise` 方案，原因如下：
- 目标浏览器环境移除 IE11 及更低版本后，Promise 已得到全面支持。
- Vue 3 使用 TypeScript 和 ESM 构建，默认面向现代浏览器。
- Promise 在微任务优先级方面的一致性最佳。
- `MutationObserver` 在非 DOM 环境（如 SSR 和 Web Worker）不可用，而 Promise 普遍可用。

## 底层原理

### 完整调用链路

从数据变化到 `nextTick` 回调执行的完整链路：

```typescript
// 1. 修改响应式数据
count.value++

// 2. 触发 setter → 通知依赖
// src/reactivity/src/reactive.ts（简化）
set(target, key, value) {
  const result = Reflect.set(target, key, value)
  trigger(target, key)
  return result
}

// 3. trigger → 调度 effect
// src/reactivity/src/effect.ts（简化）
export function trigger(target, key) {
  const depsMap = targetMap.get(target)
  if (!depsMap) return
  
  const effects = depsMap.get(key)
  if (effects) {
    for (const effect of effects) {
      if (effect.scheduler) {
        // 组件渲染 effect 有自定义 scheduler
        effect.scheduler()  // → queueJob(effect)
      } else {
        effect.run()
      }
    }
  }
}

// 4. queueJob → 入队并启动调度
export function queueJob(job) {
  // 去重后入队
  if (!queue.includes(job)) {
    queue.push(job)
  }
  queueFlush()
}

// 5. queueFlush → 调度微任务
function queueFlush() {
  if (!isFlushPending && !isFlushing) {
    isFlushPending = true
    currentFlushPromise = resolvedPromise.then(flushJobs)
  }
}

// 6. flushJobs → 批量执行更新
function flushJobs() {
  // ...执行所有 queue 中的 job
  // ...执行 post flush callbacks（nextTick 回调在这里触发）
}

// 7. nextTick 回调在 flushPostFlushCbs 中被执行
function flushPostFlushCbs() {
  const cbs = [...new Set(pendingPostFlushCbs)]
  for (const cb of cbs) {
    cb()
  }
}
```

### 深入 flushJobs 的同步特性

`flushJobs` 虽然是微任务调度，但它在执行 **组件更新的同步代码** 时是同步的：

```typescript
export function flushJobs(seen?: CountMap) {
  isFlushPending = false
  isFlushing = true
  
  // flushPreFlushCbs 同步执行
  flushPreFlushCbs(seen)
  
  // 这个 for 循环是同步的
  for (flushIndex = 0; flushIndex < queue.length; flushIndex++) {
    const job = queue[flushIndex]
    if (job && job.active !== false) {
      callWithErrorHandling(job, null, ErrorCodes.SCHEDULER)
      // job 的执行也是同步的
    }
  }
  
  // 清空队列（同步）
  flushIndex = 0
  queue.length = 0
  
  // 执行后置回调（同步）
  flushPostFlushCbs(seen)
  
  isFlushing = false
}
```

这意味着 **一旦 `flushJobs` 开始执行，它会一口气把所有任务处理完**，不会中断。这也是为什么 Vue 3 能保证在同一微任务内完成所有 DOM 更新和 `nextTick` 回调的执行。

### Promise.then 的微任务特性

Vue 3 的核心调度依赖 `Promise.then` 创建一个微任务：

```javascript
// 创建已兑现的 Promise
const resolvedPromise = Promise.resolve()

// 每次调度刷新时，创建一个新的微任务
currentFlushPromise = resolvedPromise.then(flushJobs)
```

`Promise.resolve().then(fn)` 的微任务排队机制：

```
同步执行阶段：                  微任务队列：
┌─────────────────────┐      ┌─────────────┐
│ queueJob(effect1)   │  →   │             │
│ queueFlush()        │  →   │ flushJobs   │
│   Promise.then(     │      │             │
│     flushJobs)      │      └─────────────┘
│ count.value++       │
│ queueJob(effect2)   │ ← 同一 tick 入队，无新微任务
│ count.value++       │
│ nextTick(cb)        │ → 将 cb 存入 pendingPostFlushCbs
│                     │
│ 同步代码执行完毕     │     ← 开始处理微任务
│                     │     ← flushJobs 开始执行
│                     │     ← 执行 effect1、effect2
│                     │     ← DOM 批量更新
│                     │     ← flushPostFlushCbs → 执行 cb
└─────────────────────┘
```

### nextTick 回调执行时序

假设我们有多个嵌套调用：

```typescript
import { ref, nextTick } from 'vue'

const count = ref(0)

// 同步代码 1
count.value = 1

// 同步代码 2：注册 nextTick
nextTick(() => {
  console.log('nextTick 1, count:', count.value)
})

// 同步代码 3
count.value = 2

// 同步代码 4
nextTick(() => {
  console.log('nextTick 2, count:', count.value)
})

// 同步代码 5
console.log('同步执行完毕')
```

执行结果：
```
同步执行完毕
nextTick 1, count: 2
nextTick 2, count: 2
```

**分析：**

1. 同步代码全部执行完毕后，`count.value` 的最终值是 2。
2. 两个 `nextTick` 中的回调被存入 `pendingPostFlushCbs` 队列。
3. 进入微任务阶段，`flushJobs` 执行：
   - 执行组件更新（渲染 `count` 为 2）
   - 执行 `flushPostFlushCbs`
   - 依次输出两个日志，`count.value` 都是 2

这正是 `nextTick` 的微妙之处：**它保证在 DOM 更新后执行，但多个 nextTick 回调在同一个微任务中依次执行，共享更新后的 DOM 状态**。

### currentFlushPromise 的链式设计

Vue 3 `nextTick` 的一个精妙设计是使用 `currentFlushPromise` 建立 Promise 链：

```typescript
let currentFlushPromise: Promise<void> | null = null

// 当调度刷新时
currentFlushPromise = resolvedPromise.then(flushJobs)

// nextTick 中
export function nextTick(fn?) {
  const p = currentFlushPromise || resolvedPromise
  return fn ? p.then(fn) : p
}
```

这个设计保证了以下场景的正确执行顺序：

```typescript
// 第一次数据变化 → 调度第一次刷新
count1.value = 1
// currentFlushPromise = resolvedPromise.then(flushJobs)

// 注册第一个 nextTick
nextTick(() => console.log('cb1'))

// 第二次数据变化（flushJobs 尚未执行）
count2.value = 2
// 此时 queueJob 直接将 effect 追加到 queue 中

// 注册第二个 nextTick
nextTick(() => console.log('cb2'))
```

由于 `nextTick` 持有对 `currentFlushPromise` 的引用，所有回调都在同一个 `then` 链上：
```
resolvedPromise.then(flushJobs)
  .then(() => cb1)
  .then(() => cb2)
```

如果 `flushJobs` 执行期间又产生了新的 job 和新的 `nextTick` 回调，它们会被追加到当前队列的末尾，保证执行顺序的正确性。

## 高频面试题解析

### 面试题 1：Vue 3 的 nextTick 是宏任务还是微任务？

**答：微任务。**

Vue 3 使用 `Promise.resolve().then(flushJobs)` 实现，Promise 的回调属于微任务。Vue 2 虽然有一整套降级策略，但首选也是 Promise（微任务），只有在不支持 Promise 的旧浏览器中才使用 `setImmediate` 或 `setTimeout` 这类宏任务。

**追问：为什么选择微任务而不是宏任务？**

选择微任务的关键原因是 **优先级**。微任务在同步代码执行完毕后立即执行，早于 UI 渲染和宏任务。如果使用 `setTimeout`（宏任务），DOM 更新会延迟至少一个宏任务周期，导致用户可感知的卡顿：

```javascript
// 使用宏任务（setTimeout）的后果
data.count = 1
// 宏任务调度 → 要等下一个事件循环
// 在这期间：UI 可能读取到旧的 DOM 状态
// 用户可能看到一秒的"闪烁"
```

而微任务在同步代码执行完毕、UI 渲染之前就执行了，DOM 更新的反馈更加及时。

### 面试题 2：nextTick(cb) 和 Promise 微任务的关系？

**答：本质相同，但有额外保障。**

`nextTick(cb)` 等价于 `Promise.resolve().then(cb)`，但 Vue 的 `nextTick` 有额外的设计：

1. **DOM 更新保障**：`nextTick` 持有 `currentFlushPromise` 引用，确保回调在 DOM 批量更新之后执行。
2. **回调队列管理**：支持回调去重、错误处理、队列批量执行。
3. **主动标记**：内部使用 `isFlushPending` 和 `isFlushing` 状态管理，确保调度正确性。

```typescript
// 两者的关键区别
Promise.resolve().then(() => {
  // 只是微任务，不保证 DOM 已更新
})

nextTick(() => {
  // 保证在当前 flushJobs 完成后执行
  // DOM 一定是最新状态
})
```

### 面试题 3：flushJobs 和 flushPostFlushCbs 的执行顺序？

**答：先 flushJobs（执行所有组件更新），再 flushPostFlushCbs（执行 nextTick 注册的回调）。**

```
1. flushPreFlushCbs       → 前置回调（如 watch 的 flush: 'pre' 选项）
                           ↓
2. for 循环执行 queue     → 组件渲染 effect（DOM 更新）
                           ↓
3. flushPostFlushCbs      → nextTick 回调、watch 的 flush: 'post' 选项
```

这个顺序保证了 `nextTick` 回调执行时，DOM 已经是更新后的状态。同时，如果在 `flushJobs` 过程中又产生了新的 `pre` 回调，会插入到队列适当位置；而 `post` 回调则进入 `pendingPostFlushCbs`，在最后一次 `flushJobs` 结束时执行。

### 面试题 4：Vue 3 相比于 Vue 2 在 nextTick 实现上有哪些改进？

| 维度 | Vue 2 | Vue 3 |
|------|-------|-------|
| 实现文件 | `src/core/util/next-tick.js`（约100行） | `packages/runtime-core/src/scheduler.ts`（约300行） |
| 降级策略 | Promise → MutationObserver → setImmediate → setTimeout | 仅使用 Promise |
| 回调存储 | 简单数组 `callbacks` | 分层：`pendingPreFlushCbs`、`queue`、`pendingPostFlushCbs` |
| 去重机制 | 依赖 `pending` 标志 | 对每个 job 进行 `includes` 检查 + `Set` 去重 |
| Promise 化 | 在 nextTick 函数内手动构造 Promise | 基于 `currentFlushPromise` 链式 Promise |
| 错误处理 | try-catch 包裹 | 使用 `callWithErrorHandling` 统一处理 |
| TypeScript | 无（JS 实现） | 完整类型定义 |
| 递归处理 | 简单入队 | 递归检测 + 最大递归深度限制 |

### 面试题 5：在 nextTick 回调中修改数据会怎样？

```typescript
import { ref, nextTick } from 'vue'

const count = ref(0)

nextTick(() => {
  // 在 DOM 更新后再次修改数据
  count.value = 100
  
  // 再次等待下一次 DOM 更新
  nextTick(() => {
    console.log('深层嵌套的 nextTick')
  })
})
```

**执行流程：**

1. 首次 flushJobs 执行，DOM 更新为 `count = 0`。
2. flushPostFlushCbs 执行，触发了 `count.value = 100`。
3. `count.value = 100` 触发 `queueJob` → `queueFlush`。
4. 此时 `isFlushing` 为 true，`queueFlush` 检测到当前正在刷新：
   - 新建 `currentFlushPromise = resolvedPromise.then(flushJobs)`（下一个微任务）
5. 当前 flushJobs 执行完毕，`isFlushing = false`。
6. 因为 `queue.length > 0`（有新的 job），递归调用 `flushJobs` 清空剩余队列。
7. 队列中存在新的 job（`count = 100` 的更新），执行 DOM 更新为 `count = 100`。
8. 执行第二个 `nextTick` 注册的回调。

**结论：** 在 `nextTick` 回调中修改数据会触发新一轮的异步更新，但会在同一微任务周期内完成（如果时间允许），或者推迟到下一个微任务。

### 面试题 6：如何手写一个简易的 nextTick？

```typescript
// 手写简易 nextTick（面试基础版）
class SimpleNextTick {
  private callbacks: Function[] = []
  private pending = false
  
  private flushCallbacks() {
    this.pending = false
    const copies = this.callbacks.slice(0)
    this.callbacks.length = 0
    for (let i = 0; i < copies.length; i++) {
      copies[i]()
    }
  }
  
  nextTick(cb?: Function): Promise<void> {
    return new Promise((resolve) => {
      this.callbacks.push(() => {
        if (cb) {
          try { cb() } catch (e) { console.error(e) }
        }
        resolve()
      })
      
      if (!this.pending) {
        this.pending = true
        // 使用微任务
        Promise.resolve().then(() => this.flushCallbacks())
      }
    })
  }
}

// 手写简易 nextTick（带优先级的进阶版）
enum FlushPriority {
  PRE = 'pre',
  NORMAL = 'normal',
  POST = 'post'
}

class AdvancedNextTick {
  private queue: Function[] = []
  private postQueue: Function[] = []
  private flushing = false
  
  async nextTick(cb?: Function, priority = FlushPriority.POST): Promise<void> {
    if (priority === FlushPriority.POST) {
      this.postQueue.push(cb!)
    } else {
      this.queue.push(cb!)
    }
    
    if (!this.flushing) {
      this.flushing = true
      await Promise.resolve()
      
      // 先执行普通队列
      while (this.queue.length) {
        this.queue.shift()!()
      }
      
      // 再执行后置队列
      while (this.postQueue.length) {
        this.postQueue.shift()!()
      }
      
      this.flushing = false
    }
  }
}
```

## 总结与扩展

### 核心回顾

`nextTick` 是 Vue 异步更新策略的关键组成部分，理解它的原理对于深入掌握 Vue 的响应式系统和性能优化至关重要。

**一句话记忆要点：**

> Vue 的 nextTick 利用 Promise 创建微任务，在当前同步代码执行完毕、DOM 批量更新完成后执行回调。

**核心流程图解：**

```
数据变化
   ↓
trigger → 通知 effect
   ↓
effect.scheduler → queueJob
   ↓
queueFlush → resolvedPromise.then(flushJobs) -- 调度微任务
   ↓                          ↑   ← 同步代码继续执行
同步代码执行完毕              │
   ↓                          │
开始处理微任务 ←───────────────┘
   ↓
flushJobs:
  ├─ flushPreFlushCbs()    → watch pre 回调等
  ├─ 循环执行 queue[]      → 组件渲染（DOM 批量更新）
  └─ flushPostFlushCbs()   → nextTick 回调执行
```

### 扩展思考

1. **与框架无关的泛化理解**：`nextTick` 本质上就是一个"微任务延迟执行"的工具。类似的模式在 React 中也有体现——`setState` 的批量更新、React 18 的 `flushSync`（同步更新）等。理解 JavaScript 的事件循环是前端框架异步更新的共同基础。

2. **微任务的滥用与性能**：虽然微任务优先级高，但过度使用可能导致"微任务饥饿"——微任务队列无限追加新的微任务，导致宏任务和 UI 渲染永远得不到执行机会。Vue 3 的 `flushJobs` 中的最大递归深度检测正是为了防止这种情况。

3. **并行与调度**：Vue 3 正在发展的 Vapor Mode（无虚拟 DOM 模式）可能会改变异步更新的底层实现，但 `nextTick` 作为用户接口 API 将保持兼容。

4. **其他框架的类似实现**：

```javascript
// React 的 flushSync（类 nextTick 的行为）
import { flushSync } from 'react-dom'

flushSync(() => {
  setState(newValue)
})
// DOM 已同步更新（非异步批量）

// Angular 的 ChangeDetectorRef.detectChanges
import { ChangeDetectorRef } from '@angular/core'
// 手动触发变更检测，同步更新 DOM
cdr.detectChanges()

// Svelte 的 tick
import { tick } from 'svelte'
await tick()
// DOM 已更新
```
