## 一句话概括

Vue 组件通信的本质是**数据在组件树中的流动与共享**，从父子间的基础 props/emit 到全局状态管理 Pinia，Vue 3 提供了超过十种通信方式，每种方式对应不同的作用域与耦合层级，掌握它们意味着你能在任何场景下做出最优雅的技术选型。

## 背景与意义

### 为什么需要了解多种通信方式？

在单页面应用（SPA）的开发中，组件化是 Vue 的核心思想。应用被拆分为组件树，组件之间需要交换数据——这个 "交换" 的动作就是组件通信。随着应用规模的增长，数据流的混乱程度会迅速膨胀，而选择合适的通信方式直接决定了代码的可维护性、可测试性和可扩展性。

一个常见的误解是 "组件通信 = props + emit"，但这只是冰山一角。Vue 3 的 Composition API、响应式系统、以及周边生态提供了丰富的通信工具箱：

| 通信方式 | 适用场景 | 耦合度 | 数据流方向 |
|---------|---------|-------|-----------|
| props / emit | 父子组件直接交互 | 强耦合 | 单向 / 事件回调 |
| v-model / defineModel | 表单控件 / 双向绑定 | 中耦合 | 双向 |
| $attrs / inheritAttrs | 组件封装 / 透传 | 弱耦合 | 父 → 子 |
| provide / inject | 深层祖先 → 后代 | 弱耦合 | 单向（可响应式） |
| slot / 作用域插槽 | 模板定制 / 渲染委托 | 中耦合 | 父 ↔ 子（模板级） |
| ref / expose | 直接调用子组件方法 | 紧耦合 | 命令式 |
| 事件总线（mitt） | 非层级关系组件 | 松散 | 任意方向 |
| Pinia / Vuex | 全局共享状态 | 无耦合 | 集中式 |

这些方式之间并非互斥，一个好的 Vue 应用往往同时使用其中的 3~5 种。关键在于：**用最轻量的方式解决当前的问题，不引入不必要的抽象**。

## 概念与定义

在深入每个方式之前，先厘清几个核心概念：

**组件树（Component Tree）**：Vue 应用是一个以 `#app` 为根节点的树状结构。数据天然可以从父节点流向子节点，但跨层级或跨分支的通信需要额外的机制。

**单向数据流（One-Way Data Flow）**：Vue 的核心设计原则。数据从父组件向下传递给子组件，子组件不能直接修改父组件的数据，而是通过触发事件通知父组件来变更。这保证了数据变更的可追溯性。

**响应式系统（Reactivity System）**：Vue 3 基于 Proxy 实现的依赖追踪机制。当响应式数据变化时，所有依赖它的视图和计算属性自动更新。这是所有通信方式的基础设施。

**组件通信的三大维度**：
1. **方向性**：父 → 子、子 → 父、双向、任意方向
2. **作用域**：父子间、祖孙间、兄弟间、全局
3. **触发机制**：静态传递、事件驱动、响应式依赖、指令式调用

理解这些维度后，我们来看每一种通信方式的具体实现。

## 最小示例

### 1. props / emit —— 最基础的父子通信

这是 Vue 最原始也最直接的通信方式，Vue 新手接触的第一个概念。

```vue
<!-- Parent.vue -->
<script setup>
import { ref } from 'vue'
import Child from './Child.vue'

const count = ref(0)
const handleUpdate = (val) => {
  count.value = val
}
</script>

<template>
  <Child :count="count" @update="handleUpdate" />
  <p>父组件 count: {{ count }}</p>
</template>
```

```vue
<!-- Child.vue -->
<script setup>
defineProps({ count: Number })
const emit = defineEmits(['update'])

const increment = () => {
  emit('update', count + 1)
  // ❌ 不能直接修改 props.count
}
</script>

<template>
  <button @click="increment">
    子组件 count: {{ count }}
  </button>
</template>
```

`defineProps` 和 `defineEmits` 是 `<script setup>` 中的编译宏，无需额外 import。TypeScript 支持通过泛型提供类型约束：

```typescript
defineProps<{
  count: number
  label?: string  // 可选 prop
}>()

defineEmits<{
  (e: 'update', value: number): void
  (e: 'change', id: string): void
}>()
```

### 2. provide / inject —— 跨层级数据注入

当数据需要从祖先组件传递到深层后代时，逐层 props 透传会带来严重的 "prop drilling" 问题。provide/inject 完美解决了这一痛点。

```vue
<!-- Ancestor.vue -->
<script setup>
import { ref, provide } from 'vue'
import Descendant from './Descendant.vue'

const theme = ref('dark')
// 提供响应式数据，后代可以响应式感知变化
provide('theme', theme)
// 也可以提供普通值
provide('appName', 'My App')
</script>

<template>
  <Descendant />
</template>
```

```vue
<!-- Descendant.vue -->
<script setup>
import { inject } from 'vue'

const theme = inject('theme')       // 自动 unpack ref
const appName = inject('appName')   // 普通字符串
// 带默认值
const config = inject('config', { debug: false })
// 如果没有提供且无默认值，严格模式下可以要求必须提供
const requiredValue = inject('requiredKey')
if (!requiredValue) {
  throw new Error('requiredKey must be provided by an ancestor')
}
</script>

<template>
  <div :class="theme">
    App: {{ appName }}
  </div>
</template>
```

> ⚠️ Vue 3 中，通过 `provide('key', ref(...))` 传递的 ref 会自动在 inject 时拆包，所以 inject 拿到的是响应式的原始值。但如果传递的是 `reactive` 对象，则直接引用该对象，修改其属性会触发更新。

### 3. v-model / defineModel —— 简洁的双向绑定

Vue 3.4+ 引入了 `defineModel` 宏，让双向绑定的实现变得前所未有的简洁。

```vue
<!-- Parent.vue -->
<script setup>
import { ref } from 'vue'
import MyInput from './MyInput.vue'

const username = ref('')
</script>

<template>
  <MyInput v-model="username" />
  <p>输入的内容: {{ username }}</p>
</template>
```

```vue
<!-- MyInput.vue -->
<script setup>
const model = defineModel({ type: String, required: true })
// model 是一个 ref，读取它拿到父组件的值，赋值它会触发父组件的更新
</script>

<template>
  <input v-model="model" />
</template>
```

`defineModel` 本质上是语法糖，展开后等价于：

```vue
<script setup>
const props = defineProps({ modelValue: String })
const emit = defineEmits(['update:modelValue'])

const model = computed({
  get: () => props.modelValue,
  set: (val) => emit('update:modelValue', val)
})
</script>
```

支持多个 v-model 绑定时使用带参数的 `defineModel`：

```vue
<!-- UserForm.vue -->
<script setup>
const name = defineModel('name')
const email = defineModel('email')
</script>

<template>
  <input v-model="name" placeholder="姓名" />
  <input v-model="email" placeholder="邮箱" />
</template>
```

```vue
<!-- Parent.vue -->
<UserForm v-model:name="name" v-model:email="email" />
```

### 4. 透传 Attributes（$attrs）

当一个组件包裹原生元素或其他组件时，未声明为 props 的属性会自动"透传"到根元素上。

```vue
<!-- BaseButton.vue -->
<script setup>
// inheritAttrs 的默认值为 true
</script>

<template>
  <button>
    <slot />
  </button>
</template>
```

```vue
<!-- 使用 -->
<BaseButton
  type="submit"
  class="primary-btn"
  :disabled="isLoading"
  @click="handleClick"
>
  提交
</BaseButton>
```

渲染结果：`<button type="submit" class="primary-btn" disabled>提交</button>`

更高级的用法是通过 `useAttrs()` 在组合式 API 中访问：

```vue
<script setup>
import { useAttrs } from 'vue'

const attrs = useAttrs()
console.log(attrs.class) // 可以拿到透传的属性
</script>
```

还可以通过 `v-bind="$attrs"` 手动绑定到指定元素：

```vue
<template>
  <div class="wrapper">
    <input v-bind="$attrs" />
  </div>
</template>
```

### 5. 插槽作用域（Scoped Slots）

插槽让父组件可以向子组件的特定位置插入模板内容。作用域插槽更进一步，允许子组件向插槽暴露数据。

```vue
<!-- List.vue -->
<script setup>
defineProps({ items: Array })
</script>

<template>
  <ul>
    <li v-for="item in items" :key="item.id">
      <!-- 将 item 暴露给父组件 -->
      <slot name="item" :data="item" :index="item.index">
        <!-- 默认内容 -->
        {{ item.label }}
      </slot>
    </li>
  </ul>
</template>
```

```vue
<!-- Parent.vue -->
<List :items="list">
  <template #item="{ data, index }">
    <span class="custom-item">
      {{ index + 1 }}. {{ data.label }}
      <button @click="handleEdit(data)">编辑</button>
    </span>
  </template>
</List>
```

### 6. 事件总线（mitt）

对于非父子关系的组件，Vue 3 不再内置 `$on`/`$off`，官方推荐使用独立的 mitt 库（仅 200 字节）。

```typescript
// eventBus.ts
import mitt from 'mitt'

// 定义事件类型（TypeScript 友好）
type Events = {
  'user:login': { userId: string; token: string }
  'user:logout': void
  'notification': { message: string; type: 'success' | 'error' }
}

export const emitter = mitt<Events>()
```

```vue
<!-- A.vue -->
<script setup>
import { emitter } from './eventBus'

const login = () => {
  // 触发事件
  emitter.emit('user:login', { userId: '123', token: 'xxx' })
}
</script>
```

```vue
<!-- B.vue -->
<script setup>
import { onMounted, onUnmounted } from 'vue'
import { emitter } from './eventBus'

onMounted(() => {
  emitter.on('user:login', ({ userId, token }) => {
    console.log(`用户 ${userId} 已登录`)
  })
})

onUnmounted(() => {
  // 必须清理监听，防止内存泄漏
  emitter.all.clear()
})
</script>
```

### 7. Pinia 全局状态管理

当状态需要在多个不相关的组件间共享时，全局状态管理是最可靠的方案。

```typescript
// stores/user.ts
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'

export const useUserStore = defineStore('user', () => {
  // state
  const user = ref<User | null>(null)
  const token = ref<string>('')

  // getters
  const isLoggedIn = computed(() => !!token.value)
  const displayName = computed(() => user.value?.name ?? '未登录')

  // actions
  async function login(credentials: { username: string; password: string }) {
    const res = await api.login(credentials)
    user.value = res.user
    token.value = res.token
  }

  function logout() {
    user.value = null
    token.value = ''
  }

  return { user, token, isLoggedIn, displayName, login, logout }
})
```

在任意组件中使用：

```vue
<script setup>
import { useUserStore } from '@/stores/user'

const store = useUserStore()
// 直接读取状态 —— 自动响应式
const { isLoggedIn, displayName } = storeToRefs(store)
// storeToRefs 保持响应式解构，直接解构会丢失响应性

const handleLogout = () => {
  store.logout()
}
</script>
```

## 核心知识点拆解

### props 的深层机制

props 不仅仅是属性传递。Vue 3 中 props 的底层依赖 `reactive` 实现，这意味着父组件的数据变更会即时反映到子组件的视图中。

**Props 校验**：Vue 支持运行时类型校验，这在大型团队中非常有用：

```typescript
defineProps({
  status: {
    type: String as PropType<'pending' | 'success' | 'error'>,
    required: true,
    validator: (val: string) => ['pending', 'success', 'error'].includes(val)
  },
  items: {
    type: Array as PropType<Item[]>,
    default: () => []
  },
  config: {
    type: Object as PropType<Config>,
    default: () => ({})
  }
})
```

**性能考量**：每个 props 的变更都会触发子组件的重新渲染。如果父组件的某个数据频繁变化但不影响子组件，考虑使用 `computed` 进行节流，或者在合适的时机使用 `v-once`。

### provide/inject 的响应式传递陷阱

传递 ref 和 reactive 的行为不同，这是最常见的坑：

```vue
<script setup>
import { ref, reactive, provide } from 'vue'

const count = ref(0)          // ref
const state = reactive({ x: 0, y: 0 }) // reactive

provide('count', count)       // 后代 inject('count') 拿到的是 number（ref 被自动拆包）
provide('state', state)       // 后代 inject('state') 拿到的是 reactive 对象本身

// 想要传递 ref 不被拆包？用 getter/setter 封装
provide('countRef', {
  get value() { return count.value },
  set value(v) { count.value = v }
})
</script>
```

从 Vue 3.5+ 开始，推荐使用 `provide` 传递响应式对象时配合 readonly 防止后代意外修改：

```typescript
import { readonly } from 'vue'

provide('state', readonly(state))
provide('updateState', (key: string, value: unknown) => {
  state[key] = value
})
```

### v-model 的完整演进史

Vue 2.2 引入 `.sync` 修饰符 → Vue 2.3 正式支持 `v-model` 多绑定 → Vue 3 统一为 `v-model:xxx` 语法 → Vue 3.4 `defineModel` 宏 → Vue 3.5 `defineModel` 支持 `local` 配置。

```typescript
// Vue 3.5+ defineModel 的新功能
const model = defineModel({
  // 与父组件同步，但也可以有本地回退值
  local: true,
  type: String,
  default: ''
})
// local: true 表示当父组件未传 v-model 时，model.value 可以独立存在
```

### 事件总线的替代演进

Vue 2 中内置的 `$on`/`$off`/`$once` 在 Vue 3 中被移除，原因包括：
1. 事件名称是字符串，缺乏类型安全
2. 难以追踪事件来源和去向
3. 维护不当容易导致内存泄漏

mitt 作为替代品更为纯粹，但也有其局限性。对于复杂应用，更推荐用 Pinia 替代所有事件总线的使用场景。

### Pinia vs. 其他全局状态方案

| 特性 | Pinia | Vuex 4 | 全局 reactive | 事件总线 |
|------|-------|--------|---------------|---------|
| TypeScript 支持 | 优秀 | 一般 | 一般 | 差 |
| DevTools 支持 | ✅ | ✅ | ❌ | ❌ |
| 模块化 | 原生支持 | namespaced | 手动 | 手动 |
| 服务端渲染 | ✅ | ✅ | ❌ | ❌ |
| 插件生态 | 丰富 | 成熟 | 无 | 无 |

对于大部分中大型应用，Pinia 是最佳选择。但极简场景（如主题切换）用 `provide` + `reactive` 就够了，不必引入额外依赖。

## 实战案例

### 案例一：可复用表单组件体系

设计一个企业级表单组件库，需要支持表单校验、交互反馈、数据同步。

```vue
<!-- FormInput.vue -->
<script setup lang="ts">
import { inject, computed } from 'vue'

const model = defineModel<string>()

// 从父级的 Form 组件读取校验上下文
const formContext = inject<{
  errors: Record<string, string>
  touched: Record<string, boolean>
  validate: (name: string, val: string) => void
}>('formContext', null)

const props = defineProps<{
  name: string
  label: string
  rules?: Array<(value: string) => string | null>
}>()

const errorMessage = computed(() => formContext?.errors[props.name] ?? '')
const isTouched = computed(() => formContext?.touched[props.name] ?? false)

const handleBlur = () => {
  formContext?.validate(props.name, model.value)
}
</script>

<template>
  <div class="form-field" :class="{ 'has-error': errorMessage && isTouched }">
    <label :for="name">{{ label }}</label>
    <input
      :id="name"
      v-model="model"
      @blur="handleBlur"
      :aria-invalid="!!errorMessage"
    />
    <Transition name="fade">
      <p v-if="errorMessage && isTouched" class="error-msg" role="alert">
        {{ errorMessage }}
      </p>
    </Transition>
  </div>
</template>
```

```vue
<!-- Form.vue -->
<script setup lang="ts">
import { provide, reactive } from 'vue'

const errors = reactive<Record<string, string>>({})
const touched = reactive<Record<string, boolean>>({})

const validate = (name: string, value: string): boolean => {
  // 校验逻辑...
  touched[name] = true
  return !errors[name]
}

provide('formContext', { errors, touched, validate })

const handleSubmit = () => {
  // 遍历所有字段执行校验
}
</script>

<template>
  <form @submit.prevent="handleSubmit" novalidate>
    <slot />
  </form>
</template>
```

这里我们同时使用了 defineModel（双向绑定）、provide/inject（跨层级传递校验上下文）、插槽（内容插入），三种通信方式协作构建了一个解耦的表单系统。

### 案例二：CMS 页面编辑器

在一个拖拽式页面编辑器中，组件之间的通信变得非常复杂：

```typescript
// stores/editor.ts
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'

export const useEditorStore = defineStore('editor', () => {
  const selectedId = ref<string | null>(null)
  const blocks = ref<Block[]>([])
  const clipboard = ref<Block | null>(null)
  const history = ref<Block[][]>([])
  const historyIndex = ref(-1)

  const selectedBlock = computed(() =>
    blocks.value.find(b => b.id === selectedId.value)
  )

  function addBlock(block: Block, afterId?: string) {
    const idx = afterId
      ? blocks.value.findIndex(b => b.id === afterId) + 1
      : blocks.value.length
    blocks.value.splice(idx, 0, block)
    pushHistory()
  }

  function removeBlock(id: string) {
    const idx = blocks.value.findIndex(b => b.id === id)
    if (idx !== -1) {
      blocks.value.splice(idx, 1)
      if (selectedId.value === id) selectedId.value = null
      pushHistory()
    }
  }

  function moveBlock(id: string, direction: 'up' | 'down') {
    const idx = blocks.value.findIndex(b => b.id === id)
    const target = direction === 'up' ? idx - 1 : idx + 1
    if (target >= 0 && target < blocks.value.length) {
      [blocks.value[idx], blocks.value[target]] =
        [blocks.value[target], blocks.value[idx]]
      pushHistory()
    }
  }

  function pushHistory() {
    history.value = history.value.slice(0, historyIndex.value + 1)
    history.value.push(structuredClone(toRaw(blocks.value)))
    historyIndex.value = history.value.length - 1
    // 限制历史记录数量
    if (history.value.length > 50) history.value.shift()
  }

  function undo() {
    if (historyIndex.value > 0) {
      historyIndex.value--
      blocks.value = structuredClone(history.value[historyIndex.value])
    }
  }

  function redo() {
    if (historyIndex.value < history.value.length - 1) {
      historyIndex.value++
      blocks.value = structuredClone(history.value[historyIndex.value])
    }
  }

  return {
    selectedId, blocks, clipboard,
    selectedBlock,
    addBlock, removeBlock, moveBlock, undo, redo
  }
})
```

在编辑器组件中，我们使用 Pinia 管理全局状态，使用 props 向下传递块级数据，使用作用域插槽让每个块类型的渲染组件获得自定义内容，使用 provide/inject 传递编辑器的主题配置：

```vue
<!-- EditorCanvas.vue -->
<script setup lang="ts">
import { provide, ref } from 'vue'
import { useEditorStore } from '@/stores/editor'
import BlockRenderer from './BlockRenderer.vue'
import EditorToolbar from './EditorToolbar.vue'

const store = useEditorStore()
const editorConfig = ref({
  gridSize: 10,
  snapToGrid: true,
  zoom: 1
})

provide('editorConfig', editorConfig)
</script>

<template>
  <div class="editor-layout">
    <EditorToolbar />
    <div class="canvas" :style="{ zoom: editorConfig.zoom * 100 + '%' }">
      <BlockRenderer
        v-for="block in store.blocks"
        :key="block.id"
        :block="block"
        :selected="store.selectedId === block.id"
        @select="store.selectedId = $event"
        @move="store.moveBlock(block.id, $event)"
      >
        <!-- 每个 block 可以通过作用域插槽自定义内容 -->
        <template #default="{ block }">
          <component :is="getRenderer(block.type)" :data="block.data" />
        </template>
      </BlockRenderer>
    </div>
  </div>
</template>
```

这个案例展示了如何在实际项目中组合多种通信方式：Pinia 管理核心状态、props 传递块数据、作用域插槽提供渲染灵活性、provide/inject 注入编辑器配置。

## 底层原理

### 响应式系统的基石

Vue 3 的响应式系统基于 ES6 Proxy。当我们通过 `reactive()` 包装一个对象时，Vue 会创建一个 Proxy 实例来拦截 get、set、deleteProperty 等操作：

```typescript
// 简化版实现
function reactive(target) {
  const proxy = new Proxy(target, {
    get(target, key, receiver) {
      // 依赖收集：在访问属性时，记录当前正在运行的 effect
      track(target, key)
      return Reflect.get(target, key, receiver)
    },
    set(target, key, value, receiver) {
      const oldValue = target[key]
      const result = Reflect.set(target, key, value, receiver)
      if (oldValue !== value) {
        // 触发更新：当值改变时，通知所有依赖该属性的 effect
        trigger(target, key)
      }
      return result
    }
  })
  return proxy
}
```

**这为什么重要？** 所有组件通信方式最终都依赖这个机制：

- **props**：父组件的模板编译为 render 函数时，访问了响应式数据，触发了依赖收集。子组件渲染时读取 props，建立了从父数据到子组件的依赖链。
- **provide/inject**：provide 传递的 ref 或 reactive 对象注册到子组件的依赖图中，当这些数据变化时，所有用了 `inject` 的组件都会响应式更新。
- **Pinia**：Pinia 的 state 本质上是 `reactive({})`，所有的 getters 都是 `computed`，actions 是普通函数。当你读取 store 的某个属性时，Vue 响应式系统自动追踪依赖。

### defineModel 的编译时魔法

`defineModel` 之所以是"编译宏"，是因为它在 SFC 编译阶段就被展开为标准的 props/emit 声明。展开后的代码大致如下：

```javascript
// 编译后的结果
export default {
  props: {
    modelValue: { type: String, required: true }
  },
  emits: ['update:modelValue'],
  setup(props, { emit }) {
    const model = computed({
      get: () => props.modelValue,
      set: (val) => emit('update:modelValue', val)
    })
    return { model }
  }
}
```

**为什么需要 computed？** 因为 props 本身是从父组件的响应式数据"投射"过来的，子组件不能修改 props（会报错）。通过 computed 的 get/set，我们封装了一个"看起来像 ref"的对象，读的时候读 props，写的时候触发事件让父组件修改。

### $attrs 的透传规则

透传的核心逻辑在 Vue 的虚拟 DOM 创建阶段：

```javascript
// 简化版：创建虚拟节点时处理 attrs
function createVNode(type, props, children) {
  // 区分 props 和 attrs
  const normalizedProps = normalizePropsOptions(type)
  const { attrs, props: resolvedProps } = splitProps(props, normalizedProps)

  return {
    type,
    props: resolvedProps,
    attrs,  // 未声明为 props 的属性存到这里
    children,
    // ...
  }
}

// 挂载时，attrs 会被自动设置到根元素的 DOM 属性上
function mountElement(vnode, container) {
  const el = document.createElement(vnode.type)
  for (const key in vnode.attrs) {
    if (key.startsWith('on')) {
      el.addEventListener(key.slice(2).toLowerCase(), vnode.attrs[key])
    } else {
      el.setAttribute(key, vnode.attrs[key])
    }
  }
  // ...
}
```

当 `inheritAttrs: false` 时，Vue 跳过自动设置到根元素，而是将 attrs 保留给开发者手动处理。

## 高频面试题解析

### Q1：props 为什么不能直接修改？我非要用子组件修改 props 会怎样？

**核心原因**：Vue 的单向数据流设计保证数据变更可追踪。如果子组件直接修改 props，调试时会无法判断数据到底是父组件改的还是子组件改的，导致状态混乱。

**实际表现**：在开发模式下，Vue 会对 props 使用 `shallowReadonly` 包装，直接修改会触发控制台警告。但在生产模式下，如果传递的是对象/数组等引用类型，子组件**可以**通过修改对象的属性来"绕过"约束——这正是 Vue 官方警告的情况，因为这种隐式修改难以追踪。

**正确做法**：
- 基本类型：使用 emit 触发父组件变更
- 引用类型：避免直接修改，改为 emit 新的数据让父组件替换，或者使用 v-model

### Q2：provide 和 Pinia 有什么区别？什么时候用哪个？

**核心区别**：
- **作用域不同**：provide/inject 仅对组件子树生效；Pinia 对整个应用全局生效
- **可测试性**：Pinia 的 store 可以在组件外独立测试；provide 依赖组件树上下文
- **DevTools**：Pinia 有完整的 DevTools 支持（时间旅行、状态快照）；provide 无
- **复杂度**：provide 零依赖、零额外代码；Pinia 需要安装包和注册插件

**选型建议**：
- ✅ 主题配置、本地化语言、用户权限等贯穿组件树的数据 → provide/inject
- ✅ 跨路由的全局状态、服务端缓存、复杂异步操作 → Pinia
- ✅ 极简的三四个组件间的非全局状态 → provide/inject + reactive
- ❌ 不要在 provide 中暴露可写响应式对象让后代随意修改 → 用 readonly 保护

### Q3：事件总线（mitt）还有什么值得使用的场景？

在 Vue 3 移除内置事件总线后，很多人认为它应该被彻底淘汰。但 mitt 有两个不可替代的优势：

1. **零成本跨组件通信**：不需要经过组件树、不需要建立 store，任何两个组件之间可以自由通信。对于"组件 A 的一个操作触发组件 B 的某个方法"这种强关联场景，事件总线比 Pinia 更轻量。

2. **非 Vue 环境与 Vue 组件的通信**：在一个混合前端架构中（比如微前端），mitt 可以作为桥梁连接不同框架的组件。

但必须承认：**95% 的前端场景中，Pinia 或 provide/inject 是更好的选择**。事件总线应该被视为"最后手段"而非"默认方案"。

### Q4：defineModel 和直接使用 props + emit 有什么优劣？

defineModel 的优势：
- 代码量减少 50%+，尤其多 v-model 场景
- 类型推断更完善
- 语义清晰，一看就知道是双向绑定

劣势：
- 背后隐藏了 props 和 emit 的声明，某些代码审查场景可能偏好显式声明
- 对于复杂的转换逻辑（如输入校验后再同步），需要额外处理

在实际项目中，推荐：**简单的双向绑定用 defineModel，需要复杂校验和转换逻辑时使用 props + emit 显式控制**。

### Q5：透传 Attributes 在组件封装中有什么最佳实践？

透传是构建"透明代理组件"的核心机制。最佳实践包括：

```vue
<script setup>
import { useAttrs } from 'vue'

const attrs = useAttrs()
</script>

<template>
  <div class="wrapper">
    <!-- 方法 1：自动透传到根元素（默认） -->
    <!-- 方法 2：v-bind="$attrs" 手动透传 -->
    <BaseInput v-bind="attrs" />
    <!-- 方法 3：继承部分属性 -->
    <BaseButton v-bind="{ ...attrs, class: undefined }">
      <slot />
    </BaseButton>
  </div>
</template>
```

核心原则：**让开发者感觉像是在直接使用原生 HTML 元素一样自然**。比如封装一个 `MyInput` 组件，使用者应该能直接传 `placeholder`、`disabled`、`maxlength` 等原生属性，而不需要你的组件在 props 中一一声明。

## 总结与扩展

### 技术选型决策树

面对一个组件通信需求时，可以按照以下顺序决策：

```
需要传递数据？
├─ 父子直连？
│  ├─ 简单数据 → props
│  ├─ 双向绑定 → v-model (defineModel)
│  └─ 调用子方法 → ref + expose
├─ 祖孙跨层？
│  ├─ 少量数据 → provide/inject (readonly 保护)
│  └─ 大量数据 → Pinia
├─ 兄弟/不相关组件？
│  ├─ 复杂状态 → Pinia
│  └─ 简单事件 → provide/inject (通过共同祖先) → 事件总线 (最后手段)
└─ 模板级定制？
   ├─ 简单插槽 → slot
   └─ 需要子组件数据 → 作用域插槽
```

### 未来趋势

1. **Signal 范式的影响**：Vue 3.5+ 的 `useTemplateRef` 和响应式 props 的改进，正在往更细粒度的响应式方向进化。React 的 Signal 库（如 Preact Signals）的理念正在被 Vue 社区吸收。

2. **编译时优化**：Vue Vapor Mode（无虚拟 DOM 模式）正在实验阶段，届时组件通信的底层机制会进一步优化，但上层 API 不变。

3. **TypeScript 深度集成**：`defineModel`、`defineProps` 的泛型支持越来越完善，未来的组件通信体验会是"写 JS 得到 TS 的全面推断"。

### 写在最后

组件通信方式的选择，本质上是**在"灵活"和"可预测"之间找平衡**。Props 和 emit 虽然啰嗦但数据流清晰；provide/inject 很方便但数据来源难以追踪；Pinia 功能强大但对小项目来说过度抽象。

没有"最好的"通信方式，只有"最适合当前场景"的方案。理解每种方式的底层机制、适用边界和取舍权衡，才能在不同的项目规模和组织架构下做出最合适的选择。

希望这篇文章能帮你建立起 Vue 3 组件通信的完整认知地图。无论是在面试中被问到"Vue 组件通信方式"时的从容应答，还是在项目中面对复杂的数据流交互时的淡定选型，你都已经有了自己的判断框架。
