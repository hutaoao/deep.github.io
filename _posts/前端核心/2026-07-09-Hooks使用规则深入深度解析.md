## 一句话概括

React Hooks 的两条黄金规则——"只在最顶层调用 Hooks" 和 "只在 React 函数组件或自定义 Hook 中调用 Hooks"——并非 React 团队拍脑袋的编码建议，而是源于 Hooks 底层基于链表（Fiber 节点上的 `memoizedState` 链表）的顺序调用的实现机制。任何违反这两条规则的行为都会导致链表结构错乱、状态"串位"，引发难以调试的 bug。

## 背景与意义

### 从 Class 组件到 Hooks 的范式革命

React 16.8 正式引入 Hooks 之前，React 社区长期被几个痛点困扰：

1. **状态逻辑复用困难**：Class 组件只能通过 `Higher-Order Components`（HOC）或 `Render Props` 来复用逻辑，导致"Wrapper Hell"（包装地狱）。打开一个中等规模的 React 应用的 React DevTools，组件树的嵌套层级往往深达十几层。
2. **生命周期逻辑分散**：在 `componentDidMount` 和 `componentDidUpdate` 中经常要处理同一份副作用逻辑，开发者需要手动维护这两者之间的同步关系。
3. **this 指向问题**：Class 组件中方法的 `this` 绑定需要额外的心智负担，`bind(this)` 或箭头函数的选择让团队代码风格难以统一。

Hooks 的出现彻底改变了这一切。函数组件加 Hooks 的写法让状态逻辑可以被抽取为独立的函数单元（自定义 Hook），不再受组件生命周期的约束。然而，Hooks 的易用性背后隐藏着一个严苛的实现约束——**Hooks 的调用顺序必须在每次渲染中保持严格一致**。

### 为什么需要深入了解 Hooks 规则？

很多初学者把两条规则当作"面试八股文"来背，但在实际开发中，一条 `if` 语句包裹的 `useState`、一个提前 `return` 的渲染分支，就可能悄悄摧毁整个组件树的状态一致性。更隐蔽的是，ESLint 规则也不是万能的——它无法覆盖所有运行时动态条件。理解规则底层的 Fiber 链表机制，才能从根本上避免踩坑。

## 概念与定义

### 两条黄金规则

**规则一：只在最顶层调用 Hooks**

不要在循环、条件语句或嵌套函数中调用 Hooks。这条规则的准确表述是：**在同一个组件内，每次渲染时所有 Hooks 的调用顺序必须完全一致**。

`memoizedState` 链表结构如下：

```
Fiber.memoizedState
  ├── useState_0  (第 1 个 Hook → 指向第 2 个)
  ├── useState_1  (第 2 个 Hook → 指向第 3 个)
  ├── useEffect_0 (第 3 个 Hook → 指向第 4 个)
  ├── useRef_0    (第 4 个 Hook → null)
  └── ...
```

每次渲染时，React 会从上到下遍历这个链表，把当前 Hook 的调用与链表上的节点一一对应。顺序一旦错乱，对应的 state（状态）和 effect（副作用）就会张冠李戴。

**规则二：只在 React 函数组件或自定义 Hook 中调用 Hooks**

普通 JavaScript 函数不具备 Fiber 节点上下文，没有 `currentlyRenderingFiber` 和 `memoizedState`，调用 Hooks 会直接引发错误。

### 相关核心概念

| 概念 | 说明 |
|------|------|
| `memoizedState` | Fiber 节点上挂载的单向链表，存储该组件所有 Hooks 的状态 |
| `workInProgress` | React 协调阶段正在构建的下一个 Fiber 树 |
| `currentlyRenderingFiber` | 当前正在渲染的 Fiber 节点，Hooks 的上下文载体 |
| `Hook.next` | 链表指针，指向当前组件的下一个 Hook |
| `queue` | Hook 上挂载的更新队列，存储所有待处理的 `update` 对象 |
| `pendingEffects` | `useEffect` 的副作用链表，在 commit 阶段逐一执行 |
| `StrictMode` | React 严格模式，在开发环境下故意双调用渲染函数以暴露副作用 |

## 最小示例

### 条件调用的错误示范

```jsx
function BadExample({ shouldUseName }) {
  const [count, setCount] = useState(0);

  if (shouldUseName) {
    // ❌ 违反规则一：条件调用 Hooks
    const [name, setName] = useState('Alice');
  }

  const [age, setAge] = useState(25);

  useEffect(() => {
    document.title = `${name || 'User'} - ${count}`;
  });

  return <div>{count}</div>;
}
```

渲染序列分析：

**首次渲染（`shouldUseName = true`）：**

| 调用顺序 | Hook 类型 | 链表索引 |
|---------|-----------|---------|
| 1 | `useState(0)` | hook0（count） |
| 2 | `useState('Alice')` | hook1（name） |
| 3 | `useState(25)` | hook2（age） |
| 4 | `useEffect(cb)` | hook3（effect） |

**第二次渲染（`shouldUseName = false`）：**

| 调用顺序 | Hook 类型 | 实际匹配的链表节点 |
|---------|-----------|------------------|
| 1 | `useState(0)` | hook0 ✅（count） |
| 2 | *跳过条件块* | hook1 ❌（本应是 name，但被跳过） |
| 3 | `useState(25)` | hook1 ❗（本应是 hook1，实际匹配到 name） |
| 4 | `useEffect(cb)` | hook2 ❗（本应是 hook2，实际匹配到 age） |

最终结果：`age` 的初始值 25 被塞入了 `name` 对应 hook1 中保存的 'Alice' 状态之上，渲染出的 `count` 也可能会读取到意想不到的值。更致命的是，第 4 步的 `useEffect` 从原本的 hook3 偏移到了 hook2，如果 hook2 之前有未被清理的 effect，还会导致内存泄漏。

### ESLint 触发报警

```bash
React Hook "useState" is called conditionally.
React Hooks must be called in the exact same order in every component render.
```

这条 ESLint 规则的实现本质上是一个 AST 静态分析 —— 规则追踪 `useState`、`useEffect` 等函数调用在语法树中的嵌套上下文，检查其是否被 `if`、`for`、`while` 等控制流语句包裹。

## 核心知识点拆解

### 1. 条件调用的底层问题：链表断裂

如前所述，条件调用最直接的影响就是 `memoizedState` 链表指针错位。要理解问题的根源，需要查看 React 源码中的 `mountWorkInProgressHook` 和 `updateWorkInProgressHook` 两个核心函数。

**首次渲染（Mount 阶段）：**

在 mount 阶段，React 调用 `mountWorkInProgressHook`，在 `workInProgress` Fiber 的 `memoizedState` 尾部追加新节点：

```javascript
function mountWorkInProgressHook() {
  const hook = {
    memoizedState: null, // Hook 缓存的状态值
    baseState: null,
    baseQueue: null,
    queue: null,        // 更新队列
    next: null,         // 下一个 Hook 的指针
  };

  if (workInProgressHook === null) {
    // 这是组件第一个 Hook
    currentlyRenderingFiber.memoizedState = workInProgressHook = hook;
  } else {
    // 追加到链表末尾
    workInProgressHook = workInProgressHook.next = hook;
  }

  return workInProgressHook;
}
```

**更新渲染（Update 阶段）：**

在 update 阶段，`updateWorkInProgressHook` 从当前 Fiber 的 `memoizedState` 链表头部开始，按顺序取下一个节点：

```javascript
function updateWorkInProgressHook() {
  let nextCurrentHook;

  if (currentHook === null) {
    // 第一个 Hook，从 current Fiber 的链表头开始
    const current = currentlyRenderingFiber.alternate;
    nextCurrentHook = current.memoizedState;
  } else {
    // 后续 Hook，沿链表向前推进
    nextCurrentHook = currentHook.next;
  }

  // ... 省略 clone 和 updateQueue 处理逻辑

  currentHook = nextCurrentHook;
  const newHook = currentHook;
  // 沿 workInProgress 链表也向前推进
  workInProgressHook = workInProgressHook.next = newHook;

  return workInProgressHook;
}
```

当条件分支导致第二次渲染跳过一个 `useState` 调用时，`currentHook.next` 指向的是上一个渲染的第 2 个 Hook（name），但第 3 次 `useState` 调用期望从链表的第 2 个节点获取数据——这就造成了状态错位的根本根源。

### 2. 循环调用的陷阱

在循环中调用 Hooks 同样是高危操作：

```jsx
function BadLoop({ items }) {
  items.forEach(item => {
    // ❌ 循环中调用 Hooks
    const [expanded, setExpanded] = useState(false);
  });
}
```

循环调用的问题在于：当 `items` 数组长度或顺序发生变化时，Hooks 数量会同步变化，但链表节点总数是在第一次渲染时固定的。后续渲染中，数组变长意味着多出来的 Hooks 调用找不到对应的链表节点（`currentHook` 走到 null 后继续调用会抛出 `Rendered fewer hooks than expected` 错误）；数组变短意味着部分链表节点无法被消费（`Rendered more hooks than during the previous render`）。

正确的做法是**在组件顶层确定好所有 Hook 的结构，用数组或对象来管理动态数据**：

```jsx
function GoodExample({ items }) {
  // ✅ 顶层确定 Hook 位置
  const [expandedState, setExpandedState] = useState(
    items.map(() => false)
  );

  const toggle = (index) => {
    setExpandedState(prev => {
      const next = [...prev];
      next[index] = !next[index];
      return next;
    });
  };
}
```

### 3. 闭包陷阱与依赖项规范

闭包陷阱（Closure Trap）是 Hooks 使用者最容易踩的坑，它表现为：`useEffect`、`useCallback`、`useMemo` 等 Hook 的回调函数中读取到的变量值不是最新值，而是创建该闭包时的"快照值"。

```jsx
function ClosureTrap() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const timer = setInterval(() => {
      // ❌ 每次触发时 count 始终是 0
      console.log(count);
    }, 1000);
    return () => clearInterval(timer);
  }, []); // 空依赖数组

  return <button onClick={() => setCount(c => c + 1)}>+1</button>;
}
```

**依赖项规范的五条原则：**

1. **声明所有用到的外部变量**：任何在 useEffect/useCallback/useMemo 回调中读取的 props、state、衍生值，都应声明在依赖数组中。
2. **区分引用稳定性和值稳定性**：`useRef` 的 `.current` 值变化不会触发重新渲染，且 `ref` 对象本身引用稳定，可以不放在依赖中。`dispatch`（来自 `useReducer`）和 `setState` 函数也是引用稳定的。
3. **避免对象/数组直接作为依赖**：在 JSX 中直接创建的 `{ id: 1 }` 或 `[1,2,3]` 每次渲染都是新的引用，会导致无限重渲染。应使用 `useMemo` 稳定引用。
4. **优先使用函数式更新**：当新状态依赖旧状态时，使用 `setCount(prev => prev + 1)` 替代 `setCount(count + 1)`，可以减少依赖项需求。
5. **使用 `useCallback` 稳定回调引用**：将回调函数用 `useCallback` 包裹，避免子组件因引用变化而不必要地重新渲染。

### 4. eslint-plugin-react-hooks 原理

`eslint-plugin-react-hooks` 的 `rules-of-hooks` 规则是最广泛使用的 ESLint 规则之一。其底层实现原理基于 **AST（抽象语法树）静态分析**。

**检测流程：**

```
源代码 → 解析为 AST（espree / @typescript-eslint/parser）
  → 遍历 AST 节点
  → 收集所有名为 "use" + 大写字母开头的函数调用
  → 对每个调用，向上追溯其祖先节点
  → 检查祖先链中是否包含：
      - IfStatement（条件语句）
      - ConditionalExpression（三目运算）
      - ForStatement / WhileStatement / DoWhileStatement（循环语句）
      - SwitchStatement（switch）
      - LogicalExpression 或 BinaryExpression 中的短路运算
      - try/catch/finally
      - 嵌套函数声明（普通函数内部调用 Hook，规则二）
```

关键源码逻辑（简化版本）：

```javascript
// 伪代码：eslint-plugin-react-hooks 的核心检测逻辑
function checkNode(context, node) {
  // 1. 确认是以 "use" 开头的函数调用
  if (!isHookCall(node)) return;

  // 2. 向上回溯 AST，检测是否在条件/循环中
  let current = node.parent;
  while (current) {
    if (isConditionalStatement(current)) {
      context.report({
        node,
        message: `React Hook "${node.callee.name}" is called conditionally. ` +
                 `React Hooks must be called in the exact same order in every ` +
                 `component render.`,
      });
      return;
    }
    if (isLoopStatement(current)) {
      context.report({
        node,
        message: `React Hook "${node.callee.name}" may be executed more than once. ` +
                 `Hooks must be called exactly once per render.`,
      });
      return;
    }
    current = current.parent;
  }
}
```

**静态分析的局限性：**

静态分析无法覆盖所有场景。例如以下情况 ESLint 无法捕捉：

```jsx
function ProblematicComponent({ flag }) {
  // ESLint 能检查到条件调用
  // if (flag) { useState(0); }

  // 但无法检测到以下动态条件：
  const hooks = [useState, useEffect]; // 动态选择 Hook
  hooks[flag ? 0 : 1]();               // 运行时才确定调用哪个 Hook

  // 或者通过对象属性间接调用
  const api = { useState: React.useState };
  api.useState(0); // 不再是直接调用 "useState" 标识符

  // 或者在 Proxy 劫持后调用
  return <div />;
}
```

这也是为什么官方要求开发者除了依赖 ESLint 规则之外，还需要**理解规则背后的原理**来规避 ESLint 覆盖不到的场景。

### 5. 自定义 Hook 的规则传递

自定义 Hook 本质上是将调用 Hooks 的逻辑封装到一个以 `use` 开头的函数中。这个 `use` 前缀不仅是命名约定，更被 ESLint 规则和 React 内部机制用来识别"这是一个 Hook 调用节点"。

```jsx
// ✅ 正确的自定义 Hook
function useOnlineStatus() {
  const [isOnline, setIsOnline] = useState(true);

  useEffect(() => {
    const handleOnline = () => setIsOnline(true);
    const handleOffline = () => setIsOnline(false);

    window.addEventListener('online', handleOnline);
    window.addEventListener('offline', handleOffline);

    return () => {
      window.removeEventListener('online', handleOnline);
      window.removeEventListener('offline', handleOffline);
    };
  }, []);

  return isOnline;
}
```

**规则传递的本质：**

当你调用 `useOnlineStatus()` 时，这个调用本身必须出现在组件的最顶层，不能出现在条件语句中。自定义 Hook 内部的 Hooks（`useState`, `useEffect`）会"展开"到当前组件的 `memoizedState` 链表中。也就是说，调用自定义 Hook 相当于把 Hook 内部的 `useState` 和 `useEffect` 按顺序"内联"到当前组件的链表中。

```jsx
function MyComponent() {
  // 以下调用将在链表中展开为 3 个节点：
  // hook0: useState (来自 useOnlineStatus)
  // hook1: useEffect (来自 useOnlineStatus)
  // hook2: useState (下面这个)
  const isOnline = useOnlineStatus();
  const [count, setCount] = useState(0);
}
```

更重要的是，**自定义 Hook 内部同样遵守两条规则**——这意味着你也不能在自定义 Hook 内部的条件或循环中调用 Hooks。

### 6. 严格模式下 Hooks 的故意双调用

React 的 `StrictMode` 在开发环境下会故意双调用（Double Invoke）以下函数体：

- 函数组件本体
- `useState` / `useReducer` 的初始化器（initializer）
- `useMemo` 的回调函数
- `useEffect` / `useLayoutEffect` 等副作用的回调

```jsx
function StrictModeComponent() {
  // 开发环境下此函数体被调用两次
  const [state, setState] = useState(() => {
    // 这个初始化器也会被调用两次
    const initialValue = expensiveComputation();
    return initialValue;
  });

  useEffect(() => {
    // 这个回调也会被调用两次（挂载 → 卸载 → 再挂载）
    console.log('Effect runs');
    return () => {
      console.log('Effect cleanup');
    };
  }, []);
}
```

**为什么要双调用？**

严格模式的双调用是为了检测**不纯的函数**。如果副作用回调或初始化器中包含网络请求、DOM 操作、全局状态修改等不纯操作，双调用会立即暴露问题——因为同一个操作被意外执行了两次。

**挂载流程在 StrictMode 下的实际顺序：**

```
1. 组件渲染（第 1 次）→ 产生副作用
2. 组件渲染（第 2 次）→ 副作用被覆盖
3. 实际挂载（1 次）→ effect 回调只执行 1 次
4. 卸载（模拟）→ effect cleanup 执行
5. 重新挂载 → effect 回调再次执行
```

注意：在 React 18+ 的 StrictMode 中，React 会**真正执行 effect → cleanup → effect** 的完整生命周期，用于检测 `useEffect` 中的清理函数是否正确执行。这是为什么很多开发者在升级到 React 18 后发现某些 effect 在开发环境下执行了两次的原因。

## 实战案例

### 案例一：条件 Hook 导致购物车状态错乱

**场景：** 电商平台商品列表页，根据用户是否登录展示不同的交互组件。

```jsx
// ❌ 有问题的实现
function ProductCard({ product, isLoggedIn }) {
  const [quantity, setQuantity] = useState(1);

  if (isLoggedIn) {
    // ❌ 条件调用 Hook
    const [showReview, setShowReview] = useState(false);
  }

  const [favorited, setFavorited] = useState(false);

  useEffect(() => {
    trackProductView(product.id);
  }, [product.id]);

  return (
    <div>
      <span>{quantity}</span>
      <button onClick={() => setFavorited(!favorited)}>
        {favorited ? '❤️' : '🤍'}
      </button>
    </div>
  );
}
```

**问题分析：**

当用户切换登录状态时，`showReview` 这个 Hook 的存在与否发生变化，导致 `favorited` 从链表索引 3 偏移到索引 2，`useEffect` 从索引 4 偏移到 3。最终的后果是——`favorited` 读取到了 `showReview` 残留的 `false` 状态（首次为未登录），或者更糟——`false` 覆盖了真实的收藏状态。

**解决方案：**

将所有 Hooks 放到顶层，条件逻辑用纯 JS 控制：

```jsx
// ✅ 正确的实现
function ProductCard({ product, isLoggedIn }) {
  const [quantity, setQuantity] = useState(1);
  const [showReview, setShowReview] = useState(false); // ✅ 移到顶层
  const [favorited, setFavorited] = useState(false);

  useEffect(() => {
    trackProductView(product.id);
  }, [product.id]);

  // 用条件渲染替代条件 Hook
  const reviewSection = isLoggedIn && showReview ? (
    <ReviewDialog product={product} />
  ) : null;

  return (
    <div>
      <span>{quantity}</span>
      <button onClick={() => setFavorited(!favorited)}>
        {favorited ? '❤️' : '🤍'}
      </button>
      {reviewSection}
    </div>
  );
}
```

### 案例二：闭包陷阱导致倒计时组件异常

**场景：** 一个需要精确倒计时的考试计时器组件。

```jsx
// ❌ 有问题的实现
function ExamTimer({ durationMinutes }) {
  const [timeLeft, setTimeLeft] = useState(durationMinutes * 60);

  useEffect(() => {
    const timer = setInterval(() => {
      // ❌ 闭包陷阱：timeLeft 始终是初始值
      if (timeLeft <= 0) {
        clearInterval(timer);
        alert('时间到！');
        return;
      }
      setTimeLeft(timeLeft - 1); // 每次都是 1800 - 1 = 1799
    }, 1000);
    return () => clearInterval(timer);
  }, []); // 空依赖导致闭包捕获的是第一次渲染的 timeLeft

  return <div>{formatTime(timeLeft)}</div>;
}
```

**问题根因：**

`useEffect` 的空依赖数组 `[]` 导致 effect 回调在组件整个生命周期内只创建一次。闭包捕获的是第一次渲染时的 `timeLeft` 值（假设 `durationMinutes = 30`，则 `timeLeft = 1800`）。每秒调用 `setTimeLeft(timeLeft - 1)` 实际上是在做 `setTimeLeft(1799)`——因为闭包中的 `timeLeft` 始终是初始值。倒计时永远卡在 1799 秒。

**解决方案（三种方式）：**

```jsx
// 方案一：使用函数式更新（推荐）
useEffect(() => {
  const timer = setInterval(() => {
    setTimeLeft(prev => {
      if (prev <= 1) { // prev 是当前最新值
        clearInterval(timer);
        alert('时间到！');
        return 0;
      }
      return prev - 1;
    });
  }, 1000);
  return () => clearInterval(timer);
}, []);

// 方案二：将 timeLeft 加入依赖数组（要处理 cleanup）
useEffect(() => {
  if (timeLeft <= 0) return;
  const timer = setInterval(() => {
    setTimeLeft(prev => prev - 1);
  }, 1000);
  return () => clearInterval(timer);
}, [timeLeft]); // 每次 timeLeft 变化都重建计时器

// 方案三：使用 useRef 保存最新值
const timeLeftRef = useRef(timeLeft);
timeLeftRef.current = timeLeft;

useEffect(() => {
  const timer = setInterval(() => {
    if (timeLeftRef.current <= 0) {
      clearInterval(timer);
      alert('时间到！');
      return;
    }
    setTimeLeft(prev => prev - 1);
  }, 1000);
  return () => clearInterval(timer);
}, []);
```

方案一最为简洁——利用函数式更新从队列中读取最新值，完全避免了闭包捕获陈旧值的问题。方案二也能工作但每次 `timeLeft` 变化都会重建 `setInterval`，在高频场景下可能引入时序问题。方案三通过 `ref` 绕过了 `useEffect` 的闭包限制，适合需要读取最新值但不希望 effect 重新执行的场景。

### 案例三：自定义 Hook 释放链表的条件断连

**场景：** 一个通用的分页 Hook，根据不同数据类型使用不同的分页策略。

```jsx
// ❌ 错误的抽象：条件调用内部 Hook
function usePagination(dataType) {
  if (dataType === 'cursor') {
    // ❌ 条件调用 Hooks
    const [cursor, setCursor] = useState(null);
    const [hasMore, setHasMore] = useState(true);
    return { next: () => fetchNextPage(cursor), hasMore };
  }

  if (dataType === 'offset') {
    // ❌ 条件调用 Hooks
    const [page, setPage] = useState(1);
    const [totalPages, setTotalPages] = useState(0);
    return { next: () => setPage(p => p + 1), totalPages };
  }

  return null;
}
```

**问题：** 当 `dataType` 从 `'cursor'` 切换为 `'offset'` 时，这个自定义 Hook 内部的状态链表节点数从 2（cursor, hasMore）变为 2（page, totalPages），但由于头两个节点在首次渲染的链表中已经固定，状态会完全错乱。

**解决方案：** 将可变策略设计为纯数据而非 Hook 条件：

```jsx
// ✅ 正确：统一 Hook 结构，策略层用纯对象
function usePagination(dataType) {
  const [paginationState, setPaginationState] = useState({
    type: dataType,
    cursor: null,
    hasMore: true,
    page: 1,
    totalPages: 0,
  });

  const strategy = useMemo(() => {
    if (dataType === 'cursor') {
      return {
        next: () => setPaginationState(prev => ({
          ...prev,
          cursor: fetchNextPage(prev.cursor),
        })),
        hasMore: paginationState.hasMore,
      };
    }
    // offset 策略
    return {
      next: () => setPaginationState(prev => ({
        ...prev,
        page: prev.page + 1,
      })),
      totalPages: paginationState.totalPages,
    };
  }, [dataType, paginationState]);

  return strategy;
}
```

统一为 `useState` + `useMemo` 的结构，所有 Hooks 调用点保持不变，变化的只有纯逻辑。

## 底层原理（结合 React 18+ 源码分析）

### 1. Hooks 在 Fiber 上的存储结构

React 内部通过 Fiber 架构实现组件树的增量渲染。每个函数组件对应一个 Fiber 节点，其上的 `memoizedState` 字段存储了该组件所有 Hooks 的链表。

Fiber 节点的简化类型定义：

```typescript
interface Fiber {
  // 当前组件的状态链表（Hooks 链表头）
  memoizedState: Hook | null;

  // 上一次渲染的 Fiber（双缓存机制）
  alternate: Fiber | null;

  // 组件类型（函数组件这里就是函数本身）
  type: FunctionComponent;

  // 当前 props
  pendingProps: any;
  memoizedProps: any;

  // 副作用标记
  flags: Flags;
  subtreeFlags: Flags;
  // ...
}

interface Hook {
  // Hook 当前缓存的状态值
  memoizedState: any;

  // 基础状态（用于中断恢复）
  baseState: any;

  // 基础更新队列（用于中断恢复）
  baseQueue: Update<any, any> | null;

  // 等待处理的更新队列
  queue: UpdateQueue<any, any> | null;

  // 指向下一个 Hook
  next: Hook | null;
}
```

### 2. 渲染阶段的 Hooks 调度

React 的渲染分为两个阶段：**render 阶段**（也称为 reconcile 阶段）和 **commit 阶段**。Hooks 主要在 render 阶段被调用和调度。

**Render 阶段流程：**

```
beginWork()
  → 判断组件类型（Function Component）
  → renderWithHooks()
    → 设置全局 Hooks 上下文
    → 调用函数组件本身（执行所有 Hooks 调用）
    → 收集所有 Hooks 调用结果
    → 返回子 Fiber
```

`renderWithHooks` 函数是 Hooks 调度的心脏：

```javascript
// ReactFiberHooks.old.js（简化版）
export function renderWithHooks(
  current: Fiber | null,
  workInProgress: Fiber,
  Component: Function,
  props: any,
  secondArg: any,
  nextRenderLanes: Lanes,
): any {
  // 设置当前渲染的 Fiber
  currentlyRenderingFiber = workInProgress;
  workInProgress.memoizedState = null;

  // 清空全局指针
  currentHook = null;
  workInProgressHook = null;
  didScheduleRenderPhaseUpdate = false;

  // 根据是首次渲染还是更新，设置不同的 dispatcher
  ReactCurrentDispatcher.current =
    current === null || current.memoizedState === null
      ? HooksDispatcherOnMount  // Mount 阶段
      : HooksDispatcherOnUpdate; // Update 阶段

  // 执行函数组件，内部会调用 Hooks
  // 这些 Hooks 调用实际上被 dispatcher 代理
  let children = Component(props, secondArg);

  // 检查是否在渲染中安排了更新
  if (didScheduleRenderPhaseUpdate) {
    // 处理 render 阶段触发的更新
    do {
      didScheduleRenderPhaseUpdate = false;
      numberOfReRenders += 1;
      children = Component(props, secondArg);
    } while (didScheduleRenderPhaseUpdate);
  }

  // 清理上下文
  ReactCurrentDispatcher.current = ContextOnlyDispatcher;
  // ...

  return children;
}
```

**关键细节：** React 使用 `ReactCurrentDispatcher.current` 全局变量来动态切换 Hooks 的实现。Mount 阶段和 Update 阶段的 `useState`、`useEffect` 等实现完全不同。当不在 React 渲染上下文时（如普通函数中调用 Hook），`current` 被设置为 `ContextOnlyDispatcher`，调用任何 Hook 都会抛出不友好的错误提示。

### 3. Mount 阶段的 Hooks 创建

以 `useState` 为例，Mount 阶段的实现会创建一个新的 Hook 节点并追加到链表尾部：

```javascript
// Mount 阶段 useState 的实现
function mountState<S>(
  initialState: (() => S) | S,
): [S, Dispatch<BasicStateAction<S>>] {
  const hook = mountWorkInProgressHook();

  if (typeof initialState === 'function') {
    initialState = initialState();
  }

  hook.memoizedState = hook.baseState = initialState;

  const queue = {
    pending: null,
    lanes: NoLanes,
    dispatch: null,
    lastRenderedReducer: basicStateReducer,
    lastRenderedState: (initialState: any),
  };
  hook.queue = queue;

  const dispatch = dispatchSetState.bind(
    null,
    currentlyRenderingFiber,
    queue,
  );
  queue.dispatch = dispatch;

  return [hook.memoizedState, dispatch];
}
```

### 4. 更新阶段的 Hook 读取

Update 阶段，`useState` 的实现会从当前 Fiber 的 `alternate`（上一次渲染的 Fiber）中读取对应的 Hook 节点：

```javascript
function updateState<S>(
  initialState: (() => S) | S,
): [S, Dispatch<BasicStateAction<S>>] {
  return updateReducer(basicStateReducer, (initialState: any));
}

function updateReducer<S, I, A>(
  reducer: (S, A) => S,
  initialArg: I,
  init?: I => S,
): [S, Dispatch<A>] {
  const hook = updateWorkInProgressHook();
  const queue = hook.queue;

  // 处理更新队列
  if (queue !== null) {
    // 从 pending 链表中获取所有待处理的更新
    const pending = queue.pending;
    if (pending !== null) {
      // 拼接更新链表并计算新状态
      // ...
      hook.memoizedState = newState;
    }
  }

  const dispatch = queue.dispatch;
  return [hook.memoizedState, dispatch];
}
```

至此我们可以清楚地看到：`updateWorkInProgressHook` 沿着 `currentHook.next` 依次获取节点。如果某个 Hook 在本次渲染中没有被调用，`currentHook.next` 的推进就会错位，从而导致后续所有 `useState`、`useEffect` 等取错状态。

### 5. 严格模式的双调用机制

严格模式的双调用是通过在 `renderWithHooks` 中重置 Fiber 结构实现的简化逻辑：

```javascript
// 伪代码：StrictMode 的双调用逻辑
function renderWithHooksInStrictMode(current, workInProgress, Component, props) {
  // 第一次调用：完成整个渲染
  const children1 = renderWithHooks(current, workInProgress, Component, props);

  // 重置 workInProgress Fiber 到初始状态
  resetFiberForStrictMode(workInProgress);

  // 第二次调用：完全相同的渲染
  const children2 = renderWithHooks(current, workInProgress, Component, props);

  // 使用第二次渲染的结果
  return children2;
}
```

这就是为什么在 StrictMode 下，`useMemo` 的回调会执行两次、`useEffect` 会经历挂载→卸载→再挂载的完整周期的原因。

## 高频面试题解析

### Q1：为什么 Hooks 不能放在条件语句中？

**核心答案：** React 的 Hooks 是依赖**调用顺序**与 `memoizedState` 链表上的节点一一对应的。条件语句的存在会改变调用顺序，导致链表断裂、状态错位。

**加分回答：** 可以进一步展开，说明 `updateWorkInProgressHook` 函数通过 `currentHook.next` 来推进链表。条件调用导致推进步调不一致，后续所有 Hooks 获取到的状态都是错误。此外可以提到，React 团队在设计 Hooks 时，**刻意选择了基于顺序关联而非 key 关联**的方式，是为了性能考虑——顺序关联不需要创建和维护 Map 数据结构，内存占用更小。

### Q2：为什么 `useEffect` 的回调在 StrictMode 下执行了两次？

**核心答案：** StrictMode 在开发环境下会故意模拟"挂载 → 卸载 → 再挂载"的流程，用于检测 effect 中的清理函数是否正确实现，以及 effect 回调是否包含不应重复执行的副作用。

**加分回答：** 这是 React 18 引入的"可复用 State"（Reusable State）机制的预演。React 18 在 StrictMode 下 effect 会执行 **mount → cleanup → mount** 模式，这模拟了未来可能会有的"组件卸载后状态保留并在后续重新挂载"的场景。如果 effect 中没有妥善实现 cleanup（比如忘记取消订阅、清除定时器），就会在第二次 mount 时产生重复的副作用实例。

### Q3：`useCallback` 和 `useMemo` 的依赖项写错了会导致什么后果？

**核心答案：** 依赖项不足会导致闭包捕获陈旧值（Stale Closure）；依赖项过多会导致不必要的重计算或重渲染。

**详细分析：**

- **不足的依赖（Missing Dependencies）**：闭包中读取的变量是创建时的快照值。如果 `useEffect` 中依赖了 `props.id` 却没有在依赖数组中声明，effect 将始终使用第一次渲染时的 `props.id`。
- **多余的依赖（Unnecessary Dependencies）**：导致效果在无关值变化时也被重新执行。如果是 `useMemo`，会导致浪费计算；如果是 `useEffect`，可能导致意外重发请求或重建监听器。

**解决方案：**
- 使用 `eslint-plugin-react-hooks` 的 exhaustive-deps 规则
- 使用 `useReducer` 对复杂状态进行聚合管理，减少依赖项数量
- 使用 `useRef` 存储不需要触发重渲染的"跨渲染变量"

### Q4：自定义 Hook 必须在函数名上加 `use` 前缀，这是强制规定吗？

**核心答案：** 从技术实现角度，React 本身并不强制检查函数名是否以 `use` 开头。但这个前缀对于 **eslint-plugin-react-hooks** 来说是必要的——插件通过检查函数名是否以 `use` 开头（后跟大写字母）来判断一个函数是否是自定义 Hook，从而正确应用 Hooks 规则检查。

**加分回答：** 不遵守 `use` 前缀约定的后果：
1. ESLint 不会检查该函数内部的 Hooks 规则
2. 团队成员无法通过命名快速判断这是一个"有状态"的函数
3. React DevTools 不会将其标记为 Hook 并展示内部状态
4. 未来的 React 编译器（React Forget / React Compiler）可能也会依赖 `use` 前缀来优化代码

### Q5：如何解决 `useEffect` 中读取不到最新 state 的问题？

**核心答案：** 这是一个典型的闭包陷阱问题。最常见的三种解决方案：

1. **函数式更新**：`setState(prev => prev + 1)` —— 不依赖外部变量，永远从更新队列读取最新值。
2. **将变量加入依赖数组**：`useEffect(() => {...}, [variable])` —— 变量变化时重建 effect，但要注意清理。
3. **使用 ref 存储最新值**：`ref.current = value; useEffect(() => { /* 使用 ref.current */ }, [])` —— 绕过闭包限制，适合订阅等场景。

**深入分析：** 三种方案的权衡：
- 方案 1 只对纯"基于旧状态计算新状态"的场景有效
- 方案 2 通用性强但可能引起 effect 重建过于频繁
- 方案 3 适合"需要读取最新值但不应重新订阅"的场景（如 WebSocket 连接）

## 总结与扩展

### 核心要点回顾

1. **Hooks 两条规则的根源**：Fiber 的 `memoizedState` 链表结构决定了 Hooks 必须按固定顺序、固定数量在每次渲染中被调用。这是理解所有 Hooks 使用规则的基础。

2. **条件调用和循环调用的本质危害**：改变了链表节点的访问顺序或数量，导致后续所有 Hooks 读取到错误的状态。

3. **闭包陷阱的根本原因**：JavaScript 闭包的特性决定了回调函数会捕获创建时的变量值。正确声明 `useEffect`/`useCallback`/`useMemo` 的依赖数组是避免闭包陷阱的关键。

4. **ESLint 规则的价值与局限**：静态分析能覆盖大部分显式的规则违反，但无法检测动态调用、Proxy 劫持等运行时场景。理解底层机制比依赖 lint 规则更可靠。

5. **严格模式的双调用是一种特性而非 bug**：它帮助开发者提前发现不纯的副作用和清理不完整的问题。

### 进阶阅读方向

- **React Compiler（原 React Forget）**：React 19+ 引入的新编译器，可以自动记忆组件，不再需要开发者手动为 `useMemo`、`useCallback` 填写依赖数组。但 Hooks 的两条规则仍然适用。
- **React Server Components（RSC）与 Hooks**：服务端组件中不能使用 Hooks，客户端组件（`'use client'`）可以。理解这个区分对现代化 React 开发至关重要。
- **Concurrent Mode 与 Suspense**：在并发模式下，组件可能被中断和恢复渲染，对 Hooks 的纯度和幂等性提出了更高要求。
- **useEffect 的完整生命周期**：从 React 18 开始，`useEffect` 在开发环境 StrictMode 下会执行"挂载→清理→再次挂载"，理解其完整的生命周期对编写健壮的副作用代码很重要。

以上两条规则和背后机制的知识，不仅是面试中的高频考点，更是每一位 React 开发者每天编码时都在依赖的执行契约——每次 `useState` 调用背后，都是 React 的 Fiber 引擎在默默维护一个顺序敏感的状态链表，而你越理解这个链表，就越能掌控 React。
