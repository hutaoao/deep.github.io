## 一句话概括

**Vue 2 中 v-for 优先级高于 v-if，Vue 3 中 v-if 优先级高于 v-for；两者同时使用会导致每个循环迭代都执行条件判断，引发性能浪费和逻辑歧义，应通过计算属性或 template 包裹来规避。**

---

## 背景

在 Vue 模板开发中，`v-if` 和 `v-for` 同时出现在同一个元素上是一个常见但危险的操作。许多开发者直觉上认为「先过滤再循环」或「先循环再过滤」，但 Vue 2 和 Vue 3 的编译行为完全不同，这种差异往往是 bug 的来源。

官方文档明确警告：**不建议在同一元素上同时使用 `v-if` 和 `v-for`**。理解其优先级差异及底层编译逻辑，是写出高性能 Vue 模板的关键。

---

## 概念与定义

### 基本定义

| 指令 | 作用 | 编译产物 |
|------|------|----------|
| `v-if` | 条件渲染，根据表达式真假决定是否挂载 DOM | 生成三元表达式或条件语句 |
| `v-for` | 列表渲染，基于数据源循环生成 DOM | 生成 `renderList()` 调用 |

### 优先级（核心差异）

```
Vue 2：v-for 优先级 > v-if
Vue 3：v-if 优先级 > v-for
```

这意味着两者同时出现时，Vue 2 先执行循环再判断条件，Vue 3 先判断条件再决定是否循环。

## 最小示例

### Vue 2 中的问题代码

```vue
<!-- ❌ Vue 2：每次循环都要判断 v-if，即使 list 大部分元素不需要渲染 -->
<ul>
  <li v-for="item in list" :key="item.id" v-if="item.isActive">
    {{ item.name }}
  </li>
</ul>
```

### Vue 3 中的问题代码

```vue
<!-- ❌ Vue 3：v-if 先执行，但 item 在 v-if 中不可用，直接报错 -->
<ul>
  <li v-for="item in list" :key="item.id" v-if="item.isActive">
    {{ item.name }}
  </li>
</ul>
```

### 正确写法（通用，Vue 2 / Vue 3 均推荐）

```vue
<!-- ✅ 使用计算属性预先过滤 -->
<ul>
  <li v-for="item in activeList" :key="item.id">
    {{ item.name }}
  </li>
</ul>

<script setup>
import { computed } from 'vue'

const props = defineProps(['list'])
const activeList = computed(() => props.list.filter(item => item.isActive))
</script>
```

## 核心知识点拆解

### 1. Vue 2 的编译逻辑

Vue 2 的编译器在解析同一元素上的多个指令时，按照**指令优先级**决定生成顺序。`v-for` 的优先级（1）高于 `v-if`（2），因此先生成循环结构，再在循环体内嵌入条件判断。

**编译产物（简化）：**

```js
// 模板：<li v-for="item in list" :key="item.id" v-if="item.isActive">
function render() {
  return this.list.map(item =>
    item.isActive
      ? createElement('li', { key: item.id }, item.name)
      : null
  ).filter(Boolean)
}
```

**问题：** 即使 `item.isActive` 为 false 的比例很高，`list.map()` 仍然遍历了整个数组，只是每个元素都返回 `null`，造成不必要的计算开销。

### 2. Vue 3 的编译逻辑

Vue 3 反过来，`v-if` 优先级高于 `v-for`。编译器先处理条件，再处理循环。

**编译产物（简化）：**

```js
// 模板：<li v-for="item in list" :key="item.id" v-if="item.isActive">
function render() {
  return this.list
    ? this.list.map(item => {
        // ❌ item 在 v-if 的作用域之外，访问不到！
        // 实际上 Vue 3 会直接报编译错误
      })
    : null
}
```

Vue 3 的编译器在检测到同一元素上同时存在 `v-if` 和 `v-for` 时，会**直接抛出警告或报错**，强制开发者修正写法。

### 3. 为什么 Vue 3 要反转优先级？

Vue 3 的设计者注意到 Vue 2 的行为容易引发歧义：

- 开发者往往**直觉上期望「先过滤再循环」**，但 Vue 2 实际是「先循环再过滤」
- 这种直觉与实现的背离导致大量低效代码
- 反转优先级后，`v-if` 无法访问 `v-for` 的迭代变量，编译期就能报错，提前暴露问题

### 4. 官方推荐的两种解决方案

**方案一：使用计算属性（最推荐）**

```js
// 预先过滤，只循环需要的数据
const activeList = computed(() => list.filter(item => item.isActive))
```

优点：数据层过滤，逻辑清晰，可复用，可测试。

**方案二：使用 template 包裹（需要访问迭代变量时）**

```vue
<!-- ✅ template 上写 v-for，子元素上写 v-if -->
<ul>
  <template v-for="item in list" :key="item.id">
    <li v-if="item.isActive">
      {{ item.name }}
    </li>
  </template>
</ul>
```

优点：不需要额外计算属性，适合简单场景。注意 `v-for` 写在 `template` 上，`v-if` 写在子元素上，两者不在同一元素，优先级问题不复存在。

### 5. `v-if` 与 `v-for` 在不同元素上的情况

```vue
<!-- ✅ v-if 在父元素，v-for 在子元素：完全没问题 -->
<ul v-if="list.length">
  <li v-for="item in list" :key="item.id">
    {{ item.name }}
  </li>
</ul>
<p v-else>暂无数据</p>
```

这种写法中 `v-if` 和 `v-for` 不在同一元素上，不存在优先级冲突，是合法且常见的模式。

## 实战案例

### 案例一：权限按钮渲染（反例 → 正例）

**反例（Vue 2 常见写法）：**

```vue
<!-- ❌ 每个按钮都参与循环，再判断权限 -->
<div v-for="btn in allButtons" :key="btn.id">
  <button v-if="hasPermission(btn.permission)">
    {{ btn.label }}
  </button>
</div>
```

问题：`hasPermission()` 在每次渲染时对每个按钮都调用一次，共 N 次。

**正例：**

```vue
<!-- ✅ 计算属性预先过滤 -->
<script setup>
const authorizedButtons = computed(() =>
  allButtons.filter(btn => hasPermission(btn.permission))
)
</script>

<div v-for="btn in authorizedButtons" :key="btn.id">
  <button>{{ btn.label }}</button>
</div>
```

### 案例二：大型列表的性能优化

```vue
<!-- 场景：1000 条数据，只有 10 条需要渲染 -->
<!-- ❌ 反例：1000 次循环，990 次返回 null -->
<li v-for="item in hugeList" :key="item.id" v-if="item.score > 60">
  {{ item.name }}: {{ item.score }}
</li>

<!-- ✅ 正例：计算属性只返回 10 条，循环 10 次 -->
<script setup>
const passedList = computed(() =>
  hugeList.value.filter(item => item.score > 60)
)
</script>

<li v-for="item in passedList" :key="item.id">
  {{ item.name }}: {{ item.score }}
</li>
```

**性能对比（1000 条数据）：**

| 写法 | 循环次数 | 创建 VNode 数 | 内存占用 |
|------|----------|---------------|----------|
| 反例（同一元素） | 1000 | 1000 | 高 |
| 正例（计算属性） | 10 | 10 | 低 |

## 底层原理

### Vue 2 源码中的优先级定义

在 `vue/src/compiler/codegen/events.js` 和指令处理模块中，Vue 2 定义了指令的优先级：

```js
// vue/src/platforms/web/compiler/directives/index.js
// 优先级数值越小，优先级越高（与直觉相反！）
export const directivePriority = {
  'for': 1,   // 最高
  'if': 2,
  'else': 3,
  'else-if': 3,
  'model': 4,
  'on': 5,
  'bind': 6,
  'slot': 7,
  'text': 8,
  'html': 9,
  'show': 10   // 最低
}
```

编译时，编译器按照优先级**从低到高**依次处理（数值大的先处理），所以 `v-for`（1）最后被处理，即先生成循环结构，再在其中嵌入条件。

### Vue 3 编译器中的处理逻辑

Vue 3 在 `packages/compiler-core/src/transforms/vIf.ts` 和 `vFor.ts` 中，通过 `onExit` 钩子在处理 AST 节点时检查冲突：

```ts
// vue-next/packages/compiler-core/src/transforms/vIf.ts（简化逻辑）
export const transformIf: NodeTransform = (node, context) => {
  // 在 exit 阶段检查同一元素上是否存在 v-for
  if (node.props.some(p => p.name === 'for')) {
    context.onWarn(
      `v-if and v-for cannot be used on the same element. ` +
      `Use a <template> wrapper for v-for instead.`
    )
  }
}
```

Vue 3 的编译流程中，`v-if` 的 transform 先于 `v-for` 执行（因为优先级更高），当检测到同一元素上同时存在两者时，直接发出编译期警告。

### 为什么编译产物不同？

Vue 2 的渲染函数：

```js
// v-for 优先：先 map，再在 map 的回调里判断 v-if
with (this) {
  _c('ul', _l(list, function(item) {
    return item.isActive
      ? _c('li', { key: item.id }, [_v(_s(item.name))])
      : _e()  // _e = emptyVNode
  }), 0)
}
```

Vue 3 的渲染函数（正确写法，计算属性过滤后）：

```js
// 计算属性预先过滤，直接循环过滤后的数组
import { renderList as _renderList } from 'vue'
import { Fragment as _Fragment } from 'vue'

function render(_ctx) {
  return _renderList(_ctx.activeList, (item, index) => {
    return _createElementVNode('li', { key: item.id }, item.name)
  })
}
```

## 高频面试题解析

### Q1：Vue 2 和 Vue 3 中 v-if 和 v-for 的优先级有什么区别？

**答：**
- **Vue 2**：`v-for` 优先级高于 `v-if`。先循环，再在每个循环项上进行条件判断。
- **Vue 3**：`v-if` 优先级高于 `v-for`。先判断条件，再决定是否循环（此时迭代变量不可用，编译报错）。

### Q2：为什么 Vue 3 要修改优先级？

**答：** Vue 2 的行为与开发者的直觉相悖。大多数开发者写 `v-if` 和 `v-for` 在同一元素上时，期望的是「先过滤再循环」，但 Vue 2 实际是「先循环再过滤」，导致性能问题。Vue 3 反转优先级后，访问不到迭代变量，编译期直接报错，强制开发者使用正确写法（计算属性或 template 包裹）。

### Q3：在同一元素上同时使用 v-if 和 v-for 有什么问题？

**答：**
1. **性能问题**（Vue 2）：每次循环都执行条件判断，即使大部分元素不需要渲染
2. **逻辑歧义**：代码意图不清晰，维护者难以理解期望行为
3. **Vue 3 直接报错**：编译不通过，代码无法运行

### Q4：如何正确替代「在同一元素上使用 v-if 和 v-for」？

**答：**
1. **首选计算属性**：`computed(() => list.filter(...))`，数据层过滤，逻辑最清晰
2. **使用 template 包裹**：`v-for` 写在 template 上，`v-if` 写在子元素上
3. **分层写法**：`v-if` 控制容器显示，`v-for` 渲染列表，两者在不同元素上

### Q5：以下代码在 Vue 2 和 Vue 3 中分别有什么问题？

```vue
<li v-for="user in users" :key="user.id" v-if="user.isActive">
  {{ user.name }}
</li>
```

**答：**
- **Vue 2**：能运行，但每个 `user` 都会参与循环，即使 `isActive === false`，造成性能浪费
- **Vue 3**：编译报错，`v-if` 无法访问 `v-for` 的 `user` 变量，因为 `v-if` 先执行

**正确写法：**

```vue
<li v-for="user in activeUsers" :key="user.id">
  {{ user.name }}
</li>

<script setup>
const activeUsers = computed(() => users.value.filter(u => u.isActive))
</script>
```

## 总结与扩展

### 核心要点

1. **Vue 2**：`v-for` > `v-if`，先循环再判断，有性能问题
2. **Vue 3**：`v-if` > `v-for`，先判断再循环，访问不到迭代变量直接报错
3. **通用解决方案**：计算属性过滤（最推荐）或 template 分层
4. **不同元素上分别使用**：完全没问题，是推荐写法

### 最佳实践口诀

> **「v-if 与 v-for，永远不共元素；过滤交给 computed，template 分层也可行。」**

### 扩展阅读

- **Vue 官方风格指南**：明确将「避免 v-if 和 v-for 用在同一元素」列为必要条件
- **Vue 3 迁移指南**：`v-if` 和 `v-for` 优先级变化是 Breaking Change 之一
- **虚拟 DOM 层面的影响**：不必要的 VNode 创建会增加 diff 成本，尤其在大型列表中
- **ESLint 插件**：`eslint-plugin-vue` 的 `vue/no-use-v-if-with-v-for` 规则可在编译前捕获此问题

### 相关知识点串联

| 相关主题 | 关联点 |
|----------|--------|
| 计算属性 vs 方法 | 过滤场景用 computed 有缓存，性能更优 |
| 虚拟 DOM Diff | 多余 VNode 增加 diff 成本 |
| ESLint 规则 | `vue/no-use-v-if-with-v-for` 强制规范 |
| Vue 3 迁移 Breaking Changes | 优先级反转是重点之一 |
