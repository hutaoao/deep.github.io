## 一句话概括

本文从 TypeScript 源码层面深入解析 `Record`、`Exclude`、`Extract`、`NonNullable` 等核心内置工具类型的实现原理，厘清映射类型、分布式条件类型、`never` 类型传播及 `infer` 关键字等底层机制，帮助读者从"会用"进阶到"理解其所以然"。

## 背景

TypeScript 自带了一套工具类型（Utility Types），它们定义在 `lib.es5.d.ts` 中，仅有寥寥数十行代码，却是整个类型编程体系的基石。绝大多数开发者能熟练使用 `Partial<T>`、`Pick<T, K>`，却未必理解为什么 `Exclude<'a' | 'b' | 'c', 'a'>` 返回 `'b' | 'c'`，或者为什么把 `never` 塞进条件类型会得到 `never`。

了解工具类型的源码实现，至少有几点重要价值：

- **告别"黑盒"使用**：理解为什么某个语法会报错，为什么某个类型组合不会按预期工作；
- **编写自定义工具类型**：面试和工作中常见"实现 `MyPick`""实现 `MyReadonly`"类题目，底层原理相通；
- **深化类型系统认知**：条件类型分发、`infer`、映射类型是类型编程三驾马车，工具类型是它们的最佳教学载体。

## 概念与定义

### 工具类型概述

工具类型是 TypeScript 内置的泛型类型操作函数——它们**接受一个或多个类型参数，返回一个新的类型**。定义在 `node_modules/typescript/lib/lib.es5.d.ts` 中，从 TypeScript 2.1 起随版本逐步引入。

### 前置概念

要读懂源码，需要理解三个基础概念：

**1. 泛型约束（Generic Constraint）**：`<T extends SomeType>` 表示 `T` 必须满足某个形状约束。

**2. 映射类型（Mapped Type）**：`{ [P in K]: T[P] }` 遍历联合类型 `K` 中的每个成员，逐个转换属性类型。

**3. 条件类型与分发（Conditional Types & Distributive）**：`T extends U ? X : Y` 当 `T` 是裸类型参数且为联合类型时，TypeScript 会将其拆解为_distributive_形式——分别对每个成员求值再合并。

### 最小示例

```typescript
// Record：构造一个键为 K、值为 T 的对象类型
type CatInfo = Record<'name' | 'age', string>;
// { name: string; age: string }

// Exclude：从 T 中排除可赋值给 U 的类型
type T0 = Exclude<'a' | 'b' | 'c', 'a'>;
// 'b' | 'c'

// Extract：从 T 中提取可赋值给 U 的类型
type T1 = Extract<'a' | 'b' | 'c', 'a' | 'f'>;
// 'a'

// NonNullable：从 T 中排除 null 和 undefined
type T2 = NonNullable<string | number | null | undefined>;
// string | number
```

## 核心知识点拆解

### 基础工具类型源码回顾

在深入新工具类型之前，先回顾五个"经典款"的源码，它们构成了理解后续内容的知识基线：

```typescript
// 让所有属性变为可选
type Partial<T> = {
    [P in keyof T]?: T[P];
};

// 让所有属性变为必选
type Required<T> = {
    [P in keyof T]-?: T[P];
};

// 让所有属性变为只读
type Readonly<T> = {
    readonly [P in keyof T]: T[P];
};

// 从 T 中选择部分属性
type Pick<T, K extends keyof T> = {
    [P in K]: T[P];
};

// 从 T 中排除部分属性（基于 Pick + Exclude）
type Omit<T, K extends keyof any> = Pick<T, Exclude<keyof T, K>>;
```

**关键观察**：`Partial`、`Required`、`Readonly` 通过**映射类型对属性附加修饰符**；`Pick` 约束 `K` 必须是 `T` 的键，然后映射选取；`Omit` 则是 `Exclude` 与 `Pick` 的组合——这里的 `Exclude` 在运行时排除了 `K` 中的键名。

### Record 源码解析

```typescript
// lib.es5.d.ts 中的定义
type Record<K extends keyof any, T> = {
    [P in K]: T;
};
```

**逐行解析**：

| 部分 | 含义 |
|------|------|
| `K extends keyof any` | 约束 K 必须是 `keyof any`（即 `string | number | symbol`）的子类型，确保键合法 |
| `[P in K]` | 对于联合类型 K 中的每个成员 P，生成一个属性 |
| `: T` | 每个属性的值类型都是 T |

**为什么有三种键**？JavaScript 对象的键原本只能是 `string`，但 TypeScript 允许 `number` 和 `symbol` 也同样可以作为键存在。`keyof any` 就是这三者的联合。

**对比 `{ [key: string]: T }`**：索引签名 `{ [key: string]: T }` 允许任意 string 键，`Record<K, T>` 则严格限定在 `K` 指定的键集合内，类型更精确。

**典型陷阱**：

```typescript
// 错误：K 不能是 object
type Bad = Record<object, string>;
// Error: Type 'object' does not satisfy the constraint 'string | number | symbol'

// 正确
type Good = Record<'a' | 'b', string>; // √
```

### Exclude 源码解析

```typescript
type Exclude<T, U> = T extends U ? never : T;
```

这是**分布式条件类型**的教科书式应用。四行代码，理解起来却颇有深度：

**执行逻辑**：

当 `T` 是联合类型 `'a' | 'b' | 'c'` 时，TypeScript **将联合类型拆开**，分别判断每个成员：

1. `'a' extends 'a' ? never : 'a'` → `never`
2. `'b' extends 'a' ? never : 'b'` → `'b'`
3. `'c' extends 'a' ? never : 'c'` → `'c'`

合并结果：`never | 'b' | 'c'` → `'b' | 'c'`

**关键规则**：分布式条件类型的触发条件是——条件类型**检查的类型是一个裸类型参数**（即 `extends` 左边的 `T` 直接是泛型参数，没有被 `[]` 包裹或其他类型操作包裹）。

**如果不用裸类型参数**：下面的写法不会被分发：

```typescript
// 包裹之后不再分发
type ExcludeNonDist<T, U> = [T] extends [U] ? never : T;

type Test = ExcludeNonDist<'a' | 'b', 'a'>; // 结果是 'a' | 'b'，不是 'b'
```

因为 `[T]` 整体作为一个元组类型，不再对联合成员逐个分发。

**从不变量看联合**：这段代码也揭示了一个 TypeScript 的核心规则——`never` 在联合类型中被吸收，即 `never | 'b'` 等价于 `'b'`。

### Extract 源码解析

```typescript
type Extract<T, U> = T extends U ? T : never;
```

Extract 是 Exclude 的镜像操作，仅交换了 true/false 分支：

1. `'a' extends 'a' | 'f' ? 'a' : never` → `'a'`
2. `'b' extends 'a' | 'f' ? 'b' : never` → `never`
3. `'c' extends 'a' | 'f' ? 'c' : never` → `never`

合并：`'a' | never | never` → `'a'`

**注意**：Extract 判断的是"能否赋值给 U"，而非"和 U 中的某个相等"。`any` 类型能赋值给几乎所有类型，因此 `Extract<any, number>` 的结果是 `any`。

### NonNullable 源码解析

```typescript
type NonNullable<T> = T extends null | undefined ? never : T;
```

这段代码是 Exclude 的特殊情况，等价于 `Exclude<T, null | undefined>`。

```typescript
type Demo = NonNullable<string | null | undefined | number>;
// 1. string extends null | undefined ? never : string → string
// 2. null extends null | undefined ? never : null → never
// 3. undefined extends null | undefined ? never : undefined → never
// 4. number extends null | undefined ? never : number → number
// 结果：string | number
```

**面试常见考点**：`NonNullable<never>` 为什么是 `never`？因为 `never` 是空联合，分布式条件类型对于空联合直接返回 `never`。

### ReturnType 和 Parameters 源码解析

```typescript
type Parameters<T extends (...args: any) => any> = T extends (...args: infer P) => any ? P : never;

type ReturnType<T extends (...args: any) => any> = T extends (...args: any) => infer R ? R : any;
```

这里用到了 TypeScript 类型推断的核心关键字——`infer`。

**`infer` 的作用**：在条件类型的 `extends` 子句中声明一个待推断的类型变量，TypeScript 会在匹配时尝试推导出该变量的实际类型。

**Parameters 的执行过程**：

1. 泛型约束 `T extends (...args: any) => any` 确保 T 是函数类型
2. 条件类型中 `T extends (...args: infer P) => any` 匹配函数签名
3. TypeScript 从 `T` 中提取参数元组类型赋值给 `P`
4. 匹配成功返回 `P`，失败返回 `never`

```typescript
type Fn = (x: number, y: string) => boolean;

type Params = Parameters<Fn>;      // [x: number, y: string]
type Ret = ReturnType<Fn>;         // boolean
```

**`infer` 的更多应用**：

```typescript
// 提取 Promise 的值的类型
type Unwrap<T> = T extends Promise<infer U> ? U : T;

type A = Unwrap<Promise<string>>; // string
type B = Unwrap<string>;          // string

// 提取数组元素的类型
type Flatten<T> = T extends Array<infer U> ? U : T;

type C = Flatten<string[]>;  // string
type D = Flatten<number>;    // number
```

## 实战案例

### 案例1：投票机制——基于 Exclude + Extract

假设有一组用户状态，要统计"活跃用户"：

```typescript
type UserStatus = 'online' | 'offline' | 'away' | 'busy';
type ActiveStatus = 'online' | 'busy';

// 定义活跃用户状态
type ActiveUsers = Extract<UserStatus, ActiveStatus>;
// 'online' | 'busy'

// 定义非活跃用户状态
type InactiveUsers = Exclude<UserStatus, ActiveStatus>;
// 'offline' | 'away'
```

### 案例2：深度可选——使用映射类型+条件类型

```typescript
type DeepPartial<T> = {
    [P in keyof T]?: T[P] extends object ? DeepPartial<T[P]> : T[P];
};

interface Config {
    server: {
        host: string;
        port: number;
    };
    debug: boolean;
}

type PartialConfig = DeepPartial<Config>;
// {
//   server?: { host?: string; port?: number };
//   debug?: boolean;
// }
```

这是 `Partial` 的递归版，利用条件类型判断值是否为对象，是则递归处理。

### 案例3：函数类型过滤器——RetrunType + infer

```typescript
// 从联合函数类型中提取返回值为 string 的函数
type ReturnString<T> = T extends (...args: any[]) => infer R
    ? R extends string ? T : never
    : never;

type F1 = () => string;
type F2 = () => number;
type F3 = () => Date;

type StringFuncs = ReturnString<F1 | F2 | F3>;
// () => string
```

### 案例4：构建自己的 NonNullable

```typescript
// 排除 null 和 undefined
type MyNonNullable<T> = T extends null | undefined ? never : T;

// 深度排除：递归处理嵌套对象中的 null/undefined
type DeepNonNullable<T> = {
    [P in keyof T]-?: T[P] extends object
        ? DeepNonNullable<NonNullable<T[P]>>
        : NonNullable<T[P]>;
};

type Data = {
    name: string | null;
    meta: {
        age: number | null;
        tags: string[] | undefined;
    } | null;
};

type SafeData = DeepNonNullable<Data>;
// { name: string; meta: { age: number; tags: string[] } }
```

## 底层原理

### 条件类型分发（Distributive Conditional Types）

分发机制是理解 Exclude/Extract/NonNullable 的核心：

> 当条件类型 `T extends U ? X : Y` 中的 `T` 是一个"裸类型参数"时，TypeScript 会自动将联合类型拆解为单个成员，分别计算后再合并。

**触发条件精确描述**：

```typescript
// ✅ 分发：T 是裸类型参数
type Foo<T> = T extends string ? 'yes' : 'no';

// ❌ 不分发：T 被 [] 包裹
type Bar<T> = [T] extends [string] ? 'yes' : 'no';

// ❌ 不分发：T 出现在更复杂的类型中
type Baz<T> = { key: T } extends { key: string } ? 'yes' : 'no';
```

**为什么设计分发机制**？因为对于联合类型 `'a' | 'b'`，我们通常希望分别考察每个成员。如果保留整体性，`'a' | 'b' extends 'a'` 永远不成立（除非 `'a' | 'b'` 是 `'a'` 的子集，而联合是子集的上界而不是下界）。

### never 类型的传播规则

`never` 在类型系统中的行为堪称"幽灵"：

- 在联合类型中：`T | never` → `T`（被吸收）
- 在交叉类型中：`T & never` → `never`（吞噬一切）
- 在条件类型中：若 `T` 为 `never`，整个条件类型直接返回 `never` 而不触发分发

```typescript
// 当 T = never 时，以下条件类型不会分发
type Check<T> = T extends string ? true : false;
type Result = Check<never>; // never，不是 true 也不是 false
```

这是因为 `never` 作为空联合，分发时没有任何成员可以处理，TypeScript 将其短路处理。

### 映射类型的展开与修饰符

映射类型 `{ [P in K]: T }` 在展开时：

1. **遍历联合类型 K**：K 中的每个字符串字面量成为对象的键
2. **同态 vs 异态**：如果映射类型遍历 `keyof T` 且值类型依赖于 `T[P]`，称为_homomorphic_（同态），可以保留 `readonly` 和 `optional` 修饰符
3. **修饰符操作**：`?` 和 `-?` 控制可选性，`readonly` 和 `-readonly` 控制只读性

```typescript
// 去掉所有属性的 readonly
type Mutable<T> = {
    -readonly [P in keyof T]: T[P];
};

// 去掉所有属性的可选
type Concrete<T> = {
    [P in keyof T]-?: T[P];
};
```

### 泛型约束 vs 条件类型的区别

容易混淆的一点：`K extends keyof any` 在泛型约束中表示"K 是合法键类型的子集"，但它**只做静态检查**，不改变 K 的类型。而条件类型 `T extends U ? ...` 会在运行时（类型计算时）根据 T 的实际类型进行分支选择。

```typescript
// 泛型约束 - 编译时检查
function fn<T extends string>(arg: T) {}  // T 必须是 string 的子类型

// 条件类型 - 类型计算时分支
type IsString<T> = T extends string ? 'yes' : 'no';  // 根据 T 实际类型得到不同结果
```

## 高频面试题解析

### Q1：实现 `MyPick<T, K>`

**要求**：从 `T` 中选取 `K` 指定的属性。

```typescript
type MyPick<T, K extends keyof T> = {
    [P in K]: T[P];
};
```

**解析**：约束 K 必须是 T 的键，然后用映射类型遍历 K 并取出对应值类型。

### Q2：实现 `MyExclude<T, U>`

```typescript
type MyExclude<T, U> = T extends U ? never : T;
```

**追问**：为什么不用 `[]` 包裹？追问的就是分布式条件类型的分发机制。

### Q3：实现 `MyReadonly2<T, K>`

**要求**：将 `T` 中 `K` 指定的属性设为 `readonly`，其余保持原样。

```typescript
type MyReadonly2<T, K extends keyof T = keyof T> = {
    readonly [P in K]: T[P];
} & {
    [P in Exclude<keyof T, K>]: T[P];
};
```

**解析**：利用交叉类型将只读部分和原样部分合并，默认 `K = keyof T` 时等价于 `Readonly<T>`。

### Q4：为什么 `Exclude<never, string>` 是 `never`？

**答案**：因为 `never` 是空联合类型，分布式条件类型对于空联合不进行任何分发，直接返回 `never`。

### Q5：`Parameters` 中 `infer P` 推断不出来会得到什么？

```typescript
type Parameters<T extends (...args: any) => any> = T extends (...args: infer P) => any ? P : never;
```

由于泛型约束已经确保了 `T` 是函数类型，条件类型的匹配必定成功，所以永远不会走到 `never` 分支。但如果把约束去掉：

```typescript
type BadParams<T> = T extends (...args: infer P) => any ? P : never;
type X = BadParams<number>; // never
```

## 总结与扩展

本文深入剖析了 TypeScript 内置工具类型的源码实现。核心要点可以归纳为一张对照表：

| 工具类型 | 核心机制 | 实现思路 |
|---------|---------|--------|
| `Partial/Required/Readonly` | 映射类型 + 修饰符 | 遍历 `keyof T`，添加或移除修饰符 |
| `Record<K,T>` | 映射类型 + 约束 | 以 K 为键集合，统一设置值类型 |
| `Pick<T,K>` | 映射类型 + 约束 | 从 T 的键中选取 K 子集 |
| `Omit<T,K>` | Exclude + Pick 组合 | 先排除键再选取 |
| `Exclude<T,U>` | 分布式条件类型 | 从联合中筛出不在 U 中的成员 |
| `Extract<T,U>` | 分布式条件类型 | 从联合中筛出在 U 中的成员 |
| `NonNullable<T>` | 分布式条件类型 | 排除 null 和 undefined |
| `ReturnType<T>` | `infer` 推断 | 从函数签名中解构返回值 |
| `Parameters<T>` | `infer` 推断 | 从函数签名中解构参数 |

### 推荐阅读

所有工具类型的源码都集中在 TypeScript 安装目录下的 `lib.es5.d.ts` 文件中。建议读者打开该文件（位置：`node_modules/typescript/lib/lib.es5.d.ts`），搜索 `type ` 关键字，直接阅读原始定义。此外，TypeScript 官方手册的[Utility Types](https://www.typescriptlang.org/docs/handbook/utility-types.html) 和 [Conditional Types](https://www.typescriptlang.org/docs/handbook/2/conditional-types.html) 章节也是权威参考。

进阶方向：尝试实现 `DeepReadonly`、`RequiredByKeys`、`ConditionalPick` 等更复杂的工具类型，每一次实现都是对类型编程能力的锤炼。
