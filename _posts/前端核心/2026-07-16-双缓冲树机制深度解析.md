## 一句话概括

React Fiber 的双缓冲树机制维护了两棵 Fiber 树——current 树（当前屏幕上显示的 UI 对应的树）和 workInProgress 树（正在构建中的树），通过 alternate 指针在两者之间复用 Fiber 节点，实现渲染过程中对用户无感知的状态切换，是 Fiber 架构支持可中断渲染的基石。

## 背景与意义

### 为什么需要两棵树？

在 Stack Reconciler 时代，React 只有一棵虚拟 DOM 树。更新流程是：

```
状态变更 → 递归遍历虚拟 DOM 树 → 生成新的虚拟 DOM → 与旧树 Diff → 提交 DOM 变更
```

这棵"单树"模式有一个致命问题：**如果在遍历过程中发生了中断，React 无法恢复之前的状态**。因为没有"中间的 Work In Progress"概念——要么全部完成，要么全部没做。

想象一下，你正在手写一篇文章，每一个字的改动都需要擦除整页重写。这就是 Stack Reconciler 面对更新中断时的困境。

双缓冲树解决了这个问题。它的灵感来源于图形学中的**双缓冲技术**：

### 图形学中的双缓冲

在屏幕渲染中，双缓冲将画面绘制在一个不可见的"后备缓冲区"中，绘制完成后通过指针交换让它瞬间显示在屏幕上。这样做的好处是——用户永远不会看到"绘制到一半"的画面。

React 的双缓冲树采用了同样的思想：**在不可见的 workInProgress 树上完成所有变更，然后一步切换到 current 树，用户感知到的永远是完整的 UI 状态。**

## 概念与定义

### current 树

当前屏幕上显示的 UI 对应的 Fiber 树。当 React 开始渲染时，`fiberRoot.current` 指向这棵树。它在 commit 阶段之后被更新。

```typescript
// fiberRoot.current 指向 current 树
const currentTree = fiberRoot.current;
// current 树上的所有 fiber 的 alternate 都不为空（在首次渲染后）
```

### workInProgress 树

正在构建中的 Fiber 树，尚未显示在屏幕上。React 在 beginWork 阶段从 `fiberRoot` 开始，通过 `createWorkInProgress` 函数逐节点创建或复用 Fiber 节点，形成一棵新的 workInProgress 树。

```typescript
// 在 render 阶段开始时，React 创建 workInProgress 树
// fiberRoot.current.alternate 就是 workInProgress 的根节点
```

### alternate 指针

连接 current 树和 workInProgress 树中同一节点对应关系的引用。每个 Fiber 节点都有一个 `alternate` 字段：

```typescript
type Fiber = {
  // current 树的 fiber 节点
  stateNode: any,
  // ...其他字段
  
  // 指向另一个树中对应的 fiber 节点
  alternate: Fiber | null,
  
  // 首次创建时 alternate 为 null
  // 首次 commit 后，两棵树的 alternate 互相指向
};
```

### 双缓冲切换（Root Switching）

在 commit 阶段完成后，React 将 `fiberRoot.current` 从 current 树指向 workInProgress 树，完成"页面"切换。下一次更新时，新的 workInProgress 树会基于新的 current 树创建——通过 `alternate` 进行复用。

```
前: fiberRoot.current → current树
     fiberRoot.current.alternate → workInProgress树

commit后: fiberRoot.current → 原来的workInProgress树（现在是新current树）
          新的 alternate 指向旧 current 树（现在是新的 workInProgress 基础）
```

## 最小示例

不要被名称吓到，双缓冲树其实是一条简单的规则在起作用。我们可以通过一个极简的计数器来观察它的行为：

```jsx
function Counter() {
  const [count, setCount] = useState(0);
  
  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(c => c + 1)}>+1</button>
    </div>
  );
}
```

当 Counter 首次渲染时，Fiber 树的状态变化如下：

```
首次渲染前：
  fiberRoot.current → null（还没有 current 树）

首次渲染（render 阶段）：
  React 从 fiberRoot 开始，没有旧的 alternate 可用
  createWorkInProgress() 为每个节点创建新的 Fiber → workInProgress 树
  fiberRoot.current 仍指向 null

首次渲染（commit 阶段）：
  fiberRoot.current = finishedWork（新创建的树）
  此时只有一棵树
  
第二次更新（setCount(1)）：
  fiberRoot.current → 当前显示的树（有 3 个 Fiber 节点）
  React 检查 current.alternate → null（第一次只有一棵树）
  为根节点创建新的 Fiber 作为 workInProgress
  createWorkInProgress() 遍历 current 树，为每个 Fiber 创建或复用 alternate
  
第二次更新（commit 阶段）：
  fiberRoot.current = finishedWork（workInProgress 树）
  旧的 current 树成为下一轮的 workInProgress 基础
  此时两棵树共存，alternate 互指
```

关键观察：**从第二次更新开始，current 树和 workInProgress 树通过 alternate 形成"双缓冲"状态**。

## 核心知识点拆解

### 1. createWorkInProgress 的复用逻辑

这是构建 workInProgress 树的核心函数。它通过 alternate 指针实现了高效的节点复用：

```typescript
// 源码简化：packages/react-reconciler/src/ReactFiber.js

function createWorkInProgress(current: Fiber, pendingProps: any): Fiber {
  let workInProgress = current.alternate;
  
  if (workInProgress === null) {
    // 首次创建：current 还没有 alternate
    // 创建新的 Fiber 实例
    workInProgress = createFiber(
      current.tag,
      pendingProps,
      current.key,
      current.mode,
    );
    
    // 设置相互引用
    workInProgress.alternate = current;
    current.alternate = workInProgress;
  } else {
    // 已有 alternate：复用现有的 Fiber 实例
    // 只需要重置一些可变字段
    
    // 重置 pendingProps
    workInProgress.pendingProps = pendingProps;
    
    // 重置副作用标志
    workInProgress.flags = NoFlags;
    workInProgress.subtreeFlags = NoFlags;
    workInProgress.deletions = null;
  }
  
  // 从 current 复制不可变字段
  workInProgress.type = current.type;
  workInProgress.stateNode = current.stateNode;
  workInProgress.child = current.child;
  workInProgress.sibling = current.sibling;
  workInProgress.index = current.index;
  workInProgress.ref = current.ref;
  
  // 从 current 复制 pending 的更新信息
  workInProgress.lanes = current.lanes;
  workInProgress.childLanes = current.childLanes;
  
  return workInProgress;
}
```

**关键复用策略**：
- **new fiber**（`alternate === null`）：创建全新 Fiber 实例，这是首次渲染或节点类型变化后的情况
- **复用 fiber**（`alternate !== null`）：重用现有对象，仅重置可变状态（effects flags、pendingProps），复制不可变状态（type、key）
- 这种复用策略避免了大量的对象创建和 GC 压力，是 Fiber 架构高性能的重要保障

### 2. 中断恢复的场景

双缓冲树是如何在可中断渲染中工作的？让我们用一个场景来解释：

```typescript
// 假设有一个包含 10000 个列表项的组件正在渲染
// 渲染进行到一半（已经创建了 5000 个 workInProgress fiber）
// 突然来了一个高优先级的用户输入更新

// Scheduler 检测到 shouldYield() === true，暂停当前工作
// 此时状态：
//   current 树 → 完整的旧 UI（用户看到的是完整页面）
//   workInProgress 树 → 仅构建了一半（用户看不到这个半成品）

// React 保存所有 pending 的 lanes
// 并记住 workInProgress 树的当前进度

// 高优先级更新触发新一轮渲染
// React 检查两个更新是否有冲突

// 如果有冲突（状态已变化）：
//   直接丢弃旧的 workInProgress 树
//   基于最新的 current 树，重新创建 workInProgress
//   从根节点重新开始构建

// 如果没有冲突（如兄弟组件的更新）：
//   React 可以继续使用现有的 workInProgress
//   但实际实现中通常选择重新开始，保证一致性

// 关键点：无论哪种方式，用户看到的 current 树始终是完整的
```

这就是双缓冲的核心优势——**用户永远不会看到"渲染到一半"的残缺界面**。

### 3. commit 阶段的交替切换

在 commit 阶段，`commitRoot` 会执行最重要的一步——交换根指针：

```typescript
// 源码简化：commitRootImpl

function commitRootImpl(root: FiberRoot): void {
  const finishedWork = root.finishedWork;
  
  // ... 执行各种 DOM 操作（mutation effects）...
  
  // 关键步骤：交换根指针
  // 将 current 指向 finishedWork
  root.current = finishedWork;
  
  // 交换后：
  //   - finishedWork（原来的 workInProgress）成为新的 current 树
  //   - 旧的 current 树现在成为新的 alternate
  //   - 下一次渲染时，createWorkInProgress 会基于新的 current 树创建
}
```

这个指针交换是原子操作——一旦发生，整个 UI 的 Fiber 表示就切换到了新状态。

## 底层原理（含源码分析）

### 1. completeWork 中的 alternate 更新

在 render 阶段的 `completeUnitOfWork` 中，React 会将 workInProgress 树中的更新结果"回写"到 alternate（current 树对应的节点）上：

```typescript
// 源码简化：packages/react-reconciler/src/ReactFiberCompleteWork.js

function completeWork(current: Fiber | null, workInProgress: Fiber) {
  const newProps = workInProgress.pendingProps;
  
  switch (workInProgress.tag) {
    case HostComponent: {
      // DOM 元素的构建
      const rootContainerInstance = getRootHostContainer();
      const type = workInProgress.type;
      
      if (current !== null && workInProgress.stateNode !== null) {
        // 更新已存在的 DOM 节点
        updateHostComponent(current, workInProgress, type, newProps, rootContainerInstance);
      } else {
        // 创建新的 DOM 节点
        const instance = createInstance(type, newProps, rootContainerInstance);
        appendAllChildren(instance, workInProgress);
        workInProgress.stateNode = instance;
      }
    }
  }
  
  // 关键：将 workInProgress 的 flags 冒泡到父节点
  bubbleProperties(workInProgress);
}

// bubbleProperties 将子节点的 flags 汇总到父节点
function bubbleProperties(completedWork: Fiber) {
  let newChildLanes = NoLanes;
  let newSubtreeFlags = NoFlags;
  
  let child = completedWork.child;
  while (child !== null) {
    newChildLanes = mergeLanes(newChildLanes, child.lanes);
    newSubtreeFlags |= child.subtreeFlags;
    newSubtreeFlags |= child.flags;
    child = child.sibling;
  }
  
  completedWork.childLanes = newChildLanes;
  completedWork.subtreeFlags = newSubtreeFlags;
}
```

### 2. effectList 的构建与消费

React 会在 workInProgress 树上构建一个 **effectList**（副作用链表），将所有有 DOM 变更的 Fiber 节点串联起来，避免在 commit 阶段再次遍历整棵树：

```typescript
// 源码简化
function completeUnitOfWork(unitOfWork: Fiber): void {
  let completedWork = unitOfWork;
  
  do {
    // ... completeWork ...
    
    // 构建 effectList
    if (returnFiber.flags & ShouldCapture) {
      returnFiber.subtreeFlags = completedWork.subtreeFlags & StaticMask;
      returnFiber.deletions = completedWork.deletions;
    } else {
      if (returnFiber.flags !== NoFlags) {
        // 将有 flag 的 fiber 连接到 effectList 中
        if (returnFiber.lastEffect !== null) {
          returnFiber.lastEffect.nextEffect = completedWork;
        } else {
          returnFiber.firstEffect = completedWork;
        }
        returnFiber.lastEffect = completedWork;
      }
    }
    
    completedWork = returnFiber;
  } while (completedWork !== null);
}
```

effectList 构建完成后，在 commit 阶段只需要遍历 effectList 链表即可找到所有需要操作 DOM 的 Fiber 节点，时间复杂度从 O(n) 降到了 O(k)（k 为有副作用的节点数量）。

### 3. 重置 workInProgress 树

每次 commit 后，旧 workInProgress 树会变为新的 current 树。在下一次渲染前，React 通过 `resetWorkInProgress` 重置状态：

```typescript
// 源码简化
function resetWorkInProgress(workInProgress: Fiber, renderLanes: Lanes): void {
  // 重置 pendingProps 为 current 的 memoizedProps
  workInProgress.pendingProps = workInProgress.memoizedProps;
  
  // 重置各种状态
  workInProgress.updateQueue = workInProgress.alternate.updateQueue;
  workInProgress.flags = NoFlags;
  workInProgress.subtreeFlags = NoFlags;
  workInProgress.deletions = null;
  
  // 保留 lanes 信息（标记哪些更新需要在下一次渲染中处理）
  workInProgress.lanes = workInProgress.alternate.lanes;
}
```

## 高频面试题解析

### Q1：双缓冲树如何支撑 React 的可中断渲染？

**参考答案：**

双缓冲树通过"用户永远看不到中间状态"来支撑可中断渲染：

1. **中断安全**：当 workInProgress 树被中断时，它只是一个"未完成的草稿"。current 树保持完整，用户看到的 UI 是正确的
2. **丢弃成本低**：如果中断后需要重新开始，可以直接丢弃 workInProgress 树，基于 current 树重新构建。丢弃的只有未提交的工作量，不影响用户感知
3. **无中间状态泄露**：commit 阶段的根指针交换是 O(1) 操作，从"旧 UI"到"新 UI"的切换是瞬间完成的

想象你在做一场演讲时同时准备下个月的演讲稿——你在草稿纸上写写停停，不会影响听众当前看到的投影画面。

### Q2：为什么 React 需要 alternate 指针？直接深度拷贝不行吗？

**参考答案：**

使用 alternate 指针而非深拷贝有以下优势：

1. **性能**：深拷贝所有 Fiber 属性涉及大量对象创建和内存分配，而 alternate 指针只需在两个已存在的对象之间建立引用关系，成本极低
2. **按需复用**：大多数情况下，Fiber 节点的 type、key、stateNode 等字段不会变化。通过 alternate 复用这些字段，避免了一次"复制到新对象"的工作
3. **内存节约**：两棵树的节点数最多是大 N 的两倍。如果不复用，每次更新都需要创建一棵全新的树（N + 1 倍内存），而 alternate 复用了大部分对象，实际新增对象远少于 N
4. **GC 友好**：交替复用减少了垃圾回收压力——每两次更新之间的新对象创建量大幅降低

简而言之，alternate 指针让 React 在"创建新树"和"复用旧对象"之间找到了优雅的平衡。

### Q3：双缓冲树在当前树很大（数万个节点）时更新效率会下降吗？

**参考答案：**

不会显著下降。原因在于 React 的更新是**按需遍历**的：

1. **跳过未变化子树**：如果某个子树没有任何 pending 的 lanes，beginWork 在遍历到它的根节点时返回 `bailoutOnAlreadyFinishedWork`，直接跳过整个子树，不进入其 children 的 alternate 创建流程
2. **effectList 跳过无变更节点**：即使当前只有一棵庞大的树，只要某个小分支有更新，workInProgress 树上只有那条路径上的节点会被实际处理，其余节点通过 bailout 跳过
3. **延迟渲染**：Concurrent 模式下的时间切片只切片实际有工作的部分，静止的子树不消耗任何 CPU

所以双缓冲树的开销是 O(k) 而非 O(n)，其中 k 是实际受更新影响的节点数量。

## 总结与扩展

### 总结

双缓冲树是 React Fiber 架构的中枢机制，通过交替两棵 Fiber 树来实现：

- **渲染安全性**：用户永远看不到渲染中间态
- **中断恢复**：workInProgress 树可被安全丢弃和重建
- **高效复用**：alternate 指针实现 O(1) 节点复用
- **并发基础**：支持 Suspense、Transition 等并发特性的底层数据架构

### 扩展：与其他框架的对比

- **Vue 3**：Vue 3 没有双缓冲树的概念。它的响应式系统能做到细粒度的直接 DOM 更新，不必维护两棵完整的虚拟 DOM 树。不"双缓冲"不代表落后——每种架构服务于不同的设计约束
- **Preact**：Preact 的 Fiber 实现是 React Fiber 的简化版，也使用了双缓冲树，但去掉了大量并发模式相关的字段和逻辑
- **Svelte**：Svelte 编译时直接将反应性注入 DOM 操作代码中，没有运行时虚拟 DOM 树，自然也没有双缓冲的概念

双缓冲树是"以内存换安全"的设计取舍。对于 React 这种运行时框架，这种取舍是完全值得的。
